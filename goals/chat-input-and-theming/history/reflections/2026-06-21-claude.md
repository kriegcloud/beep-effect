# Reflection — chat-input-and-theming (2026-06-21, claude)

## Summary

Shipped both packet capabilities into `apps/professional-desktop`: an app-local green "workbench" theme (near-black green dark + green/parchment light, `system` default, persisted toggle, reused `@beep/ui` `OrbBackground tone="green"`) and a feature-flagged, lobehub-style Lexical chat input in `@beep/editor` (fixed toolbar, `/` slash, `@` mentions, attachment capture, plain-Enter send, live character count, send/stop, WAI-ARIA combobox a11y), with the placeholder/cursor bug fixed. The foundation owns the generic mechanism; the app injects product meaning. After a working v1 against the SPEC, the packet owner directed a full schema-first + atom-first rewrite (superseding the SPEC's "UI config = plain TS" carve-out): `ComposerFeatures`/`SlashItem`/`MentionOption`/`ComposerAttachment` are `S.Class` (functions via `S.declare`/`Fn`, JSX via `DOMReactNode`, MIME via `@beep/schema` `ImageMimeType`); all per-composer state + Lexical bindings are `Atom.family` keyed by the `LexicalEditor` (no React hook state, `get.addFinalizer` cleanup), handlers are `composerRuntime.fn`, prop→atom seeding is `useAtomInitialValues`, and there is zero `Effect.runSync*` (sync decode via `S.decodeUnknownOption`, Effect execution via `Atom.runtime`). Verdict: complete; `bun run check` + Storybook (9/9) + live browser QA (slash/mention/Enter/combobox-a11y, both theme modes) all green; attachment send-transport intentionally degraded (stubbed send) per the Exception Ledger.

## Tooling experience

- **Worked:** Parallel `Explore` agents mapped four surfaces (editor, app wiring, POC, theme) fast; a backgrounded Lane-A agent built the (independent) theming files while I built the Lane-B foundation; the Storybook `addon-vitest` interaction tests validated six stories' play functions in a real browser; **claude-in-chrome QA was the highest-value tool** — it caught two interaction bugs that no static check could.
- **Didn't:** A recon agent's "primitive doesn't exist" claim (repo-grep instead of node_modules) was a confident false negative on a load-bearing fact.
- **Frustrating:** The `@phosphor-icons/react` / `nodenext` `exports`-condition resolution failure was opaque ("did you mean `Icon`?") and required diffing the sibling package's tsconfig to diagnose; then the test tsconfig needed the same fix separately.
- **Wished existed:** A repo invariant/lint that bundled UI packages use Bundler resolution, and a recon convention that "exists in installed package?" checks node_modules.

## Implementation improvement opportunities

- The send/typeahead Enter-collision fix and the boolean menu-open tracking are now in `@beep/editor/chat` and reusable; future composers should not re-derive them.
- The final app composer is fully atom-first: `latest` state, the initial-state decode, and the submit/stop/draft handlers all flow through atoms / `Atom.runtime` (the earlier ref + remount-by-key + `useCallback` version was replaced), and `MessageView`'s `Effect.runSyncExit` was converted to an `Atom.runtime` family. The `onError` Lexical config callback is a no-op to stay `runSync`-free — a future improvement is to route editor errors through a logging `Atom.runtime` rather than swallow them.
- The reusable atom-first Lexical-composer patterns (per-editor `Atom.family` + `get.addFinalizer` bindings, `useAtomInitialValues` seeding, `runtime.fn` handlers, boxed function atoms) are worth extracting into the `atom-reactivity-specialist` skill as a worked example — Lexical's imperative, long-lived command handlers are a non-trivial fit for strict atom-first and the bridge took several iterations to get right.

## Goal & prompt critique

The packet was unusually strong: a cited, adversarially-verified deep-research report, a concrete palette table, a live Chrome feature-map, locked grilled decisions, and an Exception Ledger. The only friction was the implicit dependency/tsconfig work needed to host an icon-bearing toolbar in `@beep/editor` (see prompt-critique finding). I would add one line to the SPEC's Target Surfaces noting that `@beep/editor` gains `@phosphor-icons/react` + `@lexical/{selection,utils}` catalog declarations and Bundler resolution.

## TODOs worth codifying

See the `todos` block above — chiefly the Bundler-resolution default for ui-system packages and the reusable send-key/typeahead interaction contract.

## Lessons (confidence-tiered)

- **HIGH — Critical:** Browser-drive every interactive composer change. The two shipped-blocking bugs (Enter sends instead of selecting; `aria-expanded` stuck via a counter) were invisible to `tsc`, biome, and jsdom story tests, and visible instantly in Chrome.
- **HIGH — Critical:** Resolve module/types issues at the `exports`/`moduleResolution` layer, not by avoiding the library. Aligning `@beep/editor` to `Bundler` (matching `@beep/ui`) was the correct, consistent fix — the inherited `nodenext` was the real defect.
- **MEDIUM — Best practice:** Treat recon agents' "X does not exist" as a hypothesis to confirm against `node_modules`, never a fact, when X is load-bearing.
- **LOW — Consideration:** Keep the green theme app-local until a second consumer appears; the promotion path to a `@beep/ui` preset is documented and cheap.

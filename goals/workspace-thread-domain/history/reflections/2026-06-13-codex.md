# Reflection - workspace-thread-domain (2026-06-13, codex)

## Summary

This packet landed the workspace conversation spine: Thread, Turn, Message,
PGlite tables/migrations, `@beep/anthropic`, the `agents` rename, and the
epistemic UsageRecord append path. The final verdict is strong: focused package
proofs passed, generated surfaces were refreshed, and full Yeet verify is green.

## Tooling experience

- **Worked:** The repo-local skills and Yeet path gave good guardrails. Focused
  package checks found typing/model issues quickly, `docgen:local -- --full`
  proved the generated docs after the rename, and the final Yeet verdict JSON
  gave a compact closeout proof.
- **Didn't:** The architecture operation-plan proof list was static enough that
  adding a new db-admin migration needed an explicit generator update before
  repo-cli tests would accept the new files.
- **Frustrating:** The first Yeet failure was late in a long run, and the
  needed fix was not in the new domain model but in the surrounding proof
  inventory. That is correct repo hygiene, but it is easy to miss during a
  large slice packet.
- **Wished existed:** An operation-plan command that derives migration proof
  files from `db-admin` target registration would have turned that static-list
  repair into normal generated drift instead of a manual allowlist edit.

## Implementation improvement opportunities

- The domain and table layer is ready for app wiring, but the next packet should
  add repository/use-case behavior around branch traversal and turn
  finalization rather than expanding the entity surface further.
- The static Anthropic price/config row is intentionally approximate. When the
  runtime wiring consumes it, add a single catalog update path so cost
  assumptions do not drift across driver and UsageRecord code.
- Keep the shared entity fixture helpers in `@beep/test-utils` as the default
  pattern for future BaseEntity-heavy slice tests.

## Goal & prompt critique

The goal was clear about source hierarchy, non-goals, PGlite-first storage, and
the split between workspace, drivers, agents, and epistemic. I would add this
line to the implementation checklist:

```text
After adding db-admin migration files, update or regenerate the architecture
operation-plan proof surface and rerun the repo-cli operation-plan test before
full Yeet.
```

I would also mention that repeated BaseEntity test fixture setup should use
`@beep/test-utils` or promote a helper before running the clone/reuse gate.

## Lessons (confidence-tiered)

- **HIGH - Critical:** Smoke PGlite early and again through Yeet integration
  after migrations land; the early proof de-risks the storage decision, while
  full Yeet catches repo-wide migration registration effects.
- **MEDIUM - Best practice:** Treat proof inventories as implementation
  surfaces when a packet adds generated migrations or package renames.
- **LOW - Consideration:** Keep one short evidence file per major packet slice;
  it makes final closeout much easier to audit than one enormous terminal log.

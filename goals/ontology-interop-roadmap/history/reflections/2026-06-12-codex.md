# Reflection - ontology-interop-roadmap (2026-06-12, codex)

## Summary

The packet turns the ontology interop discussion into an execution-capable V1
roadmap that stays inside existing foundation modeling packages. Its best
outcome is a clean spine: SKOS vocabulary, opt-in ontology profile behavior,
Markdown projection, JSON Schema sidecars, and domain-agnostic provenance.

## Tooling experience

- **Worked:** The research set did the right narrowing work before
  implementation: current surface analysis, FOLIO benchmarking, SKOS research,
  Effect JSON Schema notes, and final synthesis all point at the same V1 shape.
- **Didn't:** The SPEC still says `ACTIVE` while README, PLAN, and manifest
  mark the implementation retained or complete; later agents need to reconcile
  those lifecycle words carefully.
- **Frustrating:** The goal spans standards, legal workflow context, and package
  code, so the deferred-scope guard matters as much as the implementation list.
- **Wished existed:** A reusable foundation-modeling vocabulary placement guide
  for deciding when a term belongs in `@beep/rdf` versus `@beep/ontology`.

## Implementation improvement opportunities

- Keep SKOS profile behavior opt-in so existing RDFS/OWL class semantics do
  not silently become concept semantics.
- Preserve Markdown as the V1 documentation projection; route suffixes,
  browsers, and app/server presentation should wait for a later owner.
- Use public `@beep/*` package aliases in tests that exercise this package
  boundary, matching the repo's package test guidance.

## Goal & prompt critique

The compact launcher is useful because it names the implemented V1 spine and
the deferred surfaces. I would make one edit for consistency: change any stale
"ACTIVE" wording in the spec header during a future packet-doc cleanup so every
packet surface agrees that the roadmap is completed-retained evidence.

## Lessons (confidence-tiered)

- **HIGH - Critical:** package boundaries are the main safety rail - RDF
  vocabulary constants, ontology authoring behavior, and runtime semantic-web
  services must not collapse into one package.
- **MEDIUM - Best practice:** sidecar metadata is the right default for Effect
  JSON Schema because it serves API, editor, and RAG use cases without
  polluting RDF projections.
- **LOW - Consideration:** lifecycle words in old packet docs can drift; prefer
  manifest plus README plus final plan state when writing historical summaries.

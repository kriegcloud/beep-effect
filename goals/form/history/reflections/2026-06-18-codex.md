# Reflection - form (2026-06-18, codex)

## Summary

This packet rebuilt `@beep/form` as a schema-first, Effect v4, TanStack React
Form substrate with the complete P1/P2/P3 field inventory, additive `@beep/ui`
primitives, schema role reconciliation, Storybook coverage, and full Yeet proof.
The final verdict is strong: package checks, 29 form Storybook play tests, and
repo-wide `bun run beep yeet verify` are green.

## Tooling experience

- **Worked:** The packet's locked decisions and P0 research were specific enough
  to avoid re-grilling during implementation. Focused package gates made the
  form/ui/schema edit loop fast, Storybook play tests covered field behavior,
  and the final Yeet verdict gave a compact repo-wide proof.
- **Didn't:** Package-local proof did not catch Storybook app styling drift. The
  fields functioned, but the app needed its own PostCSS config bridge and static
  Emojibase wiring before the rendered stories looked correct.
- **Frustrating:** The final global proof surfaced generated config-sync,
  fallow boundary, dead-code, and dependency-order repairs late in the packet.
  Those were valid repo hygiene failures, but they were not visible from the
  package scope.
- **Wished existed:** A small Storybook visual smoke lane that checks "shared UI
  styles loaded" would have caught the most user-visible issue before manual
  inspection.

## Implementation improvement opportunities

- Upload previews now use revocable object URLs for image files. If another
  preview-bearing upload primitive lands, extract the object URL creation and
  cleanup into a shared helper or hook so the leak-prevention rule is not
  reimplemented by hand.
- DateTime and Time picker styling needed explicit column width tuning after
  visual QA. Future MUI-X primitive work should include narrow and wide sample
  values in Storybook assertions or screenshots.
- The field count intentionally treats standalone `Radio` as subsumed by
  `RadioGroup`. If product usage later needs a single boolean-radio field,
  add it as a small follow-up rather than reopening this packet.

## Goal & prompt critique

The goal was unusually good: it named the source hierarchy, dependency envelope,
date adapter caveats, widget pins, and exact verification gates. I would add one
line to the launcher workflow:

```text
Before starting P2/P3 edits, run `bun run beep yeet verify` once and record
whether any repo-wide generated or policy drift is already present.
```

I would also make the Storybook app styling/static-asset expectation explicit:

```text
After adding a new ui-system primitive story, inspect the rendered Storybook app
or run a visual smoke check that proves shared Tailwind/PostCSS styles and
static widget assets are loaded.
```

## Lessons (confidence-tiered)

- **HIGH - Critical:** Prove UI-system packets in the app that renders them, not
  only in the package that defines them - Storybook app wiring can break visual
  quality while package checks remain green.
- **MEDIUM - Best practice:** Run one full Yeet baseline before a late-phase
  widget/date expansion so generated repo drift is separated from feature
  fallout.
- **LOW - Consideration:** Keep upload preview object URL lifecycle code small
  and reusable if additional preview components appear.

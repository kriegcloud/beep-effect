# Reflection - chat-surface-parity (2026-06-15, codex)

## Summary

This packet closed the remaining POC parity gaps for the desktop chat surface:
observability and UX affordances, repair-before-drop behavior for streamed
blocks, and rich mermaid/table/youtube blocks across streaming, validation,
`@beep/md` persistence, and rendering. The final verdict is high confidence
because local foundation proof, aggregate Yeet proof, and the real Anthropic E2E
all passed.

## Tooling experience

- **Worked:** Focused package tests gave fast confidence for `@beep/md`,
  Lexical, editor nodes, agents domain/server, and the desktop integration
  harness. `bun run beep yeet verify` provided the aggregate proof that smaller
  package lanes did not hide repo-level failures.
- **Didn't:** The 1Password MCP path failed in multiple ways before closeout:
  first Linux peer effective GID rejection, then parent binary permissions, then
  a closed Codex MCP transport. The eventual green path was `op run` with an
  approved `op://` secret reference.
- **Frustrating:** The real-LLM acceptance item was semantically small but
  operationally fragile. It depended on host 1Password state more than on repo
  code, which made a completed implementation look blocked until the secret
  injection path was repaired.
- **Wished existed:** A repo-owned "real provider proof" wrapper that accepts a
  1Password reference, performs a non-leaking injection smoke check, and then
  runs the gated E2E command would remove repeated hand wiring.

## Implementation improvement opportunities

- The real Anthropic test should stay narrow and explicit, but it could emit a
  compact artifact describing which block kinds streamed and persisted. That
  would make future closeout evidence easier to audit than a single Vitest pass
  line.
- The repair-loop tests already cover invalid-to-repaired behavior; a future
  hardening pass could add a deliberately unrepairable rich-block fixture to
  prove the drop-and-log branch without relying on provider behavior.

## Goal & prompt critique

I would revise P4 in `GOAL.md` to include the exact sanctioned local command
shape for paid-provider proof:

- "For local real-LLM proof, inject the credential with `op run -- ...`; the
  command must only report pass/fail and must not print the resolved value or
  the secret reference."

That would preserve the security boundary while making the final acceptance path
less dependent on operator memory.

## TODOs worth codifying

- Add a 1Password MCP host-health note for Linux installs: the desktop relay
  rejects MCP unless `/opt/1Password/onepassword-mcp` has the
  `onepassword-mcp` group and setgid bit.
- Add a reusable real-provider E2E wrapper or doc snippet that performs a
  non-leaking `op run` smoke check before launching the expensive test.

## Lessons (confidence-tiered)

- **HIGH - Critical:** Keep real-provider acceptance tests gated but runnable
  locally with secret-reference injection - fixture CI proves shape, but only
  the live provider proved the final mermaid/table/youtube streaming and
  persistence contract.
- **MEDIUM - Best practice:** Record the exact live command in packet history -
  closeout evidence is much easier to trust when the command, gate variables,
  and result are all adjacent.
- **LOW - Consideration:** Consider promoting secret-injection troubleshooting
  into a shared skill or repo note - this issue can recur outside chat parity
  wherever 1Password MCP and paid-provider tests meet.

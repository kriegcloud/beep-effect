# Reflection — mcp-kit (2026-07-01, claude)

## Summary

Shipped `@beep/mcp-kit` end-to-end in one day: P0 re-verification (no drift),
seven SPEC deliverables with 15 fixture/proof tests through the real
`McpServer` pipeline, a two-round verify/fix loop, PR #288 opened mergeable
with hosted checks green except three inherited-from-main reds, and all six
bot-review findings fixed. Orchestration ran as one Claude session delegating
all research/implementation to a single resumable Codex agent — the resume
pattern preserved implementation context across four passes and worked well.

## Tooling experience

- **Worked:** `bun run beep create-package` matched the P0-derived checklist
  exactly (workspace glob skip, `$McpKitId` auto-registration, tsconfig-sync);
  yeet quality packets routed failures precisely; `@effect/vitest` `layer()`
  with `Layer.provide(ConfigProvider.layer(...))` cleanly proved build-time
  vanish semantics; resumable Codex delegation kept the main session lean.
- **Didn't:** `yeet publish` proof cannot accept recorded-unrelated red lanes,
  forcing the manual push recovery; the root `bun run test` wrapper dies with
  an unhandled `PlatformError` instead of reporting which package failed.
- **Frustrating:** transient dual-arity law failure caused by `@beep/repo-cli`
  tests running yeet stash machinery against the live worktree mid-proof; the
  `@beep/schema` identifier regression's symptom package shifts with turbo
  cache state, which made first-pass attribution look repo-utils-local.
- **Wished existed:** a per-packet "known-red ledger" consumed by yeet so
  attributed pre-existing failures don't block publish; a full-lint-gate
  checklist surface for new packages.

## Implementation improvement opportunities

- Consolidate the seven drivers' hand-rolled optional-secret reads onto
  `resolveSourceCredential` (in `mcp-host-retrofit`).
- `projectWithinBudget`'s `mintFetchableHandle` and the tier-gate audit sink
  are both consumer-owned callbacks; once `uspto-mcp` lands, evaluate whether
  a shared consumer-side helper should exist.

## Goal & prompt critique

- The packet was strong: verified `file:line` design facts in GOAL.md meant P0
  was a fast confirm rather than open research, and the license-discipline
  table in `research/SOURCES.md` made port-vs-clean-room decisions mechanical.
- Revise future launchers to name the full lint-gate composition and the two
  new-package gotchas (post-scaffold dep step; changeset-status blindness to
  uncommitted changesets). Both cost an extra verify iteration here.
- SPEC said "six fixture/proof tests" (the acceptance bar); the kit shipped 15.
  Fine, but reviewers flagged the count mismatch across packet files — future
  SPECs should phrase it as "at least".

## Lessons (confidence-tiered)

- **HIGH — Critical:** Attribute before fixing — the greptile "Context.get
  throws" finding was empirically false (`Context.Reference` defaults), but the
  fail-closed hardening it suggested was still right; and the "repo-utils test
  failure" was actually a systemic `@beep/schema` regression whose symptom
  moves with turbo cache. Both attributions changed the fix.
- **HIGH — Critical:** For new packages, run the FULL repo lint suite before
  declaring package-scoped green — five law lanes and the canonical JSDoc
  vocabulary only fire repo-wide.
- **MEDIUM — Best practice:** One resumable implementation agent beats many
  fresh ones for multi-pass work — each `--resume` pass (implement → verify-fix
  → review-fix) reused full context and each round converged in one iteration.
- **LOW — Consideration:** When a proof lane fails once and passes on
  re-check with an unchanged tree, suspect intra-proof interference (parallel
  test lanes mutating shared state) before suspecting the diff.

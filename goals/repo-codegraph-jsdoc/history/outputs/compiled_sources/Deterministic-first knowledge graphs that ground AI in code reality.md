# Deterministic-first knowledge graphs that ground AI in code reality

**The most effective architecture for reducing AI agent hallucinations in TypeScript monorepos layers three certainty tiers: deterministic facts extracted from the AST, categorically constrained classification metadata, and LLM-generated semantic enrichment.** This "deterministic-first" philosophy — maximize what can be computed, minimize what requires inference — finds strong support across static analysis research, applied category theory, and recent work on LLM grounding. A 2026 IEEE/ACM FORGE paper demonstrated that deterministic AST analysis corrected **77% of identified LLM hallucinations** in code generation tasks, and IBM Research's GraphGen4Code built knowledge graphs exceeding **2 billion RDF triples** from source code using program analysis as the foundation. No existing tool implements the full pipeline for TypeScript, but the building blocks are mature: ts-morph for AST manipulation, Effect-TS for typed error channels, and emerging hybrid pipelines like RepoAgent for structured LLM prompting.

---

## The documentation drift problem demands structural detection, not text matching

Documentation drift — where code evolves while its documentation stays frozen — is the root cause of knowledge graph decay. The industry has converged on three tiers of defense, though none yet operates at the AST-structural level this architecture requires.

**Heuristic text matching** represents the current state-of-the-art. Swimm, the most mature drift-detection platform, uses a patented algorithm that analyzes every git commit since documentation creation, comparing code-as-text-strings to determine if changes are meaningful enough to warrant updates. When structural changes exceed a configurable threshold, CI checks fail, blocking merges until documentation is updated. DeepDocs takes a more autonomous approach: installed as a GitHub App, it performs deep scans on every commit to main, identifies specific outdated sections, and opens separate PRs with minimal edits that preserve existing style. Both tools treat documentation as opaque text rather than structured metadata.

**Linting-based validation** catches format violations but not semantic drift. The `eslint-plugin-jsdoc` package offers **40+ rules** including `check-param-names` (validates @param names match actual function parameters), `require-jsdoc` (enforces documentation exists), and `informative-docs` (ensures comments aren't just restating code). The companion `eslint-plugin-tsdoc` validates conformance to the TSDoc specification. These run in standard CI pipelines and catch structural mismatches — a renamed parameter that wasn't updated in JSDoc — but cannot detect when a function's behavior changes while its description remains stale.

**AST-derived documentation eliminates drift for structural facts entirely.** AG Grid provides the best TypeScript case study: after discovering that hardcoded interface definitions in their documentation were "no longer accurate, as the underlying source code had been updated but the documentation version was missed," they built a pipeline using `ts.createSourceFile` to parse code into AST, traverse `InterfaceDeclaration` nodes, and extract property types alongside JSDoc annotations into JSON that feeds their documentation site directly. This approach makes drift impossible for anything derivable from the AST — and as detailed below, that category is far larger than most teams realize.

The "docs-as-tests" paradigm offers a complementary defense. Tools like `tsdoc-testify` extract code from `@example` blocks in TSDoc comments and generate Jest-compatible test files, while `doctest-ts` rewrites JSDoc examples as unit test assertions. TypeDoc's `{@includeCode}` directive (since v0.27.7) references tested code from external files, ensuring examples are always valid. However, the TypeScript doctest ecosystem remains fragile — multiple attempts (`danr/doctest-ts`, `yamadapc/jsdoctest`, `Widdershin/markdown-doctest`) appear abandoned, a notable gap compared to Rust's first-class `rustdoc` test support where documentation examples are compiled and run as part of the standard build.

## Category theory constrains classification without arbitrary string schemas

The second layer of the knowledge graph — classification metadata like `@category`, architectural intent, and bounded context — is where most systems introduce uncontrolled string schemas that drift and fragment. Category theory offers tools to make this layer rigorous, though a critical finding is that **no mathematical proof exists for a minimal complete set of code categories**. The structure of classification can be formalized even if the specific categories remain engineering judgment.

**Discriminated unions model code categories as coproducts.** In category theory, coproducts represent disjoint sums — the formal analog of TypeScript's discriminated unions. Defining code categories as a closed union type with a `_tag` discriminant (following Effect-TS conventions) provides compile-time exhaustiveness checking: any `switch` over categories that misses a case produces a type error. This is not merely a stylistic choice. The `never` type serving as the initial object in TypeScript's type category guarantees that exhaustiveness checks are total — every code element must be classified into exactly one category, and the compiler enforces it. This transforms what would be a runtime string-matching problem into a compile-time guarantee.

**Moggi's computational effects provide the closest formal basis for "what kinds of code exist."** Eugenio Moggi's 1991 paper "Notions of Computation and Monads" established that distinct notions of computation correspond to distinct monads: the Identity monad for pure computation, Maybe/Option for partial computation, Either for error-handling, State for stateful computation, Reader for environment-dependent computation, Writer for logging, IO for side effects, and Continuation for control flow. Plotkin and Power's 2002 follow-up proved the converse — that monads can be characterized by algebraic operations and equations — establishing a bidirectional taxonomy. This means the fundamental categories of computation have a mathematical basis, even if mapping them to architectural layers requires judgment.

**Convergence across architectural patterns suggests eight natural categories.** Analyzing Clean Architecture (Martin, 2012), Hexagonal Architecture (Cockburn, 2005), Onion Architecture (Palermo, 2008), and FP effect systems reveals consistent separation into: **domain/business logic** (entities and use cases, pure functions), **data access** (database adapters, State/IO monad), **presentation/API** (controllers, Reader monad), **infrastructure** (frameworks and drivers, IO monad), **validation** (domain rules, Either/Validation), **transformation** (data mapping, Functor/map), **side effects** (external interactions, Effect monad), and **pure computation** (algorithms, Identity monad). While this convergence is empirical rather than proven, its consistency across independently developed architectural philosophies is striking.

**Ologs provide the categorical framework for the knowledge graph itself.** David Spivak and Robert Kent's "Ologs: A Categorical Framework for Knowledge Representation" (PLoS ONE, 2012) defines ontology logs where objects are types ("a validation function," "a data access layer") and morphisms are functional relationships ("calls," "depends on," "transforms into"). Crucially, functors between ologs formalize translations between classification schemes — mapping between an AST-level view (syntax nodes and their relationships) and a JSDoc metadata view (documentation tags and their semantics). The "fusion olog" construction integrates multiple worldviews into a single coherent representation, directly applicable to merging deterministic AST facts with LLM-inferred categories.

**Galois connections formalize the observation-to-classification mapping.** Fong and Spivak's *Seven Sketches in Compositionality* (Cambridge, 2019) shows that Galois connections — adjunctions between partially ordered sets — model the relationship between observable code properties (async, generic, exported, effectful) and the categories they imply. A function that is async, performs IO, and calls an external API maps to "side-effect" and "infrastructure" categories through a formally defined, monotone connection. This replaces ad-hoc tagging rules with a structure that guarantees consistency: if code has property X and X implies category C, the classification follows automatically.

## Most JSDoc tags are deterministically derivable from the TypeScript AST

The central technical insight of this architecture is that the majority of JSDoc documentation content requires no human authoring and no LLM inference. The TypeScript Compiler API (`ts.createProgram` + `TypeChecker`) exposes enough information to generate **at least 14 standard JSDoc/TSDoc tags** with 100% certainty.

**Fully deterministic tags from AST modifier flags and node structure** include `@param` (parameter names, types, optionality, and defaults from `FunctionDeclaration`/`MethodDeclaration` nodes), `@returns` (from explicit return type annotations or `TypeChecker.getReturnTypeOfSignature()` for inferred types), `@template`/`@typeParam` (from `TypeParameterDeclaration` nodes including constraints and defaults), `@async` (from `ModifierFlags.Async`), `@implements` and `@extends` (from heritage clauses), `@export` (from `ModifierFlags.Export`), `@access` (from `ModifierFlags.Public|Private|Protected`), `@readonly`, `@abstract`, `@override` (TS 4.3+), and `@static`. The key enabling API is `checker.getTypeOfSymbolAtLocation()`, which returns full Type objects even for inferred types, and `checker.typeToString()` for human-readable serialization.

**Effect-TS unlocks @throws and @requires as deterministic tags.** In standard TypeScript, exceptions are untyped — `@throws` documentation is purely human-authored and perpetually at risk of drift. Effect-TS changes this fundamentally. The `Effect<A, E, R>` type encodes errors in the `E` channel and dependencies in the `R` channel as first-class type parameters. A function returning `Effect<User, HttpError | ValidationError, Database | Cache>` deterministically yields `@throws HttpError`, `@throws ValidationError`, `@requires Database`, and `@requires Cache`. The TypeScript compiler API can extract these via `checker.getTypeArguments()` on the return type and decompose union types into individual members. This makes Effect-TS monorepos **uniquely suited** for knowledge graph construction — the type system itself is the graph of error propagation and dependency injection.

**ts-morph is the optimal tool for programmatic JSDoc generation.** While the raw TypeScript Compiler API provides all necessary information, ts-morph wraps it with a fluent, chainable interface specifically designed for AST manipulation. Its `JSDocStructure` type enables type-safe JSDoc construction: `node.addJsDoc({ description, tags: JSDocTagStructure[] })` generates well-formed documentation blocks. All changes are held in memory until `project.save()`, enabling batch transformations across an entire monorepo. For reading existing documentation, `node.getJsDocs()` returns structured tag data that can be compared against AST-derived facts to detect drift. Microsoft's API Extractor (`@microsoft/api-extractor`) takes a complementary approach, reading compiled `.d.ts` files rather than source to extract API surfaces, and provides API review workflows that detect breaking changes — useful for the knowledge graph's change-detection layer.

## The three-layer enrichment pipeline with grounded LLM inference

With deterministic tags covering structural documentation and category theory constraining classification, the LLM's role shrinks to a well-defined, verifiable scope: generating natural-language descriptions, `@example` code blocks, `@see` cross-references, and inferring architectural categories within the fixed taxonomy.

**RepoAgent (EMNLP 2024) validates the "AST-first, LLM-second" architecture.** Developed by OpenBMB with **900+ GitHub stars**, RepoAgent implements a three-stage pipeline for Python: global structure analysis (AST parsing → project tree → reference relationship DAG), documentation generation (bottom-up topological ordering ensures dependencies are documented first, structured prompts include project tree + code snippet + reference relationships), and documentation update (git pre-commit hook integration). Applied to a **~270,000-line codebase**, reference recall was perfect through deterministic analysis, while LLM-only approaches showed degraded recall. The structured prompt grounding the LLM in actual code structure is the key mechanism — and directly portable to TypeScript via ts-morph.

**Iterative grounding eliminates the dominant failure mode.** The De-Hallucinator system (Eghbali & Pradel, 2024) demonstrated that a simple two-pass approach — initial LLM prediction → identify hallucinated APIs via static analysis → retrieve correct project-specific APIs → re-query LLM — achieved **23–50% improvement in edit distance** and **24–61% improvement in API usage recall**. For knowledge graph enrichment, this translates to: generate a description, validate that every symbol, type, and function reference in the description exists in the AST, flag mismatches, and re-prompt with corrections. The 2026 FORGE paper extended this, showing deterministic AST analysis corrected 77% of LLM hallucinations, with the highest fix rate for missing imports (**97.9%**) and lower but still substantial rates for mistyped APIs (70%).

**The layered certainty model makes trust explicit.** Each piece of documentation in the knowledge graph carries a provenance tag:

- **Layer 1 (certainty = 1.0)**: AST-derived structural tags — @param, @returns, @template, @async, @implements, @extends, @export, @access, plus Effect-TS @throws and @requires. These are computed, not authored, and cannot hallucinate.
- **Layer 2 (certainty = 0.85–0.95)**: Type-system-derived semantic information — complex type resolution, generic instantiation, conditional type evaluation. Relies on the TypeScript type checker, which is deterministic but can produce surprising results for deeply nested conditional types.
- **Layer 3 (certainty = 0.6–0.85)**: LLM-inferred content — natural-language descriptions, @example code blocks, @category classification (constrained to the fixed taxonomy), @see cross-references, architectural intent annotations. Confidence varies by task: generating a one-line function summary is higher confidence than inferring architectural intent.

This layered model enables AI agents consuming the knowledge graph to make trust-appropriate decisions: Layer 1 facts can be used without hedging, Layer 2 facts are reliable but worth double-checking against source, and Layer 3 content should be presented as inferred rather than authoritative.

## Incremental processing makes this viable for monorepo scale

Performance at monorepo scale requires incremental processing — re-analyzing only what changed. TypeScript's incremental compilation infrastructure provides the foundation.

The `--incremental` flag creates `.tsbuildinfo` files storing the project dependency graph and build state. Combined with project references (`--composite`), `tsc --build` orchestrates multi-project builds that only recompile out-of-date packages. The knowledge graph pipeline can mirror this: on file change, consult `.tsbuildinfo` to identify affected files, re-run AST extraction only for those files and their dependents, update graph nodes for changed symbols, and re-trigger LLM enrichment only for symbols whose structural signatures changed. For a single file, AST parsing plus type checking takes milliseconds. Cold-starting a 100-package monorepo takes seconds to low minutes depending on type complexity.

TypeDoc, the standard TypeScript documentation generator, lacks incremental mode entirely — it performs full rebuilds on every invocation. This is a significant limitation that the knowledge graph architecture should address. ts-morph can be pointed at specific source files rather than the entire project, and the `TypeChecker` can be reused across files within a single `Program` instance. The practical architecture would maintain a persistent graph database (Neo4j, FalkorDB, or similar), update it incrementally on each commit, and use git pre-commit hooks (à la RepoAgent) for the LLM enrichment pass.

FalkorDB's code graph approach demonstrates the target architecture: AST → graph database → GraphRAG-SDK integration for LLM-augmented code understanding. Daytona's code graph similarly combines AST parsing + knowledge graph + LLM function calling, with continuous updating as code evolves. Neither is TypeScript-specific, but both validate the architectural pattern.

## The integrated pipeline and what remains unsolved

The complete architecture flows as follows: TypeScript source files are parsed via ts-morph, which extracts all AST-derivable JSDoc tags (Layer 1) and type-system-resolved information (Layer 2). Code elements are classified using a discriminated union taxonomy grounded in Moggi's effect categories and constrained by Galois connections from observable properties to categories. LLM enrichment (Layer 3) runs with structured prompts including the full AST context, generating descriptions and examples that are validated against static facts through De-Hallucinator-style iterative grounding. The resulting knowledge graph — stored with explicit certainty metadata per node and edge — feeds both human documentation (via TypeDoc-compatible output) and AI agent context windows (via structured retrieval). CI/CD validation via eslint-plugin-jsdoc, custom AST-diff checks, and optionally Swimm or DeepDocs ensures the graph stays synchronized with code.

Several gaps remain open. First, the TypeScript doctest ecosystem is immature compared to Rust's `rustdoc` — validating @example blocks requires fragile third-party tools. Second, no existing tool implements the full Effect-TS type decomposition pipeline for extracting @throws and @requires from `Effect<A, E, R>` type parameters; this would need to be built. Third, the category-theoretic classification layer is formally motivated but practically untested — the Galois connection from code properties to categories needs empirical validation across real codebases to confirm that the eight converged categories are sufficient and non-overlapping. Fourth, RepoAgent's "AST-first, LLM-second" pipeline exists only for Python; porting it to TypeScript via ts-morph is straightforward in principle but represents meaningful engineering effort. The building blocks are mature. The integration is the unsolved problem — and the opportunity.

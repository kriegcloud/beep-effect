# Reflection — gov-legal-data-driver-codegen (2026-06-30, claude)

## Summary

Stood up the gov/legal driver substrate end-to-end: finished the keyed
`@beep/govinfo` driver on a hand-authored transport transformer applied via
`HttpApiClient.make`'s `transformClient` seam, built the keyless `@beep/ecfr`
driver on the raw `HttpClient.mapRequest` path as the second transformer
consumer, and promoted the shared transformer to
`@beep/api-transport` (`foundation/capability`) under the ≥2-consumer gate. All
nine SPEC acceptance criteria verify; build/check/test are network-free.

## Tooling experience

- **Worked:** `bun run beep create-package` scaffolded the capability package,
  registered the `$ApiTransportId` identity composer, and synced root
  tsconfig/aliases + lockfile in one shot — it made the promotion low-risk.
  Reading the pinned `effect@4.0.0-beta.92` `.ts` source shipped in
  `node_modules` was essential for getting beta HTTP primitives right without
  guessing. `HttpClient.withRateLimiter` already parses `X-RateLimit-*` and
  retries `429`, so no limiter needed reimplementing. runpod's test harness
  (a `Context.Service` fake `HttpClient` + `@effect/vitest` `layer()`) was a
  clean, copyable template.
- **Didn't:** The `@effect/openapi-generator` path did not fit eCFR — no
  Swagger-2.0 source dialect, silent `format: date` downgrade, and a
  double-emit quirk when combining `addSchema` + `generate(defs)`.
- **Frustrating:** The custom tsgo effect-diagnostics are strict and fire on
  idiomatic-looking code; each new service cost extra check→fix cycles.
- **Wished existed:** A documented cheatsheet of those diagnostics, and a way to
  read a native limiter's rate-limit state without a parallel tap.

## Implementation improvement opportunities

- Collapse the rate-limit telemetry tap into the limiter if effect ever exposes
  its state.
- Grow the eCFR bespoke renderer to cover recursive definitions.
- Add a dedicated `@beep/api-transport` unit test (the drivers exercise it
  transitively today via `passWithNoTests`).

## Goal & prompt critique

- Stop pinning an exact `effect` patch version in prose — reference the catalog.
- Reword AC#6 so the drift check is "per generator-owning package", not "per
  keyed + keyless".
- Record that eCFR/FedReg have no cleanly fetchable spec, so the checked-in-spec
  route is the expected path, not a fallback of last resort.

## TODOs worth codifying

- An "authoring an Effect driver service" recipe covering the diagnostics.
- Renderer support for nested/recursive OpenAPI definitions.
- Offline coverage of the P2 Token-header / X-API-KEY auth branches when unblocked.

## Lessons (confidence-tiered)

- **HIGH — Critical:** Read the installed beta `effect` source before writing
  against `effect/unstable/*` — the API (e.g. `withRateLimiter` options,
  `transformClient`'s strict `HttpClient.HttpClient` return type) is not
  guessable and shifts between betas.
- **MEDIUM — Best practice:** Key operation caches on the decoded
  Schema.Class payload — its structural `Equal`/`Hash` (verified, including
  array fields) makes repeat requests cache-hit deterministically, satisfying a
  call-count assertion without a stable-string key.
- **LOW — Consideration:** When a native primitive hides its state (the rate
  limiter), a thin observable `tap` is an acceptable, honest way to expose it
  for tests — but flag it as a candidate for removal.

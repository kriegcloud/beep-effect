# Reflection — rich-text-foundation (2026-06-12, claude)

## Summary

Shipped the schema-first rich-text foundation in one pass: `@beep/lexical-schema`
(v1 node union + Md ↔ Lexical codecs with a locked, tested lossiness profile,
zero runtime lexical imports, 32 dtslint pins against lexical 0.45) and
`@beep/editor` (viewer, composer primitives, runtime artifact-ref decorator on
the `@beep/ui` substrate), proven by a fixture assistant turn rendering through
schema → viewer in `apps/storybook` chromium play tests. All package gates and
the storybook suite (464/464) are green.

## Tooling experience

- **Worked:** `bun run beep create-package` (once fixed) scaffolds + registers +
  syncs + locks in one shot; `tsconfig-sync --write` wired references from
  package.json deps automatically; tstyche dtslint made the lexical 0.45 pin
  precise; the storybook vitest addon gave a real-chromium render proof with
  zero bespoke harness; the refined `scratchpad/Lexical.schemas.ts` sketch cut
  the model-writing risk dramatically.
- **Didn't:** `create-package`'s ts-morph registry update against the refactored
  `packages.ts` (fixed in `CreatePackage/Handler.ts`), and it appends registry
  segments unformatted (failed `@beep/identity:lint` until `lint:fix`); first
  `yeet verify` run (stale fallow envelopes); yeet's failure packet
  misattributed a truncated-log failure to `dual-arity` (see findings).
- **Frustrating:** effect/String dual vs data-last signatures (`startsWith`,
  `slice`) differ from native and from each other; `exactOptionalPropertyTypes`
  makes optional-key conformance pins finicky (present-`undefined` vs absent).
- **Wished existed:** an Option combinator for `key absent | null | T` without
  `undefined`; a headless render harness for decorator nodes (decorate() is
  only proven via storybook).

## Implementation improvement opportunities

- `EditorComposer.onSerializedChange` decodes the full state on every change;
  for large documents a debounced or read-phase decode would be cheaper.
- The artifact-ref chip styling is minimal (muted mono chip); revisit when the
  desktop chat surface defines real artifact affordances.
- `collectListItems` flattens nested lists; if follow-ons need depth, `@beep/md`
  would need block-children list items first (named follow-on, out of v1).

## Lessons (confidence-tiered)

- **HIGH — Critical:** Never override a parent `S.Class` field's codec in
  `.extend` — the parent constructor still validates the parent codec at
  runtime; restructure the hierarchy (untagged base + sibling leaves) instead.
- **HIGH — Critical:** Lock codec lossiness profiles with executable tests
  (round-trip identity + targeted drop/degrade/normalize cases) before
  building consumers; the README table then documents what the tests prove.
- **MEDIUM — Best practice:** For wire-format conformance pins under
  `exactOptionalPropertyTypes`, compare with surgical `Omit`s per divergence
  and a comment per omission — a full-type pin is impossible (mutable arrays,
  widened discriminants) and a loose pin is worthless.
- **MEDIUM — Best practice:** Don't export React props interfaces from
  ui-system packages — the schema-first policy flags exported pure-data
  interfaces (model as schema) and callback-bearing interfaces (non-schema
  signals); keep them module-private like `@beep/ui` does and let consumers
  use `React.ComponentProps`.
- **MEDIUM — Best practice:** Schema-heavy test files need at least one
  `S.toArbitrary` + fast-check property test or a documented inventory
  exception (SFV4-arbitrary-tests) — write the property test up front; it
  caught nothing here but the round-trip property is exactly the package's
  core law.
- **LOW — Consideration:** Registering fewer Lexical nodes (no
  CodeHighlightNode) is a feature when the schema owns the wire profile —
  the editor can't produce out-of-schema states it doesn't register.

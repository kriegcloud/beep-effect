# Reflection — oppold-corpus-pipeline (2026-06-11, claude)

## Summary

All six phases executed in one session: 31.7 GB / 8,438 files salvaged off
failing removable media with 100% SHA-256 verification, cataloged and exactly
deduplicated in DuckDB (7,330 distinct digests), extracted for real through
hardened libpff/tika subprocess engines (663,272 PST message artifacts,
6,702 text artifacts, zero failures on SPEC formats), organized into a
docket-family taxonomy (105 families) with 242 recycle-bin names restored,
and enriched against the USPTO Open Data Portal via a new product-neutral
driver. Verdict: the packet contract was satisfiable in a single pass
because the repo's file-processing contracts and Files-command precedent
were reusable as designed.

## Tooling experience

- **Worked:** `@beep/file-processing` schemas slotted in unchanged;
  `DuckDb.makeNodeLayer` + `read_json` ingested the provenance manifest in
  one statement; stub-binary tests made the subprocess drivers CI-safe; the
  Effect language-service diagnostics (strictEffectProvide, banned
  `new Date()`/`JSON.parse`) consistently pushed toward the repo idioms and
  every suggestion compiled.
- **Didn't:** `beep create-package` aborted on the identity composer
  registration; `docgen:local` demanded a full-repo proof once bun.lock
  changed, deferring the docgen gate to publish time.
- **Frustrating:** sha256sum's backslash escape marker silently polluting
  digests; multi-line Edit calls fighting biome's import re-sorting (several
  no-op string replacements until re-reading the post-format file).
- **Wished existed:** a streaming Sha256 helper in `@beep/schema` (noted in
  the packet research as a likely need — confirmed; node:crypto was used in
  the CLI instead).

## Implementation improvement opportunities

- Incremental sources/failures JSONL writes during `corpus extract`.
- Recovered-items pffexport pass (`-m all`) over the immutable raw PSTs to
  surface deleted-then-purged mail.
- Message-level diff of the two distinct `LH_Inbox_2011.pst` snapshots now
  that both are exported.
- Wire enrichment applicant names back into `organized/clients/` via a
  generated label map (grounded by USPTO data, still no guessing).

## Goal & prompt critique

The GOAL/PLAN/SPEC stack was unusually executable: source order, per-file
manifest fields, stop conditions, and `$I`/`$R` semantics were all stated up
front and survived contact with reality. Two edits would have helped: scope
the recycle-bin restoration claim to the measured 252/283 counts, and name
the USPTO credential prerequisite (ID.me-verified MyODP key) in P4 so the
human dependency is visible before the phase starts.

## Lessons (confidence-tiered)

- **HIGH — Critical:** Validate operational shell-script outputs through the
  consuming schema as early as possible — the catalog's
  `CorpusProvenanceRecord` decode caught a digest-encoding defect that two
  layers of hash comparison had structurally hidden (both sides carried the
  same escape marker, so verification passed while the value was wrong).
- **MEDIUM — Best practice:** For at-risk media, salvage with a dumb
  fail-fast script first and formalize the CLI afterward; the 26-PST USB
  copy completed before the first line of repo code was written.
- **MEDIUM — Best practice:** Process one representative per content digest
  during extraction and keep duplicate linkage in the catalog — it cut 1,108
  redundant parses including a 300 MB PST that existed three times.
- **LOW — Consideration:** Effect engines with `R = never` capture their
  platform services at construction; an effectful `make*Engine(config)`
  constructor is the clean pattern when a driver needs spawners or the
  filesystem.

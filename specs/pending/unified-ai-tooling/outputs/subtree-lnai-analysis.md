## Summary
LNAI treats `.ai` (its `UNIFIED_DIR`) as the canonical workspace that houses `config.json`, `settings.json`, `AGENTS.md`, rule Markdown, skills, and per-tool override folders, then parses everything into schema-driven `UnifiedState` before exporting to any tool-specific output, so the same source of truth flows through every step of the pipeline (`packages/core/src/constants.ts:1`, `packages/core/src/parser/index.ts:16`, `packages/core/src/schemas/index.ts:3`, `packages/core/src/types/index.ts:35`).
`runSyncPipeline` orchestrates parse → unify validation → per-tool `Plugin.export` → orphan cleanup → manifest refresh → managed `.gitignore` updates, giving dry-run/skip-cleanup toggles plus tool filtering for predictable sync runs backed by hashed manifests and cleanup helpers (`packages/core/src/pipeline/index.ts:23`, `packages/core/src/manifest/index.ts:16`, `packages/core/src/cleanup/index.ts:6`, `packages/core/src/writer/index.ts:61`, `packages/core/src/writer/index.ts:157`).
The CLI wraps the pipeline so `lnai sync`/`validate` present spinners, option parsing, error categorization, and explicit validation result reporting, while `lnai init` bootstraps the `.ai` structure and lets users pick tools/version control upfront before the sync step (`packages/core/src/commands/sync.ts:16`, `packages/core/src/commands/validate.ts:16`, `packages/core/src/commands/init.ts:17`).

## Key Patterns to Reuse
- Schema-first ingestion: `parseUnifiedConfig` feeds `UnifiedState` compiled from Zod-driven tool, MCP, permissions, rule, and skill schemas so validation invariants can be enforced before any adapter sees the data (`packages/core/src/parser/index.ts:16`, `packages/core/src/schemas/index.ts:3`, `packages/core/src/types/index.ts:35`, `packages/core/src/validator/index.ts:30`).
- Adapter contract plus shared helpers: every plugin implements the `Plugin` interface with `detect/import/export/validate`, reuses `createSkillSymlinks`, and layers overrides from `.ai/.<tool>` via `applyFileOverrides`, enabling adapters to remain stateless and focused on transforming `UnifiedState` (`packages/core/src/plugins/types.ts:8`, `packages/core/src/utils/agents.ts:33`, `packages/core/src/utils/overrides.ts:75`, `packages/core/src/constants.ts:27`).
- Manifest-driven drift detection: the sync loop keeps a `.lnai-manifest.json`, computes deletions against the new outputs, removes stale files, and rewrites a dedicated `.gitignore` section, ensuring generated directories stay tidy and the CLI can skip actual writes through dry runs (`packages/core/src/manifest/index.ts:16`, `packages/core/src/cleanup/index.ts:6`, `packages/core/src/writer/index.ts:61`, `packages/core/src/writer/index.ts:157`, `packages/core/src/pipeline/index.ts:23`).
- Directory-aware rule transformations: helpers that group rules by glob directory feed multiple adapters (e.g., the Cursor/Codex outputs) with consistent descriptions and converted MCP/permission references, so the core only needs to expose rules once (`packages/core/src/utils/rules.ts:25`, `packages/core/src/plugins/codex/index.ts:22`, `packages/core/src/plugins/cursor/transforms.ts:20`, `packages/core/src/utils/transforms.ts:12`).
- Guided scaffold: `initUnifiedConfig` creates the `.ai` tree (config + optional rules/skills directories), validates tool IDs, and lets the CLI prompt for enabled tools/version-control choices before the first sync, giving users a clear migration route (`packages/core/src/init/index.ts:27`, `packages/core/src/commands/init.ts:17`).

## Risks/Anti-patterns
- Adapters routinely emit warnings/skipped reasons whenever a feature (e.g., permissions, MCP servers, root-level rules) is unsupported, so any downstream project must surface such gaps lest users think every config is live despite being ignored (`packages/core/src/plugins/copilot/index.ts:92`, `packages/core/src/plugins/codex/index.ts:81`, `packages/core/src/plugins/windsurf/index.ts:76`).
- Running sync with `--dry-run` or `--skip-cleanup` without following up with a real sync risks orphan files lingering on disk, because manifest-driven deletions and manifest rewrites happen only when `dryRun` is false (`packages/core/src/pipeline/index.ts:23`).
- Generated outputs can be tracked if a tool opts into `versionControl`, so failing to respect that flag when updating `.gitignore` or the manifest would leave repos out of sync by ignoring files users want to commit (`packages/core/src/pipeline/index.ts:167`, `packages/core/src/writer/index.ts:157`, `packages/core/src/constants.ts:27`).

## Concrete Recommendations for beep-sync
1. Define a single canonical directory (e.g., `.beep` or `.sync`) that mirrors LNAI’s `UNIFIED_DIR` layout—config, settings, AGENTS/rules/skills, plus per-tool override subfolders—and parse it into a typed `UnifiedState` before any downstream processing so all adapters read from the same curated configuration (`packages/core/src/constants.ts:1`, `packages/core/src/parser/index.ts:16`, `packages/core/src/init/index.ts:27`).
2. Build schema-driven validators for that directory (using Zod or similar) so errors are surfaced early and every tool adapter can rely on uniform types (`packages/core/src/types/index.ts:35`, `packages/core/src/schemas/index.ts:3`, `packages/core/src/validator/index.ts:30`).
3. Define a lightweight plugin interface (detect/import/export/validate) and reuse shared helpers for symlinking AGENTS/skills and applying `.beep/.<tool>` overrides, keeping adapter code focused on transforming rules/MCP/permissions into native formats (`packages/core/src/plugins/types.ts:8`, `packages/core/src/utils/agents.ts:33`, `packages/core/src/utils/overrides.ts:75`).
4. Mirror LNAI’s sync pipeline utilities—parse → validate → plugin export → cleanup → manifest rewrite → `.gitignore` management—and expose CLI commands for `sync`, `validate`, and `init` with the same spinner/options semantics so users can preview changes, skip cleanup, and bootstrap cleanly (`packages/core/src/pipeline/index.ts:23`, `packages/core/src/manifest/index.ts:16`, `packages/core/src/cleanup/index.ts:6`, `packages/core/src/writer/index.ts:61`, `packages/core/src/commands/sync.ts:16`, `packages/core/src/commands/validate.ts:16`, `packages/core/src/commands/init.ts:17`).
5. Ensure drift detection respects a per-tool `versionControl` toggle: regenerate the manifest, delete orphan files, and rewrite a managed `.gitignore` section every non-dry-run sync so beep-sync never leaves stale outputs or silently ignores tracked files (`packages/core/src/pipeline/index.ts:167`, `packages/core/src/writer/index.ts:157`, `packages/core/src/constants.ts:27`).

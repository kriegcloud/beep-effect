# How DeepWiki turns raw code into navigable wikis

**DeepWiki generates wiki-style documentation from codebases using a RAG-first pipeline** — not compiler-grade static analysis — where repositories are cloned, chunked into text fragments, embedded as vectors, and then fed as context to large language models that produce the prose, diagrams, and hierarchical structure. The system's core innovation is its "cluster-based planning" algorithm, which groups related code files into topical clusters that become wiki pages. Cognition Labs (the $10.2B company behind Devin) built DeepWiki as the free, public version of their internal "Devin Wiki" and "Devin Search" tools, launched in April 2025. While the official product is closed-source, an independent open-source reimplementation (deepwiki-open) and extensive community reverse-engineering reveal that the architecture is fundamentally **LLM-centric with vector retrieval**, not a traditional code-intelligence system built on ASTs, type solvers, or language servers. For your `beep-effect` monorepo — a TypeScript/Effect-based project with deep type hierarchies and domain slicing — this matters: DeepWiki will understand your README's narrative and your file structure well, but may struggle with the kind of cross-package type-level relationships that Effect's tagless-final patterns create.

## The six-stage pipeline from clone to wiki

DeepWiki processes repositories through a sequential pipeline that transforms raw source code into structured documentation. Based on confirmed behavior from the official product, Cognition's documentation, and the deepwiki-open clone, the pipeline operates in six discrete stages.

**Stage 1: Repository acquisition.** DeepWiki clones the repository via Git, supporting GitHub, GitLab, and Bitbucket with platform-specific authentication. For private repos, personal access tokens are URL-encoded into the clone URL. The system maintains a clone cache at `~/.adalflow/repos/` (in the open-source version) to avoid redundant downloads.

**Stage 2: File scanning and filtering.** A recursive directory scan identifies all files, applying exclusion rules for binaries, vendor directories, lock files, and files exceeding size limits. Each file is wrapped in a `Document` object containing structured metadata — crucially, an `is_implementation` flag distinguishes test files from production code, allowing the RAG system to prioritize implementation when retrieving context. A `repo.json` configuration defines these filtering rules. This stage also performs **token counting** using `tiktoken` to ensure files don't exceed embedding model context limits.

**Stage 3: Embedding generation.** Documents are split into overlapping text chunks using a `TextSplitter` component (configured for chunk size and overlap to preserve cross-boundary context). These chunks are then embedded into dense vectors using models like OpenAI's `text-embedding-3-small` or Google's `text-embedding-004`. The resulting vectors are stored in a **FAISS index** for fast similarity search. The open-source version stores these at `~/.adalflow/databases/`. Cognition's production system almost certainly uses a more scalable vector database, but the FAISS-based approach reveals the fundamental architecture.

**Stage 4: Cluster-based structure planning.** This is DeepWiki's most distinctive step. The system sends the repository's **file tree, README content, and RAG-retrieved context** to an LLM via WebSocket. The LLM analyzes this input and returns a proposed wiki structure as structured XML — determining which sections to create, how to nest them hierarchically (up to 3 levels deep, e.g., `5.2 Available Effects`), and what each page should cover. Cognition's official documentation explicitly names this **"cluster-based planning"** as the default algorithm. When a `.devin/wiki.json` file exists in the repo root, users can bypass automatic clustering and specify exact pages. The system enforces limits: **maximum 30 pages** (80 for enterprise), maximum 100 notes, and 10,000 characters per note.

**Stage 5: Per-page content generation.** Each wiki page is generated independently using RAG-augmented LLM calls. For each page, the system retrieves relevant code chunks from the FAISS index based on the page's topic, injects them as context, and asks the LLM to produce comprehensive markdown documentation. Pages are generated serially (`MAX_CONCURRENT=1` in the open-source version) to avoid overwhelming the LLM service. Each page includes inline source citations with file paths and line ranges (e.g., `Sources: src/effects/model.cpp30-226`), linking to specific commits on GitHub.

**Stage 6: Diagram generation and caching.** The LLM generates **Mermaid.js diagrams** for architecture overviews, data flows, and component relationships. These are rendered client-side via a dedicated `Mermaid.tsx` React component. The complete wiki is cached (clone cache → embedding cache → wiki cache in a three-tier strategy), with a **7-day refresh cooldown** on the production site.

## What DeepWiki actually understands (and what it fakes)

The most important distinction about DeepWiki's approach is what constitutes "understanding." **DeepWiki does not perform AST parsing, type checking, or compiler-level analysis.** There is no confirmed use of tree-sitter, Language Server Protocol, or any compiler API. The system reads code as text, embeds it as vectors, and relies on LLMs to interpret meaning. This has profound implications for accuracy.

Community analysis from Hacker News reveals specific failure patterns. An LLVM contributor found that DeepWiki's compilation pipeline diagram was **fundamentally wrong** — misplacing Clang-AST, botching the vectorization/instruction-selection flow, and completely omitting GlobalISel with no discussion of LLVM IR or the pass manager. A LibreOffice maintainer discovered DeepWiki falsely claimed the project uses Buck as its build system — because BUCK files existed in the root directory (used for Maven publishing, not building), and the LLM misinterpreted them as the primary build system.

These failures reveal three systematic biases in DeepWiki's analysis. First, **file-size bias**: the system appears to weight larger files as more important. An LLVM analysis showed that features in single 30,000-line files received extensive coverage, while equally important subsystems split across multiple 4-5K line files were ignored entirely. Second, **root-directory bias**: configuration files at the repo root receive disproportionate attention, leading to mischaracterization when root files serve niche purposes. Third, **naming-convention dependence**: the LLM over-relies on identifier names (function names, package names, struct names) to infer purpose. When naming is legacy, misleading, or domain-specific, documentation becomes subtly wrong — "the kind of mistake that's likely to boomerang on you only a month or so later," as one developer observed.

For your `beep-effect` repository specifically, DeepWiki likely handles the monorepo structure well — it excels at identifying package boundaries from directory layout (`packages/iam/`, `packages/documents/`, etc.) and your extensive README provides rich context for the clustering algorithm. However, Effect-TS's algebraic patterns, tagless-final encoding, and the deeply nested type-level machinery that makes `@beep/common` enforce invariants at compile time would likely be described in surface-level terms without capturing the type-theoretic intent.

## Cognition Labs, Devin, and the code-understanding lineage

DeepWiki is confirmed as a product of **Cognition AI** (also known as Cognition Labs), the company that builds Devin. CEO Scott Wu and CTO Steven Hao lead the company, which has raised over **$400 million at a $10.2 billion valuation**. The product evolution is well-documented: Devin Wiki and Devin Search launched as paid features in early 2025, Devin 2.0 shipped on April 3, 2025, and DeepWiki launched as the free public version on **April 25, 2025**. A DeepWiki MCP Server followed on May 22, 2025, providing programmatic access via three tools (`ask_question`, `read_wiki_contents`, `read_wiki_structure`).

In November 2025, Cognition announced **Windsurf Codemaps**, explicitly described as "the next step" building on DeepWiki and Ask Devin technology. Codemaps is powered by Cognition's proprietary **SWE-1.5 model** (a frontier-scale, hundreds-of-billions-of-parameters model running at up to 950 tokens/second via Cerebras) and Claude Sonnet 4.5. This confirms that Cognition has invested in custom code-understanding models beyond off-the-shelf LLMs, though whether SWE-1.5 currently powers DeepWiki's wiki generation (versus just Codemaps) remains unconfirmed.

Cognition has published **no research papers, no architectural deep-dives, and no detailed technical blog posts** about DeepWiki's internals. Their blog posts are product announcements, not engineering expositions. Scott Wu has discussed DeepWiki in podcast appearances — notably on "How I AI" where he demonstrated using DeepWiki to research codebases before prompting Devin — but these focus on workflow, not architecture. The company clearly treats its code-understanding technology as proprietary competitive advantage. Wu's stated philosophy — "as an engineer, you spend a lot more time reading and understanding code than you do writing code" — frames DeepWiki as infrastructure for the broader Devin ecosystem.

## How DeepWiki compares to the competitive landscape

DeepWiki occupies a specific niche: **zero-friction, fully automated, wiki-style documentation generation from raw source code.** This distinguishes it from most tools in the adjacent space, which either require human authoring, focus on code search rather than documentation, or serve narrower use cases.

**Sourcegraph** represents the most technically sophisticated competitor but serves a fundamentally different purpose. Sourcegraph builds a genuine **code graph** using LSIF (Language Server Index Format) and their successor protocol SCIP, providing precise go-to-definition, find-references, and cross-repository navigation. Their "Normsky" architecture — coined by Steve Yegge — explicitly marries Chomsky-style deterministic parsing with Norvig-style statistical AI. Sourcegraph's AI assistant Cody uses this code graph as structured context, reducing hallucinations. But Sourcegraph does not generate wiki documentation — it's a code search and navigation platform.

**Swimm** offers the closest methodological contrast. Swimm uses a three-step pipeline: **deterministic static analysis first**, then context retrieval, then LLM generation. Their proprietary "Swimm Engine" uses relational ranking algorithms to understand code relationships and importance before any LLM touches the content. Every AI-generated output is anchored to specific code. This deterministic-first approach produces more reliable documentation but requires enterprise setup and CI integration, making it far less accessible than DeepWiki's URL-swap model.

**Google Code Wiki** (launched November 2025) is DeepWiki's most direct competitor, offering virtually identical functionality — auto-generated Wikipedia-style documentation with Gemini-powered AI, architecture diagrams, and conversational chat. Google's key differentiator is **continuous regeneration after every commit**, compared to DeepWiki's static snapshots with 7-day refresh cooldowns. Google also integrates NotebookLM for auto-generated video walkthroughs.

**Greptile** builds genuine code graphs (functions, variables, classes, files, directories with their connections) and uses graph-based RAG for code review, but focuses on PR review rather than documentation generation. **Bloop** uses tree-sitter for precise code navigation across 10+ languages and Qdrant for vector search, but has pivoted toward COBOL-to-Java legacy modernization. **Mintlify**, **GitBook**, and **ReadMe** are documentation platforms requiring human authoring with AI assistance — they are not automated code-to-wiki generators.

DeepWiki's unique advantage remains its **zero-friction access model**: replacing `github.com` with `deepwiki.com` in any URL requires no setup, no API keys, and no installation. No other tool matches this simplicity of access.

## Inside the clustering algorithm and wiki structure decisions

The most opaque — and most interesting — part of DeepWiki is how it decides what pages to create and what to emphasize. The official term is "cluster-based planning," but Cognition has published no details about the algorithm itself.

From observed behavior across many repositories, the clustering appears to follow a consistent pattern. DeepWiki typically generates **9-17 top-level sections** with 1-3 levels of nesting. The structure almost always follows this template: an overview page, a system architecture section, a getting-started section, feature-specific sections (the bulk of the wiki), implementation detail sections, an API reference, and a contributing/development guide. For the KDE/kdenlive project, for example, the wiki produced 9 top-level sections: Overview → Core Architecture → Timeline System → Monitor System → Effects System → Rendering System → Title Editor → Internationalization → Developer Guide — each with relevant subsections.

The algorithm appears to use the **file tree structure as its primary clustering signal**, supplemented by README content and sampled code context. Monorepos with clear directory boundaries (like your `beep-effect` with `apps/`, `packages/`, `tooling/`) likely produce cleaner clustering than flat repositories. The `.devin/wiki.json` configuration mechanism confirms that the automatic clustering can be overridden — suggesting Cognition recognizes it doesn't always produce optimal results for every codebase.

Each generated page lists **"Relevant source files"** at the top — typically 5-20 files per page, linked to specific commit SHAs on GitHub. Source citations appear inline throughout the text with line-number ranges (e.g., `src/effects/model.hpp30-226`). The system also generates **cross-references between wiki pages** using relative paths, creating an interlinked knowledge structure rather than isolated pages.

For diagram generation, the LLM creates Mermaid.js specifications representing its understanding of architecture, data flow, and component relationships. These diagrams are AI-generated interpretations, not extracted from existing documentation or computed from actual dependency graphs. This means they can be impressively comprehensive for well-structured projects but fundamentally wrong for complex systems where the LLM misinterprets code purpose.

## What we know for certain vs. what remains inference

The technical picture of DeepWiki is built on three tiers of evidence, and maintaining epistemic clarity about which tier supports each claim is essential.

**Confirmed by Cognition directly**: DeepWiki is the free public version of Devin Wiki and Devin Search. It uses "cluster-based planning" for wiki structure. It's "powered by Devin." The Q&A uses RAG for context grounding. The `.devin/wiki.json` format allows steering generation. Mermaid.js renders diagrams. The MCP server at `mcp.deepwiki.com` provides three tools. SWE-1.5 and Claude Sonnet 4.5 power the related Codemaps product. Wiki limits are 30 pages standard, 80 enterprise.

**Strongly inferred from the open-source clone and observed behavior**: The pipeline follows clone → scan → embed → cluster → generate → cache stages. Vector embeddings use models like `text-embedding-3-small`. FAISS or equivalent handles similarity search. Pages are generated independently with per-page RAG context. Overlapping text chunks preserve cross-boundary context. File metadata (size, path depth, implementation vs. test) influences prioritization. Three-tier caching (repo, embeddings, wiki) optimizes performance.

**Speculative or unconfirmed**: Whether DeepWiki uses tree-sitter, AST parsing, or any deterministic static analysis (no evidence found — it appears to be purely text-and-LLM based). Which specific LLMs power wiki generation (Claude models are confirmed for Codemaps; likely similar for DeepWiki). Whether it builds an explicit knowledge graph (no evidence; appears to be pure RAG). The exact clustering algorithm internals. Whether SWE-1.5 is used for DeepWiki or only for Codemaps. Infrastructure details (cloud provider, production vector database, scaling architecture).

## Conclusion

DeepWiki represents an **LLM-maximalist approach to code documentation** — betting that sufficiently capable language models, given well-retrieved code context, can produce useful documentation without the traditional code-intelligence infrastructure of AST parsers, type solvers, and semantic analyzers. This bet pays off for well-structured repositories with clear naming conventions and good READMEs, where the LLM's pattern recognition aligns with actual code semantics. It fails predictably for large, complex systems where architectural understanding requires the kind of deep structural analysis that only compiler-grade tooling provides.

For your `beep-effect` monorepo, DeepWiki will likely produce a reasonable overview of your package structure and domain slices, heavily leveraging your opinionated README. But the Effect-TS patterns, the type-level invariants, and the catamorphism over DAGs in your `knowledge` package will likely be described in generic terms. The most technically rigorous alternative for accurate documentation would be Swimm's deterministic-first approach, though it requires far more setup. The emerging competitive threat is Google Code Wiki, which matches DeepWiki's ease of access while adding continuous regeneration — a feature that matters for actively developed codebases.

The deeper insight is architectural: DeepWiki deliberately trades precision for scale and accessibility. By avoiding the complexity of language-specific parsers and type systems, it can index any repository in any language with the same pipeline. This is a product decision, not a technical limitation — and it explains both why DeepWiki can cover 50,000+ repos and why LLVM contributors find its output "distinctly worse than what I would have found just traipsing through the code myself."

## Payload and Schema Contracts

### Contract A: Tag payloads

- Payloads are discriminated by `_tag` and must deserialize via canonical union branches.
- Canonical tag names are sourced from the 113-name literal tuple.

### Contract B: Applicability payloads

- Payload values map to legal AST attachment targets.
- Out-of-surface applicability values are non-conformant.

### Contract C: Source metadata payloads

- `url` and `retrievedAt` require typed codec materialization.
- Missing or malformed provenance fields downgrade claim usability.

### Contract D: Validation payloads

- Exhaustiveness expectations for source groups are explicit numeric contracts.
- Snapshot payloads must include schema-version integrity.

## Behavior Contracts

1. Contract consumers must preserve status labels (`implemented`, `specified`, `conceptual`) when emitting or consuming normalized outputs.
2. Contract consumers must preserve certainty-tier interpretation and avoid flattening deterministic and inferred statements.
3. Contract examples are valid only when their evidence IDs resolve to source-backed facts.
4. Known reliability mismatch conditions are handled as explicit risk entries, not silent contract rewrites.

## Versioning and Ownership Expectations

1. Canonical contract changes require owner approval in the ownership matrix.
2. Breaking changes to discriminated unions, canonical literal sets, or provenance codecs require downstream compatibility notes.
3. Snapshot schema version changes require explicit migration guidance before adoption.
4. Deferred reliability concerns retain owner/disposition metadata until resolved.

## Contract Examples

### Example 1: Valid tagged-union route

A payload with `_tag` mapped to a canonical tag variant is accepted when it decodes through the declared union branch and matches canonical tag-name membership.

### Example 2: Valid provenance object

A source metadata payload is conformant when `url` and `retrievedAt` satisfy typed materialization constraints.

### Example 3: Deferred reliability contract note

Validation scripts that reference local imports inconsistent with artifact location remain under deferred conflict handling and must not be represented as resolved contract stability.

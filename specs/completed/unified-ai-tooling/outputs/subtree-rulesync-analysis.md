# Summary
- Unified `.rulesync/rules/*.md` rely on frontmatter fields such as `root`, `localRoot`, `targets`, and `globs` (plus tool-specific hints) so one canonical source can gate which instructions reach each assistant; the processor enforces a single root and separates local overrides before pushing anything downstream, keeping the canonical set authoritative for every target (see `skills/rulesync/file-formats.md:7`, `src/features/rules/rulesync-rule.ts:20`, `src/features/rules/rules-processor.ts:752`).
- The generator converts each non-root rule into the matching `ToolRule`, then prepends reference sections (TOON for "toon" targets, plain-reference for legacy targets) and simulated-convention guidance before writing the root file; local-root content is either appended to the existing root text or emitted as a dedicated file like `CLAUDE.local.md`, which keeps the final instructions concise while directing consumers to the detailed fragments they need (see `src/features/rules/rules-processor.ts:512`, `src/features/rules/rules-processor.ts:631`, `src/features/rules/rules-processor.ts:685`, `src/features/rules/rules-processor.ts:982`, `src/features/rules/rules-processor.ts:1050`).

# Risks/Anti-patterns
- Multiple or missing root rules stop generation or produce noisy warnings, and `localRoot` is rejected unless a root exists, so be explicit about which file is the canonical root before running the compiler (see `src/features/rules/rules-processor.ts:775`, `src/features/rules/rules-processor.ts:789`).
- `rulesync fetch` defaults to `overwrite`, writes files in parallel, and only skips when `--conflict skip` is passed, so any fetch-like flow in beep-sync needs the same UX safeguards (see `src/lib/fetch.ts:267`, `src/lib/fetch.ts:375`, `skills/rulesync/cli-commands.md:92`).

# Concrete Recommendations for beep-sync
- Mirror the `.rulesync/rules/*.md` layout by hosting canonical instructions in `.beep/rules/` with the same frontmatter vocabulary; this lets beep-sync parse `targets/globs/localRoot` once and route the text into multiple assistants deterministically (see `skills/rulesync/file-formats.md:7`, `src/features/rules/rulesync-rule.ts:20`, `src/features/rules/rules-processor.ts:752`).
- Build the generator pipeline around the same steps as `RulesProcessor`: convert canonical rules into per-tool files, append reference/convention sections when needed, and handle `localRoot` fragments either by appending or by writing a special file so overrides stay scoped (see `src/features/rules/rules-processor.ts:512`, `src/features/rules/rules-processor.ts:548`, `src/features/rules/rules-processor.ts:631`, `src/features/rules/rules-processor.ts:685`, `src/features/rules/rules-processor.ts:982`, `src/features/rules/rules-processor.ts:1050`).
- Offer a conflict strategy mirror of `rulesync fetch` by defaulting to `overwrite` but allowing a `--conflict skip` path (and logging the skipped files) so teams can protect hand-edited outputs when importing or regenerating instructions (see `src/lib/fetch.ts:267`, `src/lib/fetch.ts:375`, `skills/rulesync/cli-commands.md:92`).
- Keep a schema-backed `.beep/mcp.json` (with `enabledTools`/`disabledTools`, optional `targets`, `description`, etc.) and a per-target capability table like `toolMcpFactories` so beep-sync strips unsupported fields before writing native configs; include converters for tools that rename these fields (e.g., `enabled_tools` vs `enabledTools` for Codex) so the canonical file remains tool-agnostic yet expressive (see `skills/rulesync/file-formats.md:215`, `src/features/mcp/rulesync-mcp.ts:20`, `src/features/mcp/mcp-processor.ts:69`, `src/features/mcp/mcp-processor.ts:378`, `src/features/mcp/codexcli-mcp.ts:19`).

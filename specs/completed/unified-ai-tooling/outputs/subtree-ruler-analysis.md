# Subtree Ruler Analysis

## Summary
Ruler gathers `.ruler` Markdown files in a deterministic order (root `AGENTS.md` first, legacy fallbacks next, rest sorted) and concatenates them with explicit `<!-- Source: … -->` markers before handing the combined payload, per-directory MCP bundle, and optional skills bundle to each agent; nested runs discover every `.ruler` tree and treat each directory independently while sharing the same agent selection logic (`.repos/ruler/src/core/FileSystemUtils.ts:64`, `.repos/ruler/src/core/RuleProcessor.ts:3`, `.repos/ruler/src/core/apply-engine.ts:48`, `.repos/ruler/src/lib.ts:45`).

## Key Patterns to Reuse
- Per-agent adapters implement the shared `IAgent` contract, allow per-agent overrides (enabled flags, alternate output instructions/config, MCP strategy), and let `getAgentOutputPaths` gather the applicable destinations so `apply` can capture both outputs and `.bak` backups before touching files (`.repos/ruler/src/agents/IAgent.ts:8`, `.repos/ruler/src/agents/agent-utils.ts:1`, `.repos/ruler/src/core/apply-engine.ts:431`).
- Generated outputs and backups are tracked in `generatedPaths` and fed into the managed `.gitignore` block, where `updateGitignore` writes a monotonic `# START/END Ruler Generated Files` section to `.gitignore` (or `.git/info/exclude`) and `revert` removes that block when rolling back (`.repos/ruler/src/core/apply-engine.ts:440`, `.repos/ruler/src/core/GitignoreUtils.ts:15`, `.repos/ruler/src/revert.ts:174`).
- MCP propagation honors a merge-vs-overwrite strategy pulled from CLI flags, agent config, or the global config, transforms the payload for agent quirks, and defers to `mergeMcp` so overwrite simply replaces the server map while merge unions new entries (`.repos/ruler/src/cli/handlers.ts:55`, `.repos/ruler/src/core/apply-engine.ts:865`, `.repos/ruler/src/mcp/merge.ts:11`).
- Every write is preceded by `backupFile`, and the revert engine knows how to restore `.bak` files, delete generated files lacking backups, and clean extra directories/MCP targets so apply/revert stay symmetric (`.repos/ruler/src/core/FileSystemUtils.ts:187`, `.repos/ruler/src/core/apply-engine.ts:990`, `.repos/ruler/src/core/revert-engine.ts:51`, `.repos/ruler/src/revert.ts:174`).
- Skills support is opt-in/experimental: discovery inspects `.ruler/skills`, `propagateSkills` warns if non-native agents are requested, and propagation copies the tree atomically into each agent’s skills directory (`.repos/ruler/src/core/SkillsProcessor.ts:416`, `.repos/ruler/src/core/SkillsProcessor.ts:490`, `.repos/ruler/src/core/SkillsProcessor.ts:606`, `.repos/ruler/src/core/SkillsUtils.ts:1`).

## Risks/Anti-patterns
- Skills support will delete each agent’s skills directory when `skills` is disabled or `propagateSkills` exits early, so accidental toggles can leave those directories empty; the module documents the feature as experimental and logs warnings when an agent lacks native skills (`.repos/ruler/src/core/SkillsProcessor.ts:416`, `.repos/ruler/src/core/SkillsProcessor.ts:471`).
- The managed `.gitignore` block overwrites any prior `# START/END Ruler Generated Files` section on every run and `revert` removes the entire block (and the file if it becomes empty), so custom ignore entries must live outside that span or risk being deleted on rollback (`.repos/ruler/src/core/GitignoreUtils.ts:71`, `.repos/ruler/src/revert.ts:174`).
- Nested mode forces `nested = true` even when child configs declare `nested = false`, emitting a warning but still processing; teams expecting to opt out of nesting at lower levels should be ready for that behavior (`.repos/ruler/src/core/apply-engine.ts:146`).

## Concrete Recommendations for beep-sync
1. Inherit Ruler’s deterministic rule assembly: walk `.ruler` directories top-down, prioritize repo-root `AGENTS.md`, omit `.ruler/skills`, and concatenate files with `<!-- Source: … -->` markers so downstream adapters see their provenance (`.repos/ruler/src/core/FileSystemUtils.ts:64`, `.repos/ruler/src/core/RuleProcessor.ts:3`, `.repos/ruler/src/core/apply-engine.ts:48`).
2. Mirror the adapter config tableau: define a shared interface with enable flags, alternate output paths, and per-agent MCP controls; normalize user keys via substring matches and resolve the final agent list via CLI/default order before applying outputs so filters, defaults, and overrides work uniformly (`.repos/ruler/src/agents/IAgent.ts:8`, `.repos/ruler/src/core/config-utils.ts:15`, `.repos/ruler/src/core/agent-selection.ts:13`).
3. Keep every generated file backed up (`FileSystemUtils.backupFile`), update a managed `generatedPaths` list for `.gitignore`, and supply a revert path that restores `.bak`, deletes generated files without backups, and cleans empty directories/MCP files so beep-sync can undo changes reliably (`.repos/ruler/src/core/FileSystemUtils.ts:187`, `.repos/ruler/src/core/apply-engine.ts:990`, `.repos/ruler/src/core/revert-engine.ts:51`, `.repos/ruler/src/revert.ts:174`).
4. Surface MCP merge vs overwrite via CLI flags/config and reuse a single `mergeMcp` helper so overwrite simply replaces the server map and merge unions entries while agent-specific transforms (e.g., Claude/Kilo/Gemini) happen in the same flow (`.repos/ruler/src/cli/handlers.ts:55`, `.repos/ruler/src/core/apply-engine.ts:865`, `.repos/ruler/src/mcp/merge.ts:11`).
5. If beep-sync ever needs to sync auxiliary assets, follow the skills pattern: treat `.ruler/skills` as the source, warn/clean when support is off, copy with atomic temp directories, and push the same tree to each agent-specific path so updates stay atomic (`.repos/ruler/src/core/SkillsProcessor.ts:416`, `.repos/ruler/src/core/SkillsProcessor.ts:606`, `.repos/ruler/src/core/SkillsUtils.ts:1`).

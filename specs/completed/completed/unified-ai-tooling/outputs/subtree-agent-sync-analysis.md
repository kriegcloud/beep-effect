# Summary
- Agent Sync uses a hub-and-spoke canonical model with per-config metadata to translate every supported tool into a shared representation, which is then rendered into real files rather than symlinks, so every sync rewrites the destination content while preserving format-specific fields for future round-trips (see `core/canonical_models.py:1` and `core/orchestrator.py:525`).
- The UniversalSyncOrchestrator discovers paired files, consults the SyncStateManager history, resolves conflicts (interactive or via `--force`/`--yes`), and only writes when the newest version wins, with dry-run/strict guards and per-file warnings bubbled up to CLI users (see `core/orchestrator.py:284`, `core/orchestrator.py:368`, and `core/orchestrator.py:449`).
- State is persisted in `~/.agent_sync_state.json` with atomic writes and pair-specific keys so deletions and multi-project runs stay consistent, while the CLI exposes multi-mode ergonomics plus platform-aware path discovery for Claude/Copilot/Gemini profiles (see `core/state_manager.py:1` and `cli/main.py:115` with `core/profile_paths.py:30`).

# Key Patterns to Reuse
1. Canonical data structures with required fields, metadata mixins, and validation for agents, permissions, and slash commands make it easy to support new tools without N² conversions (`core/canonical_models.py:1`).
2. A plugin-ready adapter interface paired with FormatRegistry allows new formats to register a coordinator that knows how to parse, serialize, and advertise supported config types; the registry also provides detection helpers and per-type subdirectory counts (`core/adapter_interface.py:1`, `core/registry.py:19`, `core/registry.py:197`).
3. Format adapters delegate to config-type handlers (e.g., `ClaudeAdapter` maintains a handler map for each type), so format-specific parsing/serializing stays isolated while warning hooks flow back to the orchestrator (`adapters/claude/adapter.py:24`).
4. The universal orchestrator does file discovery, action selection, race-condition guards, conversion warnings, strict-mode halting, and state updates before printing concise stats, plus a merge-mode that combines bidirectional changes (see `core/orchestrator.py:284`, `core/orchestrator.py:525`, and `core/orchestrator.py:708`).
5. SyncStateManager tracks `source_mtime`/`target_mtime`, `last_action`, and pair metadata, saves via atomic rename with secure permissions, and exposes removal/clear helpers so deletions and new files are handled cleanly (`core/state_manager.py:1`, `core/state_manager.py:92`).
6. `sync_all_config_types` wraps the orchestrator to auto-detect available config types (agents, permissions, slash commands), preview counts, confirm with the user, and run each type independently while aggregating warnings/results for high-level CLI commands (`core/orchestrator.py:968`).
7. CLI ergonomics mix modes (`--convert-file`, `--sync-file`, directory sync), flexible flags (`--only`, `--dry-run`, `--force`, `--strict`, `--yes`, `--state-file`, `--no-autodiscover`), auto-discovery via FormatRegistry/profile resolver, and conversion knobs like `--add-argument-hint`, all giving scripted and interactive workflows (`cli/main.py:185` and `core/profile_paths.py:30`).

# Risks/Anti-patterns
- The single JSON state file in the user home mixes all directory pairs, so parallel runs or differing projects can step on each other unless beep-sync namespaces entries or lets teams supply a per-repo state path (`core/state_manager.py:55`).
- Default conflict resolution still prompts interactively (`core/orchestrator.py:449`), which blocks automation unless callers remember to pass `--force`/`--yes`; automation flows should default to reproducible tie-breakers or explicit flags.
- Lossy-conversion warnings are only fatal when `--strict` is enabled, so conversions that downgrade Claude deny rules to Copilot “ask” silently succeed, meaning beep-sync should make strict-like auditing easy (`core/orchestrator.py:534`).
- Auto-detection via FormatRegistry warns but still picks the first match, so ambiguous filenames can route to the wrong adapter unless beep-sync keeps deterministic registration order or requires explicit overrides (`core/registry.py:86`).
- There is no filesystem watcher; the tool only runs when explicitly invoked, so merging relies entirely on manual sync runs and state timestamps, which means beep-sync should be explicit about when rebuilds happen instead of assuming live linkage.

# Concrete Recommendations for beep-sync
1. Mirror the canonical model + metadata mixin so each config type (agent, permission, command) can be parsed from the source format, enriched with metadata, and serialized into every target format without needing symlinks; treat every sync as a generated rewrite (`core/canonical_models.py:1` and `core/orchestrator.py:525`).
2. Offer a FormatAdapter/registry/handler kit similar to `core/adapter_interface.py:25`, `core/registry.py:19`, and `adapters/claude/adapter.py:24` so new adapters plug in cleanly, advertise supported config types, and expose warning hooks for downstream orchestrators.
3. Build a universal orchestrator that (a) discovers file pairs, (b) compares mtimes against a persisted state, (c) resolves conflicts interactively/with `--force`/`--yes`, (d) writes generated files with race-guards, (e) collects warnings, and (f) updates state before printing summaries (`core/orchestrator.py:284`, `core/orchestrator.py:368`, `core/orchestrator.py:525`).
4. Persist sync history in an atomic JSON file keyed by resolved directory pair and include helpers to update/remove entries so deletions and restorations behave predictably; also allow overriding the path via CLI (`core/state_manager.py:1`, `core/state_manager.py:92`, `cli/main.py:285`).
5. Provide a CLI with multi-mode support (`--convert-file`, `--sync-file`, directory sync), auto-discovery (via FormatRegistry and a ProfilePathResolver), `--only` filtering, `--dry-run`, `--force`, `--strict`, `--yes`, and conversion tweaks (`--add-argument-hint`, `--add-handoffs`) so both interactive users and scripts get clear affordances (`cli/main.py:185`, `core/profile_paths.py:30`).
6. Surface conversion warnings and fail fast when users ask for `--strict`; make every adapter emit warnings via the orchestrator’s accumulation so the CLI can log them before writing (`core/orchestrator.py:525`).
7. Since the sync writes generated files (no symlinks), beep-sync should treat targets as derived outputs with explicit deletions, and use the state manager to detect removals/new files rather than trying to canonically map symlinks back into the source tree (`core/orchestrator.py:525`, `core/state_manager.py:55`).

## Summary
- ai-rulez centralizes every config decision under `.ai-rulez/config.yaml` (schema v3) plus the curated `rules`, `context`, `skills`, `agents`, and `domains` trees, so generated outputs are always derived from a single, validated canonical model.
- The V3 generator layers content by profile/domain, keeps generation deterministic by sorting on priority/name and hashing writes, and stamps every target with a managed header that explains how to edit the sources.
- Migration workflows (manual `migrate v3` and the `generate` command's --auto-migrate hook) convert v2 projects into the same directory-based layout while detecting presets, writing the new schema, and validating the result before allowing overwrites.

## Key Patterns to Reuse
- **Canonical schema**: `.ai-rulez/config.yaml` is versioned (`version: "3.0"`, `name`, optional `description`), lists `presets` (built-ins or custom {name,type,path,template}), can define `profiles`, `gitignore`, `includes` (remote/local with `merge_strategy`, `install_to`, etc.), `header` styles, and compression options; schema/ai-rules-v3.schema.json plus docs/configuration.md codify every field and validate automatically.
- **Profile/domain layering**: Documentation (docs/domains.md) and `config.ConfigV3`/`GeneratorV3` keep root rules/context/skills/agents always included while `profiles` selects specific `.ai-rulez/domains/{name}` subtrees; domain files override root counterparts (same filename wins) and `GeneratorV3.getContentForProfile` merges root + domain content before rendering.
- **Generation determinism**: `templates.GenerateHeader`/`sortByTitlePriority` sort rules, sections, and agents by priority (critical→minimal) and name so the output order never jumps; `Generator.shouldWriteFile` hashes existing runs and skips rewrites when content is unchanged, eliminating timestamp churn.
- **Headers & managed markers**: Every generated file prepends a header that can be `detailed`, `compact`, or `minimal` (configurable via `header.style`); the header explains that `.ai-rulez/` is the source of truth, shows folder structure, reminds agents to use the MCP server, and issues a “DO NOT EDIT” warning, giving beep-sync a clear template for marking managed zones.
- **Migration workflows**: `cmd/commands/migrate.go` (with dry-run/validate/force/backups) and `internal/migration/v3_migrate.go` create `.ai-rulez/` directories, convert rules/context/agents to markdown with frontmatter, migrate MCP servers to `mcp.yaml`, detect the correct presets (with a sorted fallback list), emit the new schema-backed `config.yaml`, and finally run a generator to validate the result; `generate` will auto-run this path when a V2 config is detected and `--auto-migrate` allows control (`ask/true/false`).

## Risks/Anti-patterns
- **Implicit default profile**: When no profile flag is passed and no `default` profile is configured, `GeneratorV3.resolveProfile` silently uses `default`, which only pulls root content; forgetting to define a default/profile per team can lead to missing domain-specific rules in generated outputs.
- **Domain collision warnings are only logs**: `config.GetContentForProfile` lets domain files override root content without muting the generator, but collisions merely log warnings; if beep-sync needs stricter ownership, it should validate collisions or namespace domain files to avoid accidental overrides.
- **Include resolution is non-fatal**: `LoadConfigV3` only warns when remote/local includes fail to resolve (errors are logged but generation continues with local content), so a mistaken include could go unnoticed in CI; beep-sync should add post-merge validation that explicitly fails when includes aren’t fetched.
- **MCP server name overlap**: Domain MCP servers override root servers solely by name, so reusing the same identifier in multiple domains silently replaces the root definition; beep-sync should keep MCP names unique or namespace them to avoid cross-domain overrides.

## Concrete Recommendations for beep-sync
- Mirror ai-rulez’s directory blueprint inside `.beep-sync/` (or whatever alias) with `config.yaml`, `rules/`, `context/`, `skills/`, `agents/`, and `domains/{domain}/` trees, validate against a schema inspired by `schema/ai-rules-v3.schema.json`, and surface the same `presets`, `profiles`, `includes`, `header`, and `compression` knobs so tooling can reuse the same parser/validator.
- Adopt profile/domain layering: keep shared guidance in the root folders, add domains per team/subsystem, expose `profiles` that select domain sets, and enforce the “domain overrides root” rule in any generation or sync step; document the default profile behavior so teams know when they must request a specific profile.
- Ensure deterministic generation by sorting content first by priority and then name (the template helpers above can be reused or ported), compute stable hashes before writing, and skip writes when the hash matches to avoid noise in git and watchers.
- Prepend every generated artifact with a header like ai-rulez’s `GenerateHeader` output, mention the editable `.beep-sync/` sources, point users toward the MCP helper (if available), and make sure the header style is configurable (detailed/compact/minimal) per schema so different outputs can keep size constraints in sync.
- Build a V1→V2 migration path (and wired `generate --auto-migrate`) that mirrors the migrator’s steps: detect legacy config, create the new directory tree, convert rules/context/skills/agents with frontmatter, infer presets via a prioritized list, write the new schema-backed config, and run a validation generation before committing; include dry-run/validate/force flags so teams can preview or protect existing `.beep-sync/` directories.

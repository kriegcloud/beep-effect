# TrustGraph V1 Capability Map

## Scope

This map is organized by `beep-effect` workflow and kernel or control-plane responsibility, not by the TrustGraph source tree.

## Capability Map

| Workflow or responsibility | Required v1 behavior | Existing `beep-effect` assets to reuse | Relevant TrustGraph capability evidence | Disposition | Rationale |
| --- | --- | --- | --- | --- | --- |

| Control-plane identity, progress, budgets, and audit | Treat these as architectural, not optional plumbing. | Shared SQLite durability service | Keep curated document state, curated processing state, repo snapshots, and query packet audit in the same SQLite substrate with separate contracts. | [RepoMemorySqlLive.ts](/home/elpresidank/YeeBois/projects/beep-effect/packages/repo-memory/sqlite/src/RepoMemorySqlLive.ts) and existing store contracts | TrustGraph uses separate infrastructure layers, but the required capability is durable local state, not matching backing technologies | `adapt` | Phase 1 is intentionally SQLite-first and single-node. |

| Semantic overlay on retrieval | Preserve best-effort semantic enrichment as non-gating and keep claim or evidence modeling for later phases. | Existing semantic overlay and semantic store boundary in repo-memory | TrustGraph graph RAG and graph query capabilities show the longer-term direction | `defer` | The capability is relevant, but not required for the Phase 1 grounded retrieval proof. |
| Claim and evidence modeling | Add first-class claim-evidence objects only after the deterministic packet slice works. | Existing semantic-web and trust docs | TrustGraph graph-centric capabilities and later expert-memory docs | `defer` | Explicitly Phase 2 or later. |
| Vector embeddings for docs | Do not use vector stores or semantic similarity search in Phase 1. | Existing SQLite lexical retrieval posture | TrustGraph document embeddings and vector backends | `defer` | The prompt explicitly prefers grounded retrieval first and SQLite-first durability. |
| General document library beyond curated repo docs | Do not support arbitrary local files, PDFs, workspace-global notes, or user-managed manifests in Phase 1. | Fixed curated path precedent in [TrustGraphRuntime.ts](/home/elpresidank/YeeBois/projects/beep-effect/tooling/cli/src/commands/TrustGraph/internal/TrustGraphRuntime.ts) | TrustGraph supports broader document ingestion | `exclude` | That would turn the slice into a broader document platform. |
| Agent runtime and tool orchestration | Do not port agent, ReAct, or generic tool-execution services. | None required for Phase 1 | TrustGraph `agent` service surfaces and related gateway APIs | `exclude` | Explicit non-goal. |
| Text completion and prompt serving | Do not port prompt or text-completion services. | None required for Phase 1 | TrustGraph `text-completion` and prompt-oriented surfaces | `exclude` | Explicit non-goal and incompatible with the deterministic Phase 1 answer path. |
| Distributed or multi-backend deployment | Do not port flow topology, object-store assumptions, Cassandra, or vector-backend portability. | Existing single-node SQLite posture | TrustGraph deployment and storage topology | `exclude` | Phase 1 is intentionally single-node and local-first. |

## Summary

The core capability choices are:

- `adapt` TrustGraph's document-library and processing-kernel behaviors into local repo-memory package laws
- `adopt` local repo-native indexing, packet law, control-plane discipline, SQLite durability, and observability
- `defer` claim-evidence and graph-heavy semantic expansion
- `exclude` agent, prompt, generalized document-platform, vector-store, and distributed-platform concerns

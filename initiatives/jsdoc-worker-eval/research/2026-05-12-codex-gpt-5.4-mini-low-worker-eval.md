# Codex GPT-5.4 Mini Low Worker Eval

## Summary

The hosted Codex baseline proved that `beep docgen quality-worker-eval` can run
read-only worker turns against a hosted Codex model with explicit provider,
model, and reasoning-effort controls.

The run completed three repo-wide remediation packets using `gpt-5.4-mini` at
low reasoning. All three worker turns completed, all three drafts were marked
`candidate`, and the worker reported zero policy violations.

## Command

```sh
bun run beep docgen quality-worker-eval --all --provider codex --model gpt-5.4-mini --reasoning-effort low --packet-limit 3 --output initiatives/jsdoc-worker-eval/history/outputs/2026-05-12-codex-gpt-5.4-mini-low-worker-eval.json
```

## Outcome

- Provider: `codex`
- Model: `gpt-5.4-mini`
- Reasoning effort: `low`
- Scope: `all`
- Source packets: 25
- Selected packets: 3
- Completed packets: 3
- Candidate drafts: 3
- Failed packets: 0
- Timed-out packets: 0
- Policy violations: 0
- Total runtime: about 27.3 seconds

Selected packets covered three `VERSION` exports from the architecture-lab
package family:

- `@beep/architecture-lab-proof`
- `@beep/architecture-lab-client`
- `@beep/architecture-lab-config`

The first two hosted attempts failed because the worker response schema used
`LiteralKit.members` objects as JSON Schema enum values. The implementation was
fixed to use `LiteralKit.literals`, after which the hosted baseline completed.

## Recommendation

Keep worker eval read-only. Hosted Codex with a small model and low reasoning is
promising enough for more evaluation and reviewer comparison, but this is not
enough evidence to unblock auto-remediation write mode.

Use this report as the hosted baseline when comparing future local GPU workers
or Runpod-hosted Qwen runs.

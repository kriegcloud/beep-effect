# RDFS/OWL reasoning engines for code knowledge graphs

**Forward-chaining RDFS/OWL reasoning over code knowledge graphs is a well-founded approach for automated JSDoc enrichment, with polynomial-time complexity guarantees under OWL 2 RL, mature incremental update algorithms, and strong empirical evidence that pre-computed KG inferences dramatically reduce LLM hallucination in documentation generation.** The system described — a TypeScript monorepo using Effect-TS with a forward chainer implementing RDFS rules (rdfs2, rdfs3, rdfs5, rdfs7, rdfs9, rdfs11) and OWL rules (sameAs, inverseOf, TransitiveProperty, SymmetricProperty) over FalkorDB — aligns directly with the OWL 2 RL profile, which was designed for exactly this pattern of rule-based materialization. The key architectural challenge is that FalkorDB has no built-in reasoning engine, requiring all inference to happen in the application layer with results written back as explicit triples. This report covers formal foundations, optimization strategies, and integration patterns across all ten requested dimensions.

## Code ontologies provide proven blueprints for modeling TypeScript as RDF

Several mature ontologies exist for representing source code as RDF triples, with **CodeOntology** (Atzeni & Atzori, ISWC 2017) being the most directly relevant. CodeOntology provides an OWL 2 ontology rooted in a `CodeElement` superclass, with disjoint subclasses for classes, methods, variables, statements, and expressions. Applied to OpenJDK 8, it produced **2+ million RDF triples** of structural information. The ontology models part-whole relations extensively — packages contain classes, classes contain methods, methods contain statements — using Ontology Design Patterns for composition. CodeOntology also applies Named Entity Disambiguation to link code comment entities to DBpedia resources, a precursor to the JSDoc enrichment pipeline described.

For broader software engineering modeling, **SEON** (Software Engineering Ontology Network, Borges Ruy et al., EKAW 2016) provides a three-layer pyramid: a foundational layer based on UFO (Unified Foundational Ontology), a core layer with the Software Process Ontology, and domain-specific ontologies covering requirements, design, coding, and testing. A separate SEON from the University of Zurich focuses on software evolution with dedicated ontologies for issue tracking, code metrics, and repository history.

The mapping from TypeScript constructs to RDF is straightforward. Class inheritance maps to `rdfs:subClassOf`, making rdfs9 (subclass entailment) and rdfs11 (subclass transitivity) directly applicable for propagating type information and documentation through class hierarchies. Interface implementations map to custom RDF properties with transitive sub-properties, as demonstrated by **SCRO** (Source Code Representation Ontology), which defines **56 OWL classes and 73 object/data properties** including transitive properties for inheritance chains. Module containment maps to part-whole composition properties, and function call graphs map to invocation edges — IBM's **GraphGen4Code** demonstrated this at scale, producing **2.09 billion triples** from 1.3 million Python files with interprocedural data flow analysis. No TypeScript-specific RDF ontology exists in the literature, but the CodeOntology approach was designed for language flexibility and provides the template.

## Incremental forward chaining avoids full recomputation on every commit

When a small set of files changes, the critical question is how to scope reasoning to only the affected subgraph. The state of the art is the **Backward/Forward (B/F) algorithm** developed at Oxford for RDFox (Motik, Nenov, Piro & Horrocks, AAAI 2015). B/F improves on the foundational **DRed (Delete/Rederive) algorithm** (Gupta, Mumick & Subrahmanian, SIGMOD 1993) by reducing overestimation during deletions. Where DRed blindly overdeletes a superset of potentially affected facts and then rederives those with alternative support, B/F uses backward chaining to check whether a deleted fact still has supporting derivations before removing it. This eliminates unnecessary work when facts have multiple derivation paths — common in code KGs where a type might be inferred through both inheritance and interface implementation.

For the RETE algorithm adapted to RDF, triple patterns become alpha nodes that filter on predicate values, and beta nodes perform joins between matched patterns. RETE provides **O(1) incremental pattern matching** when adding a new fact — the fact propagates through the existing network, activating only affected nodes. A two-phase RETE optimization (Tai, 2015) builds only the alpha network first, gathers statistics, then constructs the beta network with a "most specific condition first" heuristic to minimize intermediate join results. Node sharing between rules with identical patterns (e.g., rdfs9 and other rules sharing the `?x rdf:type ?y` pattern) reduces both memory and matching cost.

**Complexity bounds** for the system's specific rules are favorable. RDFS closure computation is **polynomial time** for finite RDF graphs. Ground RDFS entailment (no blank nodes in the target — the practical case for code KGs) is in **P**. OWL 2 RL forward chaining is also **PTime in data complexity**, since RL is translatable to Datalog. The specific rules implemented (rdfs2/3/5/7/9/11 plus sameAs/inverseOf/TransitiveProperty/SymmetricProperty) fall squarely within OWL 2 RL's ~70–80 rule rule set. An important optimization: rdfs5 and rdfs11 (subproperty and subclass transitivity) are TBox-only rules that can be computed once when the schema changes, while rdfs2, rdfs3, rdfs7, and rdfs9 are the runtime data-dependent rules that fire on every ABox update.

**Differential Dataflow** (McSherry) offers an alternative paradigm with uniform performance for both additions and deletions. **DDlog** compiles Datalog programs into incremental Rust programs, and **DYRE** provides an interpreter avoiding DDlog's slow compilation. However, DD uses 2–500× more memory than specialized approaches and struggles with deep recursion (long transitive closures), making it less suitable than B/F for code KGs with deep inheritance chains.

## Predicate-based indexing is the optimal strategy for code KG forward chaining

Production reasoners achieve million-triple-per-second materialization through a combination of dictionary encoding, multi-order indexing, and predicate-partitioned storage. **RDFox** achieves **6.1 million triples/second** reasoning rate with memory as low as **36.9 bytes per triple** by encoding each RDF resource as a sequential integer ID (usable as array indexes), maintaining sorted/clustered indexes for merge joins, and batch-oriented index maintenance during parallel materialization.

For triple indexing, three collation orders suffice to efficiently answer all eight triple pattern types: **SPO** (handles s,p,o / s,p,? / s,?,? / ?,?,?), **POS** (handles ?,p,o / ?,p,?), and **OSP** (handles ?,?,o / s,?,o). This is the approach used by Apache Jena TDB2 and Apache Rya. At the extreme, **RDF-3X** maintains 15 indexes (all 6 permutations plus 6 aggregated and 3 single-value) with B+-tree storage and LZ77 compression, trading space for query speed. For forward chaining specifically, **predicate-based indexing is critical** because rule antecedents almost always specify predicates — GraphDB uses PSO and POS indexes for exactly this reason.

For the system's scale of **1K–100K triples** (a typical code module's subgraph), in-memory hash-based approaches dominate. At 100K triples with RDFox's footprint, the entire graph fits in ~3.7 MB RAM and loads in ~0.1 seconds. At this scale, hash tables provide O(1) point lookups that outperform tree-based structures. A practical architecture for the Effect-TS forward chainer: maintain a predicate-partitioned hash map where each predicate (e.g., `rdf:type`, `rdfs:subClassOf`, `code:invokes`) maps to a set of (subject, object) pairs. This directly mirrors RETE alpha nodes and enables constant-time pattern matching per rule antecedent.

FalkorDB itself uses **GraphBLAS sparse matrices** for adjacency representation — each relationship type has its own sparse adjacency matrix, and multi-hop traversals become matrix multiplications. This is structurally similar to predicate-partitioned indexing. However, FalkorDB has **no built-in forward chaining**, so the Effect-TS reasoner must maintain its own in-memory index structures for rule evaluation and write results back via Cypher queries.

## The owl:sameAs problem demands careful canonicalization for code entities

TypeScript/JavaScript module systems create pervasive identity challenges. Re-exports (`export { Foo } from './module-a'`) create true runtime identity. Aliased imports (`import { Foo as Bar }`) create binding identity. Ambient declarations in `.d.ts` files declare types corresponding to runtime objects. Module augmentation extends classes with new members, creating entities that are "the same class" with different shapes in different contexts.

The formal **owl:sameAs problem** is well-documented: sameAs is reflexive, symmetric, transitive, and satisfies the replacement property (a congruence), meaning if a = b, ALL statements about a also hold for b. This creates a **combinatorial explosion**: for an owl:sameAs clique of size n, standard OWL 2 RL rules derive **n² sameAs triples** via **2n³ + n² + n derivations**, and each triple with terms in cliques of sizes nₛ, nₚ, nₒ gets expanded to nₛ × nₚ × nₒ triples. Empirical data from the BTC 2010 dataset shows most sameAs components are small (average 2.4 resources per component across 2.9 million components), but outlier components with thousands of resources exist.

The solution is **canonical representative rewriting**, as implemented by RDFox. The approach uses a **concurrent union-find** data structure (amortized O(1) per operation via union by rank and path compression) to maintain equivalence classes. When `a owl:sameAs b` is asserted, a canonical representative is chosen, and all occurrences of the non-canonical resource are rewritten in both data and rules. RDFox's rewriting approach reduces materialized triples by up to **7.8×** and materialization time by up to **31.1×** compared to naive sameAs expansion. The critical subtlety: rules must also be rewritten, not just data — failing to rewrite rule constants can cause missing inferences.

For code entities specifically, owl:sameAs is appropriate for re-exports and aliased imports (true runtime identity), while **skos:exactMatch** (transitive, symmetric, but without property propagation) is better for cases like module augmentation or ambient declarations where entities are "the same concept" but have legitimately different properties in different contexts.

## Documentation propagation maps naturally to RDFS inference rules

Existing documentation inheritance tools provide the behavioral specification that RDFS/OWL rules should replicate. **Javadoc's @inheritDoc** follows a precise algorithm: (1) look in each directly implemented/extended interface in declaration order, (2) use the first doc comment found, (3) if none, recursively apply to each interface, (4) if still none, look in the superclass. TSDoc's `{@inheritDoc}` can reference unrelated classes or imports from separate NPM packages, copying summary, @remarks, @param, @returns, and @typeParam blocks. The key insight from Drupal's long-running debate is that fine-grained partial inheritance (inherit most but override one @param) is **extremely difficult to implement** and no documentation system fully supports it.

The RDFS rules implemented by the system map directly to documentation propagation patterns:

- **rdfs9 (subclass entailment)** propagates documentation from parent classes to subclass instances — the foundational rule for `@inheritDoc` behavior
- **rdfs7 (subproperty entailment)** enables modeling documentation property hierarchies where a detailed-description property is a sub-property of a general-description
- **rdfs11 (subclass transitivity)** enables multi-level inheritance chains for documentation
- **owl:TransitiveProperty** models transitive documentation containment through module→submodule→class hierarchies

For **@throws propagation through call chains**, a custom rule captures: if caller invokes callee and callee throws exception E, then caller may throw E. Making the invocation property transitive via `owl:TransitiveProperty` automatically propagates exceptions through arbitrary call depths. **Deprecation cascading** follows similarly: if entity E is deprecated and dependent D depends on E, then D uses a deprecated entity — with transitivity propagating through dependency chains. Meta's **SCARF** system demonstrates this at production scale, recursively discovering all internal and external dependencies of deprecated code and determining the correct order of operations for safe deletion. Maven ecosystem studies found failure propagation chains averaging **90.80 levels deep**, underscoring the need for automated transitive reasoning.

**Design by Contract** principles from Eiffel provide the formal foundation for contract propagation: preconditions should only be weakened (or kept same) in overriding methods, postconditions can only be strengthened. This asymmetry means @throws documentation should accumulate (union) through call chains while @param constraints should intersect.

## Provenance tracking through named graphs and confidence decay

For a code KG, provenance tracking serves two purposes: explaining why a fact was inferred (for developer trust) and enabling efficient invalidation when source files change. The **W3C PROV-O** ontology provides the formal framework, with parsed facts modeled as `prov:Entity` generated by parsing `prov:Activity`, and inferred facts as entities derived from base facts via inference activities. However, PROV-O's full machinery is heavyweight for per-triple tracking.

The practical recommendation is a **hybrid approach**: **named graphs** for batch-level provenance (grouping all facts parsed from a single file or produced by a single inference run) combined with **RDF-star** annotations for statement-level metadata when needed. Named graphs add minimal overhead (one URI per group) and are well-supported by quad stores. RDF-star (`<<:classA :inheritsFrom :classB>> :confidence 0.95`) is compact and queryable, far superior to classic reification which requires 4 extra triples per annotated statement. FalkorDB, as a property graph database, naturally supports per-edge property annotations that serve the same purpose.

For **confidence scoring**, a multiplicative decay model works well:

- Directly parsed facts from source code AST: **confidence = 1.0**
- Statically analyzed facts (call graphs, data flow): **confidence = 0.95**
- Single-hop inferences: **confidence = min(premise confidences) × rule_reliability**
- Multi-hop chain decay: **confidence = base × decay_factor^depth** (e.g., 0.95 × 0.9^depth)

Rule reliability factors should vary by rule type: type hierarchy traversal (rdfs9/rdfs11) gets **0.95** (highly reliable), @throws propagation via transitive calls gets **0.85** (may over-approximate), heuristic pattern detection gets **0.70**. This approach aligns with work on UKGE (Uncertain Knowledge Graph Embedding) and the IEEE paper on continuous confidence propagation across KGs.

A **JTMS (Justification-based Truth Maintenance System)** inspired architecture is ideal for the per-commit update scenario: each inferred fact stores its list of premise fact IDs. When a file changes, trace forward from affected parsed facts to identify all dependent inferences for invalidation and recomputation. This is precisely the dependency-directed backtracking that Doyle (1979) formalized, and it connects directly to the B/F algorithm's backward checking during incremental deletions.

## Tiered partial materialization balances freshness against query speed

Full materialization (GraphDB's approach) stores all inferred triples for optimal query speed but inflates storage — track information alone can consume **90% of total storage** for large datasets. Pure backward chaining (Stardog/Virtuoso's approach) adds zero load-time overhead but incurs **1–7.43× query-time penalty** per the QueryPIE study (ISWC 2011). For a code KG that changes every commit, neither extreme is optimal.

The recommended architecture is **tiered partial materialization** with four levels. **Tier 1** (always materialized, confidence 1.0): base facts parsed from source code, stored in named graphs per source file, incrementally updated by re-parsing only changed files. **Tier 2** (eagerly materialized, confidence ≥ 0.85): schema/type hierarchy, module dependency graph, single-hop relational inferences — updated via incremental B/F-style reasoning touching only affected subgraphs. **Tier 3** (lazily materialized/cached, confidence 0.5–0.85): multi-hop transitive dependencies, cross-module impact analysis — computed on demand and cached with TTL, invalidated via JTMS-style dependency tracking. **Tier 4** (never materialized): full-codebase traversals and exploratory queries, computed via backward chaining at query time.

QueryPIE's key insight is that pre-computing only the **terminological closure** (TBox inferences like class hierarchies and property characteristics) is tiny (seconds, few triples) but reduces backward-chaining work by **an order of magnitude**. For the specific rules implemented, this means materializing the transitive closures of rdfs:subClassOf (rdfs11) and rdfs:subPropertyOf (rdfs5) eagerly, then applying rdfs2/3/7/9 incrementally as ABox data changes.

For **embedding pipelines**, the recommendation is to generate embeddings from Tier 1 + Tier 2 materialized data (stable, high-confidence) and use incremental embedding updates for changed subgraphs. Volatile Tier 3 inferences should be handled at retrieval time rather than baked into embeddings.

## Locality-based modules enable scoped reasoning over file-change boundaries

When reasoning over a subgraph, the fundamental challenge is ensuring completeness: inferred facts may depend on facts outside the scope. **Syntactic locality modules** (Cuenca Grau et al., WWW 2007) provide the formal solution. Given a signature S (set of entity names of interest) and ontology O, a locality-based module extracts the **minimal subset M ⊆ O** that completely captures O's meaning for terms in S. ⊤⊥*-modules are both self-contained (include all axioms needed for their own entailments) and depleting (the remainder of the ontology is also self-contained). Empirical results on SNOMED CT showed the largest locality module was ~1/10,000 of the full ontology size — demonstrating dramatic reductions for practical ontologies.

For a code KG, the natural scope boundary is the **transitive import closure** of changed files. When `file-a.ts` changes, the reasoning scope includes file-a's direct entities, everything file-a imports (transitively), and everything that imports file-a (transitively, for downstream propagation). This maps to **subgraph-based ABox materialization** (Zhu et al., KSEM 2022), which divides the ABox into instance-centered multi-hop subgraphs and reasons over each independently — experiments showed this approach **effectively improves reasoning efficiency and achieves superior scalability**.

**E-connections** (Cuenca Grau, Parsia & Sirin, 2006) provide a formal framework where component ontologies remain in disjoint domains connected by link properties, preserving decidability. For a code monorepo, each package could be an E-connection component with import/export relationships as link properties. **Package-based Description Logics** (Bao & Honavar, AAAI 2006) further support partial domain overlap and selective knowledge sharing between modules. Graph partitioning algorithms like **IOGP** (HPDC 2017) achieve 2× better query performance with <10% overhead through incremental online partitioning that adjusts as graphs grow.

## OWL 2 RL is the optimal profile for code knowledge graph reasoning

The system's rule set falls squarely within **OWL 2 RL**, which was designed for scalable rule-based forward chaining. RL achieves **PTime data complexity** across all standard reasoning tasks: consistency checking, subsumption, instance checking, and conjunctive query answering. This compares favorably to OWL 2 DL's N2ExpTime-complete combined complexity and even OWL 2 EL's PTime (which achieves the same complexity class but via completion-based algorithms rather than forward chaining).

The three OWL 2 profiles are **mutually incomparable** — no profile is a subset of another, and combining any two breaks tractability. OWL 2 RL's specific advantages for code KGs: it supports owl:sameAs reasoning (essential for entity deduplication), property chains (useful for transitive containment and call chains), negation via disjointness constraints (a node cannot be both a FunctionDeclaration and a ClassDeclaration), and full Datalog translation. The one limitation — no existential quantification in rule heads — is rarely needed for code KGs where entities are explicitly declared rather than existentially inferred.

The complete OWL 2 RL/RDF rule set contains approximately **70–80 rules** organized into categories: equality semantics (~6 rules for sameAs), class axiom semantics, property axiom semantics, class expression semantics (intersection, union, hasValue, cardinality), schema vocabulary semantics (functional/inverse-functional, symmetric/transitive), and inconsistency detection rules. The system's implemented subset (rdfs2/3/5/7/9/11 plus sameAs symmetry/transitivity/propagation, inverseOf, TransitiveProperty, SymmetricProperty) covers the most performance-critical rules. Adding further OWL 2 RL rules incrementally — particularly **property chain axioms** for modeling complex propagation like "overrides + hasDocumentation → inheritsDocumentation" — would extend the system's inference capability without changing its complexity class.

## Pre-computed inferences cut LLM hallucination from 63% to under 2%

The most compelling evidence for integrating KG reasoning with LLM documentation generation comes from an ontology-grounded GraphRAG study for clinical QA (PubMed 2025): hallucination rates dropped from **63% (ChatGPT-4) and 48% (DeepSeek-R1) to just 1.7%**, with accuracy rising from 37% to **98%**. While this is a medical domain result, the mechanism — providing structured, verified facts as context rather than relying on LLM inference — applies directly to code documentation.

**Microsoft's GraphRAG** architecture provides the integration template: segment the corpus into text units, extract entities and relationships, build a community hierarchy via graph algorithms (Leiden), and generate community summaries. For code KGs, the "extraction" phase is replaced by deterministic parsing and forward-chaining inference, which is more reliable than LLM-based entity extraction. The query phase fans out from specific entities to neighbors and associated context, which is precisely what the JSDoc enrichment pipeline needs — fan out from a function to its transitive call chain, inherited documentation, thrown exceptions, and deprecation status.

**CODEXGRAPH** (NAACL 2025) demonstrates the pattern specifically for code: static analysis extracts code graphs, and an LLM agent translates natural language questions into Cypher queries against the graph. The key finding was that **edges in code graphs significantly matter** — removing structural relationships degraded performance across all LLMs tested. The system described goes further by pre-computing inferred edges (transitive dependencies, inherited documentation, propagated @throws) that the LLM would otherwise have to guess.

The practical integration pattern has four layers: (1) **KG as background knowledge** — inject pre-computed transitive closures, aggregated facts, and inferred documentation as structured context into LLM prompts; (2) **KG as reasoning guide** — provide multi-hop reasoning paths the LLM follows rather than attempting its own chain-of-thought; (3) **KG as validator** — after the LLM generates JSDoc, validate against KG-known facts and flag hallucinations; (4) **KG as confidence annotator** — attach confidence scores to each inferred fact so the LLM can weight information appropriately. Current LLM-based JSDoc tools (llmdocgen, JetBrains AI) fundamentally cannot know cross-file information — a code KG with pre-computed transitive call chains, exception propagation, and dependency analysis provides exactly the missing context.

## Conclusion

The system's architecture — an Effect-TS forward chainer implementing RDFS/OWL rules over FalkorDB — is well-grounded in both theory and practice. The implemented rule set corresponds to a subset of OWL 2 RL with **PTime data complexity**, ensuring scalability. The critical implementation decisions are: use **union-find canonicalization** for owl:sameAs to avoid the n³ derivation explosion; adopt the **B/F algorithm** rather than DRed for incremental deletions to minimize overdeleting; implement **predicate-partitioned hash maps** as the in-memory index structure for sub-100K-triple subgraphs; apply **tiered partial materialization** with TBox closure computed eagerly and ABox inferences split between eager (Tier 2) and lazy (Tier 3); and track provenance via **named graphs with JTMS-inspired dependency lists** for efficient per-commit invalidation. The unique value proposition of this approach over LLM-only documentation generation is that forward-chaining inference produces deterministic, provenance-tracked, confidence-scored facts — @throws chains, documentation inheritance, deprecation cascades — that ground LLM generation in verified structural knowledge rather than probabilistic guessing.

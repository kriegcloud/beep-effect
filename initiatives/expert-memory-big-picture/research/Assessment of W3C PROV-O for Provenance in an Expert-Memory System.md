# Assessment of W3C PROV-O for Provenance in an Expert-Memory System

## Purpose

Assess whether PROV-O is the right provenance backbone for the expert-memory direction in this repo, and define the constraints required to use it without collapsing evidence, lifecycle time, or operational control into a single RDF layer.

## Executive Position

PROV-O is a strong default backbone for interoperable provenance in this architecture, but it is not the whole provenance solution.

For this repo, the right posture is:

- use PROV-O for lineage and interchange
- pair it with explicit evidence-anchor modeling
- keep lifecycle and domain time fields explicit
- generate bounded provenance views instead of treating one global provenance graph as the operational truth
- keep policy and access-control concerns outside PROV-O itself

In short: PROV-O fits the semantic and export layer of expert memory, but the full system still needs claims, evidence, lifecycle fields, control-plane state, and operational storage beyond plain PROV graphs.

## Why PROV-O Fits This Repo

### 1. It matches the run-lineage problem directly

The expert-memory work repeatedly emphasizes explainable runs, grounded retrieval packets, and auditable execution. PROV-O maps well onto that shape because it gives a stable vocabulary for:

- entities produced or consumed by the system
- activities such as extraction, enrichment, validation, or packet assembly
- agents such as humans, software components, or automated services

That is the part of provenance the repo already needs regardless of domain.

### 2. It supports the claim-and-evidence posture

The broader architecture in this repo is not trying to preserve raw graph edges alone. It is moving toward claims, evidence, provenance, and time as the durable trust surface. PROV-O fits well when:

- a claim record is treated as a provenance-bearing artifact
- an extraction or reasoning step is modeled as an activity
- derivation, revision, quotation, and primary-source relationships are kept explicit

This aligns with the repo's direction without forcing every domain concept into provenance vocabulary.

### 3. It keeps semantic-web interoperability available

The semantic-web package work wants a provenance layer that is compatible with RDF, JSON-LD, and bounded projection/export workflows. PROV-O is a better fit for that role than inventing a private provenance ontology from scratch.

## Where PROV-O Is Not Enough

### 1. Evidence anchoring

PROV-O can say that something was derived from or generated by something else, but it does not by itself give a complete answer to:

- which exact span in a document supports this claim
- which selector should the UI highlight
- how a retrieval packet should surface evidence precisely

That requires an evidence-anchor model. In this repo, the best fit is an explicit evidence layer that can optionally align with the [Web Annotation Data Model](https://www.w3.org/TR/annotation-model/) and its selector vocabulary.

### 2. Lifecycle and domain time

The expert-memory architecture needs more than one time axis. It needs to distinguish concepts such as:

- observed time
- published time
- ingested time
- asserted time
- derived time
- effective time
- superseded time

PROV activity timestamps are useful, but they should not absorb all of these domain meanings. Lifecycle fields should remain first-class rather than being forced into a single provenance timestamp story.

### 3. Policy and access control

Provenance is often more sensitive than the user-facing answer it supports. PROV-O can express lineage, but it is not the policy layer. The system still needs a separate access-control and redaction posture for:

- evidence spans
- provenance bundles
- retrieval packets
- audit exports

### 4. Operational storage and scale

The repo's architecture is not "RDF everywhere." The representation layers and control-plane documents already argue for bounded, explainable projections over mechanically grounded operational storage. That means the safest posture is:

- store operational truth in the right runtime store for the job
- emit or project PROV views per run, claim, or retrieval packet
- avoid one ever-growing provenance graph as the only runtime source of truth

## Recommended Modeling Posture

### Core mapping

Use PROV-O at the level where lineage matters most:

- `ClaimRecord` or equivalent durable assertion artifact -> provenance-bearing entity
- evidence anchor -> provenance-bearing entity linked to a span, selector, or source anchor
- extraction, enrichment, validation, reasoning, merge, or packet assembly step -> activity
- human operator, software module, model-backed step, or workflow actor -> agent

### Bounded projection rule

Prefer bounded provenance products such as:

- per-run provenance views
- per-derived-claim provenance views
- per-retrieval-packet provenance views
- focused audit exports

Do not make an unbounded global provenance graph the primary interaction surface.

### Evidence rule

Every non-trivial claim should be able to point at evidence explicitly. PROV-O should describe lineage between the claim, the run, and the source artifact, while the evidence layer should describe:

- exact target document or resource
- selectors or offsets when needed
- retrieval-ready evidence payloads

### Time rule

Keep lifecycle and business time explicit even when a claim or activity also has PROV timestamps. The system should be able to answer:

- what happened when
- when the system learned it
- when the claim became effective
- when it was corrected or superseded

without pretending those are all the same thing.

## Practical Comparison Posture

This repo does not need PROV-O to do every job.

Use it for:

- interoperable provenance vocabulary
- run and derivation lineage
- bounded semantic exports
- claim-to-source and run-to-output traceability

Do not expect it to replace:

- evidence selectors
- lifecycle modeling
- access-control policy
- operational workflow state
- domain-specific contradiction and supersession logic

## Recommendation

Treat [PROV-O: The PROV Ontology](https://www.w3.org/TR/prov-o/) as the interoperable provenance backbone for expert memory in this repo.

Adopt it with these guardrails:

1. Keep claims and evidence as first-class artifacts above raw edges.
2. Pair provenance with explicit evidence anchors.
3. Keep lifecycle time explicit and domain-shaped.
4. Generate bounded provenance projections instead of relying on one global graph.
5. Keep policy and runtime control outside the PROV-O vocabulary itself.

That combination matches both the expert-memory big-picture work and the semantic-web package direction without overclaiming what provenance alone can do.

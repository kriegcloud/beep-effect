# rrf-retrieval-layer-consumption

> Scope: validate the RRF k=60 / rank-only / weighted-renormalized / span-grounded retrieval contract this packet must **consume** (not rebuild), and define the shared-layer boundary with `rag-retrieval-projection` (RRF owner) and `goals/trustgraph-port` (FalkorDB/GraphRAG) so no third RRF is built.

## Findings

### RRF origin, formula, and the k=60 constant (PRIMARY)

- Reciprocal Rank Fusion is from Cormack, Clarke & Büttcher, *"Reciprocal Rank Fusion outperforms Condorcet and individual Rank Learning Methods,"* SIGIR '09. The canonical formula is `RRFscore(d ∈ D) = Σ_{r∈R} 1/(k + r(d))`, summed over the set of rankings `R`, where `r(d)` is the rank of document `d` in ranking `r`. Verbatim: *"where k = 60 was fixed during a pilot investigation and not altered during subsequent evaluation."* — [cormack.uwaterloo.ca/cormacksigir09-rrf.pdf](https://cormack.uwaterloo.ca/cormacksigir09-rrf.pdf) (author-hosted primary PDF; also indexed at [ir.webis.de/anthology/2009.sigirconf_conference-2009.146](https://ir.webis.de/anthology/2009.sigirconf_conference-2009.146/) and [dblp CormackCB09](https://dblp.org/rec/conf/sigir/CormackCB09.html)).
- The rationale for the additive `k`: highly-ranked documents matter more, but lower-ranked ones should not vanish (as they would under an exponential), so *"The constant k mitigates the impact of high rankings by outlier systems."* This is why `k` is **added to** rank rather than used as a decay rate. — [Cormack 2009 PDF, §1](https://cormack.uwaterloo.ca/cormacksigir09-rrf.pdf)

### k=60 robustness / the "[40,80] comparable" claim (PRIMARY — Table 1)

- The paper's pilot sweep (Table 1, RR-Fusion of 30 model systems on TREC topics 351–400) reports MAP vs k: `k=0 → .2072`, `10 → .2123`, `20 → .2134`, `30 → .2139`, `40 → .2138`, `50 → .2144`, `60 → .2145`, `70 → .2146`, `80 → .2147`, `90 → .2145`, `100 → .2142`, `500 → .2098`. The text concludes *"k = 60 was near-optimal, but that the choice was not critical."* — [Cormack 2009 PDF, Table 1 + §1](https://cormack.uwaterloo.ca/cormacksigir09-rrf.pdf). **This is the direct primary support for CAPTURE's "k in [40,80] performing comparably"**: MAP across k∈[40,80] spans only .2138–.2147 (<0.5% relative), while k→0 sharply favors top-1 (.2072) and k→500 degrades toward an equal-weight position sum (.2098). Caveat: the table's literal peak is k=80, but 60↔80 differ by ~0.1% MAP (.2145 vs .2147) — negligible; k=60 remains the canonical default.

### Industry defaults converge on k=60, rank-only, no cross-stream score normalization (PRIMARY vendor docs — adversarial cross-check)

- **Elasticsearch**: `rank_constant` (the `k`) *"Defaults to `60`"*; `rank_window_size` defaults to the `size` parameter. Its formula iterates `score += 1.0 / (k + rank(result(q), d))` over each query — i.e. rank position only, never normalized scores. The base `rrf` retriever gives *"Each child retriever ... an equal weight"* (no per-retriever weighting in the base form). — [elastic.co/docs/.../reciprocal-rank-fusion](https://www.elastic.co/docs/reference/elasticsearch/rest-apis/reciprocal-rank-fusion)
- **OpenSearch**: *"By default, the rank constant is set to `60`"*; `rankScore = Σ 1/(k + query_j_rank)`; it *"focuses exclusively on rank positions"* and *"avoids score distribution variability issues that plague traditional normalization methods."* — [opensearch.org/blog/introducing-reciprocal-rank-fusion-hybrid-search](https://opensearch.org/blog/introducing-reciprocal-rank-fusion-hybrid-search/)
- **Azure AI Search**: score is `1/(rank + k)`; *"Experiments show the algorithm performs best when you set `k` to a small value, such as 60."* The doc explicitly tabulates that each stream's native score has its own incomparable range (BM25 = "No upper limit"; vector/HNSW cosine = 0.333–1.00) — RRF sidesteps this by fusing rank, not score. — [learn.microsoft.com/.../hybrid-search-ranking](https://learn.microsoft.com/en-us/azure/search/hybrid-search-ranking) (updated 2026-06-11). **Validation: beep must fuse on rank and must NOT min-max / z-score normalize BM25 vs cosine vs graph scores into a shared scale.**

### Weighted RRF and the "weight renormalization" pattern (distinct from score normalization)

- Per-stream weighting attaches a multiplier to the reciprocal-rank term: *"Each retriever contributes: `weight × 1 / (rank + rank_constant)`"*; weights must be non-negative and default to 1.0; the engines do **not** auto-normalize the weights. — [elastic.co/search-labs/.../weighted-reciprocal-rank-fusion-rrf](https://www.elastic.co/search-labs/blog/weighted-reciprocal-rank-fusion-rrf); Azure applies the same multiplier model to weighted vector queries ([hybrid-search-ranking](https://learn.microsoft.com/en-us/azure/search/hybrid-search-ranking)).
- **Key distinction for this packet**: the agentmemory pattern this cluster wants to consume renormalizes the *fusion weights* (not scores). From the in-repo gold-intake snippet of `agentmemory src/state/hybrid-search.ts:194-219`: `effectiveVectorW = hasVector ? vectorWeight : 0; effectiveGraphW = hasGraph ? graphWeight : 0; totalW = bm25W+vectorW+graphW; if (totalW>0) each /= totalW;` then `combinedScore = effB25W*(1/(K+bm25Rank)) + effVecW*(1/(K+vecRank)) + effGraphW*(1/(K+graphRank))`. — [explorations/_gold-intake/research/per-repo/agentmemory.md](../../_gold-intake/research/per-repo/agentmemory.md). This renormalizes the *per-stream fusion weights to sum to 1 when a stream is empty* (graceful local-first degradation before vector/graph projections exist) — it is **application-level weight renormalization, NOT the cross-stream score normalization RRF deliberately avoids.** Both are sound and compose; keep them conceptually separate in the contract.

### The two concrete RRF instances to consume (values + license)

- **agentmemory** (Apache-2.0 — port-with-attribution, but plain TS + Zod + bespoke iii-sdk, so reimplement the *algorithm/shape* in Effect-Schema; DI/Layer does not transfer): three-stream BM25+vector+graph, weighted + renormalized as above, `RRF_K` constant. — [agentmemory per-repo notes](../../_gold-intake/research/per-repo/agentmemory.md).
- **doc-haus** (MIT — safe to study/adapt): `dochaus/tool/search-document.ts` fuses 3 channels over the *same* chunks — embedding cosine (meaning) + FTS5/BM25 (exact tokens) + a whole-query literal-phrase channel — with `const RRF_K = 60` and `const CANDIDATES = 20`, returning each hit as `doc / section / charStart / charEnd`. Carries the *"a literal hit must not be outscored by fuzzy hits"* insight (the literal-phrase channel). — [explorations/_gold-intake/research/per-repo/doc-haus.md](../../_gold-intake/research/per-repo/doc-haus.md). Both source repos independently pin k=60, corroborating the vendor defaults above.

### Span-grounded citation shape (already a repo invariant — consume, don't invent)

- doc-haus's per-hit shape `doc/section/charStart/charEnd` maps directly onto beep's existing char-span contract: `@beep/provenance` `TextAnchor` carries a half-open char range whose invariant is *"`text.slice(start, end)` should reproduce `quote`"* (verified live at `packages/foundation/modeling/provenance/src/TextAnchor.ts`), and epistemic `EvidenceSpan` (`packages/epistemic/domain/src/values/EvidenceSpan/EvidenceSpan.model.ts`, confirmed present) wraps it with `startChar/endChar/quote` + `Confidence = UnitInterval`. The RRF layer must therefore emit hits that *carry* a TextAnchor/EvidenceSpan, never bare chunks. The explicit anti-pattern to reject is lawyergpt's un-gated tool-calling RAG (raw chunks straight to the LLM, no span provenance) — outputs must pass `CandidateClaim + Evidence` spans through the ClaimGate. — [rag-retrieval-projection CAPTURE, lawyergpt#4](../../rag-retrieval-projection/CAPTURE.md).

### Shared-layer contract: how to avoid a third RRF (the deliverable)

- **No RRF code exists in `packages/` today** (verified: `rg -rilE "reciprocal.?rank|rrf_k|rankConstant"` over `packages/`/`apps/` returns zero non-doc hits). So the "third RRF" risk is purely a *coordination-before-build* problem across three net-new owners, not a dedup of existing code — which is the cheapest possible moment to set the boundary.
- **Owner**: `explorations/rag-retrieval-projection` is the *designated single owner* of the scored hybrid 3-channel RRF fusion (k=60) + pgvector HNSW `vector_cosine_ops` projection; it states `agent-memory-tiers-bitemporal-edges` and `goals/trustgraph-port` **CONSUME it, do not rebuild it.** — [rag-retrieval-projection CAPTURE, netNew #1 + Cautions](../../rag-retrieval-projection/CAPTURE.md).
- **This packet's consumption posture**: agent-memory tiers do not implement fusion. They (a) *supply* ranked candidate lists into the shared fuser (e.g. the semantic/episodic tier's graph-expansion over bitemporal edges is one of the input streams), and (b) *read* the fused, span-carrying hits. The only memory-specific logic that stays here is pre-fusion (per-tier candidate generation, retention/decay-weighted recall) and post-fusion (session diversification, conflict/supersede filtering against the gate) — never the RRF arithmetic.
- **Reconciliation gotcha with trustgraph-port**: `goals/trustgraph-port` Phase 1 is explicitly a **scoreless deterministic ordering** path — *"Mixed-corpus results must remain one deterministic ranked list with no numeric scoring surface,"* with *"vector embeddings or semantic similarity search in Phase 1"* a stated non-goal. — [goals/trustgraph-port/SPEC.md](../../../goals/trustgraph-port/SPEC.md). So trustgraph-port Phase 1 does **not** consume RRF at all; it is a separate retrieval mode. The boundary is therefore three-way: (1) the *scored RRF fuser* lives only in rag-retrieval-projection; (2) trustgraph-port contributes the FalkorDB/GraphRAG *graph stream* (Cypher BFS projection) as a future RRF input, but its Phase-1 scoreless packet path is intentionally a different lane; (3) this packet consumes the rag-retrieval-projection RRF and feeds it the bitemporal-edge graph stream.
- **Suggested contract surface** (to be ratified in rag-retrieval-projection, referenced here): one service, e.g. `@beep/retrieval`, exporting a pure `fuse(streams: ReadonlyArray<RankedList>, opts?: { k?: number /* default 60 */; weights?: ReadonlyArray<number> }) => ReadonlyArray<FusedHit>` where each `RankedList` is `(id, rank)` tuples only (callers must NOT pass raw scores), `weights` are renormalized to sum to 1 over the *present* streams, and `FusedHit` carries the `TextAnchor`/`EvidenceSpan` of its source chunk. This makes "rank-only, k=60 default, weight-renormalized, span-carrying" structurally enforced and impossible to fork.

## Open / Unverified

- **The exact "[40,80]" bracket is derived, not verbatim.** Cormack 2009 says only "near-optimal ... not critical"; the near-flat MAP across k∈[40,80] in Table 1 is the primary support, while the explicit bracket is codified by secondary practitioner sources (BigData Boutique). Treat k=60 as the load-bearing default and [40,80] as an empirically-safe tuning band, not a paper-stated rule.
- **Graph-stream rank is an open design question (belongs to rag-retrieval-projection).** RRF needs each stream to emit a *stable ranked list*, but graph expansion yields a set, not an intrinsic ranking. The ranking signal for the FalkorDB/bitemporal-edge stream (BFS distance? node degree? edge weight 1.0/0.5/0.1? recency/temporal-validity?) is unspecified upstream — flag for the RRF owner, since a poorly-ranked graph stream is exactly the case Cormack's "higher k for less-reliable systems" intuition addresses.
- **agentmemory source lines unverified against upstream.** The `src/state/hybrid-search.ts:194-219` snippet and weight-renormalization logic are taken from the in-repo gold-intake per-repo notes, not re-fetched from the upstream GitHub repo (not accessed this session). Formula shape is corroborated by the vendor docs and doc-haus, but exact upstream line numbers / current upstream code are UNVERIFIED.
- **Whether trustgraph-port ever feeds RRF is unresolved.** Phase 1 is scoreless-by-design; promoting its graph output into the scored RRF fuser is a post-Phase-1 reconciliation that neither packet has committed to. Needs an explicit cross-packet decision before either builds a fusion path.
- **No ratified `@beep/retrieval` contract exists yet.** The suggested `fuse(...)` signature above is a proposal for rag-retrieval-projection to own; it is not (yet) an agreed API. The single concrete guarantee today is the negative one: do not implement RRF arithmetic in this packet.

# Codex research-gate critique — uspto-patent-driver-depth (2026-06-29)

## Blocking

- Claim: "`The live search endpoint is `POST /api/v1/patent/applications/search` ... The structured POST body keys ... are ...`" ([RESEARCH.md:15-23](../RESEARCH.md:15)) versus "`Whether the patent-applications `/search` endpoint accepts the POST structured body ... confirm before wiring `searchStructured`.`" ([RESEARCH.md:380-381](../RESEARCH.md:380)). Why wrong/weak: the synthesis states POST structured search as confirmed in the executive section, then marks the exact same application-search POST as unresolved. That is a release-blocking ambiguity because the proposed `searchStructured` method could be built against a method/body combination the target endpoint rejects. Concrete fix: resolve this before shape/decompose with either a real browser Swagger read or a key-authenticated probe against `POST /api/v1/patent/applications/search`; then make one of these statements disappear. If still unverified, demote the top claim to "documented by secondary connector; not yet live-probed" and keep `searchStructured` behind a spike.

- Claim: "`Conditional/credential-gated registration: NOT FOUND` (verified `rg "Layer.empty|unwrapEffect"` across `nlp-mcp/src` + `m365-mcp/src` returned nothing) — genuinely net-new." ([RESEARCH.md:225-232](../RESEARCH.md:225)). Why wrong/weak: the source-code implementation is not built, but the repo already has an active `explorations/mcp-auth-gated-registration` packet with the exact Effect/MCP design: build-time conditional mounting, handler-time `api_key_required`, and the v4 gotcha that `Layer.unwrapEffect` is not an Effect v4 API (`node_modules/effect/src/Layer.ts` exports `unwrap` at line 1498; `rg unwrapEffect node_modules/effect/src` returned nothing). The raw research also proposes "`Layer.unwrapEffect`/`Layer.mergeAll`" ([research/extend-in-place-architecture-and-codegen.md:73](../research/extend-in-place-architecture-and-codegen.md:73)), which would fail against the pinned `effect@4.0.0-beta.91` in root `package.json`. Concrete fix: change this from "NET-NEW pattern" to "depends on `explorations/mcp-auth-gated-registration`"; import its Shapes A/B/C, use `Layer.unwrap` or `Layer.catch`/`Layer.empty` as documented there, and add a dtslint spike before committing to dynamic layer folding.

- Claim: "`Default scope stays local/ODP and privilege-safe. All three credentialed tiers (EPO OAuth2, BigQuery GCP, SerpApi) transmit search terms — which may encode unfiled invention disclosures / client work product — to third-party clouds. They are OPT-IN per matter...`" ([RESEARCH.md:353-357](../RESEARCH.md:353)). Why wrong/weak: this draws the consent boundary around EPO/BigQuery/SerpApi but treats ODP and ppubs as privilege-safe defaults. ODP search and ppubs also transmit query text to external USPTO systems; for pre-filing invention disclosures, the confidentiality risk is not eliminated merely because the endpoint is official or no-key. Concrete fix: split "official/public-source" from "privilege-safe." Default to offline/local or public-identifier-only searches for pre-filing matters, require explicit matter-level consent for any external free-text search (ODP, ppubs, EPO, BigQuery, SerpApi), and encode this as a source policy in the driver/MCP auth matrix.

## Advisory

- Claim: "`The `q` parameter is USPTO "Simplified Query Syntax" = OpenSearch/Lucene `query_string`, not raw Lucene... It inherits the full Lucene grammar...`" ([RESEARCH.md:24-29](../RESEARCH.md:24)) and "`the driver needs ... a FIELD-QUERY/composition escaper ... plus a schema-enforced length cap (~1000 chars)`" ([RESEARCH.md:30-38](../RESEARCH.md:30)). Why wrong/weak: the document itself later says the exact ODP escaping set, bare aliases, and queryable fields are unresolved behind WAF/Swagger verification ([RESEARCH.md:374-379](../RESEARCH.md:374)). Promising "full Lucene grammar" from Lucene 2.9.4 overstates what a modern OpenSearch `query_string` endpoint will accept and may cause the driver to preserve syntax that ODP rejects or interprets differently. Concrete fix: rephrase to "ODP appears query_string-like; implement a tested subset." Gate each grammar feature behind fixtures/live probes, use literal-term escaping by default, and expose advanced field-query pass-through only as an explicitly unsafe/advanced mode.

- Claim: "`The reverse-engineered handshake is triple-confirmed across `patent_client`, `patent_mcp_server`, and `swinc/fetch-ppubs-patents`: `GET /pubwebapp/` seeds cookies → `POST /api/users/me/session` ... Two-step `counts`-then-`searchWithBeFamily`...`" ([RESEARCH.md:104-113](../RESEARCH.md:104)). Why wrong/weak: the raw ppubs note says the endpoint base is split and that `swinc/fetch-ppubs-patents` is a legacy `/dirsearch-public/` implementation, while `/api/` is the maintained-path hypothesis ([research/ppubs-fulltext-fallback-tier.md:66-71](../research/ppubs-fulltext-fallback-tier.md:66)). So the synthesis overstates the exact handshake as "triple-confirmed"; at most the general public-search workflow is multi-sourced, while the base URL and session/header requirements are version-sensitive. Concrete fix: rewrite this as a matrix by implementation (`patent_client`, `riemannzeta/patent_mcp_server`, `swinc`, `specsmith`) with base URL, session call, counts step, and token-header behavior separated. Keep the existing DEFERRED decision unless a live capture proves one path.

- Claim: "`Responses are namespace-heavy XML where `family-member/classification-cpc/applicant/inventor/priority-claim/legal` must be force-arrayed.`" ([RESEARCH.md:133-136](../RESEARCH.md:133)). Why wrong/weak: the in-repo inventory misses the existing XML capability. Root `package.json` already catalogs `fast-xml-parser` and `fast-xml-validator`; `@beep/schema/Xml` exports `XmlTextToUnknown` and `decodeXmlTextAs` backed by `XMLParser`/`SyntaxValidator` ([packages/foundation/modeling/schema/src/Xml.ts:13-33](/home/elpresidank/YeeBois/projects/beep-effect/packages/foundation/modeling/schema/src/Xml.ts:13), [package.json:164-165](/home/elpresidank/YeeBois/projects/beep-effect/package.json:164)). The current helper does not force-array those EPO node names, but the dependency and schema transform home already exist. Concrete fix: add an EPO-specific parser option or extend `@beep/schema/Xml` with an `isArray`/force-array hook, then decode EPO XML through schema rather than introducing a new XML stack.

- Claim: "`ClaimLifecycle = LiteralKit(["candidate","shape_valid","consistency_checked","admitted"])` ... the shared-kernel evidence-admission gate consumed by `@beep/epistemic-*`, orthogonal to patent prosecution phase. Do not overload it." ([RESEARCH.md:254-258](../RESEARCH.md:254)). Why wrong/weak: the no-overload conclusion is right, but the inventory undercounts current consumers. The shared-domain source explicitly says the vocabulary is "deliberately shared across knowledge verticals" ([packages/shared/domain/src/values/ClaimLifecycle/ClaimLifecycle.model.ts:4-8](/home/elpresidank/YeeBois/projects/beep-effect/packages/shared/domain/src/values/ClaimLifecycle/ClaimLifecycle.model.ts:4)), and `law-practice/domain` already uses it on `Distinction.lifecycleState` ([packages/law-practice/domain/src/entities/Distinction/Distinction.model.ts:12-45](/home/elpresidank/YeeBois/projects/beep-effect/packages/law-practice/domain/src/entities/Distinction/Distinction.model.ts:12)). Concrete fix: keep the warning, but phrase it as "ClaimLifecycle is an admission-state vocabulary already reused by law-practice work product; prosecution phase still needs a separate `ProsecutionPhase`/`PatentAssetStatus` axis."

- Claim: "`Do not port the hand-curated status-code maps ... sync the canonical 225-code table from ODP `/status-codes` instead.`" ([RESEARCH.md:336-337](../RESEARCH.md:336)). Why wrong/weak: the research proves the partial MCP maps are unsafe, but it does not define how a key-authenticated, drifting 225-code table becomes a stable repo artifact. Runtime fetching couples vocab decode to secret availability; checked-in generation needs provenance, version date, and refresh gates. Concrete fix: choose one: a versioned generated `Uspto.vocab.generated.ts` from PatEx/ODP with source date + checksum + refresh command, or a runtime `/status-codes` cache with typed stale/offline behavior. Do not leave "sync" as an implementation verb without owner, cadence, and proof gate.

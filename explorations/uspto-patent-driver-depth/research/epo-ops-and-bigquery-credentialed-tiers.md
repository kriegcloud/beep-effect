# epo-ops-and-bigquery-credentialed-tiers

Scope: net-new credentialed retrieval drivers (`packages/drivers/epo` EPO OPS OAuth2, `packages/drivers/google-patents-bigquery` patents-public-data, Google Patents via SerpApi) — auth, throttling/cost gates, secret governance, and the default offline/privilege-safe vs opt-in decision.

## Findings

### A. EPO OPS — OAuth2 client-credentials driver (`packages/drivers/epo`, NEW)

- **Auth flow is OAuth2 client-credentials with HTTP Basic.** Token endpoint is `POST https://ops.epo.org/3.2/auth/accesstoken` with body `grant_type=client_credentials` and an `Authorization: Basic base64(consumerKey:consumerSecret)` header; credentials are a Consumer Key + Consumer Secret obtained by registering a free app on the EPO Developer Portal and choosing the non-paying access method. ([searched summary of EPO OPS auth](https://www.oauth.com/oauth2-servers/access-tokens/client-credentials/) corroborated by the EPO connector OAuth doc <https://docs.epoconsulting.com/en/public/epo-connector/outbound/consume-sap-api-oauth> and the Rops R client <https://github.com/AlessioNar/Rops>)
- **Access token expires ~20 minutes; expiry surfaces as HTTP 400, not 401.** The python reference client states the one case it handles is "when the access token has expired: in this case, the client will automatically handle the HTTP 400 status and renew the token." So the driver must re-acquire on 400 (and defensively on 401) and clear the cached token. The CAPTURE nugget's "19-min cached token (1-min buffer)" is a sound client-side choice: refresh ~60s before the ~20-min server expiry. ([python-epo-ops-client README](https://github.com/ip-tools/python-epo-ops-client); 20-min expiry corroborated by [Rops](https://github.com/AlessioNar/Rops) and [patent.dev Go client](https://patent.dev/epo-ops-v3-2-go-client-library/))
- **`X-Throttling-Control` response header — exact shape and parse.** The python client parses it with: system-status regex `^(\w+) \(` (leading word = `idle` / `busy` / `overloaded`), and per-service regex `{service}=(\w+):(\d+)` where `SERVICES = ("images", "inpadoc", "other", "retrieval", "search")`. So the header looks like `idle (images=green:200, inpadoc=green:60, other=green:1000, retrieval=green:200, search=green:30)`: a global status word followed by, per service, a `color:limit` pair where the color is `green|yellow|red|black` and the integer is the allowed requests-per-minute for that bucket. ([python-epo-ops-client storages/sqlite.py, fetched verbatim](https://raw.githubusercontent.com/ip-tools/python-epo-ops-client/main/epo_ops/middlewares/throttle/storages/sqlite.py))
- **`black` means limit 0 → fall back to `Retry-After`.** The reference `delay_for(service)` logic: when the per-service limit is `0` (i.e. `black`), wait `timestamp + Retry-After(ms)`; otherwise space requests by `60.0 / limit` seconds (a one-minute sliding window). The CAPTURE nugget's "x-throttling-control 'black' → throw" (patents-mcp-server) is a stricter variant of the same signal — black = bucket exhausted. Recommended beep behavior: model `black` as a typed `EpoThrottled` retryable error carrying the `Retry-After`, and use Effect `Schedule` to honor `60/limit` spacing on non-black colors. ([python-epo-ops-client throttler.py](https://raw.githubusercontent.com/ip-tools/python-epo-ops-client/main/epo_ops/middlewares/throttle/throttler.py) + [storages/sqlite.py](https://raw.githubusercontent.com/ip-tools/python-epo-ops-client/main/epo_ops/middlewares/throttle/storages/sqlite.py))
- **Fair-use quota: ~4 GB/week for free registered users; quota types are layered.** The python client raises distinct exceptions per quota tier — `AnonymousQuotaPerMinuteExceeded`, `AnonymousQuotaPerDayExceeded`, `IndividualQuotaPerHourExceeded`, `RegisteredQuotaPerWeekExceeded` — driven by the `X-IndividualQuotaPerHour` / `X-RegisteredQuotaPerWeek` response headers. Non-paying registered usage is capped at 4 GB/week. ([python-epo-ops-client PyPI](https://pypi.org/project/python-epo-ops-client/); 4 GB/week corroborated by [Apify EPO OPS actor](https://apify.com/ryanclinton/epo-patent-search))
- **Responses are namespace-heavy XML; repeated elements must be force-arrayed.** OPS returns EPO XML (e.g. `ops:world-patent-data`) where elements like family-member, classification-cpc, applicant, inventor, priority-claim, and legal-status events repeat 1..n. A naive XML→JSON parse collapses single-element cases to objects; the driver must force-array those node names (CAPTURE nugget patents-mcp-server#2 lists exactly: `family-member, classification-cpc, applicant, inventor, priority-claim, legal`). EPO also offers JSON via `Accept: application/json` on many endpoints, but legal-status / INPADOC family endpoints are most reliably XML. (CAPTURE nugget `patents-mcp-server#2`; XML namespace behavior per [python-epo-ops-client](https://github.com/ip-tools/python-epo-ops-client))
- **Reference library to port from is `ip-tools/python-epo-ops-client`.** It is the de-facto canonical OPS client (also the basis for PatZilla). License: it ships under the Apache-2.0 license per the repo — porting its throttle/auth *logic* (not code) into Effect is clean. The TS source actually named in CAPTURE (`patents-mcp-server/src/clients/epo-ops.client.ts`) is the closer shape match but its repo license is UNVERIFIED. ([python-epo-ops-client repo](https://github.com/ip-tools/python-epo-ops-client))

### B. Google Patents BigQuery — `patents-public-data` (`packages/drivers/google-patents-bigquery`, NEW)

- **Canonical table is `patents-public-data.patents.publications`; claims/descriptions are nested, not separate tables.** Full-text lives in REPEATED RECORD columns `claims_localized`, `description_localized`, `abstract_localized` on the publications row. Each element has subfields `{ text: STRING, language: STRING, truncated: BOOL }`. Verified against the official example notebook's verbatim SQL: `FROM \`patents-public-data.patents.publications\` AS pubs, UNNEST(claims_localized) AS claims ... WHERE claims.language = 'en'`. ([google/patents-public-data claim_text_extraction.ipynb, fetched verbatim](https://raw.githubusercontent.com/google/patents-public-data/master/examples/claim-text/claim_text_extraction.ipynb))
- **Full text is US-only; the rest is bibliographic.** The dataset carries worldwide bibliographic data for 90M+ publications across ~17 countries plus **US full text** (claims/description); non-US rows are bibliographic-only. So BigQuery is the global *bibliographic + US-full-text* tier; EPO OPS remains the path to non-US legal-status/family detail. ([Google Cloud blog: Google Patents Public Datasets](https://cloud.google.com/blog/topics/public-datasets/google-patents-public-datasets-connecting-public-paid-and-private-patent-data); [IFI Claims](https://www.ificlaims.com/news/public-patent-data-now-available-on-google-bigquery/))
- **On-demand pricing: $6.25 per TiB scanned, first 1 TiB/month free, 10 MB minimum billed.** Charged on bytes *scanned across referenced columns* (columnar — selecting `claims_localized.text` is expensive), rounded up, minimum 10 MB per query and per referenced table. Estimated cost = `totalBytesProcessed / 2^40 * $6.25`. ([BigQuery pricing](https://cloud.google.com/bigquery/pricing); $6.25/TiB + 1 TiB free + 10 MB min corroborated across [StackScored](https://www.stackscored.com/pricing/data-warehouse/bigquery/) and [CheckThat.ai](https://checkthat.ai/brands/bigquery/pricing))
- **The mandatory dry-run cost gate is real and free.** Set `dryRun: true` (and `useQueryCache: false`) on the job config; BigQuery validates the SQL and returns `statistics.query.totalBytesProcessed` *without executing, without consuming slots, and at no charge*. Belt-and-suspenders pattern for the driver: (1) run dry-run first, compute estimated cost, reject if it exceeds a configured ceiling/`maxBytesBilled`; (2) set the *real* job's `maximumBytesBilled` so BigQuery hard-fails any query that would exceed the cap even if the estimate was wrong. ([BigQuery best-practices: estimate and control costs](https://docs.cloud.google.com/bigquery/docs/best-practices-costs); [dry-run sample](https://docs.cloud.google.com/bigquery/docs/samples/bigquery-query-dry-run))
- **Auth: Application Default Credentials (ADC) is the documented best practice; service-account JSON key via `GOOGLE_APPLICATION_CREDENTIALS` (or Node `keyFilename`) is the fallback.** "As a best practice, use Application Default Credentials." For a server driver: prefer ADC/workload-identity in prod; for local/dev, point `GOOGLE_APPLICATION_CREDENTIALS` at an SA key JSON whose secret material is stored in 1Password and materialized at runtime. ([Authenticate to BigQuery](https://cloud.google.com/bigquery/docs/authentication); [create client with SA key file](https://cloud.google.com/bigquery/docs/samples/bigquery-client-json-credentials))
- **License: CC BY 4.0; data maintained by IFI CLAIMS + Google.** "Google Patents Public Data by IFI CLAIMS Patent Services and Google is licensed under a Creative Commons Attribution 4.0 International License." Attribution is required if results are surfaced. ([Kaggle dataset page](https://www.kaggle.com/datasets/bigquery/patents); [GCP Marketplace](https://console.cloud.google.com/marketplace/product/google_patents_public_datasets/google-patents-public-data))
- **DEPRECATION nuance (2026-04-18): the GitHub *examples* repo `google/patents-public-data` was archived (read-only); the BigQuery *dataset* is unaffected and remains queryable/maintained by IFI CLAIMS.** Do not read the archive as a dataset sunset — port the SQL patterns from the archived notebooks but target the live dataset. ([google/patents-public-data repo, archived banner](https://github.com/google/patents-public-data))

### C. Google Patents via SerpApi (`packages/drivers/google-patents`)

- **It is a paid scraping API, not an official Google API; `api_key` + `engine=google_patents` required.** SerpApi explicitly labels it a "paid scraping API" whose results are *scraped from Google Patents*, with free cached results (1-hour expiration). This carries Google-Patents-ToS/scraping risk that BigQuery (a sanctioned public dataset) does not. ([SerpApi Google Patents API](https://serpapi.com/google-patents-api))
- **Filter taxonomy (port the *shape*).** `q` (terms split by `;`); `before`/`after` = `type:YYYYMMDD` where `type ∈ {priority, filing, publication}` (e.g. `before=publication:20231231`); `country` = comma CSV (e.g. `WO,US`); `status ∈ {GRANT, APPLICATION}`; `type ∈ {PATENT, DESIGN}`; `language` CSV (ENGLISH/GERMAN/...); `inventor`, `assignee` (CSV, parenthesize names containing commas); `sort ∈ {new, old}`; `clustered=true`; `dups=language`; `page`; `num` 10–100. ([SerpApi Google Patents API parameter reference, fetched](https://serpapi.com/google-patents-api))

### D. Secret governance (1Password) + the default-scope decision

- **All three tiers are credentialed/billed and should be OPT-IN, not in the offline/privilege-safe default.** Mirror the existing `@beep/uspto` `UsptoConfigInput` pattern (`S.optionalKey` + `S.String.pipe(S.RedactedFromValue)` for the key — see `/home/elpresidank/YeeBois/projects/beep-effect/packages/drivers/uspto/src/Uspto.config.ts`). Define `EpoConfigInput { consumerKey, consumerSecret }` and `GooglePatentsBigQueryConfigInput { projectId, credentials, maxBytesBilled }` and `GooglePatentsSerpConfigInput { apiKey }` with Redacted secret fields; have each driver `Layer` fail-fast with a typed `ConfigError` when the secrets are absent. Absence-of-secret → driver simply not constructed = the opt-in gate is structural, not a runtime flag. (repo pattern: `packages/drivers/uspto/src/Uspto.config.ts`)
- **Privilege-safety rationale for a solo IP firm:** EPO OPS, BigQuery, and especially SerpApi all transmit the search terms (which may encode unfiled invention disclosures / client work product) to third-party clouds. For privileged/pre-filing work the default must stay local/ODP; the credentialed tiers are enabled *per matter* with explicit consent. This matches the CAPTURE cautions (EPO + BigQuery "out of offline/privilege-safe default scope"; keep dry-run gate mandatory; ppubs/SerpApi best-effort). (CAPTURE.md §Cautions; privilege framing from MEMORY legal-ai-stack baseline)
- **1Password wiring:** store EPO `consumerKey`/`consumerSecret`, the GCP service-account JSON (single multiline secret), and the SerpApi `api_key` as `op://` secret references; materialize them into env via the onepassword Developer-Environment MCP / local-env-file flow (the `onepassword-secret-refs` skill), never commit. For BigQuery prefer GCP workload identity in prod over a long-lived JSON key. ([Authenticate to BigQuery — best practice ADC](https://cloud.google.com/bigquery/docs/authentication); onepassword-secret-refs skill)

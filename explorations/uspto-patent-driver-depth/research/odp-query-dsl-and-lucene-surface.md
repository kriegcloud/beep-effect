# odp-query-dsl-and-lucene-surface

Scope: design the extend-in-place search surface for `@beep/uspto` — ODP structured-search POST body (filter/rangeFilter/sort), Lucene metacharacter escaping policy + length cap, friendly→API nested field mapping, smart identifier disambiguation, and how to port the PatentsView `_or/_and/_text_any/_begins/_gte/_lte` DSL *shape* onto live ODP fields without the sunset `api.patentsview.org` endpoint.

## Findings

### 1. ODP search endpoint + structured POST body (canonical target shape)

- The live search endpoint is `POST /api/v1/patent/applications/search` on host `api.uspto.gov` (mirrored under `data.uspto.gov/apis/patent-file-wrapper/search`). `GET /applications/search` accepts the same params as query strings and "returns the top 25 records by default when no parameters are specified"; `POST` performs "an advanced search using a structured JSON payload with filters, range filters, pagination, and field selection." There is also `POST /applications/search/download` returning json or csv via a similar payload. (https://learn.microsoft.com/en-us/connectors/uspatenttrademarkoff/)
- The structured POST body keys (all optional) are `q`, `filters`, `rangeFilters`, `sort`, `fields`/`fieldList`, `pagination`, `facets`, `downloadType`. Canonical example body: `{"q":"...", "filters":[{"name":"applicationStatusDescriptionText","value":["Patented Case"]}], "rangeFilters":[{"field":"issueDate","valueFrom":"2010-08-04","valueTo":"2022-08-04"}], "sort":[{"field":"filingDate","order":"desc"}], "fieldList":{"action":"include","fields":["applicationNumberText","firstNamedInventor","patentNumber"]}, "pagination":{"offset":0,"limit":25}, "facets":["applicationTypeLabelName","applicationStatusCode"], "downloadType":"json"}` — confirmed by both the ODP-API-Query-Spec summary (https://data.uspto.gov/apis/patent-file-wrapper/search) and a second library (https://patent-client.readthedocs.io/en/latest/user_guide/open_data_portal.html), whose OpenSearch body shows `{"q":"Nanobody","filters":[{"name":"applicationTypeLabelName","value":["Utility"]}],"rangeFilters":[{"field":"filingDate","valueFrom":"2022-01-01","valueTo":"2023-12-31"}],"pagination":{"offset":0,"limit":25},"sort":[{"field":"filingDate","order":"Desc"}]}`.
- Shape contract per element: `filters` = `[{name, value:[...]}]` (exact-match, multi-value OR within one filter); `rangeFilters` = `[{field, valueFrom, valueTo}]` (date/numeric inclusive range); `sort` = `[{field, order}]` with `order` `asc`/`desc` (case-insensitive — examples show both `desc` and `Desc`); `fieldList` = `{action:"include"|"exclude", fields:[...]}`; `pagination` = `{offset, limit}`. (https://patent-client.readthedocs.io/en/latest/user_guide/open_data_portal.html)
- Auth: every request requires an API key sent as header `x-api-key: {key}` (lowercase in the connector doc; the existing `@beep/uspto` driver already sends `X-API-KEY` — HTTP header names are case-insensitive so both work). Keys are obtained at `data.uspto.gov/myodp`; the interactive OpenAPI/Swagger is at `data.uspto.gov/swagger/index.html`. (https://learn.microsoft.com/en-us/connectors/uspatenttrademarkoff/)
- The `GET` form encodes `rangeFilters` as a single string `field valueFrom:valueTo` (colon-separated), e.g. `petitionMailDate 2021-01-01:2025-01-01`; the `POST` JSON form splits it into `{field, valueFrom, valueTo}`. Design the driver around the POST JSON form (unambiguous, no string assembly). (https://learn.microsoft.com/en-us/connectors/uspatenttrademarkoff/)

### 2. ODP field namespace + friendly→API nested mapping (`patentNumber` → `applicationMetaData.patentNumber`)

- The ODP response (and therefore its underlying OpenSearch index) nests bibliographic fields under `patentFileWrapperDataBag[].applicationMetaData.*`. Confirmed dotted paths from the response schema: `applicationMetaData.{filingDate, applicationTypeCode, applicationStatusCode (integer), firstApplicantName, firstInventorToFileIndicator, entityStatus, groupArtUnitNumber, docketNumber, publicationCategoryBag, examinerName, patentNumber}`; `applicationNumberText` and `requestIdentifier` sit at the envelope/top level (NOT under `applicationMetaData`). The search response envelope is `{count, patentFileWrapperDataBag:[...], requestIdentifier}`. (https://learn.microsoft.com/en-us/connectors/uspatenttrademarkoff/)
- This confirms the gold-nugget mapping design (`uspto_pfw_mcp#3`): a friendly→API field mapper detects a literal dot — if the field name already contains `.` it is passed through, otherwise a bare friendly name is rewritten to its nested API path (e.g. `patentNumber:` → `applicationMetaData.patentNumber:`, `filingDate:` → `applicationMetaData.filingDate:`, `examinerName:` → `applicationMetaData.examinerName:`). The field-detection regex in the ported source is `(\w+(?:\.\w+)*)\s*:`. The exception set that must stay top-level (no `applicationMetaData.` prefix): `applicationNumberText`. (CAPTURE.md `uspto_pfw_mcp#3`; field namespace per https://learn.microsoft.com/en-us/connectors/uspatenttrademarkoff/)
- Caveat: `patent_client` issues simplified `q` like `inventionTitle:Ball AND filingDate:2024-01-01` using BARE field names (no `applicationMetaData.` prefix), implying ODP's "Simplified Query Syntax" may accept some bare aliases server-side OR `patent_client` applies its own mapping. The safe, self-documenting design is to always emit the fully-qualified dotted path (`applicationMetaData.inventionTitle`) — it matches the index and is unambiguous. (https://patent-client.readthedocs.io/en/latest/user_guide/open_data_portal.html) [exact set of server-accepted bare aliases is UNVERIFIED — see Open.]

### 3. The `q` parameter is Lucene/OpenSearch query_string ("Simplified Query Syntax")

- The ODP `q` is free-form full-text plus field-scoped Lucene query_string. USPTO publishes it as "Simplified Query Syntax" (https://data.uspto.gov/documents/documents/ODP-API-Query-Spec.pdf). Working examples use `field:value`, boolean `AND`/`OR`/`NOT`, and date equality, e.g. `q="inventionTitle:Ball AND filingDate:2024-01-01"`. (https://patent-client.readthedocs.io/en/latest/user_guide/open_data_portal.html)
- Lucene query-parser grammar this inherits: `field:value` scoping applies only to the immediately following term; wildcards `?` (single char) and `*` (multi char), no leading wildcard; fuzzy `term~` (0–1, default 0.5); proximity `"a b"~10`; inclusive range `[a TO b]` and exclusive `{a TO b}`; boost `term^4`; booleans `AND OR NOT + -` (operators are case-sensitive uppercase); grouping `(...)`. (https://lucene.apache.org/core/2_9_4/queryparsersyntax.html)

### 4. Lucene/Elasticsearch metacharacter escaping policy + length cap

- Classic Lucene reserved set (v2.9.4): `+ - && || ! ( ) { } [ ] ^ " ~ * ? : \` — escape any with a leading backslash; verbatim example `\(1\+1\)\:2`. Note this classic list does NOT include `/`. (https://lucene.apache.org/core/2_9_4/queryparsersyntax.html)
- Modern Elasticsearch/OpenSearch `query_string` (what ODP actually runs on) reserves a BROADER set: `+ - = && || > < ! ( ) { } [ ] ^ " ~ * ? : \ /`. Two ES-specific gotchas: (a) `<` and `>` **cannot be escaped at all** — they must be stripped from the query or they force a range query; (b) inside a JSON request body the backslash itself must be doubled (`\\`) because JSON treats `\` as its own escape, e.g. `"kimchy\\!"`. (https://www.elastic.co/docs/reference/query-languages/query-dsl/query-dsl-query-string-query)
- Design consequence: there are two distinct escaping modes and the driver needs both. (i) LITERAL-TERM mode — when the caller supplies a raw user string to match literally, escape the FULL modern reserved set (and strip `<`/`>`). (ii) FIELD-QUERY/COMPOSITION mode — the ported `uspto_pfw_mcp#2` policy deliberately escapes only `+ & | ! ( ) { } ^ ~` plus backslash and leaves `: " [ ] - * ` UNescaped so the user can still field-scope (`:`), phrase (`"`), range (`[ ]`), negate (`-`), and wildcard (`*`). The two modes are not interchangeable — pick per call-site (build-a-DSL vs pass-through-a-term). (CAPTURE.md `uspto_pfw_mcp#2`; reserved sets per the two sources above)
- Anti-injection length cap: the ported source raises if the query length exceeds 1000 characters. Adopt an explicit, schema-enforced cap (effect/Schema `S.maxLength`) on the assembled `q` string as a DoS/injection guard, independent of any server-side limit. (CAPTURE.md `uspto_pfw_mcp#2`)

### 5. Porting the PatentsView DSL *shape* onto ODP (NEVER the dead endpoint)

- The PatentsView query language is a JSON DSL "similar to Lucene". Operator set (verified across two sources): logical `_and`, `_or`, `_not`; comparison `_eq`, `_neq`, `_gt`, `_gte`, `_lt`, `_lte`, `_begins` (string-only), `_contains` (string, case-insensitive); full-text `_text_any`, `_text_all`, `_text_phrase`. Request envelope is `{q (required JSON criteria), f (fields array), s (sort), o (options/pagination)}`. (https://dev.patentsview.org/api/query-language.html → redirects to https://data.uspto.gov/support/transition-guide/patentsview ; corroborated verbatim by https://docs.ropensci.org/patentsview/articles/writing-queries.html)
- Canonical nested example (ropensci): `{"_and":[{"_gte":{"patent_date":"2007-03-01"}},{"_or":[{"_contains":{"patent_title":"dog"}},{"_text_all":{"patent_abstract":"dog"}}]},{"_or":[{"_text_phrase":{"inventors.inventor_country":"US"}},{"_text_phrase":{"inventors.inventor_country":"CA"}}]}]}`. Comparison ops are leaf objects `{op:{field:value}}`; logical ops take arrays of such leaves; data-type rules apply (`_begins` string fields only, `_text_*` full-text fields only, `_gte/_lte` date/numeric). (https://docs.ropensci.org/patentsview/articles/writing-queries.html)
- PORT PLAN: keep the PatentsView DSL *interface* (an internal, type-safe `{_and|_or|_not|_gte|_lte|_text_any|_begins|...}` AST authored in effect/Schema) but COMPILE it to ODP, not to PatentsView. Two viable compile targets, pick per operator: (a) leaves whose op maps cleanly to a Lucene clause compile into the ODP `q` string (`_text_any:{inventionTitle:q}` → `applicationMetaData.inventionTitle:(...)`, `_gte/_lte:{filingDate}` → `applicationMetaData.filingDate:[from TO to]`, `_begins` → `field:prefix*`, `_eq` → `field:"value"`); (b) leaves that are exact-match or range map more cleanly to the structured body (`_eq` → `filters[{name,value}]`, `_gte/_lte` pair → `rangeFilters[{field,valueFrom,valueTo}]`, sort → `sort[{field,order}]`). Field names in every leaf are rewritten through the §2 friendly→API mapper. The PatentsView field names (`patent_title`, `patent_date`, `assignees.assignee_name`, `inventors.inventor_name`) are NOT ODP fields and must be remapped to ODP equivalents (`applicationMetaData.inventionTitle`, `applicationMetaData.filingDate`/`grantDate`, applicant/inventor fields). (Mapping design from CAPTURE.md `mcp-uspto#5`, `uspto-patents-mcp#1`; ODP field names per §2)

### 6. Smart identifier disambiguation (app vs patent vs publication + confidence)

- USPTO application number = `XX/YYY,YYY` → a 2-digit **series code** (01–18+) followed by a 6-digit serial = 8 digits normalized. The series code also encodes era and type: utility series climb ~`08`(1991)…`13`(2012–14)…`16`/`17`/`18`(2018/2020/2022+); provisional applications use series `60/61/62/63`; designs use `29`; reissue/PCT-national-stage have their own ranges. (https://www.uspto.gov/web/offices/pac/mpep/s503.html ; https://www.finnegan.com/en/insights/blogs/prosecution-first/what-type-of-application-is-this-using-series-codes-to-identify-the-type-of-uspto-application.html)
- Publication number = `US` + 4-digit year + 7-digit sequence + kind code, e.g. `US20230012345A1` ("2023" year, "0012345" sequence, "A1" kind). Pre-grant publication kind codes are `A1/A2/A9`; granted-patent kind codes are `B1` (granted, no prior pre-grant pub) and `B2` (granted, had a prior pre-grant pub), plus `E`(reissue)/`P,P1..`(plant)/`S`(design). (https://www.uspto.gov/patents/apply/applying-online/publication-number ; https://www.uspto.gov/patents/search/authority-files/uspto-kind-codes)
- Disambiguation ladder (emit the correct ODP field per type): (1) kind code `A1/A2/A9` present → **publication number**; (2) kind code `B1/B2/E/P/S` present → **granted patent**; (3) 11 digits beginning with a 4-digit year (2001–present) → **publication number**; (4) letter prefix `D/RE/PP/H/T` → **patent number** (design/reissue/plant/SIR/statutory); (5) exactly 8 digits whose first two are a known series code (≈`08`–`19`) → **application number** — but this is the documented ambiguity (`uspto_pfw_mcp#1`'s "series 08–17" heuristic) because 8-digit utility patent numbers now exist (utility crossed 11,000,000 in 2021); (6) 7 digits / comma-grouped 7–8 digits → **patent number**. Attach a confidence score and surface it. (CAPTURE.md `uspto_pfw_mcp#1`; series/kind facts per the USPTO sources above)
- Correct ODP search field per resolved type: application → `applicationNumberText:{cleaned}` (top-level field, confirmed by response schema); patent → `applicationMetaData.patentNumber:{cleaned}`; publication → publication-number field on `applicationMetaData` (the existing `@beep/uspto` model exposes `earliestPublicationNumber`; exact ODP query field for publication is UNVERIFIED — see Open). Normalization first strips a leading `US`/`USPTO` and a trailing kind code `[A-Z]\d*$`, plus separators `/ , . space`, before classifying — the existing `normalizeUsptoApplicationNumber` / `normalizeUsptoPatentNumber` helpers already do most of this and should be reused, not re-implemented. (CAPTURE.md `uspto_pfw_mcp#1`; existing `packages/drivers/uspto/src/Uspto.models.ts`)

### 7. The NEVER-PatentsView guard (dates + migration)

- Legacy PatentsView `api.patentsview.org` support was announced to end February 2025 (page title "Support for Legacy API to End in February 2025") but actual discontinuation slipped to **May 1, 2025**, after which requests return **HTTP 410 Gone**. Replacement is the PatentSearch API at `search.patentsview.org`. (https://patentsview.org/data-in-action/support-legacy-api-end-february-2025-switch-patentsearch-api-now) [refines CAPTURE's "sunset Feb-2025" to the enforced May-1-2025 / 410 date]
- PatentsView itself migrates onto USPTO Open Data Portal `data.uspto.gov` on **March 20, 2026**, with expected temporary interruptions to the PatentSearch API and no committed relaunch date for PatentsView functions. This matches the CAPTURE caution "ODP key reissue 2026-03-20". Net: do not target ANY `patentsview.org` host from `@beep/uspto`; target only `api.uspto.gov`/`data.uspto.gov` ODP. The DSL above is ported as a *shape*, executed against ODP. (https://patentsview.org/data-in-action/support-legacy-api-end-february-2025-switch-patentsearch-api-now ; https://data.uspto.gov/support/transition-guide/patentsview)
- Independently, PEDS (Patent Examination Data System) was retired **March 14, 2025**, redirecting users to ODP's Patent File Wrapper — reinforcing ODP as the single live source. (https://learn.microsoft.com/en-us/connectors/uspatenttrademarkoff/)

### 8. Adjacent ODP endpoints that the search surface should know about

- Per-application detail endpoints (all `GET /{applicationNumberText}/...`): `meta-data`, `adjustment`, `assignment`, `attorney`, `continuity`, `foreign-priority`, `transactions`, `documents`, `associated-documents`. Reference data: `GET`+`POST /patent/status-codes` (status-code dictionary lookup/search). Also `POST /patent/applications/text-to-search` — a natural-language→search endpoint that exists in Swagger but is NOT on the public site (treat as experimental). Petition decisions live under `/petition/decisions/search` (GET+POST), and dataset products under `/datasets/products/search`. (https://learn.microsoft.com/en-us/connectors/uspatenttrademarkoff/)
- Pagination default is `limit:25`; the connector and patent_client both decline to document a hard max — treat the page-size ceiling as UNVERIFIED and discover it empirically, while capping in-driver. USPTO may rate-limit by volume and complex filters / large date ranges can time out. (https://learn.microsoft.com/en-us/connectors/uspatenttrademarkoff/ ; https://patent-client.readthedocs.io/en/latest/user_guide/open_data_portal.html)

### 9. Repo grounding — the extend-in-place gap

- Current `@beep/uspto` `searchApplications` issues only `GET .../applications/search?q=${encodeURIComponent(query)}` and decodes the `patentFileWrapperDataBag` envelope; there is no structured `filters`/`rangeFilters`/`sort`/`pagination`/`facets` body, no Lucene escaping, no friendly→API field map, and no identifier disambiguation. The net-new search surface (a `Uspto.search.ts`-style module with an effect/Schema-validated request model, an escaping helper, a friendly→API field map, and a DSL→ODP compiler) bolts onto the existing service exactly as the CAPTURE routes it. Reuse the existing envelope decoders (`MetadataEnvelope`), `UsptoApplicationMetadata` model, `UsptoError` reasons, and the same-origin `X-API-KEY` scoping already in `Uspto.service.ts`. (`packages/drivers/uspto/src/Uspto.service.ts`, `Uspto.models.ts`)

### 10. Licensing (porting note)

- What is being ported is *design shape*, not bulk code: the PatentsView DSL grammar and the ODP field namespace are USPTO/PatentsView public-record material (US-government, effectively public domain for the data/spec). The concrete code patterns (escaping policy, friendly→API map, identifier ladder, parameter-object validation) originate in third-party MCP repos (`uspto_pfw_mcp`, `mcp-uspto`, `uspto-patents-mcp`, `us-gov-open-data-mcp`); their software licenses were NOT fetched in this pass and MUST be checked before any verbatim copy.

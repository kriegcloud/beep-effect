# layer-unwrap-effect-provider-selection

Scope: validate `Layer.unwrapEffect` against vendored effect-v4 (beta.91) and design a provider-agnostic, config-driven, lazily-constructed tag selecting `anthropic|openai-compat|xai|venice-ai` (+ swappable gov-data drivers) at build time, deciding `Layer.die` vs typed error for unimplemented providers.

## Findings

### A. The vendored-API reality: `Layer.unwrapEffect` does NOT exist in effect-v4 ‚Äî it is `Layer.unwrap`

- The repo vendors **effect `4.0.0-beta.91`** (`.repos/effect-v4/packages/effect/package.json`). Its `Layer.ts` exports **`unwrap`**, not `unwrapEffect` and not `unwrapScoped`. Authoritative repo source: `/.repos/effect-v4/packages/effect/src/Layer.ts:1126`. Grep for `unwrapEffect|unwrapScoped` across that file returns **NONE**.
- v4 signature (scoped-aware, single function):
  `export const unwrap = <A, E1, R1, E, R>(self: Effect<Layer<A, E1, R1>, E, R>): Layer<A, E | E1, R1 | Exclude<R, Scope.Scope>>` (`.repos/effect-v4/packages/effect/src/Layer.ts:1126-1131`). Docstring: "Unwraps a `Layer` from an `Effect`, flattening the nested structure‚Ä¶ the combined error and dependency types from both the outer Effect and the inner Layer."
- By contrast the **published 3.x** API ref documents BOTH `Layer.unwrapEffect: <A,E1,R1,E,R>(self: Effect<Layer<A,E1,R1>,E,R>) => Layer<A, E|E1, R|R1>` (@since 2.0.0) AND `Layer.unwrapScoped: ‚Ä¶ => Layer<A, E|E1, R1 | Exclude<R, Scope.Scope>>` (@since 2.0.0) ‚Äî https://effect-ts.github.io/effect/effect/Layer.ts.html . Adversarial cross-check: the two primary sources agree v3 has `unwrapEffect`+`unwrapScoped`; vendored v4 has neither and instead one `unwrap` whose signature (`Exclude<R, Scope.Scope>`) equals the old `unwrapScoped`. **Conclusion: v4 merged `unwrapEffect`+`unwrapScoped` into `unwrap`. The CAPTURE seed name "`Layer.unwrapEffect`" must be written `Layer.unwrap` in this repo.**
- A WebSearch could not surface an *official changelog line* spelling out this exact rename (Effect's own tracker notes migration docs are incomplete ‚Äî https://github.com/Effect-TS/effect-smol/issues/1378 ), so the rename is established here from the two primary API sources above, not from prose docs. Marked accordingly under Open/Unverified.

### B. CAPTURE premise correction: `Layer.unwrap` is ALREADY battle-tested in this repo for build-time driver selection

- The CAPTURE nugget says provider selection via this combinator has "zero occurrences" / is net-new. That is true only for the **v3 spelling**. The **v4 `Layer.unwrap`** already has 8 in-repo call sites (grep `Layer.unwrap` under `packages/**/src`). Two are direct blueprints for this subtopic:
  - **Typed-error variant** ‚Äî `packages/drivers/m365/src/M365.service.ts:1176`: `static readonly layer: Layer.Layer<M365, M365Error> = Layer.unwrap(loadEnvConfig().pipe(Effect.map(M365.makeLiveLayer), Effect.mapError((cause) => M365Error.fromReason("config", { cause }))))`. Config failure surfaces as a **typed `M365Error` in the Layer error channel** ‚Äî the canonical "misconfigured provider is a recoverable typed error" precedent.
  - **Switch-based lazy selection variant** ‚Äî `packages/tooling/test-kit/test-utils/src/SqlTest.ts:1340`: `Layer.unwrap(Effect.gen(function*(){ const mode = options.mode ?? "auto"; ‚Ä¶ if (shouldUseExternalPgliteLayer(...)) return makeConfiguredSqlTestLayer(PgExternalTestDriver,‚Ä¶); if (shouldUseTestcontainersPgliteLayer(...)) return makeConfiguredSqlTestLayer(PgliteTestcontainersTestDriver,‚Ä¶); return makeConfiguredSqlTestLayer(PgliteInProcessTestDriver,‚Ä¶); }))`. This is exactly the requested shape: read config, branch, **return only the chosen driver's layer** so unused drivers never build. The `anthropic|openai-compat|xai|venice-ai` tag is a one-for-one analogue (external-pg/testcontainers/in-process ‚áí four LLM providers).
  - **Net-new is therefore consolidation, not invention**: a shared provider-agnostic LLM `Context.Service` tag + a single `Layer.unwrap` selector, mirroring `SqlTest`'s and `M365`'s proven shapes.

### C. `Layer.die` / `Layer.fail` constructors were REMOVED in v4 ‚Äî decide die-vs-typed-error with the surviving primitives

- v3 has both as top-level constructors: `Layer.die: (defect: unknown) => Layer<unknown>` ("Constructs a layer that dies with the specified defect", @since 2.0.0) and `Layer.fail: <E>(error: E) => Layer<unknown, E>` (@since 2.0.0) ‚Äî https://effect-ts.github.io/effect/effect/Layer.ts.html .
- The vendored v4 `Layer.ts` exports **neither** `die` nor `fail` (full export list grep shows `succeed, succeedContext, empty, sync, syncContext, effect, effectContext, effectDiscard, suspend, unwrap, mergeAll, merge, provide, provideMerge, flatMap, tap, tapError, tapCause, orDie, catchTag, catchCause, updateService, fresh, launch, mock, ‚Ä¶, span` ‚Äî `.repos/effect-v4/packages/effect/src/Layer.ts`). Surviving error primitives in v4: **`Layer.orDie`** (`A,E,R ‚Üí A,never,R`, "Translates effect failure into death of the fiber", line 1782), **`Layer.catchTag`** (line 1854), **`Layer.catchCause`** (line 1962), and **`Layer.catch`** (the `catch_` export aliased at line 1804-1819). `Effect.die` / `Effect.fail` / `Effect.orDie` remain (`.repos/effect-v4/packages/effect/src/Effect.ts:1612/1478/3622`).
- **v4 idioms for the two unimplemented-provider strategies:**
  - *Typed error* (caller can handle; recommended at the config boundary): inside the `Layer.unwrap` generator do `return yield* new ProviderNotImplemented({ provider })` (or `Effect.fail`). The error joins `unwrap`'s `E` channel ‚áí `Layer<ProviderTag, ProviderNotImplemented | ‚Ä¶>`. This is exactly the **M365 precedent** above (`Effect.mapError ‚Üí M365Error`). v4 docstring shows the same shape: `Layer.effect(Config, Effect.fail(new ConfigError()))` then `Layer.catchTag("ConfigError", ‚Ä¶)` (`.repos/effect-v4/packages/effect/src/Layer.ts:1840-1846`).
  - *Defect / crash* (truly unreachable branch, programmer error): `yield* Effect.die(new ProviderNotImplemented(...))` in the generator, or build the branch then `.pipe(Layer.orDie)` to strip the error to a fiber defect. v4 docstring demonstrates `flakyLayer.pipe(Layer.orDie)` ‚áí `Layer<‚Ä¶, never, never>` (`.repos/effect-v4/packages/effect/src/Layer.ts:1759-1777`).
- **Recommendation (die vs typed error):** if the provider literal set is *pinned by `Config.literals([...])`* (D below), Config validation already rejects unknown values, so an unmatched `switch`/`$match` branch is genuinely unreachable ‚Üí **`Effect.die`/`orDie`** is correct (fail-fast as a defect, no noise in the success-path type). For providers that are *valid config values but not yet wired* (e.g. a future `gov-data` driver listed in the literal set), prefer a **typed `ProviderNotImplemented` error** surfaced through the `unwrap` E channel ‚Äî it documents intent, is testable, and lets the app render a clear startup message ‚Äî matching the repo's existing M365 typed-config-error convention. (This is a design recommendation, not a sourced fact.)

### D. Config-driven selection: effect-v4 `Config.literals` + repo `LiteralKit` vocab

- effect-v4 adds **`Config.literals(literals, name)`** (@since 4.0.0) = `Config.schema(Schema.Literals(literals), name)`, restricting an env var to a fixed allowed set with a decode error otherwise (`.repos/effect-v4/packages/effect/src/Config.ts:1044`). `Config.literal(value, name)` (single value, @since 2.0.0) also exists (line 1016). Idiomatic selector: `const provider = yield* Config.literals(["anthropic","openai-compat","xai","venice-ai"], "AI_PROVIDER")` inside the `Layer.unwrap` generator.
- The repo already has provider-vocabulary precedent via **`LiteralKit`** and **`$match` dispatch**: `packages/drivers/ai-provider-cli/src/AiProviderCli.models.ts:27` (`export const AiProviderCliProvider = LiteralKit(["claude","codex"])‚Ä¶`) consumed by `AiProviderCliProvider.$match(provider, {‚Ä¶})` at `packages/drivers/ai-provider-cli/src/AiProviderCli.service.ts:75`. A unified LLM-provider `LiteralKit` (`["anthropic","openai-compat","xai","venice-ai"]`) gives both the `Config.literals` source schema AND the exhaustive `$match` branch selector inside `Layer.unwrap` ‚Äî repo-native, no new dependency.

### E. Canonical provider-agnostic pattern = `@effect/ai` `LanguageModel` (the official analogue, already used by the anthropic driver)

- Effect's official AI stack is built on **provider-agnostic programming**: write business logic against base `@effect/ai` service tags (`LanguageModel`, `EmbeddingsModel`), and "defer choosing the underlying provider (OpenAI, Anthropic, local models, mocks, etc.) until runtime"; provider impls ship as separate packages (`@effect/ai-openai`, `@effect/ai-anthropic`, `@effect/ai-amazon-bedrock`, `@effect/ai-google`) so you can "swap between or combine providers‚Ä¶ without initializing unused provider packages" ‚Äî https://effect.website/docs/ai/introduction/ .
- Concrete wiring shapes (primary blog): provider models `OpenAiLanguageModel.model("gpt-4o")` / `AnthropicLanguageModel.model("claude-3-7-sonnet-latest")`; client layer from Config `OpenAiClient.layerConfig({ apiKey: Config.redacted("OPENAI_API_KEY") })` then `Layer.provide(OpenAi, NodeHttpClient.layerUndici)`; multi-provider fallback via `ExecutionPlan.make({ provide: OpenAiLanguageModel.model("gpt-4o"), attempts: 3, schedule: Schedule.exponential("100 millis"), while: (e)=>e._tag==="NetworkError" }, { provide: AnthropicLanguageModel.model("claude-3-7-sonnet-latest") })` applied with `Effect.withExecutionPlan(effect, plan)` ‚Äî https://effect.website/blog/effect-ai/ .
- The repo's **anthropic driver already follows this exact pattern**: `AnthropicClient.layerConfig({ apiKey: Config.redacted(ANTHROPIC_API_KEY_ENV) }).pipe(Layer.provide(FetchHttpClient.layer))` and retry via `ExecutionPlan.make({ schedule: Schedule.exponential(Duration.millis(...), 2) })` + `Stream/Effect.withExecutionPlan` (`packages/drivers/anthropic/src/Anthropic.service.ts:36-37,103-106`; `Anthropic.repair.ts:102-105,156`). So the provider-agnostic selector should produce one of these client/model layers per branch ‚Äî the unified tag sits one level above `LanguageModel`, choosing which provider client layer to `Layer.provide`.

### F. Lazy construction ‚Äî satisfied natively; courts-db is only a conceptual analogue (BSD)

- `Layer.unwrap` is inherently lazy in the right way: the generator runs at **build time**, evaluates config, and **returns only the selected branch's `Layer`**; the non-selected providers' layers are never constructed, so unused provider clients/HTTP pools/credentials are never initialized. This directly meets the "without initializing unused providers" requirement with no extra machinery (`.repos/effect-v4/packages/effect/src/Layer.ts:1095-1131`; behavior demonstrated by `SqlTest.ts:1340` returning exactly one driver layer). For extra deferral, `Layer.suspend` (line 1091) and `Effect.suspend` defer evaluation of a chosen branch until build.
- The CAPTURE seed `courts-db#9` (`courts_db/__init__.py:23-38`, Python `def __getattr__(name): ‚Ä¶ load_courts_db(); globals()[name]=value`) is the Python module-attribute lazy-load idiom for deferring a heavy reference-data build (2,809-entry regex compile) until first access. It is a **conceptual reference only (P3)**, not a port ‚Äî the Effect-native equivalent is the `Layer.unwrap` branch + `Layer.suspend`/`Layer.fresh`, or memoization via the layer MemoMap. courts-db is published by Free Law Project under a **permissive BSD license** (https://github.com/freelawproject/courts-db , https://free.law/projects/courts-db/ ); if its court/reporter *data tables* were ever vendored for the gov-data drivers, BSD attribution applies (exact 2- vs 3-clause unverified ‚Äî see Open).

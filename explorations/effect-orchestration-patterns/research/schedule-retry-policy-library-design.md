# schedule-retry-policy-library-design

Scope: design a net-new centralized retry-policy/`Schedule` foundation module (named policies, jittered backoff, Retry-After/429 honoring, retry budgets, idempotency gating, effect-v4 `Schedule`/`ExecutionPlan` composition) and the migration path off the inline `Schedule.exponential` in the LLM drivers (seed: research-squad#3).

### effect-v4 Schedule API â€” what is and isn't available (migration-critical)

- `Schedule.exponential(base: Duration.Input, factor?: number): Schedule<Duration>` â€” `factor` defaults to 2; delays are `base * factor^n`. The d.ts example itself shows the canonical retry idiom `Schedule.exponential("50 millis").pipe(Schedule.both(Schedule.recurs(5)))`. Source: `node_modules/effect/dist/Schedule.d.ts:3172` + jsdoc lines ~3145-3178 (primary).
- **`Schedule.compose` does NOT exist in v4-beta.91.** Neither do `union`/`intersect`/`whileInput`/`whileOutput`/`check`/`maxElapsed`/`upTo`/`recurWhile` (grep returned nothing). The CAPTURE seed snippet `Schedule.exponential(...).pipe(Schedule.compose(Schedule.recurs(2)))` is **external effect-v2/v3 code (research-squad)** and will not compile here. v4 replacement: bound count with `Schedule.both(Schedule.recurs(n))` (intersection/AND â€” recurs only while *both* want to continue, taking the **max** of the two delays) or use the `attempts`/`times` option on `ExecutionPlan`/`Effect.retry`. Source: `node_modules/effect/dist/Schedule.d.ts:1012` `both` jsdoc ("Continues only while BOTH schedules want to continue (intersection/AND logic)") + absence grep (primary). MIGRATION GOTCHA.
- `Schedule.jittered(self): Schedule` is **unary** in v4 â€” it scales each delay by a random factor between **0.8 and 1.2** (Â±20%), with no `{min,max}` argument form. Source: `node_modules/effect/dist/Schedule.d.ts:3623` jsdoc ("Each recurrence delay is scaled by a random factor between `0.8` and `1.2`") (primary). This is **not** AWS "Full Jitter" (see below) â€” a deliberate design choice to document.
- `Schedule.modifyDelay(self, (output, delay) => Effect<Duration.Input>)` is the v4 hook for **adaptive / Retry-After delays**: the effectful transform can read schedule output to override the next delay. This is the composition point for honoring a server `retry-after`. Source: `node_modules/effect/dist/Schedule.d.ts:3536` jsdoc (primary). Other available combinators: `addDelay`, `either` (union/OR), `take`, `delays`, `passthrough`, `during`, `elapsed`, `spaced`, `fixed`, `recurs`, `fibonacci`.
- **Predicate gating lives on the consumer, not the Schedule.** `Effect.retry(effect, { schedule?, while?, until?, times? })` where `while`/`until` are `(error: E) => boolean | Effect<boolean,...>`. `ExecutionPlan.make({ provide, attempts?, while?: (input) => Effect<boolean,...>, schedule? })`. So "retry only retryable errors" (the anthropic `isRetryable` predicate) is an option on `retry`/`ExecutionPlan`, while the named `Schedule` carries only timing/count. Source: `node_modules/effect/dist/Effect.d.ts:6483-6486,6541` + `ExecutionPlan.d.ts:108-112` (primary). This shapes the module API: named policies export `Schedule`s; the *predicate* is a separate composable (`isRetryable`-style) the caller supplies.

### Backoff + jitter algorithm choices (AWS canon)

- AWS "Exponential Backoff And Jitter" gives the three reference formulas: **Full Jitter** `sleep = random(0, min(cap, base * 2^attempt))`; **Equal Jitter** `sleep = base + random(0, min(cap, base * 2^attempt))`; **Decorrelated Jitter** `sleep = min(cap, random(0, base) * last_sleep * 3)`. Conclusion: **Full Jitter performs best** (least client work; Equal Jitter is the loser; Decorrelated uses slightly less time but more work). Source: https://aws.amazon.com/blogs/architecture/exponential-backoff-and-jitter/ (primary, AWS) â€” corroborated https://brooker.co.za/blog/2015/03/21/backoff.html (the author's companion post).
- **Design implication / gotcha:** effect's built-in `Schedule.jittered` (Â±20% multiplicative) is closer to a narrow Equal-Jitter band, NOT Full Jitter (which goes all the way to 0). If the module wants AWS Full Jitter semantics it must build it via `Schedule.exponential(base).pipe(Schedule.modifyDelay((_, d) => Random.next... ))` capped, not `Schedule.jittered`. Recommendation: ship both â€” `jittered<name>` (cheap, Â±20%, default) and a `fullJitter` helper for high-contention fan-out. Cross-check: AWS Builders' Library "Timeouts, retries, and backoff with jitter" reiterates jitter is the load-spreading mechanism, separate from the backoff curve â€” https://aws.amazon.com/builders-library/timeouts-retries-and-backoff-with-jitter/ (primary, AWS).

### Retry-After / 429 honoring (standards + provider specifics)

- `Retry-After` is defined in **RFC 9110 Â§10.2.3** (June 2022); value is either delay-seconds (integer) or an HTTP-date. **429 Too Many Requests** is **RFC 6585 Â§4** (April 2012). Canonical client rule: if `Retry-After` is present on a 429/503, honor it exactly; if absent, fall back to capped exponential backoff + jitter. 429 = "you are over quota"; 503 = "server unavailable for everyone". Source: https://developer.mozilla.org/en-US/docs/Web/HTTP/Reference/Status/429 + https://http.dev/retry-after (secondary, both cite the RFCs).
- **Anthropic (primary):** on a 429 the API returns a `retry-after` header (seconds) plus token-bucket rate-limit headers: `anthropic-ratelimit-requests-{limit,remaining,reset}`, `anthropic-ratelimit-tokens-{limit,remaining,reset}`, `anthropic-ratelimit-input-tokens-*`, `anthropic-ratelimit-output-tokens-*` (resets in RFC 3339); plus `anthropic-priority-*` and `anthropic-fast-*` variants. Rate limiting uses the **token-bucket algorithm** (continuous replenishment, not fixed-window reset). Honor `retry-after` exactly. Source: https://platform.claude.com/docs/en/api/rate-limits (primary; note `docs.anthropic.com/en/api/rate-limits` now 301-redirects here). Separately, **`529` "overloaded"** is a server-wide capacity signal (not your quota) and should be retried with backoff â€” corroborated https://www.respan.ai/articles/anthropic-api-rate-limits (secondary).
- **OpenAI (primary cookbook):** exposes `x-ratelimit-limit-requests`, `x-ratelimit-remaining-requests`, `x-ratelimit-reset-requests` (and `-tokens` analogues); `retry-after` appears only on 429 (seconds or HTTP-date) and is "more precise than anything you'd calculate yourself." Recommended remediation: automatic retry with **random exponential backoff**. Source: https://developers.openai.com/cookbook/examples/how_to_handle_rate_limits + https://platform.openai.com/docs/guides/rate-limits (primary).
- **Design implication:** the LLM driver errors must surface the parsed `retry-after`/reset so the policy can override the computed backoff. Effect-AI's `AiError` carries `isRetryable` (already used by anthropic's `while`) but the module needs a `retryAfterMillis` extractor (per-provider) feeding `Schedule.modifyDelay`. UNVERIFIED whether `@effect/ai`'s `AiError` exposes the raw `retry-after` header â€” must be checked against the vendored `effect/unstable/ai` source before relying on it.

### Retry budgets (preventing retry-amplified overload)

- **Google SRE** uses a **per-client retry budget**: each client tracks the ratio of requests that are retries and only retries while that ratio stays **below 10%**, layered on a **per-request budget of 3 attempts**. Combined, this caps request amplification at ~**1.1x** during partial outages (vs. unbounded with naive per-request retries). Source: https://sre.google/sre-book/handling-overload/ (primary, Google SRE book, ch. "Handling Overload").
- **gRPC** encodes the same idea as a **token bucket**: `retryThrottling: { maxTokens, tokenRatio }`; each failed RPC debits 1 token, each success credits `tokenRatio`, and retries pause once `token_count` drops below `maxTokens/2`. Example `{ "maxTokens": 10, "tokenRatio": 0.1 }`. Source: https://grpc.io/docs/guides/retry/ + https://github.com/grpc/proposal/blob/master/A6-client-retries.md (primary).
- **Design implication:** a budget is **stateful** and cannot live in a pure `Schedule`; it needs a small Effect service (a `Ref`/`MutableRef` token bucket) wrapping the retry, or a `Schedule.tapInput`-driven counter. This is adjacent to the Juris.AI#5 quota-limiter nugget (sliding-window per-key) and the uspto#5 bulkhead/circuit-breaker stack in CAPTURE â€” the budget/limiter belongs in the same resilience package, layered *over* the named `Schedule`s, not inside them.

### Idempotency gating (which operations may auto-retry)

- The **IETF HTTPAPI WG** `Idempotency-Key` header draft (`draft-ietf-httpapi-idempotency-key-header`, latest **-07**; **still a Working-Group Internet-Draft, NOT yet an RFC** as of 2026-06) lets clients make non-idempotent methods (POST/PATCH) safely retryable by sending a client-generated unique key (e.g. a UUID) that the server uses to dedupe retries. GET/PUT/DELETE are already idempotent per HTTP semantics. Source: https://datatracker.ietf.org/doc/draft-ietf-httpapi-idempotency-key-header/ (primary, IETF).
- **Design implication:** the retry library must gate auto-retry on idempotency. Safe-by-default policy: auto-retry only idempotent ops (reads, GETs, LLM *completions* which are side-effect-free) and any op carrying an idempotency key; require explicit opt-in for un-keyed mutations. Stripe is the canonical production model of client-supplied idempotency keys on POST â€” corroborated by the draft's own motivation section. This argues for an `idempotent: boolean` (or `idempotencyKey?`) field on the policy-application API, not on the `Schedule` itself.

### Migration path off inline `Schedule.exponential`

1. Land the named policies in `@beep/utils/Retry` (+ barrel export), each as a `Schedule` plus a co-located default `attempts`/`while`-predicate helper (`isRetryableAiError`) so callers reconstruct the existing `ExecutionPlan` behavior.
2. In `Anthropic.service.ts`/`Anthropic.repair.ts`, replace the inline `Schedule.exponential(Duration.millis(BASE), 2)` inside `ExecutionPlan.make` with `Retry.llmRetry` (keep `attempts`/`while`/`provide` on the plan unchanged), preserving the `Stream.withExecutionPlan(..., { preventFallbackOnPartialStream: true })` call site. The constants `ANTHROPIC_DEFAULT_RETRY_*` either move to the policy or stay as the driver's tuning override.
3. **Close the openai-compat gap** by adding an `ExecutionPlan`/`Effect.retry` wrapper using `Retry.llmRetry` (net-new resilience, not a refactor).
4. Layer the retry-budget/limiter service over these policies for the gov/data drivers (USPTO/CourtListener/GovInfo) per the uspto#5 / Juris.AI#5 nuggets. Source: in-repo driver code + CAPTURE.md (primary/synthesis).

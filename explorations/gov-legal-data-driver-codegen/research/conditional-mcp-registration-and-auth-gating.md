# conditional-mcp-registration-and-auth-gating

Scope: credential-gated MCP tool registration so a driver's tools surface only when its key is present (key-gated CourtListener/GovInfo/DOL vs always-on keyless eCFR/FedReg), and how that composes with beep's two Effect-native MCP servers (`nlp-mcp`, `m365-mcp`) and operationId-derived tool naming. Primary sources: the actual Effect source shipped in this repo (`effect@4.0.0-beta.91`, `effect/unstable/ai/*`), plus the MCP spec, Orval `@orval/mcp` docs, and the five upstream API docs.

## Findings

### A. The gating decision: which drivers need a key (the binary that drives registration)

- **eCFR — keyless / always-on.** "No API keys are needed; all you need is an HTTP client or browser." Public unauthenticated REST at base `https://www.ecfr.gov`. Source: eCFR developer docs — https://www.ecfr.gov/developers/documentation/api/v1 (and reader-aids developer resources https://www.ecfr.gov/reader-aids/ecfr-developer-resources).
- **Federal Register — keyless / always-on.** "FederalRegister.gov APIs do not require API keys; all you need is an HTTP client or browser." IP-based rate limiting on aggressive querying; a descriptive `User-Agent` header is recommended. Source: https://www.federalregister.gov/developers/documentation/api/v1 and https://www.federalregister.gov/reader-aids/developer-resources/rest-api. (Note: the docs host 302-redirects bots to `unblock.federalregister.gov`; the page self-identifies as "An official website of the United States government" — CAPTURE's "unofficial prototype" framing is not supported by the current docs, see Open/Unverified.)
- **GovInfo — KEY-GATED.** "The govinfo API requires an existing api.data.gov key, or you can request one for free." api.data.gov key passed as `api_key`. Source: https://www.govinfo.gov/features/api and api.data.gov developer manual https://api.data.gov/docs/developer-manual/.
- **DOL — KEY-GATED.** "you must supply a valid token as the 'KEY' parameter" and "register at https://devtools.dol.gov/developer and request an API key for each application" (free). Exception: the dataset-catalog endpoint needs no key. Sources: DOL developer beginner's guide http://usdepartmentoflabor.github.io/DOLAPI/beginners-guide.html ; DOL API User Guide PDF https://www.dataportal.dol.gov/pdf/dol-api-user-guide.pdf ; https://developer.dol.gov/.
- **CourtListener V4 — KEY-GATED (hard).** "The v4 API endpoints enforce authentication, and anonymous requests now receive a 401 Unauthorized." Token auth header is literally `Authorization: Token <your-token-here>` (NOT `Bearer`). Token lives on the CourtListener profile; env `COURTLISTENER_API_TOKEN`. Sources: https://www.courtlistener.com/help/api/rest/ and https://wiki.free.law/c/courtlistener/help/api/rest/v4/overview.

Resulting registration matrix: `{ecfr, federal-register}` register unconditionally; `{courtlistener, govinfo, dol}` register only when their respective env key resolves.

### B. Tool naming is operationId-derived because the Effect `Tool.make` name IS the MCP tool name

- In Effect's MCP server, `registerToolkit` emits each tool with `name: tool.name` directly as the wire-level `McpTool.name` — there is no transformation. Source (primary, in-repo): `node_modules/effect/dist/unstable/ai/McpServer.js` `registerToolkit` body (`new McpTool({ name: tool.name, description: Tool.getDescription(tool), inputSchema: Tool.getJsonSchema(tool), ... })`).
- `Tool.make(name, options)` takes the literal tool name as its first arg: `export declare const make: <const Name extends string, ...>(name: Name, options?: {...})`. Source: `node_modules/effect/dist/unstable/ai/Tool.d.ts:810`. beep's two servers prove both casings: `Tool.make("Tokenize", …)` (`packages/foundation/capability/nlp/src/Tools/Tokenize.ts`) and `Tool.make("m365_list_drives", …)` / `"m365_get_site"` (`packages/drivers/m365-mcp/src/M365Tools.ts:93+`).
- Therefore OpenAPI→MCP codegen lands operationId→tool-name 1:1 by emitting `Tool.make(operationId, {...})`. This mirrors Orval's `@orval/mcp`, where the OpenAPI `operationId` is the default tool/function name and "Orval uses unique operationId values for every operation." Sources: https://orval.dev/docs/guides/mcp/ and Orval output config https://www.orval.dev/reference/configuration/output.
- MCP spec defines `name` only as "Unique identifier for the tool" — no published regex on the 2025-06-18 tools page. Source: https://modelcontextprotocol.io/specification/2025-06-18/server/tools. (Sanitization/namespacing caveat in Open/Unverified.)
- **Do not adopt Orval itself.** `@orval/mcp` "currently only works in `single` mode" (one spec per config) and emits axios + Zod + its own `server.ts`/`handlers.ts`/`http-client.ts`/`tool-schemas.zod.ts`. Port the operationId→tool-name pattern onto Effect `Tool.make`/`Toolkit` + `McpServer`, not the generated code. Sources: https://orval.dev/docs/guides/mcp/ (config `{ mode:'single', client:'mcp', baseUrl, target:'src/handlers.ts', schemas:'src/http-schemas' }`; "The `mcp` client currently only works in `single` mode").

### C. The canonical Effect-native conditional-registration mechanism (server-credential gating)

- Two registration entry points exist in `effect/unstable/ai/McpServer`:
  - `toolkit(toolkit): Layer<never, never, HandlersFor<Tools> | …>` — static layer registration (what beep uses today). Implemented as `Layer.effectDiscard(registerToolkit(toolkit)).pipe(Layer.provide(McpServer.layer))`. Source: `McpServer.js:522` and `McpServer.d.ts:289`.
  - `registerToolkit(toolkit): Effect<void, never, McpServer | HandlersFor<Tools> | …>` — imperative, callable inside an Effect program. Source: `McpServer.d.ts:282`, `McpServer.js:469`.
- **Gating recipe (boot-time, env-key):** read the key as an optional config and choose the registration layer with `Layer.unwrap`:
  - `Config.redacted(name): Config<Redacted<string>>` (source: `node_modules/effect/dist/Config.d.ts:1134`) wrapped by `Config.option(self): Config<Option<A>>` (source: `Config.d.ts:500`) → `Config<Option<Redacted<string>>>`.
  - `Layer.unwrap(self: Effect<Layer<…>>): Layer<…>` (source: `node_modules/effect/dist/Layer.d.ts:1263`) lets the registration layer be chosen by an effect; the absent-key branch returns `Layer.empty` (source: `Layer.d.ts:890`). Shape: `Layer.unwrap(Effect.gen(function*(){ const key = yield* Config.option(Config.redacted("COURTLISTENER_API_TOKEN")); return Option.isSome(key) ? McpServer.toolkit(CourtListenerToolkit).pipe(Layer.provide(CourtListenerHandlersLive)) : Layer.empty }))`.
- **Why boot-time gating is clean:** registration runs at layer build, before `run` forks the server and before the client's `initialize`. The advertised `tools` capability and every `tools/list` are computed from `server.tools`, which only contains registered tools — so a gated-out driver is invisible *and* uncallable. The capability block sets `capabilities.tools = { listChanged: true }` only `if (server.tools.length > 0)`. Source: `McpServer.js` initialize handler (`if (server.tools.length > 0) { capabilities.tools = { listChanged: true } }`).
- `effect` is MIT-licensed and already a direct dependency (`effect@4.0.0-beta.91`); this is direct use, no porting. (License: MIT, well-established for the Effect project.)

### D. Composition with beep's two existing Effect-native MCP servers

- Both existing servers use the identical shape: `McpServer.toolkit(X).pipe(Layer.provide(Handlers), Layer.provide(McpServer.layerStdio({name,version})), Layer.orDie)`. Sources: `packages/drivers/m365-mcp/src/Server.ts` and `packages/drivers/nlp-mcp/src/Server.ts`.
- **`nlp-mcp` already proves N-toolkits→one-server merge:** `Layer.mergeAll(McpServer.toolkit(NlpToolkit).pipe(Layer.provide(WinkNlpToolkitLive)), McpServer.toolkit(StreamingToolkit).pipe(Layer.provide(StreamingToolkitHandlersLive)))` over one `layerStdio`. Source: `packages/drivers/nlp-mcp/src/Server.ts` `makeServerLayer`. `Layer.mergeAll` is the published combinator (source: `Layer.d.ts:1308`).
- So a gov-legal MCP server is just `Layer.mergeAll(<keyless toolkits>, <Layer.unwrap key-gated toolkits>)` over one `McpServer.layerStdio`. Keyless eCFR/FedReg toolkit layers are unconditional; CourtListener/GovInfo/DOL are `Layer.unwrap(...)` branches that collapse to `Layer.empty` when the key is absent. This is purely additive to the two existing servers; per the live-tree snapshot the gov/legal work is "patterns, not a 3rd server" — but a separate `gov-legal-mcp` bin following the `nlp-mcp` template is the natural landing (align-stage decision, not forced here).
- `Toolkit` is the unit that bundles tools + typed handler requirements; `McpServer.toolkit`/`registerToolkit` require both `McpServer` and `Tool.HandlersFor<Tools>` in context. In a `Layer.unwrap` branch each gated toolkit must `Layer.provide` its own handler layer; the `Layer.empty` branch provides nothing (correct, since it registers nothing). Source: `McpServer.d.ts:282,289` (the `R` includes `Tool.HandlersFor<Tools>`).

### E. Effect's built-in `EnabledWhen` annotation — a *second, orthogonal* gating axis (and why it is NOT the credential-gating tool)

- Effect ships `McpSchema.EnabledWhen`: a `Context.Service` whose value is `Predicate.Predicate<{ protocolVersion: string; capabilities: ClientCapabilities; clientInfo: Implementation; _meta?: … }>`. "Annotation to conditionally enable or disable tools based on client information." Source: `node_modules/effect/dist/unstable/ai/McpSchema.d.ts:3427-3442` and `McpSchema.js:1996`.
- It is attached per-tool via `tool.annotate(McpSchema.EnabledWhen, predicate)` (the `annotate<I,S>(tag, value)` method, source `Tool.d.ts:274`), and evaluated by `filterByClient` on every list endpoint: a tool/resource/prompt is omitted from `tools/list` when its `EnabledWhen` predicate returns false for the initialized client. Source: `McpServer.js` `filterByClient` (`const enabledWhen = Context.getOrUndefined(item.annotations, EnabledWhen); if (!enabledWhen || enabledWhen(client)) out.push(...)`).
- **Works on both transports.** `initialize` stores the client's params keyed by `sessionId` (HTTP, when an `HttpServerRequest` is in context) or by `String(client.id)` in the `else` branch (stdio); `getInitializedClient` resolves either. Source: `McpServer.js:853-864` (`if (httpRequest) { clientSessions.set(sessionId, params) } else { clientSessions.set(String(client.id), params) }`).
- **Critical limit:** `filterByClient` is applied only to the *list* endpoints (`tools/list`, `resources/list`, `prompts/list`, `resources/templates/list`) — NOT to `tools/call`. So `EnabledWhen` hides a tool from discovery but does not block invocation; it is discovery-shaping, not access control. Source: `McpServer.js` request-handler map (only the `*/list` handlers call `filterByClient`; `tools/call` calls `server.callTool` directly).
- **Conclusion:** `EnabledWhen` gates on the *client's* declared identity/capabilities, not the *server's* possession of an API key, and it doesn't prevent calls. For "surface this driver only when the server holds its key," conditional layer registration (section C) is strictly better: unregistered tools are absent from discovery *and* uncallable. `EnabledWhen` is the right tool for an orthogonal need (e.g., per-tenant / per-client-capability exposure within a single registered toolset).

### F. Runtime/dynamic gating + `list_changed` (if a key arrives after startup)

- MCP spec: servers declare `{ "capabilities": { "tools": { "listChanged": true } } }` and SHOULD send `{"jsonrpc":"2.0","method":"notifications/tools/list_changed"}` when the tool set changes; clients re-call `tools/list` on receipt. Revision 2025-06-18. Source: https://modelcontextprotocol.io/specification/2025-06-18/server/tools.
- Effect implements this: `addTool` (called once per tool by `registerToolkit`) fires `notifications/tools/list_changed`, and the notification layer debounces bursts via a per-tag `setTimeout` that coalesces duplicate `*/list_changed` tags within a tick. Source: `McpServer.js` (`addTool: options => Effect.suspend(() => { … return notifications.client["notifications/tools/list_changed"]({}) })` and the `listChangedHandles` setTimeout coalescing block). The full notification RPC set (`ToolListChangedNotification = Rpc.make("notifications/tools/list_changed", …)`) is in `McpSchema.js`/`ServerNotificationRpcs`.
- So: to enable a driver mid-session (key supplied via config reload or `McpServer.elicit`), call `McpServer.registerToolkit(DriverToolkit)` inside the running server; each tool add emits a (debounced) `list_changed`. For the env-key-at-boot use case this dynamic path is unnecessary — boot-time `Layer.unwrap` is simpler and sufficient.

### G. Error/result surface for auth failures (so a missing/invalid key fails gracefully when a tool IS registered)

- MCP distinguishes protocol errors (JSON-RPC `error`, e.g. unknown tool / invalid args) from in-band tool-execution errors (`result.isError = true`). API failures like rate limits are spec-recommended to use `isError: true`. Source: https://modelcontextprotocol.io/specification/2025-06-18/server/tools ("Tool Execution Errors … API failures").
- Effect maps any handler failure to `new CallToolResult({ isError: true, content: [{ type:"text", text: Cause.pretty(cause) }] })` automatically (and logs the cause). Source: `McpServer.js` `registerToolkit` handler `Effect.matchCause({ onFailure: cause => new CallToolResult({ isError:true, … }) })`. So a registered CourtListener tool that gets a 401 at call time returns a clean tool error, not a transport crash — useful when a key is present-but-invalid (distinct from the present/absent registration decision).
- Tool behavior hints relevant to gov read-only drivers: `Tool.Readonly` defaults to `false` and `Tool.Destructive` defaults to `true`; annotate GET-only gov tools with `.annotate(Tool.Readonly, true).annotate(Tool.Destructive, false)` so clients can present them as safe. These emit MCP `readOnlyHint`/`destructiveHint`. Source: `Tool.d.ts:1233-1269` and the `annotations` block of `registerToolkit` in `McpServer.js`.

## Open / Unverified

- **MCP tool-name charset/length constraint UNVERIFIED at spec level.** The 2025-06-18 tools page defines `name` only as "unique identifier," with no regex. Some MCP SDK JSON schemas have historically constrained names to `^[a-zA-Z0-9_-]+$` (and ~64–128 char caps). Since OpenAPI `operationId`s can contain disallowed characters and collide across the 5 drivers in one merged server, codegen should (a) sanitize operationIds to a safe charset and (b) namespace per driver (e.g. `courtlistener_…`, mirroring m365's `m365_` prefix) — but the exact upstream constraint needs a check against the live MCP SDK schema, not just the prose spec.
- **Federal Register "official vs unofficial prototype":** CAPTURE calls FedReg an "unofficial prototype"; the current federalregister.gov docs present it as an official US-government API and keyless. The keyless fact is confirmed; the "prototype" characterization appears stale/incorrect for the current API — flag for the auth-matrix subtopic.
- **api.data.gov shared-key semantics for GovInfo/DOL:** both accept an api.data.gov key, but whether one DEMO_KEY/shared key spans both vs. per-service keys, and the exact param name per service (GovInfo `api_key` query vs DOL `KEY`), should be pinned by the per-driver auth-matrix subtopic; only the binary "key required" is load-bearing here and is confirmed.
- **Effect AI module stability:** these APIs live under `effect/unstable/ai` (beta.91) and are marked `@since 4.0.0`; `unstable` implies the `registerToolkit`/`toolkit`/`EnabledWhen`/`Layer.unwrap` surface may shift before GA. Pin the effect version in the codegen template and re-verify on bumps.
- **No native multi-spec MCP codegen exists off-the-shelf** that targets Effect; the repo's own `runpod`/`acp` `scripts/generate.ts` precedent is the porting base. Whether the gov toolkits merge into one `gov-legal-mcp` bin or attach to an existing server is an align-stage decision, not settled by this research.

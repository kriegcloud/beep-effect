# inrepo-codegen-precedent-and-licensing

Scope: inventory beep's runpod/acp Effect-native codegen precedent + the govinfo partial scaffold to reuse, resolve licensing for the two donor repos (us-legal-tools, us-gov-open-data-mcp), and pin the USPTO/PatentsView deprecation cliff affecting any sibling patents driver. (researched 2026-06-29)

## Findings

### A. In-repo codegen precedent — there are TWO distinct styles already shipping

- **The repo already depends on an Effect-native OpenAPI generator.** `@effect/openapi-generator` is pinned at `4.0.0-beta.91` in the root catalog (`package.json:32`) and consumed by `@beep/acp` (`packages/drivers/acp/package.json:86` → `"@effect/openapi-generator": "catalog:"`). The npm package is **MIT**, latest `4.0.0-beta.92`, beta.91 published `2026-06-28` (one day before this research), described as "Generate Effect Schema types, HTTP clients, and HttpApi modules from OpenAPI specifications" (https://registry.npmjs.org/@effect/openapi-generator). This is the canonical "reuse the repo's own Effect codegen path — do NOT introduce Orval/axios/Zod" lever. Verified in-repo: `grep -rE '"(orval|@orval/|axios|zod)"' packages apps` over real `package.json` deps returns **only** a test fixture (`packages/tooling/library/repo-utils/test/fixtures/mock-monorepo/...`), i.e. no production Orval/axios/Zod anywhere.

- **acp style = library-driven + remote pinned-spec download (the "fetch-specs" deterministic pattern).** `packages/drivers/acp/scripts/generate.ts` imports `make as makeJsonSchemaGenerator` from `@effect/openapi-generator/JsonSchemaGenerator`, downloads upstream `schema.unstable.json` + `meta.unstable.json` from a **pinned** GitHub release (`const CURRENT_SCHEMA_RELEASE = "v0.11.3"`, base URL `https://github.com/agentclientprotocol/agent-client-protocol/releases/download/${tag}`), normalizes nullable JSON-Schema unions, and rewrites package-private `src/_generated/{schema.gen.ts,meta.gen.ts}` via the TypeScript AST. It is an `effect/unstable/cli` `Command` with a `--skip-download` flag, runs on `FetchHttpClient` + `ChildProcessSpawner` + `FileSystem`, and emits `generator.generate("openapi-3.1", normalizedDefinitions, false)` (`acp/scripts/generate.ts:6,14-16,18,159,632`). `AGENTS.md` law: "Root `codegen` wiring may invoke this package generator, but build/check must remain offline." (source: in-repo files cited)

- **runpod style = bespoke effect/Schema renderer over a checked-in spec (cleanest, no library dep).** `packages/drivers/runpod/scripts/generate.ts` (982 lines) decodes a hand-modeled OpenAPI subset with `effect/Schema` classes (`S.decodeEffect(S.fromJsonString(OpenApiDocument))`), then renders `src/_generated/Runpod.generated.ts` containing: per-component `S.Class` models, request DTO classes, an `RunpodOperationDescriptor` model + one `*.make({...})` descriptor per operation, a `RUNPOD_OPERATION_SPECS` table, and a typed `RunpodOperationsShape<E>` interface. It uses `@beep/identity` ($I composer with `$I.annote`/`$I.annoteSchema`), `@beep/schema` (`LiteralKit`), `@beep/utils` (`A`/`Str`/`Struct`/`R`/`O`/`P`), and `effect` `Match` for JSON-Schema→Schema dispatch. Source spec is a **checked-in** `openapi.json` (151 KB) — there is no fetch step. Dynamic-valued fields (`gpuType`, `dataCenterId`, ...) are emitted as advisory `const` arrays (`RUNPOD_*_VALUES`) instead of frozen literal unions, so the spec drifting doesn't break decode (`runpod/scripts/generate.ts:176-205,438-454,548-580,781-826,890-949`).

- **Both generators keep `_generated/*` package-private and run codegen as a pre-build step.** `runpod/package.json` nulls the export (`"./_generated/*": null`) and defines `"generate": "bun run scripts/generate.ts && bunx biome check --write src/_generated/Runpod.generated.ts"`; the audit chain runs `generate` before `build`/`check` (`runpod/package.json:20,32,41-43`). This is the in-repo equivalent of the us-legal-tools "codegen-as-build-step / generate before build" Turbo pattern, achieved without Turbo `generate` task wiring.

- **The hand-written transport/service layer is separate from generated models (the part codegen does NOT produce).** runpod ships `Runpod.service.ts` on `effect/unstable/http` (`HttpClient` + `FetchHttpClient` + `HttpClientRequest`), `Runpod.config.ts` (`RunpodConfigInput` with `apiKey: S.optionalKey(S.String.pipe(S.RedactedFromValue))` so the key is never logged), and `Runpod.errors.ts`. Layer entry is `Runpod.makeLayer(new RunpodConfigInput({ apiKey: Redacted.make(...) }))` (`runpod/src/Runpod.service.ts:16-21,117-120`; `runpod/src/Runpod.config.ts:60-69`; `runpod/README.md`). Takeaway: codegen emits Schema/descriptors; **auth + retry + cache + service wiring is hand-authored on top** — exactly the boundary the five gov drivers need.

- **Recommendation for the five gov drivers (grounded, not prescriptive):** acp-style `@effect/openapi-generator` is the more direct fit where a clean OpenAPI 3.x spec exists (it emits Schema + HTTP clients + HttpApi modules in one shot), with runpod's bespoke-renderer-over-checked-in-spec as the fallback when an upstream spec is partial/non-conformant. Both are MIT and already proven in `packages/drivers/`.

### B. The govinfo partial scaffold — finish, don't restart

- **govinfo is domain-only (27 src files), no transport.** `packages/drivers/govinfo/src/` = `index.ts` + `domain/{values,contracts}`. Values are `S.Class` models: `SearchResult`, `PackageInfo`, `GranuleMetadata`, `GranuleContainer`, `CollectionContainer`, `CollectionSummary`, `SummaryItem`, `SearchBody`, `Sort`, `SearchResponse` (barrel `govinfo/src/domain/values/index.ts`). The single contract `domain/contracts/Search/` has `Search.contract.ts` (`Payload extends SearchBody`, `Success extends SearchResponse` at status 200, `Failure` = `S.toTaggedUnion` of `FailureBadRequest`/`FailureNotFound`/`FailureInternalServerError` built with `@beep/schema` `TaggedErrorClass` + `HttpStatus2XX/4XX/5XX`) and `Search.http.ts` (`HttpApiEndpoint.post("search", "/search", { payload, error, success })` from `effect/unstable/httpapi`). (sources: in-repo files)

- **The scaffold is placeholder-grade, not finished.** `SearchResult.model.ts` fields carry `/** change me */` comments and empty `description: ""` annotations; `Search.contract.ts` `@since 0.0.0` blocks are `TODO`. So "finish govinfo" = (1) tighten the placeholder value-model annotations, (2) **add the missing transport layer** — client/config/auth/retry/cache — on top of the existing `Search` HttpApi contract, runpod-service-style. GovInfo auth is an `api.data.gov` `api_key` **query param** (per the cluster CAPTURE; treat the exact query shape as owned by the auth-matrix subtopic).

- **The four siblings are bare VERSION-only skeletons.** `packages/drivers/{courtlistener,ecfr,dol,federal-register}/src/index.ts` each export only `export const VERSION = "0.0.0" as const`. These are net-new builds, not extends.

### C. Licensing — both donor repos resolve to MIT (one caution overturned)

- **us-legal-tools = `beshkenadze/us-legal-tools`, MIT (confirmed via GitHub API).** `gh api repos/beshkenadze/us-legal-tools/license` → `{"license":"MIT","path":"LICENSE"}`; README badge also "License: MIT" (https://github.com/beshkenadze/us-legal-tools). Its `packages/` are exactly `courtlistener-sdk, dol-sdk, ecfr-sdk, federal-register-sdk, govinfo-sdk, orval-config, scalar-ui, tsconfig` — i.e. the same five gov/legal APIs this beep cluster targets, each shipping an MCP server (`eCFRSDKServer`, `FederalRegisterServer`, `CourtListenerRESTAPIServer`, `GovInfoServer`, dol). It is a Turborepo+Bun monorepo using **Orval** for codegen (`orval-config` package, `turbo generate`). Last pushed `2025-08-06` (≈10 months stale). **MIT permits copying, but the value here is the PIPELINE (single-spec → dual SDK+MCP target, per-spec one-line config, codegen-as-build-step), not the Orval/axios/Zod OUTPUT** — port the architecture onto `@effect/openapi-generator` + `effect/Schema` + `effect/unstable/httpapi`. (https://github.com/beshkenadze/us-legal-tools)

- **us-gov-open-data-mcp = `lzinga/us-gov-open-data-mcp`, ALSO MIT — this resolves the CAPTURE "unverified" caution.** `gh api repos/lzinga/us-gov-open-data-mcp/license` → `{"license":"MIT","path":"LICENSE"}` (https://github.com/lzinga/us-gov-open-data-mcp). It is "MCP server + TypeScript SDK for 40+ U.S. government data APIs — 250+ tools", actively maintained (pushed `2026-06-11`). **Correction to CAPTURE/cautions: the license is NOT unverified — it is MIT, so its declarative `createClient(config)` factory (auth query/header/body, disk-TTL cache, retry+backoff, token-bucket rate limiter) is legally portable/copyable with attribution.** The reimplement-don't-copy guidance still holds, but on **architecture** grounds (plain-TS class/factory → Effect `Context.Service`/`Layer` + `effect/unstable/http`), not licensing.

- **Orval `@orval/mcp` single-mode caution (only relevant if Orval were used directly).** Confirmed: the MCP client requires `mode: 'single'` and cannot combine with `split`/`tags`/`tags-split` (one spec per config) (https://orval.dev/docs/guides/mcp/, https://www.npmjs.com/package/@orval/mcp). Moot for beep since the recommendation is `@effect/openapi-generator`, not Orval.

### D. USPTO / PatentsView deprecation cliff — NOT in this repo; flag for any sibling patents driver

- **Legacy PatentsView API (`api.patentsview.org`) is dead: support ended `May 1, 2025`, now returns HTTP `410 Gone`.** The Feb-2025 date in CAPTURE was the *originally announced* cutoff; it was extended. Sunset was announced Oct 2024; replacement `PatentSearch API` shipped early 2024. (https://patentsview.org/data-in-action/patentsview-ends-support-legacy-api ; corroborated by the Feb-2025 announcement page https://patentsview.org/data-in-action/support-legacy-api-end-february-2025-switch-patentsearch-api-now). **Correction to CAPTURE: "PatentsView ended Feb 2025" → legacy API actually ended `2025-05-01` and 410s.**

- **PatentsView (`www.patentsview.org`) → USPTO Open Data Portal (`data.uspto.gov`) migration starts `March 20, 2026`.** The old patentsview.org article URLs now `301`-redirect to `https://data.uspto.gov/support/transition-guide/patentsview` (observed during this research), confirming the migration is in flight. (https://data.uspto.gov/support/transition-guide/patentsview)

- **Mandatory key reissue: previously-issued PatentSearch API keys are NOT compatible with ODP.** Users must obtain new ODP API keys via the ODP "Getting Started" page; ODP public data "will require registration to access." Effective `August 18, 2026`, users must also provide four additional fields in their USPTO.gov account settings under the "Open Data Portal" section. (https://data.uspto.gov/support/transition-guide/patentsview ; search-surfaced ODP support text)

- **Legacy USPTO Developer Hub (`developer.uspto.gov`) was decommissioned `June 5, 2026`** (matches CAPTURE), and **temporary interruptions are expected** to the PatentSearch API (previously `search.patentsview.org/api`) during the ODP transition, with no firm relaunch date. Adjacent ODP decommissions: PTAB API v2 (`Jan 6, 2026`), OCE Patent Litigation Cases (`Jan 30, 2026`), Enriched Citation API/v2 (`Jan 30, 2026`). (https://developer.uspto.gov/ ; https://data.uspto.gov/support)

- **Implication for beep:** matches the live-tree routing note that the existing `packages/drivers/uspto` (hand-rolled, no codegen) targets ODP `api.uspto.gov` and must be **ODP-only, NEVER PatentsView**. Any sibling patents driver must (1) target `data.uspto.gov` ODP, (2) require an ODP-reissued key, (3) never call `api.patentsview.org` (410) and treat `search.patentsview.org/api` as transitional/unstable.

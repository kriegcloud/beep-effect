# openapi-to-effect-sdk-mcp-codegen

Scope: how to port the single-spec â†’ dual (Effect SDK + MCP server) two-target codegen idea onto beep's runpod/acp Effect-native precedent (`openapi.json` + `scripts/generate.ts` â†’ effect/Schema + `effect/unstable/ai` MCP), instead of Orval/axios/Zod â€” covering spec sourcing/committing, effect-Schema emission, and deterministic-CI generate-before-build ordering.

## Findings

### A. The beep precedent already in-tree (port onto this, not Orval)

- **Two distinct codegen precedents exist in `packages/drivers`.** `runpod` is the "checked-in `openapi.json` + hand-rolled generator" precedent: `packages/drivers/runpod/openapi.json` (OpenAPI `3.0.3`, server `https://rest.runpod.io/v1`) is committed, and `packages/drivers/runpod/scripts/generate.ts` (982 LOC, `#!/usr/bin/env bun`) decodes it and emits `src/_generated/Runpod.generated.ts`. `acp` is the "download-upstream-release + library generator" precedent: `packages/drivers/acp/scripts/generate.ts` uses `@effect/openapi-generator/JsonSchemaGenerator` and `effect/unstable/cli` `Command`/`Flag` to fetch a pinned upstream schema release (`const CURRENT_SCHEMA_RELEASE = "v0.11.3"`) over `FetchHttpClient` and emit `src/_generated/schema.gen.ts`. (Source: local files `packages/drivers/runpod/scripts/generate.ts`, `packages/drivers/acp/scripts/generate.ts`; `runpod/openapi.json`.)
- **The generated file is internal, never hand-edited, and re-formatted post-emit.** `runpod/package.json` `exports` maps `"./_generated/*": null` (the generated module is package-private), and the `generate` script is `bun run scripts/generate.ts && bunx biome check --write src/_generated/Runpod.generated.ts` â€” i.e. generate then auto-format so the committed artifact is lint-clean and diff-stable. The generated header literally says "Do not edit this file by hand." (Source: local `packages/drivers/runpod/package.json`, `runpod/scripts/generate.ts` `renderHeader`.)
- **The runpod generator emits effect/Schema-first artifacts, not axios/Zod.** It renders: `S.Class` model classes per OpenAPI component, `S.Struct`/`S.Record`/`S.Array`/`S.Finite`/`S.NullOr` field expressions, `LiteralKit(...)` for enums, per-operation `Request` classes, a `RunpodOperationDescriptor` `S.Class`, a const descriptor per operation, a `RUNPOD_OPERATION_SPECS` table, and a typed `RunpodOperationsShape<E>` interface â€” all annotated through the repo's `$RunpodId.create(...)` identity composer + `$I.annote`/`$I.annoteSchema`. (Source: local `runpod/scripts/generate.ts` lines ~456â€“888.)
- **The generated specs feed a hand-written Effect service (the "SDK target" runtime).** `runpod/src/Runpod.service.ts` imports the generated module as `* as G` and builds the driver on `effect/unstable/http` `HttpClient`/`HttpClientRequest` + `FetchHttpClient`, with a `RunpodRawRequest` escape hatch for endpoints "ahead of the checked-in OpenAPI document." So in beep the split is: generator emits schemas + an operation-spec table; a thin hand-authored service turns that table into the typed client. (Source: local `runpod/src/Runpod.service.ts`.)

### B. Effect-native two-target architecture (replaces Orval's sdk+mcp targets)

- **SDK target = `@effect/openapi-generator` (already a repo dependency, `4.0.0-beta.91`).** Its `OpenApiGenerator.generate(spec, options)` returns generated TS source as an `Effect.Effect<string>`, and `OpenApiGenerateOptions.format` is one of `"httpclient" | "httpclient-type-only" | "httpapi"` â€” i.e. it can emit either an `HttpApiClient`-style SDK or a full `HttpApi` definition directly from an OpenAPI document. The package ships `OpenApiGenerator`, `HttpApiTransformer`, `JsonSchemaGenerator`, `OpenApiTransformer`, `OpenApiPatch`, `ParsedOperation`, and a `bin`. (Source: local `node_modules/@effect/openapi-generator/dist/OpenApiGenerator.d.ts`, `.../dist` module listing; version `node_modules/@effect/openapi-generator/package.json`.)
- **MCP target = Effect's native `effect/unstable/ai` toolkit, NOT `@modelcontextprotocol/sdk`.** The repo's existing MCP servers (`packages/drivers/nlp-mcp/src/Server.ts`, `m365-mcp/src/Server.ts`) build on `import * as McpServer from "effect/unstable/ai/McpServer"`. A toolkit is `Toolkit.make(Tool.make("Name", { success, ... }), ...)` (`effect/unstable/ai/Toolkit`, `.../Tool`), and it is served via `McpServer.layerStdio({...})`, `McpServer.layerHttp({...})`, `McpServer.toolkit(toolkit)` (a `Layer`), or `McpServer.registerToolkit(toolkit)`. (Source: local `node_modules/effect/dist/unstable/ai/McpServer.d.ts` exports `layerStdio`/`layerHttp`/`toolkit`/`registerToolkit`; `Toolkit.d.ts` `make`/`toLayer`/`of`; `packages/drivers/nlp-mcp/src/Server.ts`.) Cross-checked: Effect v4's unstable modules include `ai`, and `McpServer` "enables implementing MCP servers using Effect"; `layerStdio` "sends all log output to stderr, separating it from MCP protocol communication on stdout" (https://github.com/Effect-TS/effect-smol/blob/main/MIGRATION.md, https://deepwiki.com/Effect-TS/effect/10.1-ai-integration-architecture).
- **The dual-target win is even cleaner in Effect than in Orval.** The same `Tool`/`Toolkit` abstraction is what `@effect/ai` uses for LLM function-calling, so one generated `Toolkit` simultaneously serves an MCP server (`McpServer.toolkit`) and in-process LLM tool-calling â€” the single-sourceâ†’two-target idea from us-legal-tools, but with one shared Effect primitive instead of two unrelated Orval generators. (Source: local `effect/dist/unstable/ai/Toolkit.d.ts` jsdoc examples; https://deepwiki.com/Effect-TS/effect/10-ai-and-external-services.)
- **Both pillars are pre-1.0 beta and share the train.** `effect` and `@effect/openapi-generator` are both pinned at `4.0.0-beta.91` in this repo; Effect v4 beta launched 2026-02-18 and was still beta as of 2026-05-28, and `unstable/` modules "may receive breaking changes in minor releases" (https://effect.website/blog/effect-v4beta-launch-to-may-recap/, https://github.com/Effect-TS/effect-smol/blob/main/MIGRATION.md). Treat the generator output and the `McpServer` API as moving targets; pin exact versions.

### C. Spec sourcing & committing â€” varies sharply per driver

- **GovInfo â€” official OpenAPI, public-domain, directly committable.** Interactive OpenAPI/Swagger docs live at `https://api.govinfo.gov/docs/`; the API requires an api.data.gov key (header or query param; `DEMO_KEY` for testing). The `usgpo/api` repo's `LICENSE.md` is a US Government work under 17 U.S.C. 105 + CC0 1.0 â€” "in the public domainâ€¦ copy, modify, distributeâ€¦ even for commercial purposes, all without asking permission" â€” so its spec is freely vendorable into the repo. (Sources: https://www.govinfo.gov/developers, https://api.govinfo.gov/docs/, https://raw.githubusercontent.com/usgpo/api/main/LICENSE.md.)
- **eCFR â€” downloadable OpenAPI JSON at the docs `.json` endpoint.** Interactive Swagger docs are at `https://www.ecfr.gov/developers/documentation/api/v1`, and the machine-readable document is served at `https://www.ecfr.gov/developers/documentation/api/v1.json` (the Swagger-UI `.json` convention). eCFR is keyless. This matches the us-legal-tools precedent that committed an `./v1-openapi3.json` for ecfr. (Sources: https://www.ecfr.gov/developers/documentation/api/v1, search-surfaced spec endpoint `https://www.ecfr.gov/developers/documentation/api/v1.json`; CAPTURE nugget us-legal-tools#2.)
- **Federal Register â€” interactive docs exist; it is an unofficial prototype, keyless.** Docs at `https://www.federalregister.gov/developers/documentation/api/v1` and `https://www.federalregister.gov/reader-aids/developer-resources/rest-api`; same Swagger-UI `.json` convention is the likely committable-spec source. FedReg self-describes as an unofficial prototype (per CAPTURE caution; GovInfo is the official source for the same documents). (Sources: https://www.federalregister.gov/developers/documentation/api/v1; CAPTURE caution.)
- **CourtListener â€” NO machine-readable OpenAPI; spec must be hand-authored.** The v4 docs do not publish an OpenAPI/Swagger file or a schema endpoint; the API is "open source" with GitHub-issue-driven changes. So the SDK-codegen path for CourtListener requires either hand-authoring a small OpenAPI subset to feed the generator, or hand-writing the `HttpApi`/service directly (the runpod `RunpodRawRequest` escape-hatch pattern generalizes here). Auth is literally `Authorization: Token <token>` (the docs warn "a common mistake is to forget the word 'Token'"), base `https://www.courtlistener.com/api/rest/v4/`. (Source: https://wiki.free.law/c/courtlistener/help/api/rest/v4/overview.)
  - Note a discrepancy to resolve in align: the CAPTURE caution says "V4 ENFORCES auth (anonymous = 401)," but the official v4 overview says "To encourage experimentation, many of our APIs are open by default" and anonymous works for many endpoints (with lower rate limits). The header format (`Token`, not `Bearer`) and base URL are consistent across both; the anonymous-access claim should be re-verified per-endpoint. (Source: same overview URL.)
- **DOL â€” legacy XML-first API, no clean OpenAPI found.** The DOL API (`api.dol.gov`, V1) defaults to XML and returns JSON only when sent `Accept: application/json`; the dev community lives at `usdepartmentoflabor.github.io/DOLAPI` and a newer Open Data Portal at `data.dol.gov`/`dataportal.dol.gov` (user guide PDF dated 2024-08-05). No machine-readable OpenAPI/Swagger spec surfaced â€” like CourtListener, DOL likely needs a hand-authored spec/HttpApi. (Sources: https://developer.dol.gov/beginners-guide/, https://www.dataportal.dol.gov/pdf/dol-api-user-guide.pdf, https://usdepartmentoflabor.github.io/DOLAPI/.)
- **Committing rule that transfers from the patents-mcp-server precedent:** download upstream spec into the package (`fetch-specs`/`download` step), commit both the spec and the generated output, and treat the generated dir as a read-only dependency. Hey API's own guidance: "the output folder should be treated as a dependencyâ€¦ do not directly modify its contents" and pin an exact generator version. (Source: https://heyapi.dev/openapi-ts/migrating; CAPTURE nugget patents-mcp-server#14.)

### D. effect-Schema emission patterns (two viable shapes)

- **Option 1 â€” reuse `@effect/openapi-generator` (acp precedent).** `JsonSchemaGenerator.make()` and `OpenApiGenerator.generate(spec, { format })` already produce effect/Schema + HttpApi/HttpClient TS as a string you write to `_generated/`. This is the lowest-LOC path and is what `acp` does. Its `format: "httpapi"` emission lines up with beep's existing HttpApi usage (e.g. the partially-scaffolded `@beep/govinfo` Search HttpApi contract). (Source: local `OpenApiGenerator.d.ts`; `acp/scripts/generate.ts`.)
- **Option 2 â€” hand-rolled generator (runpod precedent).** When you want bespoke output (operation-spec tables, `LiteralKit` enums, advisory-enum constants, `$Id`-composed annotations, a typed `OperationsShape<E>`), copy runpod's ~1k-LOC `scripts/generate.ts`. It decodes the OpenAPI document itself through an effect/Schema model (`S.Class` `OpenApiDocument` + `S.decodeEffect(S.fromJsonString(...))`), then string-renders the module. Trade-off: full control vs. you own the OpenAPI-subset coverage. (Source: local `runpod/scripts/generate.ts`.)
- **Alternative library (note, don't necessarily adopt): `fortanix/openapi-to-effect`** generates effect/Schema from OpenAPI 3.1 JSON (v0.9.3, 2025-11-28), but it is **MPL-2.0** (weak copyleft) vs. the Effect org's MIT, supports a single document only, JSON-only input, and `$allOf` on object schemas only. Prefer the first-party `@effect/openapi-generator` to keep the license surface MIT and stay on the Effect release train. (Source: https://github.com/fortanix/openapi-to-effect.)

### E. Deterministic-CI generate-before-build ordering

- **Turborepo canonical pattern (matches CAPTURE us-legal-tools#10):** put codegen on a `cache: false` task and make `build` depend on it â€” `"build": { "dependsOn": ["codegen", "^build"] }`, `"codegen": { "cache": false }`. Turbo "assumes tasks are deterministic"; a generator that emits identical output for identical inputs is required for caching downstream to be sound, and you should NOT use `dependsOn: []` (it breaks invalidation). Track generated files via the build task's input globs so build re-runs when generated output changes. (Sources: https://turborepo.dev/docs/crafting-your-repository/configuring-tasks, https://turborepo.dev/docs/reference/configuration, https://turborepo.dev/docs/crafting-your-repository/caching.)
- **Where beep stands today (the gap to close):** root `turbo.json` already defines a `codegen` task with `"cache": false` and `"dependsOn": ["^codegen"]`, but the `build` task currently `dependsOn` only `["^build"]` â€” it does NOT depend on `codegen`/`generate`. Determinism is instead enforced per-package inside the audit chain: e.g. `runpod`'s `beep:audit` = `bun run generate && bun run beep:build && bun run beep:check && â€¦ && bun run beep:lint`, and the package-level `generate` script also re-formats the output. So porting the "build dependsOn generate" guarantee repo-wide is a real net-new wiring decision (either wire `build â†’ codegen` in `turbo.json`, or keep the generate-first audit convention and rely on the committed `_generated/` artifact being up to date in CI). (Source: local `turbo.json` tasks `build`/`codegen`; `runpod/package.json` scripts.)
- **CI safety net to add:** because the generated code is committed, the determinism check is "re-run generate and assert `git diff --exit-code` is clean" â€” this catches drift between the committed spec and the committed output. (Pattern inferred from runpod's commit-the-output + reformat design; the `cache:false` generate task makes re-running cheap. Not yet wired in-repo â€” see Open/Unverified.)

### F. Reference precedents â€” learn the shape, do not copy the code

- **Orval `@orval/mcp`** is the originating idea (one config â†’ `sdk` axios client + `mcp` server). Orval is MIT (Copyright 2022â€“present Victor Bury), and it generates "type-safe API clients, TanStack Query/SWR hooks, Zod schemas, MSW mocks, Hono handlers, MCP servers" from OpenAPI. **Hard constraint to remember:** the `mcp` client "currently only works in single mode" (`mode: 'single'`, one spec per config; `tags`/`split` unsupported) â€” so the Orval route is one-config-per-driver. beep does not adopt Orval (axios/Zod runtime is off-stack); it ports the architecture. (Sources: https://github.com/orval-labs/orval/blob/master/LICENSE, https://orval.dev/docs/guides/mcp/, https://orval.dev/docs/reference/configuration/output/.)
- **Hey API `@hey-api/openapi-ts`** (the patents-mcp-server precedent for OpenAPIâ†’Zod validation codegen) is MIT (reverted to MIT in commit `9cec9e8` after a licensing change), used by Vercel/PayPal/AWS, and advises pinning an exact version and treating generated output as a dependency. Same governance lessons (commit spec + output, pin version) transfer; the runtime (Zod) does not. (Sources: https://www.npmjs.com/package/@hey-api/openapi-ts, https://github.com/hey-api/hey-api, https://heyapi.dev/openapi-ts/migrating.)
- **Licensing bottom line:** us-legal-tools (Orval/axios/Zod) and Hey API are MIT, but the value is the PIPELINE/PATTERNS, not their generated runtime output; port onto Effect/HttpApi/effect-Schema. Avoid vendoring MPL-2.0 `fortanix/openapi-to-effect` source. GovInfo/eCFR/FedReg specs are US-Gov public domain and freely committable. (Sources: CAPTURE cautions; license URLs above.)

### G. Gotchas (concrete, dated)

- **`@effect/openapi-generator` `httpclient` format is lossy on multi-representation responses (open bug).** Issue #1978: "The `httpclient` format is too lossy when a single HTTP status has multiple representations" â€” it keeps only the first JSON-compatible representation, emits only streaming variants for binary success (regular method falls to `unexpectedStatus`), and flattens response metadata. Affected `@effect/openapi-generator@4.0.0-beta.43`â†’current `main`; fix in PR #1979 (consolidating #1912/#1916, issues #1911/#1915). Validate generated SDKs against any endpoint that returns both `application/problem+json` and `application/json` on the same status. (Source: https://github.com/Effect-TS/effect-smol/issues/1978.)
- **MCP stdio purity:** anything written to stdout that isn't JSON-RPC corrupts the stream â€” `McpServer.layerStdio` deliberately routes logs to stderr. Generated MCP handlers must never `console.log` to stdout. (Source: https://github.com/Effect-TS/effect-smol/blob/main/MIGRATION.md.)
- **`unstable/` API churn:** `effect/unstable/ai/McpServer`, `effect/unstable/cli`, and `effect/unstable/http` can break in minor releases â€” the acp generator already pins both the upstream schema release (`v0.11.3`) and the effect version. (Source: https://github.com/Effect-TS/effect-smol/blob/main/MIGRATION.md.)
- **Spec-coverage drift:** a checked-in `openapi.json` lags the live API; runpod handles this with a `RunpodRawRequest` escape hatch for endpoints "ahead of the checked-in OpenAPI document." Plan an equivalent raw-request path per driver. (Source: local `runpod/src/Runpod.service.ts`.)
- **DOL XML default:** DOL V1 returns XML unless the request sends `Accept: application/json` â€” codegen/clients must set that header, and the spec (if hand-authored) should model JSON. (Source: https://developer.dol.gov/beginners-guide/.)
- **CourtListener auth wording:** must be `Authorization: Token <token>` (DRF TokenAuthentication), not `Bearer`; the docs flag forgetting the literal word "Token" as the most common mistake. (Source: https://wiki.free.law/c/courtlistener/help/api/rest/v4/overview.)

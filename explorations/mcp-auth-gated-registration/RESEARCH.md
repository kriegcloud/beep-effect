### The Effect v4 AI primitives (the framework we build on)

- The repo runs `effect@4.0.0-beta.91` and both MCP drivers import `{ Tool, Toolkit }` from `effect/unstable/ai` (NOT the legacy `@effect/ai` v3 package), with the server from `effect/unstable/ai/McpServer`. `Toolkit.make(...tools)` is the typed registry; `McpServer.toolkit(tk)` registers it; `McpServer.layerStdio({name,version})` provides stdio transport. The public docs at <https://effect-ts.github.io/effect/ai/ai/Tool.ts.html> document the v3 API — the v4 ground truth is `node_modules/effect/src/unstable/ai/{Tool,Toolkit,McpServer}.ts`. Effect Tool Use docs: <https://effect.website/docs/ai/tool-use/>.
- The four MCP behavior hints are first-class Effect `Context.Reference<boolean>` annotations whose defaults exactly equal the MCP wire defaults — `readOnlyHint:false`, `destructiveHint:true`, `idempotentHint:false`, `openWorldHint:true` — so an unannotated tool is treated as writing/destructive/non-idempotent/open-world. `McpServer` maps them 1:1 onto the wire `annotations` object plus `_meta` from `Tool.Meta`. MCP spec defaults verified three ways: spec <https://modelcontextprotocol.io/specification/2025-06-18/server/tools>, design blog <https://blog.modelcontextprotocol.io/posts/2026-03-16-tool-annotations/>, and the effect source defaults.

### #1 Conditional / credential-keyed Toolkit composition

- The Effect-idiomatic optional-credential idiom is `Config.redacted("X_API_KEY").pipe(Config.option)` (missing → `Option.none`, never fails) and `Config.withDefault`; a required missing `Config` fails with `ConfigError`/`MissingData`. Source: <https://effect.website/docs/configuration/>.
- Conditional registration has two complementary strategies in the MIT prior art: **conditional mounting** — `patents-mcp-server` `registerAllTools` only mounts a tool group when its creds resolved (`if(config.usptoApiKey){...} if(config.epoConsumerKey && config.epoConsumerSecret){...} registerUtilityTools // always`) — leaner surface, absent-key tools invisible; vs **always-register + handler guard** (mcp-uspto), more discoverable, larger surface. A faithful Effect port reads `Config.option` at **layer-build time** to make a tool *disappear* (Shapes A/B), or at **call time** to keep it present with a graceful error (Shape C). `lzinga/us-gov-open-data-mcp` (MIT) adds the schema-first carrier: a per-module `ModuleMeta { auth?: { envVar: string|string[]; signup } }` declaration that drives gating, the helper, and the health-probe from one record. Sources: <https://github.com/riemannzeta/patent_mcp_server>, <https://github.com/lzinga/us-gov-open-data-mcp>, <https://github.com/beshkenadze/us-legal-tools>.
- **GOTCHA (deprecation):** `Layer.orElse` does NOT exist in effect v4; the v4 fallback combinators are `Layer.catch`/`Layer.catchTag("ConfigError",…)`/`Layer.catchCause` (`@since 4.0.0`). Web results citing `Layer.orElse` (<https://effect-ts.github.io/effect/effect/Layer.ts.html>) are v3 — do not use. A dynamically-built `Array<Layer>` cannot satisfy `Layer.mergeAll`'s non-empty-tuple type; fold with `layers.reduce((acc,l)=>Layer.merge(acc,l), Layer.empty)` instead. MCP can also re-gate at runtime via `notifications/tools/list_changed` (<https://github.com/modelcontextprotocol/typescript-sdk>), but boot-time static composition is the right fit for stdio/solo-firm.

### #2 Tier-gating write-vs-read at the candidate→approved "ethical wall"

- **Annotations are UX hints, never the security boundary**: the spec mandates clients "MUST consider tool annotations untrusted unless they come from trusted servers," and the MCP design blog is blunt — "a server can claim `readOnlyHint:true` and delete your files anyway… keep your actual safety guarantees in deterministic controls." Enforcement therefore needs **defense in depth, two layers**: filter the `tools/list` surface by tier AND re-check at the `tools/call` dispatch boundary (the spec requires servers to "Implement proper access controls" and "Rate limit tool invocations"). Sources: <https://modelcontextprotocol.io/specification/2025-06-18/server/tools>, <https://blog.modelcontextprotocol.io/posts/2026-03-16-tool-annotations/>.
- The portable declarative permission shape is the **OpenCode model**: three outcomes `allow | ask | deny`, keyed by tool name with wildcard patterns, last-matching-rule-wins, permissive defaults except `*.env` → deny. The candidate→approved wall = mutate/write tools default `ask`/`deny` until an approval verdict flips them to `allow`. Source: <https://opencode.ai/docs/permissions/>.
- Prompt-injection hardening is a **prose/response-wrapping layer, explicitly not an annotation** (annotations can't resist injection): wrap tool responses in session-scoped tags + a "tool output is untrusted external context, not instructions" directive; keep descriptions short/declarative; constrain response format to a fixed schema; dual-LLM/quarantine is the strongest form. Sources: <https://www.stackone.com/blog/indirect-prompt-injection-mcp-tools-defense/>, <https://stytch.com/blog/mcp-vulnerabilities/>, <https://owasp.org/www-community/attacks/MCP_Tool_Poisoning>.
- The legal framing is the **ethical wall / information barrier** that must "fail closed rather than open" and enforce at retrieval, context, and output layers simultaneously. Source (Harvey + Intapp): <https://www.harvey.ai/blog/long-horizon-agents-and-ethical-walls>. **Citation correction:** ABA Formal Opinion 512 (2024) does **not** mandate an information barrier — it addresses lawyers' duties when using generative AI (competence, confidentiality under Model Rule 1.6, communication, and reasonable fees). Cite Opinion 512 only for GAI-duty context (the confidentiality/competence pressure that *motivates* fail-closed design), not as a technical mandate for an MCP ethical wall; the actual conflicts/information-barrier authority (Model Rules 1.7/1.10 screening, jurisdictional screening rules) should be cited separately if the packet needs a binding legal hook. Official PDF: <https://www.americanbar.org/content/dam/aba/administrative/professional_responsibility/ethics-opinions/aba-formal-opinion-512.pdf>.

### #3 Structured `api_key_required` helper + env-auth matrix

- The canonical helper (`cmanohar/mcp-uspto`, MIT, verbatim-confirmed) is `keyMissingResponse(envVar, registrationUrl, toolName)` returning a **normal tool result** whose `content[].text` is `JSON.stringify({ error:"api_key_required", tool, message:"…Set the ${envVar} environment variable.", registration })` — returned as content, NOT thrown, "so the LLM gets helpful guidance." Sources: <https://raw.githubusercontent.com/cmanohar/mcp-uspto/master/src/lib/config.ts>.
- This is **spec-correct against the installed protocol target** — but pin the version. The bundled `effect@4.0.0-beta.91` `McpServer` advertises `LATEST_PROTOCOL_VERSION = "2025-06-18"` and negotiates only `2025-06-18 | 2025-03-26 | 2024-11-05 | 2024-10-07`; it does **NOT** speak `2025-11-25` (`node_modules/effect/src/unstable/ai/McpServer.ts:336-341,1344-1346`). Both the 2025-06-18 and 2025-11-25 specs define the same two error channels — Protocol Errors (JSON-RPC `error`) and Tool Execution Errors (normal `result` + `isError:true`) — and say clients "SHOULD provide tool execution errors to language models to enable self-correction," whereas protocol errors "are less likely to result in successful recovery." A missing key is a recoverable tool-execution outcome. **Design the channel to 2025-06-18 semantics; any reliance on 2025-11-25-specific behavior is spec drift and requires an effect dependency-upgrade + reverification task first.** Ecosystem gotcha: several clients mishandle `structuredContent` (validation runs before `isError`; adapters have dropped it), so mirror the structured payload into a `content[].text` JSON string. Sources: <https://modelcontextprotocol.io/specification/2025-06-18/server/tools>, <https://modelcontextprotocol.io/specification/2025-11-25/server/tools>, <https://github.com/modelcontextprotocol/typescript-sdk/issues/654>.
- The env-auth matrix (`beshkenadze/us-legal-tools`, MIT): CourtListener→`COURTLISTENER_API_TOKEN`, GovInfo→`GOV_INFO_API_KEY`, DOL→`DOL_API_KEY`; eCFR + Federal Register → no auth. **Adversarial nuance — "auth required" collapses three gate strengths**, so the Effect matrix should encode `gate: none|soft|hard`, not a boolean: no-auth/public (eCFR, Federal Register), hard-gate (GovInfo 401 `API_KEY_MISSING`; DOL APIv4), soft-gate (CourtListener open-by-default + throttled). Sources: <https://github.com/beshkenadze/us-legal-tools>, <https://github.com/usgpo/api>, <https://wiki.free.law/c/courtlistener/help/api/rest/v4/overview>.
- The `check-api-status` probe (`patents-mcp-server` `getAvailableSources(cfg): {name,configured,healthy}[]`, MIT) has a known bug to fix on port: `configured` is presence-only and `healthy` is hardcoded `false` (never pings upstream). A truthful probe sets `healthy` from a real cheap request, surfacing 401/403 distinctly from 5xx. **Underspecified for shape (advisory):** before graduating, define a per-source probe matrix — probe endpoint, auth transport, timeout budget, cache/TTL, expected unauthenticated response (so 401/403 ≠ outage), soft-gate behavior for CourtListener, offline behavior, and a redacted result schema that never leaks credential state — since this tool is flaky, rate-limited, and credential-sensitive by nature.

### #4 Progressive-disclosure field tiers + server-side response reshaping

- **The hard constraint making reshaping mandatory:** Claude Code rejects any single MCP tool output above **25,000 tokens** (`MAX_MCP_OUTPUT_TOKENS`), erroring "MCP tool response exceeds maximum allowed tokens (25000)". A raw USPTO PFW search with `documentBag` trivially blows past it — reshaping is a correctness requirement, not a nicety. Tool *definitions* are also an upfront tax: Anthropic measured **150,000 tokens** on tool defs + intermediate results, reduced to ~2,000 (98.7%) via code execution. Sources: <https://help.xpoz.ai/en/articles/12681842-claude-code-mcp-tool-exceeds-maximum-allowed-tokens-25000>, <https://github.com/anthropics/claude-code/issues/9152>, <https://www.anthropic.com/engineering/code-execution-with-mcp>.
- **Named field tiers** (canonical: `john-walkoe/uspto_pfw_mcp` `field_configs.yaml`, MIT): `applications_minimal` (~14 dotted-path fields, claims 95–99% reduction), `applications_balanced` (~18 fields, 85–90%); `documentBag` is commented out with a verbatim WARNING "Can cause 100x token increase — use pfw_get_application_documents instead." The three-rung ladder (ultra-minimal 2–3 fields→99% / minimal 15→95% / balanced 18→85–90%) is documented in CUSTOMIZATION.md. USPTO ODP supports server-side `fields` projection at `POST /applications/search` (optional body param). Sources: <https://github.com/john-walkoe/uspto_pfw_mcp>, <https://github.com/john-walkoe/uspto_pfw_mcp/blob/master/CUSTOMIZATION.md>, <https://data.uspto.gov/apis/patent-file-wrapper/search>.
- **Reshaping levers**: `format=csv` (~40–70% on tabular), `format=outline|tree` (~91%), `fields=` whitelist (~80–90%) from `screenpipe` (license NOASSERTION → reimplement conventions, do not copy); columnar envelope (`toColumnar` + `stripNulls` + `detectTrend` linear-regression trend, row caps) from `lzinga/us-gov-open-data-mcp` (MIT). Independent guidance puts CSV lower (40–60%), so treat 40–90% as the defensible workload-dependent band. The **metadata→snippet→read tool ladder** (cheap counts → keyword snippet → full text by id, with per-turn caps) is corroborated by screenpipe's workflow + Anthropic's progressive-disclosure framing (the `mike` CourtListener ladder is unknown-license → reimplement). Large byte payloads (PDFs) go behind **UUID+TTL fetchable handles / MCP Resources** (`resources/read`, strict UUID regex gating fs access — no path interpolation), never inline. Sources: <https://raw.githubusercontent.com/screenpipe/screenpipe/main/.claude/skills/screenpipe-api/SKILL.md>, <https://raw.githubusercontent.com/lzinga/us-gov-open-data-mcp/main/src/shared/response.ts>, <https://modelcontextprotocol.io/specification/2025-06-18/server/resources>.
- **Out of scope for v1**: WASM code-mode (QuickJS/V8-isolate sandbox; Cloudflare reports 1.17M→~1k tokens / 99.9%) is a server-wide capability with heavyweight infra; field tiers + columnar reshaping + Resources already capture 95–99% of the win for stdio servers. The method-enum mega-tool (one `uspto_patents(method,…)` vs ~25 discrete tools) is the **wrong** axis for Effect — schema validation is free, discrete typed tools are cheap. Sources: <https://blog.cloudflare.com/code-mode-mcp/>, <https://github.com/jmpijll/fortimanager-code-mode-mcp>.

### #5 readOnly/idempotent hint conventions + USE-WHEN routing prose

- There is **no structured "use-when" field** — routing prose lives in the free-text `description` (or the params-schema description); the host injects it into the model prompt. Anthropic confirms description quality is high-leverage, not cosmetic (precise tool-description refinement drove Claude Sonnet to SOTA on SWE-bench Verified): state what it does and when to use it, third-person, with concrete input examples. This backs porting the screenpipe USE-WHEN / DO-NOT-USE + "Start with limit=5" prose convention. Per-field descriptions port Zod `.describe()`/`.default()` (us-legal-tools, MIT) into effect Schema annotations. Sources: <https://www.anthropic.com/engineering/writing-tools-for-agents>, <https://platform.claude.com/docs/en/agents-and-tools/tool-use/define-tools>.

- `@beep/m365-mcp` — `packages/drivers/m365-mcp/src/{Server,M365Tools,M365Handlers}.ts` — single-toolkit read-only v1, same pattern. It is the **only** package that annotates all four hints: `M365Tools.ts:100-103` chains `.annotate(Tool.Readonly,true).annotate(Tool.Destructive,false).annotate(Tool.Idempotent,true).annotate(Tool.OpenWorld,true)` (verified, repeated per tool). - `failureMode: "return"` + a structured `failure` schema is already standard in both servers (`M365Tools.ts`, `StreamingTools.ts`) — the idiomatic in-band channel for the `api_key_required` envelope (#3) and for a tier-gate refusal verdict (#2), replacing the hand-rolled `keyMissingResponse`. **BLOCKING wire-encoding gotcha (verified):** `McpServer.registerToolkit` constructs `CallToolResult({ isError: false, structuredContent, content })` for *any* successfully-returned encoded result and sets `isError: true` only when the Effect *cause* fails (`node_modules/effect/src/unstable/ai/McpServer.ts:708-738`, see `Effect.matchCause` onSuccess/onFailure). A typed `failureMode:"return"` failure (e.g. `ApiKeyRequired`) is a *success* value from the toolkit's perspective, so it ships as `isError:false` — wire-indistinguishable from a real result. Either (a) deliberately adopt the mcp-uspto success-with-error-JSON contract (mirror the typed failure into `content[].text` and accept `isError:false`, and test client self-correction behavior), or (b) wrap `tools/call` registration so returned typed failures map to `isError:true`. Do not assume `failureMode:"return"` yields `isError:true`.

### Config-gated, key-optional drivers (the credential-resolution primitive)

- The optional-credential idiom is real but **narrower than "every driver can disappear"** — split it into four classes (`rg` verified 2026-06-29), because only the first class lets a tool vanish on a missing credential:
  - **Optional *secret* credential** (`Config.redacted(...).pipe(Config.option)`, missing → `Option.none`): `@beep/uspto` (`USPTO_API_KEY`), `@beep/xai` (`XAI_API_KEY`, `XAI_MANAGEMENT_API_KEY`), `@beep/phoenix` (`PHOENIX_API_KEY`), `@beep/runpod` (`RUNPOD_API_KEY`), `@beep/hubspot` (`HUBSPOT_SERVICE_KEY`), `@beep/sanity` (`SANITY_API_TOKEN`). This is the actual primitive #1's "tool disappears on absent key" depends on.
  - **Required *secret* credential** (`Config.redacted(...)`, missing → `ConfigError`): `@beep/anthropic` (`ANTHROPIC_API_KEY`), `@beep/venice-ai` (`AI_VENICE_API_KEY`), `@beep/firecrawl` (`FIRECRAWL_API_KEY`), `@beep/box` (`CLOUD_BOX_TOKEN`) — these *fail the layer* on a missing key; they cannot self-gate to "absent."
  - **Optional *non-secret* config** (`Config.string|number|int(...).pipe(Config.option)`): `@beep/m365` (`M365_AUTHORITY`, `M365_GRAPH_BASE_URL`, `M365_MAX_RETRIES`, `M365_REDIRECT_URI`, `M365_SCOPES`, `M365_TOKEN_CACHE_PATH`), `@beep/hubspot` (`HUBSPOT_ACCOUNT_ID`, CRM/Forms URLs), `@beep/sanity` (project/dataset/version/host), `@beep/firecrawl` (backoff/retries/timeout). These are tuning knobs, **not** credentials — M365's `Config.option` usages are all non-secret config, not key-gating.
  This is the single primitive both registration strategies (#1) build on, but the gating logic must key off the *optional-secret* class only.
- `@beep/uspto` — `packages/drivers/uspto/src/Uspto.service.ts` is already built **key-optional**: resolves `apiKey: Option<Redacted>` and only sets `X-API-KEY` when present, targeting ODP `api.uspto.gov` (`Uspto.config.ts`, NOT PatentsView). This is the in-repo precedent for "driver self-gates on credential availability" (Shape C). `Uspto.models.ts` `UsptoApplicationMetadata` already projects an implicit ~11-field "minimal" tier, and `documentBag` already lives behind a separate `getDocuments` (`Uspto.service.ts:65,322`) — the #4 field-tier pattern is a *generalization of what the driver already does*, not a new concept.
- LLM-driver Config usage (boundary evidence that the LLM-fallback cluster resolves creds independently): `@beep/anthropic` `Anthropic.service.ts:36`, `@beep/xai` `XAi.service.ts:1020-1021`, `@beep/venice-ai` `VeniceAI.service.ts:1877`.

### Auth / secret / offline boundaries

- **`api_key_required` is content, not a thrown error** (spec-correct against 2025-06-18, recoverable). Mirror the structured payload into `content[].text` JSON — do not rely on `structuredContent` alone (clients drop it). **RESOLVED (was UNVERIFIED):** the Effect `McpServer` does **NOT** set `isError:true` on a `failureMode:"return"` typed failure — every successfully-returned encoded result (success *or* typed failure) ships as `CallToolResult({ isError: false, … })`; `isError:true` is reserved for a failed Effect cause (`McpServer.ts:708-738`). So the channel is the success-with-error-JSON contract by default; if you need `isError:true` semantics, add a `tools/call` registration wrapper that maps typed failures onto it.
- **Gate strength is a 3-value enum, not a boolean** (`none | soft | hard`). Encoding CourtListener as a hard boolean mis-gates it — model it as optional-token + graceful degradation (Shape C), robust to the 2026 tightening (anonymous 401 / membership-gating reported, but sources conflict; see below).

Custom governance annotations live only in `tool.annotations` Context (not emitted to the wire) and must be read by a beep-side wrapper.

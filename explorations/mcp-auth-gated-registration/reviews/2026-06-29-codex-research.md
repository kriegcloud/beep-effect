# Codex research-gate critique — mcp-auth-gated-registration (2026-06-29)

## Blocking
1. Claim: `RESEARCH.md:40` says the structured `api_key_required` plan is "spec-correct" against MCP "rev 2025-11-25." -> Problem: the installed Effect MCP server in this repo does not advertise that protocol version. `node_modules/effect/src/unstable/ai/McpServer.ts` sets `LATEST_PROTOCOL_VERSION = "2025-06-18"` and supports only `2025-06-18`, `2025-03-26`, `2024-11-05`, and `2024-10-07`. Citing the newer MCP spec as the implementation contract creates spec drift for any behavior that depends on 2025-11-25 semantics. -> Required fix: pin the research recommendations to the installed `effect@4.0.0-beta.91` MCP protocol target, or add an explicit dependency-upgrade/reverification task before relying on 2025-11-25 behavior.
2. Claim: `RESEARCH.md:63` says `failureMode: "return"` plus a structured failure schema is the idiomatic in-band channel for `api_key_required` and tier-gate refusal; `RESEARCH.md:113` leaves `isError:true` encoding as "UNVERIFIED." -> Problem: local source answers this, and the answer is unfavorable for the stated spec channel. `Toolkit.handle` tracks `isFailure`, but `McpServer.registerToolkit` discards that flag and constructs `CallToolResult({ isError: false, structuredContent, content })` for any returned encoded result; `isError:true` is used only for a failed Effect cause. So a typed `failureMode:"return"` `ApiKeyRequired` will look like a successful MCP tool result unless wrapped. -> Required fix: update the research to require a custom MCP call wrapper / `addTool` registration that maps returned typed failures to `isError:true`, or explicitly accept the mcp-uspto-style success-with-error-JSON contract and test client behavior.
3. Claim: `RESEARCH.md:131` says no public hook was found for filtering emitted `tools/list` by a runtime predicate. -> Problem: Effect already has a partial list-filtering hook: `McpSchema.EnabledWhen` is an annotation read by `McpServer`'s `filterByClient`, and `tools/list` returns only items whose predicate passes for the initialized client. This is a missed existing approach. It does not solve the security boundary by itself because `callTool` looks up the handler by name and does not re-check `EnabledWhen`. -> Required fix: document `EnabledWhen` as the list-surface filter, state its client-initialization limits, and pair it with the separate `tools/call` authorization wrapper required by `RESEARCH.md:32`.
4. Claim: `RESEARCH.md:116` says span hygiene is "already enforced (counts/sizes/paths only, never raw content/tokens)." -> Problem: that is false for the generic Effect toolkit path. `node_modules/effect/src/unstable/ai/Toolkit.ts` annotates every tool span with the full `parameters: params`, and `@beep/nlp` tools such as `Analyze`, `Tokenize`, `TransformText`, `ExtractEntities`, and `WordCount` accept raw `text` parameters. The streaming and M365 handlers add count/path/size annotations, but the upstream toolkit annotation can still capture raw user content. -> Required fix: add an explicit telemetry/redaction risk, design a sanitized wrapper or disable full-parameter span annotations for MCP calls, and add a proof test before claiming audit/span hygiene.
5. Claim: `RESEARCH.md:74` says `Activity` plus `UsageRecord` form the audit sink and "no new audit table is needed." -> Problem: verified paths show only half of that persistence exists. `packages/epistemic/domain/src/entities/Activity/Activity.model.ts` exports the `Activity` model, but `packages/epistemic/tables/src` exports only `UsageRecord`, `DbSchema` contains only `usageRecord`, and the only epistemic usage migration creates `epistemic_usage_record`. There is no `Activity` table/converter in `@beep/epistemic-tables`, while `UsageRecord` contains an `activity_id` column. -> Required fix: either add/reuse persisted `Activity` support as part of this work, or revise the design to store the tool-invocation snapshot in `UsageRecord.metadata` and stop promising one persisted `Activity` per gated call.
6. Claim: `RESEARCH.md:121` says "Two MCP servers already exist — do NOT scaffold a third" and that these patterns layer onto `@beep/nlp-mcp` and `@beep/m365-mcp`; `RESEARCH.md:83` then names CourtListener, eCFR, DOL, Federal Register, GovInfo, USPTO, and EPO as the consumers. -> Problem: filesystem verification shows only `packages/drivers/nlp-mcp` and `packages/drivers/m365-mcp` are MCP packages, and `rg "McpServer|Tool.make|Toolkit.make"` finds no MCP toolkit in `packages/drivers/uspto`, `courtlistener`, `govinfo`, `dol`, `ecfr`, or `federal-register`. The legal/government data tools have no named MCP surface if a third server is forbidden; the supporting research even mentions a "new gov-legal MCP makeServerLayer," contradicting the main synthesis. -> Required fix: choose the concrete target surface before decompose: extend an existing MCP server, create a gov-legal/USPTO MCP package, or keep this packet as reusable patterns consumed by later driver-specific MCP work.

## Advisory
1. Claim: `RESEARCH.md:67` says the canonical optional-credential idiom `Config.redacted(...).pipe(Config.option)` / `Config.withDefault` is used across 14 driver files, listing `@beep/anthropic`, `@beep/venice-ai`, `@beep/firecrawl`, and others. -> Concern: the optional secret precedent is real but narrower. `rg` shows optional redacted secrets in `sanity`, `uspto`, `hubspot`, `phoenix`, `xai`, and `runpod` plus runpod tests; `anthropic`, `venice-ai`, `firecrawl`, and `box` use required redacted credentials, while M365's `Config.option` examples are non-secret config fields. -> Suggested fix: split the inventory into optional secret credentials, required secret credentials, optional non-secret config, and URL/default config so implementation does not assume every driver can disappear on missing credentials.
2. Claim: `RESEARCH.md:115` says custom governance annotations live only in `tool.annotations` and are not emitted to the wire; `RESEARCH.md:132` expects arbitrary custom annotations to be dropped. -> Concern: arbitrary custom `Context.Reference` annotations do appear internal-only, but Effect also has `Tool.Meta`, which `McpServer.registerToolkit` emits as MCP `_meta`, and `Tool.make` has a `needsApproval` option. I found no `McpServer` enforcement for `needsApproval`, so this is not a ready-made gate, but it is relevant prior art the main synthesis omits. -> Suggested fix: distinguish three channels: internal enforcement metadata, host-visible advisory metadata via `Tool.Meta`, and `needsApproval` as an Effect API to evaluate or deliberately reject.
3. Claim: `RESEARCH.md:35` says ABA Formal Opinion 512 (2024) "requires the barrier but prescribes no technique." -> Concern: that overstates the legal citation. ABA Formal Opinion 512 is about lawyers' duties when using generative AI, especially competence, confidentiality, communication, and fees; it is not itself a technical mandate for an MCP ethical wall (official PDF: <https://www.americanbar.org/content/dam/aba/administrative/professional_responsibility/ethics-opinions/aba-formal-opinion-512.pdf>). The fail-closed ethical-wall framing may still be sound, but it needs the right authority. -> Suggested fix: rewrite this as a confidentiality/conflicts risk argument, cite Formal Opinion 512 only for GAI duty context, and cite the relevant conflicts/information-barrier authorities separately if the packet needs a legal mandate.
4. Claim: `RESEARCH.md:106` says MIT sources make the port path unblocked, including `uspto-patents-mcp`, `patents-mcp`, and related structural patterns; `RESEARCH.md:109` separately says the TypeScript `patents-mcp-server` LICENSE was not independently re-resolved. -> Concern: the licensing section is directionally cautious but internally too broad. The register-all, availability-matrix, and UUID+TTL bodies are still tied to at least one unresolved TS repo identity/license, so "clean to port" should not cover those bodies. -> Suggested fix: separate verified-MIT repos from CAPTURE-derived or unresolved-license repos, and mark any unresolved bodies as reimplement-only until the exact repository and LICENSE are confirmed.
5. Claim: `RESEARCH.md:42` says a truthful `check-api-status` probe should set `healthy` from a real cheap request and distinguish 401/403 from 5xx. -> Concern: the implementation shape is underspecified for a tool that can be flaky, rate-limited, or credential-sensitive. It does not define per-source probe endpoints, timeout budgets, cache/TTL, soft-gate behavior for CourtListener, offline behavior, or how much detail can be safely returned without leaking credential state. -> Suggested fix: add a per-source probe matrix with endpoint, auth transport, timeout, cache TTL, expected unauthenticated response, and redacted result schema before graduating.

# structured-api-key-required-helper-and-env-auth-matrix

Scope: the shared MCP auth-gating helper that returns a structured `{error:api_key_required, tool, envVar, registrationUrl}` content block instead of throwing, the per-source env-auth matrix convention (none / `COURTLISTENER_API_TOKEN` / `GOV_INFO_API_KEY` / `DOL_API_KEY`), a `check-api-status` health-probe tool, and the Effect-typed boundary that cross-links — but does not absorb — the separate multi-provider LLM dispatch/fallback cluster.

## Findings

### The canonical helper (mcp-uspto `keyMissingResponse`) — verbatim, MIT-licensed

- The canonical source is verbatim-confirmed. `keyMissingResponse(envVar, registrationUrl, toolName)` returns a normal MCP tool result `{ content: [{ type: "text", text: JSON.stringify({ error: "api_key_required", tool: toolName, message: \`This tool requires an API key. Set the ${envVar} environment variable.\`, registration: registrationUrl }, null, 2) }] }` — note the field is `registration` (the URL), and the response is *not* an error result (no `isError`). Source comment: "This is returned as content (not thrown), so the LLM gets helpful guidance." (https://raw.githubusercontent.com/cmanohar/mcp-uspto/master/src/lib/config.ts) — identical on both `master` and `main` branches (https://raw.githubusercontent.com/cmanohar/mcp-uspto/main/src/lib/config.ts).
- In the SAME file, key resolution is presence-only memoized env reads, NOT a typed boundary: `getConfig()` caches `{ odpApiKey: process.env.USPTO_API_KEY ?? null, patentsviewApiKey: process.env.USPTO_PATENTSVIEW_API_KEY ?? null, tsdrApiKey: process.env.USPTO_TSDR_API_KEY ?? null }`. So mcp-uspto's real env var for ODP is `USPTO_API_KEY` (free key from https://data.uspto.gov/apis/getting-started), with separate `USPTO_PATENTSVIEW_API_KEY` / `USPTO_TSDR_API_KEY` — these are USPTO-specific and distinct from the CourtListener/GovInfo/DOL matrix this packet targets (https://raw.githubusercontent.com/cmanohar/mcp-uspto/master/src/lib/config.ts; https://data.uspto.gov/apis/getting-started).
- The in-repo gold catalog confirms the snippet and pins `mcp-uspto` license = **MIT** (portable: reimplement-or-port both OK) (`explorations/_gold-intake/research/gold-catalog.json`, nugget `mcp-uspto#2`).

### Why "content, not throw" is spec-correct (MCP 2025-11-25)

- The current MCP spec (rev 2025-11-25) defines TWO error channels: **Protocol Errors** (JSON-RPC `error` object — unknown tool, malformed request, server error) and **Tool Execution Errors** (a normal `result` with `content` + `isError: true`). The spec lists "API failures" and "Input validation errors" explicitly as Tool Execution Errors, and states clients "**SHOULD** provide tool execution errors to language models to enable self-correction," whereas protocol errors "are less likely to result in successful recovery" (https://modelcontextprotocol.io/specification/2025-11-25/server/tools#error-handling).
- A missing-API-key is therefore correctly modeled as a tool-execution outcome (recoverable: the model/user can set the env var), NOT a thrown protocol error. mcp-uspto goes one step further than the spec minimum and returns the guidance **without** `isError:true` (a plain success result whose JSON text body carries `error:"api_key_required"`). Tradeoff: omitting `isError` means clients won't flag it as a failure; the structured `error` field inside the text is the only signal. A stricter design sets `isError:true` AND carries the structured payload so both code paths and LLMs see it (https://raw.githubusercontent.com/cmanohar/mcp-uspto/master/src/lib/config.ts vs https://modelcontextprotocol.io/specification/2025-11-25/server/tools#error-handling).
- Ecosystem gotcha: several clients mishandle the structured channel — `structuredContent` validation can run *before* the `isError` check and block error reporting, and adapters (langchain-mcp-adapters, MS agent-framework) have shipped bugs dropping `structuredContent` entirely. So the safest portable shape is to mirror the structured payload into a `content[].text` JSON string (exactly what `keyMissingResponse` does) rather than rely on `structuredContent` alone (https://github.com/modelcontextprotocol/typescript-sdk/issues/654).

### The env-auth matrix (us-legal-tools, MIT) — verbatim

- The matrix convention comes from `beshkenadze/us-legal-tools` (MIT). Verbatim README matrix: CourtListener → `COURTLISTENER_API_TOKEN` (auth **Yes**); GovInfo → `GOV_INFO_API_KEY` (auth **Yes**); DOL → `DOL_API_KEY` (auth **Yes**); eCFR → **None**; Federal Register → **None** ("No authentication required - these are public APIs"). The `mcpServers` config registers each source as a separate `bunx @us-legal-tools/<src>-sdk/mcp` command with its env key (https://github.com/beshkenadze/us-legal-tools; `explorations/_gold-intake/research/gold-catalog.json` nugget `us-legal-tools#9`, license MIT).
- ADVERSARIAL NUANCE — "auth required" in a matrix is not one thing; it collapses three distinct gate strengths:
  - **No-auth / public**: eCFR, Federal Register — work with zero credentials (https://github.com/beshkenadze/us-legal-tools).
  - **Hard-gate (401 without key)**: GovInfo genuinely rejects keyless requests — "users will receive a 401 error stating `API_KEY_MISSING`" (https://github.com/usgpo/api). DOL APIv4 likewise requires the key.
  - **Soft-gate (open-by-default, throttled)**: CourtListener's REST API is **open by default** — "To encourage experimentation, many of our APIs are open by default" and warns you when you call unauthenticated; the token is *recommended for production*, not a hard 401 for most endpoints ("Some of our APIs are only available to select users") (https://wiki.free.law/c/courtlistener/help/api/rest/v4/overview). So us-legal-tools marking CourtListener `auth: Yes` is an SDK *policy* choice (token-or-throttled), not an API hard-requirement. A faithful Effect matrix should encode gate *strength* (`none | soft | hard`), not a boolean, or it will mis-gate CourtListener tools as unavailable when they would actually run unauthenticated.

### Per-source credential transport (for the driver layer, second-source verified)

- **CourtListener**: `Authorization: Token <token>` HTTP header (the literal word `Token`, a space, then the key) — env `COURTLISTENER_API_TOKEN`; open-by-default, throttled when anonymous (https://wiki.free.law/c/courtlistener/help/api/rest/v4/overview; https://www.courtlistener.com/help/api/rest/).
- **GovInfo**: uses an **api.data.gov** key (not a GovInfo-proprietary key). Pass it as `?api_key=<key>` query param, or via api.data.gov's standard transports: `X-Api-Key` header, `api_key` query param, or HTTP Basic (key as username, empty password). Signup: https://www.govinfo.gov/api-signup. Keyless → `401 API_KEY_MISSING` (https://github.com/usgpo/api; https://api.data.gov/docs/developer-manual/; https://api.govinfo.gov/docs/).
- **DOL — version split (deprecation flag)**: the LEGACY `api.dol.gov/V1` API passes the key as a `KEY` query-string param (`?KEY=<token>`) per the old developer docs (http://usdepartmentoflabor.github.io/DOLAPI/beginners-guide.html). The MODERN portal — DOL Open Data Portal + APIv4 at `dataportal.dol.gov`, launched ~2026-02-18 — uses **header-based** token auth, with keys registered at https://dataportal.dol.gov/api-keys (up to 5 keys/user) and an explicit ban on committing keys to public repos (https://www.dataportal.dol.gov/pdf/dol-api-user-guide.pdf; https://dataportal.dol.gov/). The `DOL_API_KEY` env name and exact APIv4 header name are SDK conventions — verify the header before porting (see Open/Unverified).

### The `check-api-status` health-probe tool (patents-mcp-server, MIT)

- The probe pattern is `getAvailableSources(cfg): ApiStatus[]` returning one `{ name, configured, healthy }` row per source, e.g. `{ name:"USPTO ODP", configured: cfg.usptoApiKey !== undefined, healthy: false }`, `{ name:"EPO OPS", configured: cfg.epoConsumerKey !== undefined && cfg.epoConsumerSecret !== undefined, healthy: false }`, `{ name:"Google BigQuery", configured: (...creds...) && cfg.googleCloudProject !== undefined, healthy: false }` (`explorations/_gold-intake/research/gold-catalog.json` nugget `patents-mcp-server#12`, license MIT).
- GOTCHA worth fixing on port: `configured` is **presence-only** (env var set ≠ valid/healthy), and `healthy` is **hardcoded `false`** in the source — it never actually pings the upstream. A real `check-api-status` tool must (a) report `configured` from Config resolution and (b) set `healthy` from an actual cheap probe request (and surface 401/403 distinctly from 5xx). The presence matrix is the cheap path; the live probe is the truthful path (`explorations/_gold-intake/research/gold-catalog.json` nugget `patents-mcp-server#12`).

### Conditional registration vs always-register+guard (two complementary strategies)

- **Conditional registration** (patents-mcp-server, MIT): `registerAllTools` only mounts a tool group when its creds resolved — `if (config.usptoApiKey){ registerOdpTools; registerPtabTools; registerCitationsTools; registerOfficeActionsTools } if (config.epoConsumerKey && config.epoConsumerSecret){ registerEpoTools } registerUtilityTools(server) // always`. Leaner tool surface; absent-key tools are simply invisible (`explorations/_gold-intake/research/gold-catalog.json` nugget `patents-mcp-server#1`).
- **Always-register + handler guard** (mcp-uspto): every tool is registered; the handler checks the key and returns `keyMissingResponse(...)` when absent. More discoverable to the LLM (the tool exists and tells you how to enable it) at the cost of a larger surface (https://raw.githubusercontent.com/cmanohar/mcp-uspto/master/src/lib/config.ts).
- **Tier-gate variant** (uspto-patents-mcp, MIT): orthogonal axis — `listTools(tier)` filters `!t.premium || tier==="team"||tier==="pro"`, and `tools/call` returns JSON-RPC `rpcError(id, -32000, "Tool '<name>' is premium-only...")`. Note this uses a *protocol error* for the gate, which the MCP spec marks as less recoverable for models — for a key/tier guard, the structured-content path is the more model-friendly choice (`explorations/_gold-intake/research/gold-catalog.json` nugget `uspto-patents-mcp#4`; https://modelcontextprotocol.io/specification/2025-11-25/server/tools#error-handling).
- **Schema-first auth metadata** (us-gov-open-data-mcp, MIT): the per-module `ModuleMeta` contract carries `auth?: { envVar: string | string[]; signup: string }` — i.e. the env var name + registration URL are *data on the module*, which is exactly the input both `keyMissingResponse` (envVar, registrationUrl) and `getAvailableSources` (configured) need. This is the clean shape for a single source-registry that drives gating, the helper, and the health-probe from one declaration (`explorations/_gold-intake/research/gold-catalog.json` nuggets `us-gov-open-data-mcp#5/#6`).

### The Effect-typed boundary (how this lands in beep)

- beep's Effect-native equivalent of `process.env.X ?? null` is `Config`. Confirmed semantics: `Config.redacted("KEY")` wraps the value in `Redacted<string>` (logs as `<redacted>`, read via `Redacted.value`); `Config.option(...)` turns a MISSING value into `Option.none` (no failure); a *required* `Config` that is missing fails with a `ConfigError` whose op is `MissingData` ("Expected NAME to exist in the process context"); `Config.withDefault(x)` substitutes a fallback (https://effect.website/docs/configuration/).
- The two registration strategies map cleanly onto Effect lifecycle:
  - **Build-time / conditional**: read `Config.redacted(envVar).pipe(Config.option)` once at Layer build; branch `Layer.mergeAll(...)` to include only `McpServer.toolkit(group).pipe(Layer.provide(handlers))` whose key resolved to `Some`. This is the conditional-registration analog and matches the repo's existing composition idiom — `nlp-mcp/Server.ts` already `Layer.mergeAll`s two `McpServer.toolkit(...)` layers under one `McpServer.layerStdio` (`packages/drivers/nlp-mcp/src/Server.ts`).
  - **Handler-time / structured guard**: register the tool always; inside the handler `yield* Config.redacted(envVar).pipe(Config.option)` and, on `None`, return a typed `ApiKeyRequired` error through the toolkit's `failureMode: "return"` channel. beep's MCP tools already use `Tool.make(..., { failureMode: "return" })` mapping to a typed `AiToolError`, which the Effect `McpServer` encodes into the MCP `CallToolResult` — the Effect-native, schema-typed replacement for the hand-rolled `keyMissingResponse` (fields `error/tool/envVar/registration`) (`packages/drivers/nlp-mcp/src/StreamingTools.ts`, `failureMode: "return"`; `packages/drivers/nlp-mcp/CLAUDE.md`).
- Recommended shape: a single `@beep/schema` model `ApiKeyRequired` ({ `_tag:"api_key_required"`, `tool`, `envVar`, `registrationUrl` }) returned via `failureMode:"return"`, plus a `SourceAuth` registry record ({ `name`, `envVar`, `gate: "none"|"soft"|"hard"`, `signupUrl` }) that simultaneously drives (1) conditional `Layer.mergeAll`, (2) the structured helper, and (3) `check-api-status`. This unifies us-gov-open-data-mcp's `ModuleMeta.auth`, mcp-uspto's `keyMissingResponse`, and patents-mcp-server's `getAvailableSources` into one Effect-typed source of truth.

### Boundary WITH — not absorption OF — the multi-provider LLM dispatch/fallback cluster

- This packet and the LLM-fallback cluster share ONE primitive (`Config.redacted` credential resolution) but sit on different axes and must stay separate (per CAPTURE caution + tree-snapshot routing).
  - **This packet = build-time capability gating** of MCP *tools* by *source* credential presence. Lifecycle: resolved once at Layer build (or per-call guard). Failure model: omit the tool group / return `api_key_required`. Surface: which tools exist.
  - **LLM dispatch/fallback = runtime request-time selection** of a *model provider* per prompt, with key-precedence + automatic failover across `@beep/anthropic`, `@beep/openai-compat`, `@beep/xai`, `@beep/venice-ai`. Lifecycle: every request. Failure model: retry/failover to the next provider. Surface: which model answers.
- Repo evidence they already resolve creds independently (so fusion would couple unrelated lifecycles): `packages/drivers/anthropic/src/Anthropic.service.ts:36 Config.redacted(ANTHROPIC_API_KEY_ENV)`; `packages/drivers/xai/src/XAi.service.ts:1020-1021 Config.redacted("XAI_API_KEY"|"XAI_MANAGEMENT_API_KEY").pipe(Config.option)`; `packages/drivers/venice-ai/src/VeniceAI.service.ts:1877 Config.redacted("AI_VENICE_API_KEY")` (in-repo grep).
- Boundary contract: the MCP registration layer consumes the SAME `Config.redacted`/`Config.option` primitive but owns only the *source* gating registry; if a tool needs an LLM, it depends on a provider-selection PORT (a `Context.Tag` service) supplied by the separate cluster, and never embeds precedence/fallback logic itself. The GOLD_SYNTHESIS routes the LLM-fallback work to a proposed `multi-provider-llm-dispatch-fallback` packet (gapStatus partial/P2) — cross-link, do not pull it in (CAPTURE.md "Scope note"; `scratchpad/routing/tree-snapshot.md` line 15 + proposed-slug list).

## External Landscape

### Deterministic legal extractors — only the Free Law Project (FLP) stack is reusable; everything precision-grade is AGPL or abandoned

The OSS survey ([raw](research/deterministic-legal-regex-prior-art.md)) splits cleanly on license and maintenance:

- **LexNLP** (LexPredict/ContraxSuite) is the closest functional match — it extracts defined terms with coordinates (`get_definitions(return_coords=True)` → `(name, text, (start,end))`), section headings, and corporate-suffix companies via a deterministic `RegexpBased` locator. But it is **dual-licensed AGPLv3 + commercial**, last released **2.3.0 on 2022-11-30** (dormant). AGPL is a copyleft trap for a hosted service: **cannot vendor — clean-room reimplement the regex families.** (https://github.com/LexPredict/lexpredict-lexnlp, https://lexpredict-lexnlp.readthedocs.io/en/latest/about.html, https://github.com/LexPredict/lexpredict-lexnlp/blob/master/lexnlp/extract/en/definitions.py)
- **Blackstone** (ICLR&D) is a spaCy statistical model (PROVISION/CITATION/COURT NER), **Apache-2.0** but trained on **England & Wales** case law, v0.0.1, **last commit 2021-01-31 (~5 yr abandoned)**, and non-deterministic. Wrong jurisdiction + ML + abandoned → **reference only, do not port.** (https://github.com/ICLRandD/Blackstone, https://research.iclr.co.uk/blackstone)
- **eyecite** (FLP) is the strongest reuse candidate for *citations*: **BSD-2-Clause**, actively maintained, tested against 50M+ citations, powers CourtListener/CAP, tuned regex (no model in the matching loop), and every citation exposes `span()`/`fullSpan()` → `[start,end]` offsets. (https://github.com/freelawproject/eyecite, https://free.law/projects/eyecite/) A **TypeScript port already exists** — **eyecite-js** (`@beshkenadze/eyecite`), **BSD-2-Clause**, self-reported full parity with Python v2.7.6. **Version refresh (2026-06-29):** the npm `latest` dist-tag is now a **stable `2.7.6`** (license `BSD-2-Clause` per the npm registry metadata), so the alpha-tagged `v2.7.6-alpha.28` (2025-08-21) recorded earlier is a stale snapshot, not the current publish. **Adapt the TS port rather than re-port from Python**, but re-check the exact npm `latest`/GitHub tag + tarball license at adopt time and pin a specific version before running its parity suite. (https://www.npmjs.com/package/@beshkenadze/eyecite, https://github.com/beshkenadze/eyecite-js)
- **courts-db** (700+ court IDs, 2,100+ regexes) and **reporters-db** (1,167 reporters + 2,102 variations) are **BSD-2 JSON catalogs** — **reuse the data directly**, decode through `effect/Schema`, no code port. (https://github.com/freelawproject/courts-db, https://github.com/freelawproject/reporters-db)

Per-pattern verdict ([table](research/deterministic-legal-regex-prior-art.md)): citation extraction → **REUSE eyecite-js**; court/reporter catalogs → **REUSE FLP JSON**; defined terms, Section/Article/Exhibit cross-refs, corporate-suffix parties → **REIMPLEMENT** (LexNLP AGPL; doc-haus regexes are the design seed); amendment recitals + the versioned re-extraction migration → **NET-NEW** (no OSS coverage). The CAPTURE source `doc-haus` is **not a discoverable public repo** — its `VERSION`/regex snippets are design seeds, not a dependency. "Source/provenance on every field" is corroborated as an industry norm by OpenContracts (MIT). (https://github.com/Open-Source-Legal/OpenContracts)

### Court-PDF layout extraction — the canonical source is FLP `doctor` (BSD-2), but its filter cannot ride the existing Tika layer

The doctor#4/#5 nuggets come from **`freelawproject/doctor`**, **BSD-2-Clause** ([raw](research/court-pdf-layout-extraction.md), https://github.com/freelawproject/doctor/blob/main/LICENSE) — permissive, so the *algorithm* may be ported with attribution (not the Python runtime). Live functions on `main`: `get_header_stamp` + `get_document_number_from_pdf` (`doctor/tasks.py`) and `adjust_caption_lines` (`doctor/lib/text_extraction.py`).

- **Header-stamp isolation** is a pdfplumber `.filter()` predicate: keep chars where `fontname` contains `LiberationSans` **OR** `y0 > 750`. CM/ECF stamps the header in a distinct LiberationSans face absent from the body; FLP independently confirmed "only the tag for LiberationSans changes" across PDF instances (https://github.com/freelawproject/courtlistener/issues/6634). `y0 > 750` is the **ca5 (Fifth Circuit)** positional fallback — pdfplumber's origin is bottom-left, US-Letter is 792 pt, so >750 selects the top ~0.58 inch (https://github.com/jsvine/pdfplumber).
- **Document-number regex** `Document:|Document|Doc:|DktEntry:` recovers the CM/ECF document/docket-entry number — the `filing → docket` provenance edge (AO documents the "unique electronic document stamp" in the NEF/NDA; https://pacer.uscourts.gov/help/faqs/files-pleading-court-automatically-serve-notification, https://www.alsd.uscourts.gov/news/pageid-feature-cmecf-updated). The raw pattern has bugs to fix on port (stray captured leading `.`, duplicated `.` in the class, alternation order). `get_document_number_from_pdf` returns `""` on miss — the "a miss is an absent row" posture is already in the source.
- **Caption-line alignment** tries separators in order `[")", "§", ":"]` aligning to the first separator with ≥3 hits in the central band [30,70]. **Pre-req: spatially-faithful text** (`extract_text(layout=True, keep_blank_chars=True)`) — without layout-mode text the column-index logic is meaningless. **Empirical, NOT codified:** the `§`=TX / `:`=NY / `)`=generic mapping is grounded **only in doctor's own docstring** — it is **not** confirmed against an official Texas/NY court style rule, so treat it as the FLP authors' empirical heuristic, not a standard (`research/court-pdf-layout-extraction.md:102`).
- **TS-port reality:** pdfplumber is pure-Python (MIT, v0.11.10, 2026-06-15) and **not portable** into the TS runtime. The JS-native route is **pdf.js `getTextContent`** (each item exposes `str`/`fontName`/`transform[4,5]`=x,y), but pdf.js `fontName` is an internal style id (`g_d0_f1`, **not** `"LiberationSans"` — needs a `commonObjs` lookup) and is documented to **lose some fontName transitions** (https://github.com/mozilla/pdf.js/issues/7297, https://github.com/mozilla/pdf.js/issues/12031). Higher-fidelity non-Python routes: poppler `pdftotext -bbox-layout` (**GPL-family copyleft — see Licensing gravity**) or PDFBox `TextPosition` (JVM, Apache-2.0). **Decide the extractor backend before adopting the predicate**, and route that decision through the existing deferred OCR/diagnostics lane (`goals/file-processing-capability`).
- **Empirical caveats to verify against real PACER samples before locking constants** (`research/court-pdf-layout-extraction.md:101-105`): the **literal stamp formats** (district `Case … Document N Filed … PageID #: n` vs the ~11-digit appellate `Document: 00############`) are field-confirmed but verbatim-layout **UNVERIFIED**; **`DktEntry:` vs `Document:`** per-court distribution is unconfirmed; **`y0 > 750`** assumes 792-pt US-Letter and must be **computed from `page.height`** (legal-size = 1008 pt, rotated pages shift the band); the **`LiberationSans` font branch over-captures** body text on docs that use that face, so AND it with a top-band `y0` check rather than OR-ing. Pull real PACER PDFs to confirm the regex/positional constants before committing them.

### Span/offset model — Anthropic's own primitives corroborate the half-open-offset + verbatim-quote + fail-on-ambiguity design

The provenance-wall design is independently validated by vendor primitives ([raw](research/char-offset-provenance-span-model.md)):

- **Anthropic Citations API** grounds answers in source docs and returns the exact supporting passages: `type:"char_location"` with `start_char_index` (0-indexed) + `end_char_index` (exclusive) + `cited_text` — structurally identical to the repo's `TextAnchor` (`startChar`/`endChar` half-open + `quote`). (https://platform.claude.com/docs/en/build-with-claude/citations)
- **Anthropic text-editor `str_replace`** is the canonical unique-anchor resolver: `old_str` "must match exactly, including whitespace"; the app should ensure exactly one match; >1 → `"Found N matches … provide more context"`; 0 → `"No match found"`. This is exactly mike#5's ambiguous/not-found contract, made canonical by the vendor. (https://platform.claude.com/docs/en/agents-and-tools/tool-use/text-editor-tool)
- **Google LangExtract** is the prior art for the alignment ladder: a two-step `WordAligner` (exact `difflib` blocks → fuzzy ≥ `fuzzy_alignment_threshold`, reported default **0.75**), `AlignmentStatus` ∈ {MATCH_EXACT, MATCH_LESSER, MATCH_FUSZY, None}, unaligned extractions filtered out (`[e for e in … if e.char_interval]`). (https://github.com/google/langextract/blob/main/langextract/resolver.py, https://deepwiki.com/google/langextract) — note: whether `main` ships 3 or 4 status members and the exact `MATCH_LESSER` semantics are **UNVERIFIED** (one source listed a fourth `MATCH_GREATER`).
- **Mechanics:** `String.prototype.matchAll(/…/g)` yields `match.index` + `match[0]`, span `[index, index+len)`; the `d` flag (`RegExp.prototype.hasIndices`) adds per-capture-group `match.indices[n]` for sub-span anchoring and is safe to add to any regex ("only provides additional information"). (https://developer.mozilla.org/en-US/docs/Web/JavaScript/Reference/Global_Objects/String/matchAll, https://developer.mozilla.org/en-US/docs/Web/JavaScript/Reference/Global_Objects/RegExp/hasIndices, https://github.com/tc39/proposal-regexp-match-indices)
- **Offset-unit gotcha:** JS strings index in **UTF-16 code units**; Python/LangExtract index in **Unicode code points** — the same numeric span diverges on astral characters. Foreign citation offsets (LangExtract-Python, model citations) are **not byte-safe** to feed into JS `slice`; declare a canonical unit (recommend UTF-16, since `TextAnchor`'s invariant is `slice`-based) and re-validate `slice===quote` on ingest. (https://developer.mozilla.org/en-US/docs/Web/JavaScript/Reference/Global_Objects/String, https://docs.python.org/3/howto/unicode.html)

### Heuristic-first cascade — a validated production architecture, with a calibration trap

The "deterministic wall first, fallible model second" discipline (harvest-mcp#3) is a measured production pattern, not a novelty ([raw](research/heuristic-first-candidate-gate.md)):

- **Meta's June 2026 asset-classification case study** (PRIMARY): **~85% of requests resolve through deterministic rules; only ~15% fall through to the LLM**, which is "slower (seconds) and roughly 400× the compute cost" vs a deterministic path at "single-digit milliseconds / ~40ms." Rules are **mined and promoted** through holdout → shadow → `rules.yaml` gates, and human review found the heuristic-first pattern "more consistent, debuggable, and auditable." (https://engineering.fb.com/2026/06/25/security/privacy-aware-infrastructure-in-the-ai-native-era-an-asset-classification-case-study/)
- **LLM-cascade economics** corroborate independently: cheap-first/escalate-on-low-confidence cuts cost **45–85% while keeping ~95% of frontier quality**, routing ~85% to cheaper models; **cascading ≠ routing** (this packet is a *cascade* — produce candidate-with-confidence, then conditionally escalate). (https://tianpan.co/blog/2025-11-03-llm-routing-model-cascades, https://mbrenndoerfer.com/writing/model-routing-selection-ab-testing-cascades-strategies)
- **Confidence bands** (Box Extract API, PRIMARY): **≥0.90 high / 0.70–0.89 medium / <0.70 low**, explicitly "tune per document type / risk tolerance." Box computes confidence by consistency-sampling the LLM. (https://blog.box.com/confidence-scores-box-extract-api-know-when-rely-your-extractions)
- **THE CALIBRATION TRAP:** Juris.AI#3's hand-tuned per-type confidences (statute 0.85, case 0.90) and harvest-mcp#3's literals (0.95/0.9/0.8) are **priors / pattern-strength weights, NOT calibrated probabilities** — raw self-assessed confidence is poorly calibrated. Fix with a cheap post-hoc monotonic map (temperature / Platt / **isotonic**, the most flexible for selective prediction), learned per entity type from candidate→admitted outcomes. (https://www.kdnuggets.com/a-deep-dive-into-calibration-of-language-models-platt-scaling-isotonic-regression-temperature-scaling, https://arxiv.org/html/2411.02988v2)
- **"A miss is an absent row" is formally selective prediction with abstention** (Chow's 1970 optimal reject rule; El-Yaniv & Wiener risk-coverage). An explicit `abstain`/`defer` outcome is cheaper *and* more accurate than a low-confidence guess ("13% cost reduction, 5% error reduction" from early abstention). The escalation trigger must be **confidence/abstention-based, not "regex returned nothing"** — a measured clinical study showed naive regex-then-LLM-on-empty *underperformed* regex-only (F1 .152 vs .283). (https://aclanthology.org/2402.15610v2, https://arxiv.org/html/2505.15008v2, https://www.medrxiv.org/content/10.64898/2026.01.21.26344531.full.pdf) Rule/regex-driven `{nodes,links}` KG construction without LLM cost is an established first pass (https://neo4j.com/blog/developer/entity-linking-relationship-extraction-relik-llamaindex/, https://arxiv.org/pdf/2505.00039).

### Streaming gate vs the langextract lock — the conflict is reconcilable by construction

([raw](research/streaming-gate-vs-langextract-lock.md)) The langextract V1 lock bans *raw chunks as public API* but **explicitly permits streaming surfaced as schema-backed domain events** — so the TalentScore Partial/Complete gate instantiates the *permitted exception*; it cannot attach to `goals/langextract-capability` for **scope** reasons (V1 is non-streaming), not semantic contradiction. Upstream Google langextract's "streaming" is chunk-batching parallelism (`max_workers`), orthogonal to single-extraction partial reveal — so the gate is genuinely net-new vs both. (https://github.com/google/langextract, https://developers.googleblog.com/introducing-langextract-a-gemini-powered-information-extraction-library/)

- **Effect v4 beta.91 is the target runtime** (announced 2026-02-18; Schema rewrite; evolving APIs under `effect/unstable/*` "may receive breaking changes in minor releases"). The TalentScore `Stream.async((emit)=>…emit.single()…emit.end())` snippet is **Effect v3 shape, removed in v4** — no `Stream.async`/`asyncPush`/`asyncEffect` exists; the callback constructor is `Stream.callback` and hands you a `Queue`. The web docs at https://effect.website/docs/stream/creating/ are **stale** for this repo. (https://effect.website/blog/releases/effect/40-beta/, https://github.com/Effect-TS/effect-smol/issues/1378)
- The cleanest v4 bridge for an async-generator extraction stream is **`Stream.fromAsyncIterable(iterable, onError)`** + `Stream.concat`/`append` for the terminal Complete (not a manual `emit.end()`).
- **BAML's Partial/Final dichotomy** (TalentScore#4) maps onto `effect/Schema`: BAML auto-generates a Partial type (all fields nullable, filled as tokens arrive) while `getFinalResponse()` returns the original validated type; `@stream.done` makes a field atomic. → Partial = `S.Struct({ field: S.optionalWith(S.NullOr(T), {exact:true}) })`; Complete = strict `S.Class`. (https://docs.boundaryml.com/guide/baml-basics/streaming, https://boundaryml.com/blog/semantic-streaming) Effect RPC streaming precedent (per-element decoded domain value, client `Stream.runForeach`): https://dev.to/titouancreach/part-2-how-i-replaced-trpc-with-effect-rpc-in-a-nextjs-app-router-application-streaming-responses-566c.

### Confidence + candidate→approved lifecycle (the admission gate — REUSE, do not reinvent)

- **`UnitInterval` — TWO non-identical schemas; the gate is NOT automatically type-identical.** The repo has (a) the **canonical branded** `@beep/schema/UnitInterval` — `S.Finite.check([0,1]).pipe(S.brand("UnitInterval"))`, documented as "the canonical, domain-agnostic shape for probabilities, confidences, ratios," re-exported by epistemic as `Confidence` and carried by `EvidenceSpan` (**Verified:** `packages/foundation/modeling/schema/src/UnitInterval.ts` L32-35); and (b) an **unbranded** `@beep/nlp internal/numbers` `UnitInterval` — `S.Finite.check(UnitIntervalChecks)` with **no `S.brand`** — which is what `GroundedExtraction`/`ExtractionCandidate` `confidence: S.optionalKey(UnitInterval)` actually use (**Verified:** `packages/foundation/capability/nlp/src/internal/numbers.ts` L45-49). A boundary from `GroundedExtraction.confidence` straight into `EvidenceSpan.confidence`/`Confidence` is therefore **not** type-identical — it needs an explicit decode/rebrand step. **Recommendation:** name `@beep/schema/UnitInterval` (`Confidence`) as canonical for epistemic evidence; require the regex/heuristic seeder and the LLM path to both decode into that branded type before the gate; and plan a small shared-kernel cleanup so `@beep/nlp`/`@beep/langextract` reuse `@beep/schema/UnitInterval` rather than the local unbranded copy.
- **`ClaimLifecycle`** — `LiteralKit(["candidate","shape_valid","consistency_checked","admitted"])`, owned by `@beep/shared-domain`, re-exported through `@beep/epistemic-domain`. The task's "candidate→approved" IS the existing `candidate→admitted` shared-kernel lifecycle — **no new vocabulary needed.** **Verified:** `packages/shared/domain/src/values/ClaimLifecycle/ClaimLifecycle.model.ts` (four literals confirmed at L38).
- **`ClaimGateResult`** — `ClaimGateVerdict(["admitted","rejected"]).toTaggedUnion("verdict")`; `rejected` carries `violations` projected to product-agnostic primitives (`focusNode`/`path`/`message`/`severity`), NO RDF types. - **`ClaimLifecycleTransition`** — `S.Class({ from, to, reason: S.String })`, a pure audit value object ("records a transition, does not authorize one"). The audit primitive for the replayable decision trace (matched pattern id + per-type confidence + extractor version → `reason`).

# heuristic-first-candidate-gate

Scope: the deterministic-wall-before-fallible-model pipeline — layered heuristics decide first with per-type confidence, LLM only as fallback/refinement; confidence scoring, the non-LLM `{nodes,links}` first pass, and the candidate→approved lifecycle in `@beep/epistemic-domain`, optimized for cost and auditability.

## Findings

### A. The deterministic-wall-before-LLM pattern is a validated production architecture (not a novelty)

- Meta's June 2026 asset-classification case study is a direct PRIMARY confirmation of harvest-mcp#3's "deterministic wall first, fallible model second" discipline: **~85% of requests resolve through deterministic rules; only the remaining ~15% fall through to LLM inference**, and the LLM path is "slower (on the order of seconds) and roughly 400 times the compute cost, budgeted separately," versus a deterministic path that resolves in "single-digit milliseconds" / "~40 ms including context assembly." (https://engineering.fb.com/2026/06/25/security/privacy-aware-infrastructure-in-the-ai-native-era-an-asset-classification-case-study/)
- The same source shows the rules are *mined and promoted*, not hand-frozen: staged mining (Stage 1 single-field patterns at min-support 2 / 80% purity; Stage 2 conjunctions at 95% purity / 10+ examples; Stage 3 optional LLM-*assisted* rule proposals), then every candidate rule passes "holdout validation → shadow mode → promote to rules.yaml" behind quality gates. This is the template for a *versioned, re-extractable* heuristic layer (matches doc-haus#3's `VERSION` const discipline). (https://engineering.fb.com/2026/06/25/security/privacy-aware-infrastructure-in-the-ai-native-era-an-asset-classification-case-study/)
- Independent corroboration of the cheap-first/expensive-fallback economics from the LLM-cascade literature: cascades route the query to the cheapest model first and **escalate only when confidence is below threshold**, cutting cost **45–85% while maintaining ~95% of frontier quality** and routing ~85% of queries to cheaper models (citing RouteLLM); frontier vs efficient token pricing is roughly a 5x ratio (~$5/$25 vs ~$1/$5 per M tokens). Two independent sources (Meta, RouteLLM-via-TianPan) both land on ≈85% cheap-path coverage. (https://tianpan.co/blog/2025-11-03-llm-routing-model-cascades)
- A distinction the repo design must encode: **routing is a one-shot pre-execution decision; cascading is sequential escalation that uses the produced output's confidence as the escalation signal.** harvest-mcp#3 / this packet is a *cascade* (deterministic pass produces a candidate-with-confidence, then conditionally escalates), not a router. (https://tianpan.co/blog/2025-11-03-llm-routing-model-cascades) and (https://mbrenndoerfer.com/writing/model-routing-selection-ab-testing-cascades-strategies)

### B. Confidence scoring — bands, computation, and the calibration trap

- Canonical three-band routing (PRIMARY product doc, Box Extract API): **≥0.90 = high (minimal review / auto-accept); 0.70–0.89 = medium (light review); <0.70 = low (manual review)**, with the explicit caveat that "optimal thresholds depend entirely on your risk tolerance, document types, and the criticality of the extracted data" (a 0.70 cut that works for content tagging fails for financial/legal data). Use these as *defaults to tune per entity type*, not constants. (https://blog.box.com/confidence-scores-box-extract-api-know-when-rely-your-extractions)
- Box computes confidence by *consistency sampling*: "requesting multiple responses from the LLM and analyzing consistency" — high agreement → high confidence. This is a usable per-candidate confidence signal when the LLM refinement stage does run, distinct from the heuristic layer's per-type prior. (https://blog.box.com/confidence-scores-box-extract-api-know-when-rely-your-extractions)
- THE CRITICAL CAVEAT for Juris.AI#3's hand-tuned per-type confidences (statute 0.85, case 0.90, court/legal-term …) and harvest-mcp#3's literal heuristic confidences (0.95 search, 0.9 pagination, 0.8 auth-header): **these are priors / pattern-strength weights, NOT calibrated probabilities.** Raw self-assessed or hand-assigned confidence is "a raw model self-assessment" and is poorly calibrated — "models can produce fluent but factually wrong answers with high token probabilities." Confidence-based routing should use *calibrated* scores where calibration is enabled, "rather than the raw number." (https://engineering.fb.com/2026/06/25/security/privacy-aware-infrastructure-in-the-ai-native-era-an-asset-classification-case-study/) and (https://tianpan.co/blog/2025-11-03-llm-routing-model-cascades)
- Calibration is a solved, cheap post-hoc step: fit a monotonic map from raw score → empirical correctness on a held-out labeled set. Three standard methods — **temperature scaling, Platt scaling (parametric sigmoid), isotonic regression (non-parametric monotonic)**; isotonic is the most flexible and "most consistently improves probability estimates," and "top-class isotonic regression" is the most relevant choice for *selective prediction* (threshold on max-class probability). Practical design: keep the regex/heuristic confidence as a raw weight, then learn one isotonic map per entity type from the eventual candidate→admitted outcomes so thresholds mean what they say. (https://www.kdnuggets.com/a-deep-dive-into-calibration-of-language-models-platt-scaling-isotonic-regression-temperature-scaling) and (https://arxiv.org/html/2411.02988v2)
- The repo already has the confidence primitive: `UnitInterval` (`S.Finite` checked to `[0,1]`, `@beep/nlp/internal/numbers` → re-exported via `@beep/nlp/Handoff`) is the established `[0,1]` confidence type, and `ExtractionCandidate`/`GroundedExtraction` already carry `confidence: S.optionalKey(UnitInterval)` (`packages/foundation/capability/langextract/src/Extraction/index.ts:178,215`). A heuristic seeder should emit the *same* `UnitInterval` confidence so candidates from regex and from the LLM are type-identical at the gate. (repo-grounded: `packages/foundation/capability/nlp/src/internal/numbers.ts:45`)

### C. The "a miss is an absent row, never a wrong fact" rule is formally *selective prediction with abstention*

- doc-haus#3's invariant and the deterministic guarantee in CAPTURE Cautions map exactly to the classical **selective prediction / reject-option** frame: a system abstains on low-confidence inputs to "maximize coverage … while minimizing risk (the percentage of predicted answers that were incorrect)"; full coverage = high usability/low safety, total abstention = zero risk/zero value. The deterministic wall deliberately sits at the high-precision / low-coverage end and hands the abstained remainder to the LLM. (https://aclanthology.org/2402.15610v2 — "Selective 'Selective Prediction'") and (https://arxiv.org/html/2505.15008v2 — "Know When to Abstain")
- Theoretical backing for the per-type threshold: **Chow's (1970) optimal reject rule** — a confidence threshold minimizing misclassification risk given an abstention cost — and El-Yaniv & Wiener's risk-coverage analysis give conditions for guaranteeing low selective risk. This is the principled basis for "if no heuristic clears its per-type threshold, emit no row and defer." (https://arxiv.org/html/2505.15008v2)
- An escalation refinement worth porting: *early abstention* (the cheap stage explicitly signals "I can't decide" instead of guessing) gave "a 13% reduction in cost and 5% reduction in error rate" with a 4.1% higher abstention rate — i.e., an explicit `abstain`/`defer` outcome from the heuristic layer is cheaper and more accurate than forcing a low-confidence guess. (https://tianpan.co/blog/2025-11-03-llm-routing-model-cascades)

### D. The non-LLM `{nodes,links}` first pass (LegalEase#4) and where it lands

- LegalEase#4's shape — regex obligation/party/date patterns emitting `nodes` + typed `links` with a `strength` weight (e.g. `links.append({source, target, label:"obligated to", strength:0.6})`) — is a recognized, cost-avoiding pattern: rule/regex-driven KG construction "without the high costs typically associated with deploying LLMs," using handcrafted rules, regexes, and dependency-parse patterns as the first pass, with LLMs reserved for coreference/relation gaps. (https://neo4j.com/blog/developer/entity-linking-relationship-extraction-relik-llamaindex/) and (https://arxiv.org/pdf/2505.00039 — "Ontology-Driven Graph RAG for Legal Norms: … Deterministic Approach")
- Port target in-repo: the `strength` on a link is the same `UnitInterval` confidence used everywhere else; the `{nodes,links}` graph is the non-LLM seed that feeds the OWL/SHACL proof and the FalkorDB projection layer (per CAPTURE routing) — i.e., heuristic links become `candidate` edges, the SHACL claim gate is the structural admission check. Reuse `@beep/nlp` *conventions* (`$NlpId` identity, Schema-first models, `VERSION` discipline) — but **do NOT route raw-text regex through `Core/Pattern*` (`PatternParsers.ts`, `PatternBuilders.ts`, `PatternOperations.ts`)**: that infra parses **bracket token patterns** (`[ADJ|NOUN]`) over a tokenized wink `Document`, not `RegExp.matchAll`/`d`-flag char-offset matching, so feeding raw regex through it is an impedance mismatch (superseded by the RESEARCH.md synthesis). Build the raw-regex char-offset span extractor as a **sibling helper** that can optionally feed wink downstream, and emit `GroundedExtraction` rows with `Contract.Span` char offsets so every node/link is provenance-anchored. (repo-grounded: `packages/foundation/capability/nlp/src/Core/PatternParsers.ts`; `packages/foundation/capability/langextract/src/Extraction/index.ts:211`)

### E. The candidate→approved lifecycle already exists in `@beep/epistemic-domain` (do not reinvent)

- The admission lifecycle is shared-kernel vocabulary, **not** a free-text status: `ClaimLifecycle = LiteralKit(["candidate", "shape_valid", "consistency_checked", "admitted"])`, owned by `@beep/shared-domain/values/ClaimLifecycle` and re-exported through `@beep/epistemic-domain`. `Options` is the canonical forward order; the epistemic slice owns the *mechanism* (gate/transition/projection), shared owns the *vocabulary*. Map heuristic output to `candidate`; "approved" is the terminal `admitted` state. (repo-grounded: `packages/shared/domain/src/values/ClaimLifecycle/ClaimLifecycle.model.ts`; `packages/epistemic/domain/src/values/ClaimLifecycle/ClaimLifecycle.model.ts`)
- The gate verdict is a typed tagged union, not a boolean: `ClaimGateResult = ClaimGateVerdict(["admitted","rejected"]).toTaggedUnion("verdict")` where `admitted` carries no violations and `rejected` carries `violations: S.Array(ClaimGateViolation)` — each violation is a SHACL violation projected to product-agnostic primitives (`focusNode`, `path`, `message`, `severity ∈ {info,warning,violation}`), carrying NO RDF/engine types. The heuristic-first gate should produce exactly this verdict shape so a regex-seeded candidate and an LLM-seeded candidate are admitted by the identical structural check. (repo-grounded: `packages/epistemic/domain/src/values/ClaimGate/ClaimGateResult.model.ts:70-106`)
- Each transition is an auditable value object: `ClaimLifecycleTransition extends S.Class({ from: ClaimLifecycle, to: ClaimLifecycle, reason: S.String })` — "Pure value object — it records a transition, it does not authorize one. Legality … is enforced by the owning slice's transition service." This is the audit-record primitive: every advance (and the *reason*, e.g. "claim gate admitted" / "regex seeder, statute pattern, conf 0.85") is captured. (repo-grounded: `packages/shared/domain/src/values/ClaimLifecycle/ClaimLifecycle.model.ts`)
- The candidate entity is `CandidateClaim extends BaseEntity.Class(... Epistemic.CandidateClaimId, { fields: { fixtureKey, lifecycle: ClaimLifecycle, snapshot: UnknownRecord } })` with a persisted `lifecycle` literal column and a `snapshot` jsonb — the row a heuristic seeder writes (lifecycle `candidate`, snapshot = the `{nodes,links}`/`GroundedExtraction` payload). (repo-grounded: `packages/epistemic/domain/src/entities/CandidateClaim/CandidateClaim.model.ts:31-54`)
- NOTE the lifecycle has FOUR stages, not a bare candidate/approved binary — `candidate → shape_valid → consistency_checked → admitted`. The heuristic-first gate seeds `candidate`; `shape_valid` is the SHACL structural pass; `consistency_checked` is cross-checking (where the LLM refinement / second-source verification can attach as a *gate input*, never as the authority); `admitted` is approved-into-authoritative-store. Per the tree-snapshot, `rejected`/`superseded` states and the bitemporal store are *deferred* (→ `agent-memory-tiers-bitemporal-edges`), so do not assume a `rejected` lifecycle literal exists yet. (repo-grounded: `/tmp/.../routing/tree-snapshot.md` L10; shared `ClaimLifecycle.model.ts`)

### F. Cost minimization + auditability are the same mechanism: a replayable decision trace

- Auditability requirement (PRIMARY): a decision must carry "a decision trace showing which evidence influenced the result," "the rule that matched," and "version information"; "A useful decision trace should tell us what evidence was used, which rule or prompt version was active, and how the decision can be replayed." Human review independently found the heuristic-first pattern produced "more consistent, debuggable, and auditable decisions." This is why a deterministic-first wall is *cheaper AND more auditable* than an LLM-first design — the 85% common path is replayable from rule-id + version, no model in the trace. (https://engineering.fb.com/2026/06/25/security/privacy-aware-infrastructure-in-the-ai-native-era-an-asset-classification-case-study/)
- The cascade literature gives the matching observability contract: "structured logging of every routing decision: the model selected, the classifier's confidence, the query features used, and ideally the downstream quality signal." In-repo this is exactly `ClaimLifecycleTransition.reason` + the `Activity`/`UsageRecord` epistemic entities (the slice already has `Activity` and `UsageRecord` per tree-snapshot) — the heuristic decision (matched pattern id, per-type confidence, version) becomes the transition reason; LLM escalations become a separate `UsageRecord` so cost attribution is per-stage. (https://tianpan.co/blog/2025-11-03-llm-routing-model-cascades) and (repo-grounded: tree-snapshot L10)
- Cost-minimization design that follows: (1) heuristic pass over the full corpus is effectively free (single-digit ms); (2) only candidates that *abstain* or land in a low-confidence band are escalated to the LLM refinement stage; (3) the LLM stage's output must still pass the *same* SHACL `ClaimGateResult` gate and resolve to an exact char span (mike#5 unique-anchor resolver) before it can advance — the LLM never writes authority directly, it only proposes a candidate that the deterministic gate admits or rejects. This keeps the provenance wall intact (every admitted claim has a `GroundedExtraction.span`). (CAPTURE Cautions: "PROVENANCE WALL … no char-offset-less LLM prompts")

### G. Effect-TS realization and porting/licensing constraints

- The fallback chain is idiomatic Effect: run the deterministic heuristic, and on abstention/low-confidence use `Effect.orElse` / `orElseSucceed` (lazy fallback evaluated only on failure) or `catchTag` to escalate to the LLM stage, with `Schema.decode` validating each stage's output; "a common pattern chains timeout, retry, and orElse fallback." Model the heuristic abstention as a typed error (e.g. `HeuristicUndecided`) so `catchTag` routes precisely to the LLM stage and the trace records which branch fired. (https://effect.website/docs/schema/transformations/) and (https://effect-ts.github.io/effect/effect/Effect.ts.html)
- LICENSING / clean-room (from CAPTURE Cautions, confirmed): **harvest-mcp license is UNKNOWN and the `ParameterClassificationAgent` source is not publicly locatable** — a GitHub search for `harvest-mcp ParameterClassificationAgent` returned only unrelated Harvest time-tracking MCP servers (taiste, southleft, NetApp, …), none containing a parameter-classification agent. Reimplement the pattern clean-room (it is trivial: string-membership/regex test → `{classification, confidence, pattern}`); do NOT vendor. LegalEase#4, doctor, Legal-AI_Project are Python and TalentScore is BAML → port to Effect/TS + effect/Schema, do not vendor the runtimes; `mike` (mike#5 anchor resolver) is AGPL → clean-room only. (https://github.com/southleft/harvest-mcp and CAPTURE.md L106-107)
- Second-source on the regex-first/LLM-fallback combination being *measured* (so the team sizes expectations): a hybrid CMR/CPET clinical extractor used "regular expression based extraction … with a fallback to an LLM if nothing is returned" — useful prior art that the naive "regex-then-LLM-on-empty" wiring can *underperform* regex-only on some fields (reported F1 .152 combined vs .283 regex-only in that study), i.e. the escalation trigger must be confidence/abstention-based, not merely "regex returned nothing." (https://www.medrxiv.org/content/10.64898/2026.01.21.26344531.full.pdf)

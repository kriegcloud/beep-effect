# streaming-gate-vs-langextract-lock

Scope: resolve the Partial(candidate)/Complete(authoritative) Effect-Stream gate (TalentScore#3/#4/#6) vs the langextract V1 streaming-deferral lock — survey Effect v4 Stream tagged-union / async-bridge + schema-backed domain-event streaming, decide the candidate→admitted boundary into `@beep/epistemic-domain`, and keep the gate reconcilable with langextract V1 (schema-first events, never raw AI chunks).

## Findings

### The conflict is a beep V1 *scoping* lock, not an upstream limitation — reconcilable by construction

- The langextract V1 lock is exactly: "Streaming is deferred unless exposed as schema-backed LangExtract domain events; raw AI stream chunks are not public API." (PRIMARY: `goals/langextract-capability/SPEC.md` L88-89). The lock is conditional, not absolute: it bans *raw chunks as public API*, and explicitly *permits* streaming if it is surfaced as schema-backed domain events. The TalentScore gate satisfies that condition, so it does not actually violate the lock — it instantiates the permitted exception. The reason it cannot attach to `goals/langextract-capability` is process/scope (V1 is non-streaming, non-goal list bans CLI/rendering/streaming for V1), not a semantic contradiction.
- Upstream Google `langextract` (the namesake) *does* ship streaming, but on a different axis: document-chunk batching/parallelism for long documents (`max_workers` up to ~10 chunks in parallel), not partial-result token streaming of one extraction (https://github.com/google/langextract, https://developers.googleblog.com/introducing-langextract-a-gemini-powered-information-extraction-library/). So "streaming" upstream = throughput over many chunks; the TalentScore gate = progressive partial reveal of a *single* extraction. These are orthogonal — the gate is genuinely net-new relative to both beep V1 and the upstream lib, which is why it needs this sibling home.

### Effect v4 (beta.91) is the target runtime — the TalentScore `Stream.async` snippet does NOT port verbatim

- The repo pins `effect@4.0.0-beta.91` (PRIMARY: root `package.json`, `node_modules/effect/package.json`). Effect v4 beta was announced 2026-02-18; Stream/Schema/RPC are consolidated into the core `effect` package, Schema is a full rewrite, and evolving APIs (AI, RPC, Schema, HTTP, SQL) live under `effect/unstable/*` which "may receive breaking changes in minor releases" (https://effect.website/blog/releases/effect/40-beta/). The repo's `effect/unstable/rpc/Rpc` imports confirm v4.
- The CAPTURE/TalentScore snippet `Stream.async((emit) => { for await (...) emit.single(...); ...; emit.end() })` is **Effect v3 (BAML port) shape and is removed in v4**. In beta.91 there is **no** `Stream.async` / `asyncPush` / `asyncEffect` export; the emit-callback constructor is `Stream.callback`, and it hands you a **`Queue`**, not an `emit` object (PRIMARY: `node_modules/effect/src/Stream.ts` L777 `export const callback = <A,E,R>(f: (queue: Queue.Queue<A, E | Cause.Done>) => Effect<unknown,E,R|Scope>, options?) => ...`). You push with `Queue.offerUnsafe(queue, value)` and terminate with `Queue.endUnsafe(queue)` (per the in-source `@example`). The web docs at https://effect.website/docs/stream/creating/ still show v3 `Stream.async`/`emit.single` — those are stale for this repo. v4 renamed/removed many constructors and migration docs lag (https://github.com/Effect-TS/effect-smol/issues/1378). VERIFIED two ways: primary source grep shows no `async` export; the migration issue corroborates `asyncEffect → callback`.

### Cleanest v4 bridge for an external async-generator extraction stream is `Stream.fromAsyncIterable`, not `Stream.callback`

- The BAML stream in TalentScore#6 is an async generator (`for await (const partial of bamlStream)`). v4 ships `Stream.fromAsyncIterable(iterable, onError)` (PRIMARY: `node_modules/effect/src/Stream.ts` L1458, signature `<A,E>(iterable: AsyncIterable<A>, onError: (error: unknown) => E) => Stream<A, E>`), which wraps any async generator and folds thrown errors into a typed stream error. This is the idiomatic v4 port of `for await … emit.single`. The terminal "Complete" emit (TalentScore's post-loop `getFinalResponse()` single) is then appended with `Stream.concat`/`Stream.append` rather than a manual `emit.end()`, since `fromAsyncIterable` ends on iterator exhaustion. `Stream.callback`/`Stream.fromQueue` remain the fallback for non-iterable callback sources.

### The repo already owns the canonical schema-backed streaming pattern — this sibling reuses it, does not invent it

- `@beep/agents` already streams **schema-validated domain values** over Effect RPC, never raw chunks: `SendMessageRpc = Rpc.make("SendMessage", { payload, success: AssistantBlock, error: ChatActionError, stream: true })` (PRIMARY: `packages/agents/use-cases/src/processes/Chat/Chat.rpc.ts` L93-98). The `success` schema is the *per-element* schema and each emitted element is a decoded domain value (`AssistantBlock` / `IndexedBlock`). This is the exact "schema-backed domain events, never raw AI chunks" discipline the langextract lock demands — so the reconciliation pattern is a repo precedent, not a new bet.
- v4 RPC semantics confirm the envelope shape for a gate: `stream: true` "wraps the success and error schemas in a stream schema and sets the normal error to never"; the immediate RPC exit success is `void` and **stream errors are folded into the stream**, not the RPC envelope (PRIMARY: `node_modules/effect/src/unstable/rpc/Rpc.ts` L321-345, L383-397, L894-947). Consequence for the gate: a failed extraction surfaces as a typed *stream failure* mid-Partial, while the RPC call itself still "succeeds" by returning the stream. Second source confirms per-element-domain-value semantics + `Stream.runForeach` client consumption (https://dev.to/titouancreach/part-2-how-i-replaced-trpc-with-effect-rpc-in-a-nextjs-app-router-application-streaming-responses-566c).
- The tagged `ParseEvent` union is also a repo precedent: `S.Union([...]).pipe(S.toTaggedUnion("role"))` (PRIMARY: `packages/agents/use-cases/.../AssistantTurn.contracts.ts` L78-83, `S.tag("user")` discriminant) and `LiteralKit([...]).toTaggedUnion("verdict")` for `ClaimGateResult` (PRIMARY: `packages/epistemic/domain/src/values/ClaimGate/ClaimGateResult.model.ts` L99-106). So the gate's `Partial`/`Complete` union is `S.Union([PartialEvent, CompleteEvent]).pipe(S.toTaggedUnion("_tag"))` — idiomatic, already-used machinery.

### The Partial-vs-Final dichotomy (TalentScore#4) is a principled BAML concept that maps onto effect/Schema

- BAML *generates* a "Partial" version of every return type for streaming functions — "all Class fields into nullable fields, filled as much as possible given tokens received so far" — while the final return type is unchanged: "The return type of a function is not affected by streaming attributes!" `getFinalResponse()` "returns the original, fully-validated type — distinct from partial streaming values" (https://docs.boundaryml.com/guide/baml-basics/streaming). `@stream.done` makes a value atomic (only appears once fully formed); BAML auto-treats numbers/literals as `@stream.done` (https://boundaryml.com/blog/semantic-streaming).
- This validates the schema-first port: the **Partial/candidate** model is `S.Struct({ field: S.optionalWith(S.NullOr(T), { exact: true }) })` (progressively-filling, all nullable), and the **Complete/authoritative** model is a strict `S.Class` with all fields required (TalentScore#4's `ResumeData` vs `PartialResumeData`). `@stream.done`-atomic fields map to fields *omitted from the Partial schema entirely* (or kept non-optional only on the Complete) so they never leak half-formed. The candidate confidence/span fields stay optional on Partial, required on Complete — matching langextract's own `ExtractionCandidate` (span-less, `confidence: S.optionalKey`) vs `GroundedExtraction` (carries `span: Contract.Span`).

### DECIDED candidate→admitted boundary surface into `@beep/epistemic-domain`

The sibling owns the **transport gate**; epistemic owns the **admission gate**. They are distinct and must not be conflated:

- **Transport gate (this sibling, `@beep/epistemic-*`-adjacent or a new streaming value module):** an Effect `Stream` of a tagged `ExtractionStreamEvent = S.Union([Partial, Complete]).pipe(S.toTaggedUnion("_tag"))`. `Partial` carries a `PartialExtraction` (nullable, span-optional, presentation-only, **never persisted, never crosses into epistemic**); `Complete` carries the authoritative aligned payload. The `Complete` payload schema should be **identical to langextract V1's batch result** — i.e. `GroundedExtraction[]` + `LangExtractDiagnostics` (PRIMARY: `packages/foundation/capability/langextract/src/Extraction/index.ts` L169-224, L270+). That identity is what makes the gate reconcilable: if streaming is ever pulled into langextract, V1's existing non-streaming result IS the `Complete` event; the only addition is the `Partial` envelope.
- **Admission gate (already built, `@beep/epistemic-domain`):** the terminal `Complete` does **not** mean "admitted". It seeds a `CandidateClaim` at `lifecycle: "candidate"` (PRIMARY: `packages/epistemic/domain/src/entities/CandidateClaim/CandidateClaim.model.ts` — `lifecycle: ClaimLifecycle`, `snapshot: UnknownRecord`), with each `GroundedExtraction.span` → `EvidenceSpan` (TextAnchor + `Confidence`/UnitInterval; PRIMARY: `packages/epistemic/domain/src/values/EvidenceSpan/EvidenceSpan.model.ts`). The existing SHACL `ClaimGate` then returns a `ClaimGateResult` (`admitted` | `rejected` with violations; PRIMARY: `ClaimGateResult.model.ts` L99-106), and only an `admitted` verdict advances the `ClaimLifecycle` `candidate → shape_valid → consistency_checked → admitted` (PRIMARY: `packages/shared/domain/src/values/ClaimLifecycle/ClaimLifecycle.model.ts`). So: **stream `Complete` ⟶ epistemic `candidate` ⟶ ClaimGate ⟶ `admitted`.** The task's "candidate→approved" is the existing `candidate→admitted` shared-kernel lifecycle; no new vocabulary needed.
- **Net effect:** the streaming gate's authority boundary is one-directional — only `Complete` payloads instantiate `CandidateClaim`s; `Partial` events are a UI/transport concern that never touch `@beep/epistemic-domain` or the persisted store. This preserves both the langextract lock (no raw chunks; the on-wire `Complete`/`Partial` are decoded schema values) and the epistemic admission invariant (nothing is authoritative until the SHACL gate admits it).

- Source nuggets are BAML (TalentScore) and Python — port to effect/Schema + effect v4 Stream; do not vendor BAML/Python runtimes (CAPTURE Cautions). `mike` is AGPL, `harvest-mcp` license UNKNOWN → clean-room reimplement (not in this subtopic's direct scope but bounds the sibling). The gate itself (TalentScore) is BAML/TS reference shape only.
- Provenance wall: every authoritative `Complete`/`GroundedExtraction` row must carry source char offsets (`span`/`EvidenceSpan`); the deterministic "a miss is an absent row, never a wrong fact" guarantee and versioned re-extraction (`VERSION` const) apply to the deterministic seeders that may feed the same gate, keeping Partial events advisory and Complete events grounded.

# char-offset-provenance-span-model

Scope: how deterministic-extractor rows and resolved LLM-proposed spans carry verbatim text + char offsets onto `@beep/provenance` TextAnchor / `@beep/epistemic-domain` EvidenceSpan + GroundedExtraction.span / CandidateClaim, enforcing the provenance wall, the matchAll-index→span mapping, and a unique-anchor span resolver (mike#5).

## Findings

### A. The repo already owns the span substrate — three offset shapes, one half-open convention

- `@beep/provenance/TextAnchor` is pure provenance: `{ startChar, endChar: NonNegativeInt, quote: S.String }`, a half-open range `[startChar, endChar)` with the exact quoted substring, plus `isWellOrdered` (`startChar <= endChar`). Documented invariant: given the source, `text.slice(startChar, endChar)` reproduces `quote`. No confidence, no claim semantics — those belong to the consuming slice. (`packages/foundation/modeling/provenance/src/TextAnchor.ts`)
- `@beep/epistemic-domain` `EvidenceSpan` literally spreads `TextAnchorFields` and adds `confidence: UnitInterval` (`[0,1]`). The epistemic layer = a provenance anchor + the extraction judgement; the offset shape stays in foundation so any slice can ground without depending on epistemic. (`packages/epistemic/domain/src/values/EvidenceSpan/EvidenceSpan.model.ts`)
- `@beep/nlp/Handoff/Contract.Span` is a third copy of the same idea: `{ start, end: NonNegativeInt }`, "half-open character span `[start, end)` … zero-based offsets", with a cross-field filter (`start <= end`) and a `start + length` factory. (`packages/foundation/capability/nlp/src/Handoff/Contract.ts:152-207`)
- `@beep/langextract/Extraction.GroundedExtraction` carries `{ alignmentStatus, label, text, span?: Contract.Span, matchedText?, confidence?, attributes? }`. `AlignmentStatus = LiteralKit(["match_exact","match_lesser","match_fuzzy","unaligned"])`; `span` is OPTIONAL (unaligned rows have none). (`packages/foundation/capability/langextract/src/Extraction/index.ts:118, 211-221`)
- `CandidateClaim` is a `BaseEntity` with `{ fixtureKey, lifecycle: ClaimLifecycle, snapshot: UnknownRecord }` — the resolved span rides inside the candidate's evidence/snapshot into the candidate→approved gate; CandidateClaim itself does not re-declare offsets. (`packages/epistemic/domain/src/entities/CandidateClaim/CandidateClaim.model.ts`)
- DESIGN IMPLICATION: there are already three structurally-identical-but-distinctly-named offset shapes (`startChar/endChar`, `start/end`, and Anthropic's `start_char_index/end_char_index` below). The model needs ONE canonical conversion (all half-open, all the same unit — see G) so a deterministic regex row, a resolved LLM span, and a vendor citation all map onto `TextAnchor` without ad-hoc re-indexing.

### B. Anthropic's own primitives corroborate the design (half-open offsets + verbatim quote + fail-on-ambiguity)

- The Citations API grounds answers in source documents and "return[s] the exact passages that support each claim, so you can verify answers." Character-based citations use `type: "char_location"` with `start_char_index` ("0-indexed") and `end_char_index` ("exclusive"), plus `cited_text` (the verbatim passage) and `document_index`. This is structurally identical to `TextAnchor` (`startChar`/`endChar` half-open + `quote`) and is the vendor statement of the provenance-wall goal. (https://platform.claude.com/docs/en/build-with-claude/citations — fields confirmed in the doc's char_location example block)
- The text-editor `str_replace` tool is the canonical unique-anchor resolver: `old_str` "must match exactly, including whitespace and indentation"; the app "should ensure that there is exactly one match." Multiple matches → `"Error: Found 3 matches for replacement text. Please provide more context to make a unique match."`; zero matches → `"Error: No match found for replacement. Please check your text and try again."`; success message is "Successfully replaced text at exactly one location." (https://platform.claude.com/docs/en/agents-and-tools/tool-use/text-editor-tool) — this is exactly mike#5's ambiguous/not-found failure modes, made canonical by the vendor.

### C. LangExtract is the prior art for the alignment ladder — and the repo already ported it (with divergences)

- Google LangExtract grounds every extraction to char offsets: a two-step `WordAligner` tries exact match first, then fuzzy. `AlignmentStatus` enum = `MATCH_EXACT`, `MATCH_LESSER`, `MATCH_FUZZY`, with `None` for unaligned; the exact phase uses `difflib.SequenceMatcher` matching blocks, `MATCH_LESSER` = extraction text longer than the matched span (partial), fuzzy accepts when token overlap ≥ `fuzzy_alignment_threshold` (default **0.75**). Unaligned extractions get `char_interval = None` / `alignment_status = None` and are filtered with `[e for e in result.extractions if e.char_interval]`. (https://github.com/google/langextract/blob/main/langextract/resolver.py ; https://github.com/google/langextract/issues/246 ; https://deepwiki.com/google/langextract)
- The repo's `@beep/langextract/Alignment.alignCandidate` is the Effect/TS port of this ladder: `findExact` (`sourceText.indexOf(query)`) → `match_exact`; `findLesser` (lowercase-normalized with a per-source-offset remap) → `match_lesser`; `findFuzzy` (word-window + Levenshtein `similarity` ≥ threshold, default **0.82**) → `match_fuzzy`; else `unaligned`. It is bounded (`MAX_FUZZY_SOURCE_LENGTH 100_000`, `MAX_FUZZY_QUERY_LENGTH 4_096`, `DEFAULT_MAX_EXTRACTIONS 256`) and "fails closed to `unaligned`" past those bounds. (`packages/foundation/capability/langextract/src/Alignment/index.ts:17-247`)
- DIVERGENCES to reconcile, not re-derive: (1) the repo REDEFINED `match_lesser` to mean *case-insensitive normalized match*, whereas LangExtract's `MATCH_LESSER` means *extraction longer than matched text* — same label, different semantics; (2) threshold 0.82 vs LangExtract 0.75; (3) Levenshtein-ratio similarity vs LangExtract's `difflib`/LCS. None of these is wrong, but the span model should document which semantics `match_lesser` carries so downstream consumers (law-practice `IrToLaw`) interpret it correctly.

### D. THE GAP this subtopic must close: the exact rung silently takes the FIRST match (no uniqueness / no ambiguous failure)

- `alignCandidate` resolves the exact rung via `sourceText.indexOf(query)`, which returns the FIRST occurrence and never checks for a second. There is no `ambiguous` member in `AlignmentStatus` and no not-found-vs-ambiguous distinction (both collapse to `unaligned` / first-hit). (`packages/foundation/capability/langextract/src/Alignment/index.ts:77-80, 231-247`)
- mike#5 and Anthropic `str_replace` both REQUIRE uniqueness: a `find` string that occurs 0 times → `not_found`; >1 times → `ambiguous`, resolved by widening with `context_before` / `context_after` until the anchor is unique. The unique-anchor resolver therefore needs: (a) count occurrences of the normalized `find`; (b) if exactly 1 → span; (c) if >1 → try `context_before + find + context_after` as the search key and re-count; if still >1 → fail closed with the explicit ambiguous error ("Add longer context_before / context_after so the anchor is unique"); (d) if 0 → `not_found` ("Re-read … copy context verbatim including punctuation & whitespace"). This is netNew #10 and is the piece NOT in the current port. Recommend adding `ambiguous` + `not_found` (or a richer status union) rather than overloading `unaligned`.

### E. matchAll index → span mapping (the deterministic-regex path; Juris.AI#3 / doc-haus#3)

- `String.prototype.matchAll(/…/g)` yields match objects exposing `match.index` (start offset) and `match[0]` (full match); the span is `[match.index, match.index + match[0].length)` — which maps straight onto `Contract.Span.make({ start, end })` or the existing `start + length` factory. The `g` flag is mandatory (a non-global regex throws `TypeError`), and matchAll internally clones the regex so `lastIndex` is isolated and the iterator is single-use. (https://developer.mozilla.org/en-US/docs/Web/JavaScript/Reference/Global_Objects/String/matchAll)
- To anchor a *sub-span* (e.g. the defined term INSIDE the curly quotes in doc-haus's `TERM` regex, not the quotes), add the `d` flag (`RegExp.prototype.hasIndices`): the match array then carries an `indices` property where `match.indices[n]` is `[start, end]` for capture group `n` and `match.indices.groups.name` for named groups. The `d` flag "does not change … matching behavior … only provides additional information," so it is safe to add to every extractor regex. (https://developer.mozilla.org/en-US/docs/Web/JavaScript/Reference/Global_Objects/RegExp/hasIndices ; https://github.com/tc39/proposal-regexp-match-indices) — note the repo currently has NO `matchAll`/`d`-flag span extractor; this is genuinely net-new helper surface (a `spanFromMatch(regex, source) => ReadonlyArray<{ span, quote }>` over the `d` flag), not a reuse.

### F. Whitespace / punctuation tolerance via normalized-text-with-offset-map (preserve the TextAnchor invariant)

- The canonical tolerant-match technique (mirroring the repo's own `lowerWithSourceOffsets`): build a normalized copy of the source while recording, for every normalized character, the ORIGINAL source `start`/`end` offset; run the match in normalized space; remap the normalized hit back to original offsets so the returned `quote` re-slices verbatim from the ORIGINAL text (`text.slice(startChar,endChar) === quote` must still hold). The repo already does exactly this for lowercasing (`lowerWithSourceOffsets`, `findLesser`); the design extends the same index-map to whitespace-collapse (multiple spaces/newlines/tabs → single space) and optional punctuation-fold. (`packages/foundation/capability/langextract/src/Alignment/index.ts:82-128`; technique corroborated generally — pre-fuzzy normalization of "case, whitespace, and punctuation" is standard, e.g. https://medium.com/@praster1/text-normalization-1-whitespace-differences-9fc979487013)
- Hard rule from Anthropic `str_replace`: the matcher itself is whitespace/indentation-SENSITIVE; tolerance must therefore be implemented by normalizing on a COPY and always returning the original offsets — never by mutating the source or returning normalized offsets, which would corrupt the anchor. (https://platform.claude.com/docs/en/agents-and-tools/tool-use/text-editor-tool)

### G. Offset-unit portability gotcha: UTF-16 code units (JS) vs Unicode code points (Python/LangExtract/most tools)

- JS strings index in UTF-16 code units: `.length`, `indexOf`, and `slice` all count code units, so an astral character (emoji, rarer CJK-ext, some math symbols) is 2 units and occupies 2 indices. Python 3 (LangExtract) indexes in Unicode CODE POINTS, where the same character is 1 index. The same numeric span therefore points at different logical positions across the two runtimes whenever astral characters are present. (https://developer.mozilla.org/en-US/docs/Web/JavaScript/Reference/Global_Objects/String ; https://docs.python.org/3/howto/unicode.html)
- Consequences for this model: (1) a char span produced by LangExtract-Python or returned in a model citation is NOT byte-for-byte safe to feed into a JS `text.slice` — the design must DECLARE its canonical unit (recommend UTF-16 code units, since `TextAnchor`'s invariant is JS-`slice`-based and JS-native) and re-validate `slice===quote` on ingest rather than trusting foreign offsets; (2) the repo's own Alignment is internally INCONSISTENT — `levenshtein`/`similarity` operate on code points (`[...value]` / `toCodePoints`) while `findExact`/`findLesser`/`matchAll` operate on UTF-16 (`indexOf`, `.length`) — so similarity scoring and offset math disagree on astral input. Pick one unit and apply it on both axes.

### H. The provenance wall is half-built; generalize the existing filter

- The wall already exists at one boundary: `@beep/langextract/Handoff.definedExtractions` filters to only extractions where `span !== undefined` before minting NLP `Contract.Entity` rows — an unaligned LLM extraction never becomes an entity. (`packages/foundation/capability/langextract/src/Handoff/index.ts`)
- Generalize this into THE wall: any boundary that mints a `CandidateClaim` (or any authoritative row) must require a resolved span (`TextAnchor`/`EvidenceSpan`); an LLM-proposed claim that resolves to `ambiguous`/`not_found`/`unaligned` is REJECTED with a typed error (`alignment-failed` already exists in `LangExtractErrorReason`), never persisted with a synthesized or null offset. This encodes doc-haus's "a miss is an absent row, never a wrong fact" and CAPTURE's "no char-offset-less LLM prompts": the LLM may propose `find` text, but only the deterministic resolver may mint the offset, and only a unique resolution crosses the wall.

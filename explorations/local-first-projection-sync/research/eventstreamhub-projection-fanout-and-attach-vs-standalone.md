# eventstreamhub-projection-fanout-and-attach-vs-standalone

Scope: external landscape for refreshing local-first FalkorDB/UI projections after an authority write — server-push fan-out hub (scoped Effect.Service + per-user mailbox registry) vs Electric SQL / PowerSync / Postgres LISTEN-NOTIFY / CRDT-Yjs / FalkorDB keyspace notifications — plus the load-bearing DECISION: attach the hub to the existing professional-desktop RPC/ThreadStore surface vs stand up a standalone packet.

## Findings

### The design shape being ported (TalentScore EventStreamHub, port shape only)

- The captured nugget describes `EventStreamHub` as a scoped `Effect.Service` holding a `SynchronizedRef<MutableHashMap<UserId, ActiveConnection[]>>` of Mailboxes, with `register`/`unregister`/`notifyUser`/`notifyCurrentUser`; `notifyUser` fans a typed event out to all of a user's live WebSocket connections via `conn.mailbox.offer(event)` and prunes dead mailboxes (`Clock.currentTimeMillis`-gated). Upstream TalentScore is commercial-licensed — port the design shape, not the code (source: `explorations/local-first-projection-sync/CAPTURE.md` lines 33, 49).
- This is a **targeted server-push fan-out hub**: it is NOT a sync engine, NOT a CRDT, and NOT a database change-feed. It is an in-process registry of live client connections plus a typed `notify(userId, event)` entry point that the authority write path calls after it commits. Everything below is evaluated against that role.

### Effect-native primitives the shape maps onto (all in-repo or first-party)

- `effect/Mailbox` is an experimental "asynchronous queue that can have a done / failure signal," added in Effect 3.8, with `offer()`/`offerAll()`, `takeAll` (returns messages + completion status), `end` (success completion), `fail()` (error completion), and `Mailbox.toStream()` to expose it as a consumable Stream; "useful when you want to communicate between effects, and have a way to determine that the communication is complete" (source: https://effect.website/blog/releases/effect/38/). This is exactly the `conn.mailbox.offer(event)` primitive in the captured shape — one Mailbox per live connection, drained as a Stream by that connection's streaming RPC handler.
- `PubSub` (the alternative) broadcasts every published message to **all** subscribers (bounded applies backpressure; dropping/sliding/unbounded trade delivery for publisher throughput), and `PubSub.subscribe` yields a scoped `Dequeue`; it differs from `Queue`, where each value goes to exactly one consumer (source: https://effect.website/docs/concurrency/pubsub/). Implication: a single global `PubSub` would broadcast-then-filter every event to every connection, whereas the captured `MutableHashMap<UserId, Mailbox[]>` registry does **targeted** fan-out (`notifyUser` reaches only that user's connections) — the per-user registry is the right primitive for authority→projection refresh; `PubSub` is the right primitive only if you later want topic/broadcast semantics.
- The concurrency primitives the shape needs already ship; only `MutableHashMap` is in-repo-proven. `MutableHashMap` appears in `packages/foundation/capability/nlp/src/Graph/*`, `packages/tooling/library/repo-utils/src/TSMorph/TSMorph.service.ts`, and `packages/foundation/modeling/schema/test/MutableHashMap.test.ts` (source: repo ripgrep, 2026-06-29). **Correction (Codex gate-1, 2026-06-29):** `SynchronizedRef` is *not* used anywhere in repo `src` (zero matches via `rg -n "SynchronizedRef" packages apps --glob '**/src/**/*.{ts,tsx}'`); the module exists in Effect v4 (`effect/SynchronizedRef`) but the hub would be its first repo user. `Clock.currentTimeMillis` gating maps to a periodic prune fiber, but the idiomatic Effect teardown is Scope-based: register in the streaming-RPC handler's `Scope`, `unregister` via a finalizer on connection close, with the time-gate kept only as a half-open-socket backstop.

### The existing transport this would attach to (load-bearing local fact)

- The professional-desktop chat surface **already speaks streaming server-push RPC**: `SendMessageRpc` and `EditMessageRpc` are declared with `stream: true` in `packages/agents/use-cases/src/processes/Chat/Chat.rpc.ts` (the `ChatRpcs` `RpcGroup`), and the client transport is `RpcClient.layerProtocolSocket()` over ndjson framing on a Tauri IPC socket (`apps/professional-desktop/src/transport/IpcChatClient.ts`). So a proven server→client streaming channel exists today.
- Effect RPC supports streaming responses and server push as a first-class feature: the `@effect/rpc` rewrite "add[ed] support for streaming responses," a handler can return a `Stream`, and the server exposes WebSocket transport via `layerProtocolWebsocket` (raw sockets via `layerProtocolSocketServer`) while the client uses `RpcClient.layerProtocolSocket`; the WebSocket protocol "enables bidirectional streaming and server push" (sources: https://github.com/Effect-TS/effect/blob/main/packages/rpc/README.md ; https://effect.website/blog/releases/effect/23/ ; https://www.npmjs.com/package/@effect/rpc). A `SubscribeProjectionEvents` streaming RPC whose handler returns a queue-backed `Stream` (the queue registered into the hub for that connection's Scope) reuses the existing **transport and serialization** with no new transport machinery. **Correction (Codex gate-1, 2026-06-29):** "drop-in / no scope-teardown machinery" overstates it — connection lifecycle (`Scope`-based register/unregister + finalizer tests), identity, event schema/authorization, backpressure, and the post-commit hook across both `ThreadStore` impls are real design work; see the implementation-surface checklist in `RESEARCH.md` Constraints.
- Effect platform WebSocket transport (`BrowserSocket`/`layerWebSocket`) is already exercised in-repo (`packages/foundation/capability/observability/src/server/DevTools.ts`), confirming the websocket fan-out substrate exists if the desktop ever needs a real WS instead of the Tauri IPC socket (source: repo ripgrep, 2026-06-29).
- Verified-absent today: zero `EventStreamHub`, zero `PubSub`, zero `Mailbox`, and no `notifyUser` exist in `packages/**/src` or `apps/**/src` (source: repo ripgrep, 2026-06-29). The hub is genuinely net-new; its primitives are not.

### Alternative 1 — ElectricSQL (read-path sync engine)

- Electric is a **read-path** Postgres sync engine: an Elixir service consumes the Postgres logical-replication stream and fans it out into "Shapes" (declarative filtered subsets of a table) that clients consume over an HTTP API that is CDN-cacheable for fan-out to "millions of concurrent users"; **writes go through your own backend API — Electric does not handle the write path at all** (sources: https://electric-sql.com/products/postgres-sync ; https://electric-sql.com/docs/guides/shapes ; https://github.com/electric-sql/electric).
- Status/licensing: GA 1.0 released 2025-03-17; Apache-2.0 licensed; `@electric-sql/client` latest 1.5.16; self-hostable via Docker; pairs natively with PGlite for the local store (sources: https://electric-sql.com/blog/2025/03/17/electricsql-1.0-released ; https://github.com/electric-sql/electric ; https://www.npmjs.com/package/@electric-sql/client ; https://pglite.dev/docs/sync).
- Fit for this wedge: Electric keeps a **SQL (PGlite/SQLite) projection** fresh, not a FalkorDB graph projection, and requires (a) Postgres logical replication and (b) a separate Elixir service in the loop. The desktop authority store today is PGlite-in-sidecar (`apps/professional-desktop/src/runtime/Pglite.ts`), single-workspace, single-user — Electric's CDN fan-out value is irrelevant at one user, and it does nothing for the graph projection. Heavyweight relative to the appetite; relevant only if the architecture later centralizes on a shared Postgres with many devices.

### Alternative 2 — PowerSync (bidirectional partial-replication sync engine)

- PowerSync is a service + client-SDK sync engine: the service consumes a Postgres logical-replication slot (row-level INSERT/UPDATE/DELETE), partitions data into "buckets" via YAML **Sync Rules**, and streams to a client SQLite DB using a server-authoritative **checkpoint** model for consistency; unlike Electric it also handles **write-back** (bidirectional) (sources: https://docs.powersync.com/architecture/powersync-service ; https://www.powersync.com/blog/sync-rules-from-first-principles-partial-replication-to-sqlite ; https://powersync.com/).
- Status/licensing: the server-side Service + CLI are under the **Functional Source License (FSL)** — source-available, "eventually open-source," converting to Apache-2.0 on the second anniversary of each release; the client SDKs are Apache-2.0/MIT; PowerSync Service v1.23.1 is the latest release (2026-06-25), following v1.23.0 (2026-06-20) and v1.22.0 (2026-06-04); SOC 2 + HIPAA reached Jan 2026 (sources: https://powersync.com/legal/fsl ; https://powersync.com/blog/new-open-era-for-powersync ; https://powersync.com/open-source ; https://releases.powersync.com/announcements/powersync-service). [Versions refreshed Codex gate-1, 2026-06-29.] The FSL non-compete clause matters if PowerSync were ever embedded in a competing commercial offering.
- Fit: same shape as Electric (SQL projection, Postgres logical replication, separate service), plus write-back you do not need for one-way authority→projection refresh. Over-scoped for the current desktop; a candidate only under a multi-device, multi-user roadmap.

### Alternative 3 — Postgres LISTEN/NOTIFY (DB-native trigger, no new infra)

- Built into Postgres, no new service. Hard constraint: in the default configuration the NOTIFY **payload must be < 8000 bytes**, and the docs explicitly recommend "put it in a database table and send the key of the record" for anything larger (source: https://www.postgresql.org/docs/current/sql-notify.html ; corroborated https://www.stacksync.com/blog/beyond-listen-notify-postgres-request-reply-real-time-sync).
- Pooling gotcha: LISTEN/NOTIFY is **incompatible with PgBouncer transaction-pooling mode** — LISTEN state is connection-specific, so it works only in session pooling (which gives near-zero pooling benefit), or you dedicate a session-mode connection to the listener (sources: WebSearch synthesis of https://github.com/pgbouncer/pgbouncer/issues/655 ; PgBouncer pooling-mode docs). Notifications are also delivered only to currently-connected listeners (events during a disconnect are lost).
- Fit: an excellent **server-side trigger** to wake the hub (authority commit → `NOTIFY projection_changed, '<key>'` → a single session-mode listener fiber → `hub.notifyUser`), but it does NOT by itself reach the desktop client; it terminates inside Postgres. It is complementary to, not a replacement for, the fan-out hub. Note: the desktop authority store is PGlite, not a pooled Postgres, so today the hub can be invoked directly from the `ThreadStore` write path without LISTEN/NOTIFY at all — LISTEN/NOTIFY only earns its keep once writes originate in a shared server Postgres.

### Alternative 4 — CRDT / Yjs (multi-writer convergence, wrong problem)

- Yjs is the production-default CRDT (~920K weekly downloads, ~17K stars) with a `y-websocket` provider for a central authenticated server, plus a separate **Awareness CRDT** for ephemeral presence/cursors (sources: https://github.com/yjs/yjs ; https://docs.yjs.dev/ecosystem/connection-provider/y-websocket ; https://www.pkgpulse.com/guides/yjs-vs-automerge-vs-loro-crdt-libraries-2026).
- Fit: CRDTs solve **concurrent multi-writer conflict-free convergence**, not single-authority read-projection invalidation. The desktop chat SPEC explicitly scopes out that problem: "No collaboration; no multi-user presence" (source: `goals/desktop-chat-surface/SPEC.md` Non-Goals). Adopting Yjs would import a merge/awareness model with no current consumer; it is the wrong shape for "refresh my local projection after my own authority write." Park as a future option only if real-time co-editing of threads/graphs is ever in scope.

### Alternative 5 — FalkorDB / Redis keyspace notifications (cannot be relied on for graph mutations)

- FalkorDB is a Redis module that stores a graph under a custom module datatype; its query command `GRAPH.QUERY` has **no streaming/subscribe/change-feed semantics** (it caches execution plans and returns result sets) — the design docs cover GraphBLAS adjacency-matrix internals and do not mention keyspace notifications, change feeds, pub/sub, or subscription to mutations (sources: https://docs.falkordb.com/commands/graph.query.html ; https://docs.falkordb.com/design/ ; https://github.com/FalkorDB/FalkorDB).
- Redis keyspace notifications are **disabled by default** (`notify-keyspace-events`), and Redis Pub/Sub is **fire-and-forget**: "if your Pub/Sub client disconnects, and reconnects later, all the events delivered during the time the client was disconnected are lost" (source: https://redis.io/docs/latest/develop/pubsub/keyspace-notifications/).
- Decisive limitation for graph projections: for **custom module datatypes**, keyspace notifications are NOT auto-generated the way they are for built-in commands — the module must explicitly call `RedisModule_NotifyKeyspaceEvent()` in its command handlers to emit events (source: https://github.com/redis/redis/issues/8782 plus the Redis modules keyspace-notification API). There is no documented evidence FalkorDB emits per-node/per-edge events on `GRAPH.QUERY` writes; at best a coarse key-level event for the whole graph key, and only if FalkorDB opts in. Conclusion: **you cannot use FalkorDB as the change source** — the application's authority write path must emit the projection-refresh event itself. This is the single strongest argument that the refresh signal belongs in an app-level hub, not a DB-native feed.

### Bonus alternative — Rocicorp Zero (query-driven sync, heaviest adoption)

- Zero reached 1.0 in June 2026; it is query-driven (its own ZQL language, its own `zero-cache` server, its own protocol, server-run permissions, and rebase-on-conflict), pairing a client lib with a read-only Postgres cache (sources: https://www.infoq.com/news/2026/06/zero-version-1/ ; https://zero.rocicorp.dev/ ; https://zero.rocicorp.dev/docs/when-to-use). Treated by adopters as a "multi-quarter platform commitment." Out of scope for the current wedge; noted for landscape completeness.

### The load-bearing DECISION — attach vs standalone

- **Recommendation: ATTACH** a thin `EventStreamHub` as a scoped server-side `Effect.Service` in `@beep/workspace-server` (next to `ThreadStore`, the authority write boundary), exposed via a `stream: true` subscription RPC added to the existing agents/workspace RPC surface (sibling to `ChatRpcs`), and wired into the app-local live Layer (`apps/professional-desktop/src/runtime/Layer.ts`). Do **not** stand up a standalone packet, and do **not** adopt a sync engine (Electric/PowerSync/Zero) at this stage. Rationale below.
- The transport already exists and is proven: `ChatRpcs` already carries `stream: true` server-push turns over the Tauri IPC socket (`Chat.rpc.ts`; `IpcChatClient.ts`), and Effect RPC streaming returns a `Stream` from a handler (https://github.com/Effect-TS/effect/blob/main/packages/rpc/README.md). A queue-backed (`Stream.fromQueue`) subscription RPC reuses all of it — the *transport* is proven, so standing up a standalone transport packet is duplication; attach remains the right call, but "a few files" understates the lifecycle/identity/authorization design work (see `RESEARCH.md` implementation-surface checklist, Codex gate-1).
- The authority boundary already exists: `ThreadStore` (`packages/workspace/use-cases/src/aggregates/Thread/ThreadStore.ts`) is the single persistence boundary for `appendTurn`/`createThread`/`setTitleIfEmpty`. Per the authority/projection/cache concern flagged in CAPTURE, the hub must be a **consumer of** that committed write (call `hub.notifyUser` after the transaction), not a second source of truth — co-locating it in `@beep/workspace-server` keeps that wiring local and honors repo law "Prefer service composition over global state," "no God Layers," and app-local live Layer composition (`goals/desktop-chat-surface/SPEC.md` Constraints).
- The appetite is small and single-user today: the in-repo `local-first-voice` research loop established that "professional-desktop is single-workspace chat only," and the same exploration deliberately **right-sized away from a full slice** to an attached spike ("Right-size … the highest value × proximity is voice in the chat composer"; "Prototype in a spike branch; do NOT graduate a goal packet yet") (source: `explorations/local-first-voice/DECISIONS.md` 2026-06-27 loop entries). That is direct precedent for attaching this wedge rather than graduating a standalone packet. At one user with N windows, the per-`UserId` fan-out map is degenerate (one key) — a thin hub suffices; a sync engine is unjustified.
- Slice scoping is satisfied by attach: per auto-memory, frontend slice code lives in `packages/{slice}/client`+`ui` and `apps/todox`/`professional-desktop` is "shell, routing, and cross-slice composition only." The hub's **service + RPC contract** belong in the workspace slice (`server` + `use-cases`); only the **live Layer wiring** and the client subscription belong in `apps/professional-desktop`. This is an attach across two existing surfaces, not a new home.
- **When standalone (or a sync engine) would be warranted instead** — keep the hub behind a small port so the swap is cheap: (a) multi-device / multi-user fan-out where one user's write must reach other users' devices → reconsider PowerSync (bidirectional) or Electric (read-path) over a shared Postgres; (b) a generic cross-slice event bus consumed by many slices → promote the hub to a shared-kernel service with its own packet; (c) real-time co-editing of threads or the graph → Yjs/CRDT, a different architecture entirely. None of these is in the current appetite, so they are explicit no-gos for now, logged as the removal/escalation conditions.
- Coordinate explicitly with: `goals/desktop-chat-surface` (owns the RPC chat surface + app-local Layer; the subscription RPC lands beside `ChatRpcs`), `goals/workspace-thread-domain` (owns `ThreadStore`, the write hook site), `apps/professional-desktop` (PGlite runtime + transport + client subscription), and the authority/projection/cache standard (hub is the invalidation signal, downstream of the committed authority write). The FalkorDB projection refresh rides the same `notifyUser` event — since FalkorDB cannot self-emit reliable change events, the authority write path emits one typed event that both the UI projection and the FalkorDB projection consume.

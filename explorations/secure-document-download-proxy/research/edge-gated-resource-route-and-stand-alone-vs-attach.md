# edge-gated-resource-route-and-stand-alone-vs-attach

Scope: map the edge-gated `GET /resources/:file` security model (strict v4 UUID + `.pdf` guard, `cache-control: private, no-store`, 404-on-expired) onto a Tauri/Effect sidecar, and decide whether the secure-download proxy should STAND ALONE as a new packet or ATTACH to an existing surface (m365-driver / file-processing-capability / professional-desktop / @beep/uspto).

## Findings

### A. The edge-route security model (what the nugget actually encodes)

- **Strict v4 UUID guard before any fs access.** The canonical strict regex is `^[0-9a-f]{8}-[0-9a-f]{4}-4[0-9a-f]{3}-[89ab][0-9a-f]{3}-[0-9a-f]{12}$`: the version nibble is pinned to `4` and the variant nibble to `[89ab]`, so non-canonical / wrong-version UUIDs are rejected (https://www.dev-toolbox.tech/tools/regex-cheat-sheet/examples/regex-for-uuid-validation). The version/variant nibble positions are normative in RFC 4122 (updated by RFC 9562 for v6–v8) (https://www.dev-toolbox.tech/tools/uuid-generator/examples/uuid-regex). **Adversarial caveat (verified):** the same source warns "regex can validate the format, but it cannot guarantee uniqueness, security or that a UUID was generated correctly… for strict validation, always parse using a trusted library" — so the regex is a cheap gate, not the authorization itself. In Effect this maps to a branded `Schema` refinement (pattern + brand), with decode failure short-circuiting to 404.
- **`.pdf` suffix guard before fs access** narrows the filename to a single content type and, combined with the UUID-only basename, removes any attacker-controlled path component — the structural defense against directory traversal. This is the *first* check, before touching the filesystem ("validates `.pdf` suffix + strict v4 UUID before any fs access", CAPTURE.md nugget patents-mcp-server#11).
- **`Cache-Control: private, no-store`** — RFC 9111 normative text: `no-store` = "a cache MUST NOT store any part of either the immediate request or the response and MUST NOT use the response to satisfy any other request"; unqualified `private` = "a shared cache MUST NOT store the response (i.e., the response is intended for a single user)" (https://www.rfc-editor.org/rfc/rfc9111.html). The load-bearing directive for private PDFs is `no-store`; `private` is belt-and-suspenders against shared/intermediary caches. `no-store` also disables the browser back/forward cache (BFCache), because restoring from BFCache requires keeping a memory copy that contradicts "do not store" (https://developer.mozilla.org/en-US/docs/Web/HTTP/Reference/Headers/Cache-Control). Note `no-store` ≠ `no-cache`: `no-cache` permits storage but forces revalidation (same MDN/RFC 9111 sources).
- **404 on expired/missing** is deliberate opacity: an expired link and a never-existed link return the same status, leaking no existence signal. Combined with the opaque UUID (no business identifiers in the URL), this is the "access control at the edge" half.
- **"Access control at the edge; UUID+TTL is in-app defense-in-depth"** describes a two-layer model: the network boundary enforces who-can-reach-the-route; the UUID+TTL lookup enforces what-resolves even if the boundary is bypassed. In the original Python implementation the "edge" was a real HTTP proxy server — `SecureLinkCache.generate_persistent_link(..., base_url="http://localhost:8080")` mints URLs against a localhost HTTP listener (john-walkoe/uspto_pfw_mcp `secure_link_cache.py`, https://github.com/john-walkoe/uspto_pfw_mcp). That repo is **MIT-licensed** (verified via `gh api .../license` → `MIT`), so the link-cache half is portable; the edge-route half is generic Express-style middleware that does not need verbatim porting.

### B. Mapping the "edge" onto a Tauri/Effect sidecar (the boundary type changes)

- **The desktop today has NO HTTP edge.** `apps/professional-desktop/src-tauri/Cargo.toml` declares only `tauri-plugin-{shell,updater,log}` — no `tauri-plugin-localhost`, no `register_uri_scheme_protocol`, no HTTP listener (grep over `src-tauri/` for `register_uri_scheme|asset_protocol|http` returned nothing). The renderer↔sidecar transport is **IPC stdio (ndjson rpc frames)**: `apps/professional-desktop/server/IpcStdoutGuard.prelude.ts` documents that the Bun sidecar's stdout *is* the protocol frame stream the Tauri Rust shell parses, and `src/transport/{TauriIpcSocket.ts,IpcChatClient.ts}` carry the channel. So the Express `GET /resources/:file` does not map 1:1 — there is no edge router to hang middleware on.
- **Three candidate boundaries in Tauri, with different security postures:**
  1. **Custom URI-scheme protocol** (`register_uri_scheme_protocol` / async variant) — registers a scheme handler available to all webviews via `setURLSchemeHandler` (macOS), `AddWebResourceRequestedFilter` (Windows), `webkit-web-context-register-uri-scheme` (Linux); the handler returns `Result<HttpResponse, …>` and **can set response headers (Cache-Control) and return 404** (https://deepwiki.com/tauri-apps/wry/4.1-custom-protocols, https://docs.rs/tauri/latest/tauri/struct.Builder.html). The sync form blocks the main thread on I/O; `register_asynchronous_uri_scheme_protocol` spawns async tasks for streaming reads (https://github.com/tauri-apps/tauri/issues/5759). This is the closest analog to the Express edge: the strict-UUID + `.pdf` guard and the TTL lookup run in the handler *before* fs access, exactly preserving the nugget's ordering.
  2. **Asset protocol + `convertFileSrc`** — static file exposure gated by `app.security.assetProtocol.{enable,scope}` globs in `tauri.conf.json` (https://v2.tauri.app/security/asset-protocol/). Wrong tool for this job: it exposes *filesystem-path scopes* to the webview rather than gating per-request; it has no TTL, no opaque-ID indirection, and no per-response `Cache-Control` control. Tauri docs explicitly caution against permissive scopes like `**/*` and default `requireLiteralLeadingDot: true` to avoid leaking dot-dirs — confirming it is built for known static assets, not dynamic private PDFs.
  3. **`tauri-plugin-localhost`** — runs an actual localhost HTTP server, which would host the Express-style route most directly, but Tauri's own docs say it "brings considerable security risks… If in doubt, use the default custom protocol implementation" (https://v2.tauri.app/plugin/localhost/). Avoid for privileged document serving.
- **Effect side already has the route shape in-repo.** `effect/unstable/httpapi` `HttpApiEndpoint` is used by `packages/drivers/govinfo/src/domain/contracts/Search/Search.http.ts`, and `HttpApiBuilder`/`HttpServerResponse` by `packages/foundation/capability/observability/src/server/*`. So a schema-validated path-param route (branded strict-UUID decode → typed 404, `HttpServerResponse` with `Cache-Control: private, no-store`) is idiomatic; whether it is hosted behind a Tauri custom protocol or the IPC channel, the guard logic is the same Effect pipeline.
- **The origin-fetch half already exists in `@beep/uspto`.** `Uspto.service.ts` exposes `downloadDocument(downloadUrl: string): Effect<Uint8Array, UsptoError>` and `getDocuments(...) → UsptoDocumentReference[]` where each ref carries an optional `downloadUrl`; the API key is kept server-side via `Config`/`Redacted` (imports `Redacted` from `effect`). What is missing is (a) minting opaque, TTL-gated local handles and (b) the gated local serve that hands `Uint8Array` to the webview without leaking the USPTO `downloadUrl`/IDs. **No keychain/DPAPI/Fernet analog exists in-repo** (grep for `keychain|keyring|dpapi|encryptedsecret|aes.?gcm` → none); the repo only uses in-memory `Redacted`. The encrypted SecureLinkCache is therefore net-new (and is the sibling subtopic).

### C. Routing inventory — STAND ALONE vs ATTACH

- **ATTACH → goals/m365-driver "document route": poor fit.** `goals/m365-driver/SPEC.md` is a *read-only Microsoft Graph driver*; it lists "download `driveItem` content" as a fetch verb, but its Non-Goals explicitly defer delivery: "No document-portal ingest wiring, matter mapping, or persistence (that is the `m365-document-ingest` follow-on)." There is no serve-route or local edge in m365 — only a cloud content-download verb. Attaching a local TTL-gated serve here couples a delivery concern to one cloud driver and crosses a Non-Go.
- **ATTACH → goals/file-processing-capability: poor fit.** `goals/file-processing-capability/SPEC.md` is an *extraction substrate* (Tika/libpff → typed extraction artifacts); P1 vertical proof complete, remaining work is parser breadth/hardening. It has zero download/serve/route scope (grep for `download|serve|/resources|cache-control|ttl` → none). Different concern: turning bytes into artifacts, not delivering authoritative PDFs to a UI.
- **ATTACH → @beep/uspto File-Wrapper path: loose-couple only, don't host.** USPTO is the *first concrete origin* (it already has `downloadDocument`), but the gating pattern is provider-agnostic — the identical opaque-link + edge-serve would front m365 `driveItem` content and govinfo PDFs. Burying the proxy inside the `@beep/uspto` driver would couple a cross-cutting delivery/edge capability to one origin and violate the repo's driver-boundary rule (a driver wraps an external API; it does not own a desktop serve route).
- **ATTACH → apps/professional-desktop "secure route": the wiring host, not the home.** No such route exists yet (net-new boundary: Tauri custom-protocol/IPC). The desktop should *wire* the capability (register the protocol handler / IPC verb), but the reusable mint+gate logic should not live as bespoke app code.
- **RECOMMENDATION — STAND ALONE (small, provider-agnostic capability packet).** Make the secure-document-download proxy its own thin capability: (1) opaque, TTL-gated handle mint + lookup (the SecureLinkCache analog, encrypted-at-rest — net-new), (2) the edge-gate (strict-UUID + `.pdf` + `private, no-store` + 404-on-expired) expressed as an Effect route, consuming the existing `@beep/uspto` download verb first and `@beep/m365` content verb later, with `apps/professional-desktop` wiring the Tauri custom protocol (or IPC) boundary. This keeps the driver boundaries clean, lets one gate serve all origins, and avoids the two Non-Go attachments (m365 portal wiring, file-processing scope). It pairs with — but is distinct from — the sibling "encrypted opaque TTL link cache" subtopic.

## Open / Unverified

- **patents-mcp-server source repo identity UNVERIFIED.** The CAPTURE nugget cites `patents-mcp-server src/resources/routes.ts:17-36` (TypeScript), but `gh search` over candidate repos (riemannzeta/patent_mcp_server, openpharma-org/patents-mcp, etc.) found no TS `resources/routes.ts`; those are Python FastMCP servers. The repo is likely private/renamed. The edge-route *pattern* is fully captured in the nugget and is generic Express middleware, so re-porting from source is unnecessary — but the exact upstream license for that route file is UNVERIFIED (the link-cache half from john-walkoe/uspto_pfw_mcp is confirmed MIT).
- **Tauri custom-protocol `Cache-Control` reach into webview cache UNVERIFIED for the specific WKWebView/WebView2/WebKitGTK matrix.** Setting `no-store` on a custom-protocol response is correct in principle, but whether each platform webview honors `no-store`/BFCache suppression for custom-scheme responses (vs http(s)) was not confirmed against a primary per-platform source; treat as needs-a-spike.
- **IPC vs custom-protocol choice for the desktop is a design decision, not yet made.** Returning `Uint8Array` over the existing ndjson IPC channel and wrapping it in a renderer `Blob`/object URL would avoid registering a new Tauri scheme entirely (and sidesteps the asset-protocol scope surface), but loses the per-response header semantics; the custom-protocol path preserves them. Trade-off to resolve in shape/decompose.
- **`m365-document-ingest` follow-on** is referenced in `goals/m365-driver/SPEC.md` Non-Goals but no such goal directory was inventoried here; if it exists, confirm it does not already claim a serve-route before finalizing STAND ALONE.

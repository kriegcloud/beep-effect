# court-jurisdiction-reporter-vocabulary-schema

Scope: design the court / jurisdiction / reporter controlled vocabularies as `effect/Schema` literals + codecs (court entity fields, the F/FD/FB/S/TRS/MA jurisdiction taxonomy, the reporter-type enum) and align them to the abstract OWL/SKOS Court & Jurisdiction nodes owned by `ip-law-knowledge-graph` — without rebuilding those graph nodes.

## Findings

### A. There are TWO distinct "jurisdiction" taxonomies — do not conflate them

- **F-A1 (load-bearing).** The word "jurisdiction" denotes two different controlled vocabularies in the source corpus, and the schema must model both separately:
  1. **courts-db `jurisdiction` field** = a *state/territory token* (e.g. Alabama → `"A.L."`, California → `"C.A."`), i.e. a USPS-style two-letter postal code rendered with periods. Verified by fetching the first records of the live dataset: `{"id":"alacirct","jurisdiction":"A.L.","system":"state",...}` / `{"id":"alactapp","jurisdiction":"A.L.",...}` — https://raw.githubusercontent.com/freelawproject/courts-db/main/courts_db/data/courts.json
  2. **CourtListener `Court.jurisdiction` field** = a *system+level composite code* (e.g. `"F"`=Federal Appellate, `"FD"`=Federal District, `"TRS"`=Tribal Supreme). Verified from the model source — https://raw.githubusercontent.com/freelawproject/courtlistener/main/cl/search/models.py
- **F-A2.** courts-db decomposes what CourtListener packs into one `jurisdiction` code into three orthogonal fields — `system` (federal/state/tribal/special/extraterritorial), `type` (trial/appellate/bankruptcy/ag), `level` (colr/iac/gjc/ljc). This is effectively a normalized form of the CL composite code. Source (field values observed in live data): https://raw.githubusercontent.com/freelawproject/courts-db/main/courts_db/data/courts.json — so the design should treat `(system, type, level)` ⇄ `CL jurisdiction code` as a derivable crosswalk codec, not two unrelated enums.

### B. courts-db court-entity schema (the `court` vocabulary node shape)

- **F-B1.** Per-court object fields, verbatim from the live dataset (Alabama Court of Appeals): `id` (string, CourtListener court id), `name` (full name), `name_abbreviation` (often `null`), `citation_string` (Bluebook court reporter string, e.g. `"Ala. Ct. App."`, may be empty `""`), `jurisdiction` (state token, see F-A1), `system`, `type`, `level`, `location` (e.g. `"Alabama"`), `dates` (array of `{start, end}` ISO-date-or-null objects), `regex` (array of match templates, may reference `${var}` from variables.json), `examples` (array). The README also documents optional `court_url`, `case_types`, `notes`; `parent` (array of parent court ids) is present only on child courts. Sources: https://raw.githubusercontent.com/freelawproject/courts-db/main/README.rst and https://raw.githubusercontent.com/freelawproject/courts-db/main/courts_db/data/courts.json
- **F-B2.** `type` value set (verified distinct values in live data): `"trial"`, `"appellate"`, `"bankruptcy"`, `"ag"`. `level` value set: `"colr"` (Court of Last Resort), `"iac"` (Intermediate Appellate Court), `"gjc"` (General Jurisdiction Court), `"ljc"` (Limited Jurisdiction Court). `system` value set: `"state"`, `"federal"`, `"tribal"`, `"special"`, `"extraterritorial"`. README corroborates COLR/IAC/GJC/LJC and Trial/Appellate/Bankruptcy/AG — https://raw.githubusercontent.com/freelawproject/courts-db/main/README.rst
- **F-B3.** `citation_string` is the authoritative court→Bluebook crosswalk token (the IP anchor: the CAFC entry carries `"citation_string":"Fed. Cir."`). It is independent of both jurisdiction taxonomies and should be its own optional string field (empty-string-as-absent is a real state — see Alabama Circuit Courts `""`). Source (CAPTURE nugget courts-db#8; corroborated by field presence in live data): https://raw.githubusercontent.com/freelawproject/courts-db/main/courts_db/data/courts.json

### C. CourtListener jurisdiction taxonomy (F/FD/FB/S/TRS/MA + grouping lists)

- **F-C1.** Full constant set from `cl/search/models.py` `Court` model (code → label), reproduced verbatim from source — https://raw.githubusercontent.com/freelawproject/courtlistener/main/cl/search/models.py :
  - `FEDERAL_APPELLATE="F"`, `FEDERAL_DISTRICT="FD"`, `FEDERAL_BANKRUPTCY="FB"`, `FEDERAL_BANKRUPTCY_PANEL="FBP"`, `FEDERAL_SPECIAL="FS"`
  - `STATE_SUPREME="S"`, `STATE_APPELLATE="SA"`, `STATE_TRIAL="ST"`, `STATE_SPECIAL="SS"`, `STATE_ATTORNEY_GENERAL="SAG"`
  - `TRIBAL_SUPREME="TRS"`, `TRIBAL_APPELLATE="TRA"`, `TRIBAL_TRIAL="TRT"`, `TRIBAL_SPECIAL="TRX"`
  - `TERRITORY_SUPREME="TS"`, `TERRITORY_APPELLATE="TA"`, `TERRITORY_TRIAL="TT"`, `TERRITORY_SPECIAL="TSP"`
  - `MILITARY_APPELLATE="MA"`, `MILITARY_TRIAL="MT"`, `COMMITTEE="C"`, `INTERNATIONAL="I"`, `TESTING_COURT="T"` (23 codes total).
- **F-C2.** Grouping lists from the same model: `FEDERAL_JURISDICTIONS=[F,FD,FS,FB,FBP]`, `STATE_JURISDICTIONS=[S,SA,ST,SS,SAG]`, `BANKRUPTCY_JURISDICTIONS=[FB,FBP]`, `TRIBAL_JURISDICTIONS=[TRS,TRA,TRT,TRX]`, `TERRITORY_JURISDICTIONS=[TS,TA,TT,TSP]`, `MILITARY_JURISDICTIONS=[MA,MT]`. These are `skos:broader` parents / `skos:Collection` candidates (Federal / State / Bankruptcy / Tribal / Territory / Military). Source: https://raw.githubusercontent.com/freelawproject/courtlistener/main/cl/search/models.py
- **F-C3 (independent verification).** The labels group exactly as the public help page lists them (Federal Appellate/District/Bankruptcy/Bankruptcy Panel/Special; Military Appellate; State Supreme/Appellate/Trial/Special/Attorney General; tribal/territory tribunals) — https://www.courtlistener.com/help/api/jurisdictions/ — and the live API independently returns the rare tribal code: `GET /api/rest/v4/courts/?jurisdiction=TRS` yields 6 tribal supreme courts each with `"jurisdiction":"TRS"` (e.g. `cherokee` = Cherokee Nation Supreme Court) — https://www.courtlistener.com/api/rest/v4/courts/?jurisdiction=TRS&format=json . This cross-checks the single-file `models.py` read against two other sources.

### D. Reporter-type taxonomy — two encodings + a crosswalk

- **F-D1.** CourtListener `BaseCitation.CITATION_TYPES` integer enum (verbatim from `cl/search/models.py`): `FEDERAL=1, STATE=2, STATE_REGIONAL=3, SPECIALTY=4, SCOTUS_EARLY=5, LEXIS=6, WEST=7, NEUTRAL=8, JOURNAL=9`, each with a descriptive label (e.g. `FEDERAL` → "A federal reporter citation (e.g. 5 F. 55)"). Source: https://raw.githubusercontent.com/freelawproject/courtlistener/main/cl/search/models.py
- **F-D2.** reporters-db encodes the *same* taxonomy as **string** `cite_type` values per reporter: `state | federal | neutral | specialty | specialty_west | specialty_lexis | state_regional | scotus_early` (8 values; first reporter `"A."` = Atlantic Reporter has `cite_type:"state_regional"`). Sources: https://raw.githubusercontent.com/freelawproject/reporters-db/main/reporters_db/data/reporters.json and the documented enum in https://github.com/freelawproject/reporters-db
- **F-D3 (inferred crosswalk, see Open/Unverified).** The two encodings map: `federal→1, state→2, state_regional→3, specialty→4, scotus_early→5, specialty_lexis→6 (LEXIS), specialty_west→7 (WEST), neutral→8`; CourtListener adds `JOURNAL=9` with no reporters-db string counterpart, and reporters-db splits `SPECIALTY` into bare `specialty` plus `specialty_west`/`specialty_lexis`. Recommendation: model reporter-type as ONE LiteralKit vocabulary keyed on the reporters-db string form (the data-bearing source) and provide a transform codec to/from the CL integer for CourtListener interop.
- **F-D4.** reporter entry top-level shape (reporters-db): `name`, `cite_type`, `editions` (date-bounded regex/edition map), `variations` (variant→canonical abbreviation map), `mlz_jurisdiction`, optional `examples`/`href`/`publisher`/`notes`/`cite_format`. Source: https://raw.githubusercontent.com/freelawproject/reporters-db/main/reporters_db/data/reporters.json

### E. Normalization vocabulary (variables.json regex template dictionary)

- **F-E1.** courts-db `variables.json` is a flat map of short keys → regex fragments substituted (`${key}`) into `courts.json` `regex` arrays to tolerate OCR/abbreviation variation. First entries verbatim: `"md":"M(id(dle)?)? ?D(ist)?..."`, `"sd"`, `"ed"`, `"nd"`, `"cd"`, `"d"`, `"usa":"(((U\\.? ?S\\.? ?)|(Unite(d|s)? State(s|d)?)) ?)"`, `"usdc"`, `"uscoa"`, `"coa":"Courts? (ofr?t?)? ?App?eals?..."`, `"usbap"`, `"usbc"`. Source: https://raw.githubusercontent.com/freelawproject/courts-db/main/courts_db/data/variables.json — this belongs to the *resolver* sibling subtopic; for the *vocabulary-schema* it only matters that `regex`/`examples` are opaque `S.Array(S.String)` fields carried on the court entity, not parsed at the schema layer.

### G. effect/Schema design recommendation (repo-grounded)

- **F-G1.** Use the repo's existing `LiteralKit.make(literals, mappings)` — `packages/foundation/modeling/schema/src/LiteralKit/LiteralKit.schema.ts` — for every closed enum here. It takes a non-empty literal tuple plus an optional `EnumMappings` of `[literal, key]` pairs and emits an `Enum` object, per-member `is`-guards, an exhaustive `$match`, and `thunk` constructors, with compile-time coverage/duplicate checks (`ValidEnumMapping`). The `[code, label]` shape of `JURISDICTIONS` and `CITATION_TYPES` maps directly onto `EnumMappings` (literal=code, key=label). This avoids hand-rolling `S.Literal` unions + lookup maps. (Confirmed present; no existing court/jurisdiction/reporter vocabulary exists in `packages/law-practice/domain` or the `courtlistener` driver — the driver is a bare `index.ts` stub — so this is genuinely net-new.)
- **F-G2.** Recommended schema surface (four LiteralKits + two structs + crosswalk codecs):
  - `CourtJurisdictionCode` — LiteralKit over the 23 CL codes (F-C1) with labels.
  - `CourtSystem` (`state|federal|tribal|special|extraterritorial`), `CourtType` (`trial|appellate|bankruptcy|ag`), `CourtLevel` (`colr|iac|gjc|ljc`) — LiteralKits (F-B2).
  - `ReporterType` — LiteralKit over reporters-db strings (F-D2) + `S.transform` codec to the CL integer (F-D1/D3).
  - `Court` entity — `S.Struct`/`Model.Class` with `id`, `name`, `nameAbbreviation: S.NullOr(S.String)`, `citationString: S.String` (empty = absent), `jurisdiction` (state token, narrow later), `system/type/level` (the three LiteralKits), `location`, `parent: S.Array(CourtId)`, `dates: S.Array(S.Struct({start: NullOr(Date), end: NullOr(Date)}))`, `regex`/`examples`: `S.Array(S.String)`.
  - Crosswalk codec `CourtSystem × CourtType × CourtLevel ⇄ CourtJurisdictionCode` (F-A2) implemented as a `S.transformOrFail` backed by a static HashMap, so the two jurisdiction taxonomies stay reconcilable.
- **F-G3.** Court ids and reporter keys should be branded EntityId-style schemas (the repo's `schema-model-specialist`/`EntityId` pattern), not bare `S.String`, so they can serve as the stable join key the KG nodes reference (F-H2). Source: repo memory / `schema-model-specialist` skill; CLAUDE.md schema-first rules.

### H. SKOS/OWL alignment to ip-law-knowledge-graph (don't rebuild)

- **F-H1.** `ip-law-knowledge-graph` (PENDING stub) owns the *abstract* graph nodes: its P1 design defines a 15-branch `NodeKind` tagged union including `Court` (`_tag:"Court"`, Source S4) and `Jurisdiction` (`_tag:"Jurisdiction"`, Source S1/S6), plus edges `DecidedBy` (Judgment→Court) and `GovernedBy` (IPRight→Jurisdiction). Each node carries `_tag`, `@source`, and an "OWL Class Mapping" annotation slot. Source: `/home/elpresidank/YeeBois/projects/beep-effect/goals/ip-law-knowledge-graph/history/outputs/p1-schema-design.md`. The vocabulary packet must NOT redefine these nodes — it supplies the *reference-data values* those nodes point at.
- **F-H2 (boundary contract).** Model the vocabularies as a SKOS code list that the KG references by key: each enum member is a `skos:Concept` with `skos:notation` = the code (e.g. `"F"`, `1`, `"state_regional"`), `skos:prefLabel` = the label, `skos:broader` = its grouping parent (F-C2: Federal/State/Bankruptcy/Tribal/Territory/Military), `skos:inScheme` = one `skos:ConceptScheme` per vocabulary. This is the W3C-recommended pattern for code lists/enumerations (notation for codes, prefLabel for labels, broader for hierarchy, inScheme for membership). Sources: https://www.w3.org/TR/skos-reference/ and https://www.w3.org/TR/skos-primer/ . The KG `Court`/`Jurisdiction` OWL classes then relate to these `skos:Concept` instances (e.g. a `Court` node's jurisdiction property `owl:`-points-to / `skos:notation`-matches a `CourtJurisdictionCode` concept), so the controlled vocabulary is the ABox/value layer and the KG is the TBox/relationship layer.
- **F-H3.** Emission mechanics: the repo has `@beep/rdf` + `@beep/semantic-web` (`packages/foundation/capability/{rdf,semantic-web}`) for RDF/triples, but a grep found Web-Annotation vocab helpers there, NOT a ready SKOS `Concept`/`ConceptScheme` emitter — so SKOS serialization of these code lists is itself net-new (small) work, best done as a derivation FROM the effect/Schema LiteralKits (single source of truth) rather than authoring SKOS by hand. Carry the OWL/SKOS class IRI on each LiteralKit via the repo's `$I.annote`/schema-annotation pattern (the same "OWL Class Mapping" slot the KG uses) so traceability is declared at the schema, matching CLAUDE.md's annotation law and the `jsdoc-annotation-specialist` `$I.annoteSchema` convention.

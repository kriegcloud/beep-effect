# verbatim-span-verification-and-straddle

Scope: how to extend the existing `@beep/langextract` Alignment (normalizedâ†’raw offset map, findExact/findLesser/findFuzzy) into a verbatim-quote verifier that handles cross-chunk + cross-page `[[PAGE_BREAK]]` straddle, returns VERIFIED-with-location or NOT-FOUND (never a near match), and round-trips plainâ†”markup offsets to annotate Lexical source without corrupting markup.

## Findings

### A. What the repo already owns (the substrate to extend)

- **`@beep/langextract` Alignment is the existing verbatim grounder.** `alignCandidate` runs a fixed ladder: `findExact` (`String.indexOf`) â†’ `findLesser` (case-folded match via `lowerWithSourceOffsets`) â†’ `findFuzzy` (Levenshtein word-window, default threshold `0.82`), emitting `match_exact | match_lesser | match_fuzzy | unaligned` plus a `Contract.Span {start,end}`. It matches against **a single source string only** â€” no chunk/page reconstruction. Source: repo `packages/foundation/capability/langextract/src/Alignment/index.ts` (read directly).
- **The normalizedâ†’raw offset map already exists** as `lowerWithSourceOffsets(sourceText)`, which walks code points and returns parallel `starts[]`/`ends[]` arrays mapping each normalized-string index back to its source char offset. This is exactly the structure verbatim verification + straddle needs; `findLesser` already uses it to recover the *raw* source slice from a normalized match. Source: repo `Alignment/index.ts` lines 82-128 (read directly).
- **The result/span shape already exists and matches the canonical contract.** `Contract.Span` is `{start,end}` (`NonNegativeInt`, refined `end >= start`); `@beep/provenance` `TextAnchor` is half-open `[startChar, endChar)` + verbatim `quote` with the invariant "`text.slice(startChar, endChar)` should reproduce `quote`"; `@beep/epistemic-domain` `EvidenceSpan` = `TextAnchorFields` + `confidence`. Sources: repo `packages/foundation/capability/nlp/src/Handoff/Contract.ts` (Span), `packages/foundation/modeling/provenance/src/TextAnchor.ts`, `packages/epistemic/domain/src/values/EvidenceSpan/EvidenceSpan.model.ts` (read directly).
- The repo's `@beep/langextract` package is itself **Apache-2.0** (`packages/foundation/capability/langextract/package.json`), matching upstream â€” so it is a clean in-repo base to extend rather than a third-party port question.

### B. Upstream langextract canonical alignment â€” and how the repo port diverges

- **Upstream `google/langextract` is Apache-2.0** and frames source grounding as "Maps every extraction to its exact location in the source text" with `char_interval = None` for ungrounded extractions. https://github.com/google/langextract
- **Upstream `WordAligner` aligns on TOKEN sequences via `difflib.SequenceMatcher`**, not raw `indexOf`. Exact/lesser/greater come from matching token blocks (`S[i:i+n] == E[j:j+n]`); `MATCH_EXACT` when extraction length == matched block, `MATCH_LESSER` when "extraction longer than matched text." Char positions are recovered from each token's embedded `char_interval` (`start_pos = char_offset + start_token.char_interval.start_pos`). Sources: https://github.com/google/langextract/blob/main/langextract/resolver.py and https://deepwiki.com/google/langextract
- **Upstream fuzzy is OPTIONAL and gated** (`enable_fuzzy_alignment=True`, `fuzzy_alignment_threshold` default **0.75**, token-overlap ratio). A newer `_lcs_fuzzy_align_extraction` adds a coverage gate (0.75) + "min density" 1/3. Cross-verified by DeepWiki summary and resolver.py fetch. Sources: https://deepwiki.com/google/langextract , https://github.com/google/langextract/blob/main/langextract/resolver.py
- **`AlignmentStatus` enum = `MATCH_EXACT, MATCH_GREATER, MATCH_LESSER, MATCH_FUZZY`; `CharInterval` is `start_pos` (inclusive) / `end_pos` (exclusive)** â€” half-open, matching the repo's `TextAnchor`. Source: https://raw.githubusercontent.com/google/langextract/main/langextract/core/data.py
- **Divergence to note:** the repo port does NOT mirror upstream's token+`SequenceMatcher` design. It substitutes (a) raw `indexOf` exact, (b) a *case-folded* `lowerWithSourceOffsets` lesser match, (c) a Levenshtein word-window fuzzy at threshold 0.82 â€” and has **no `MATCH_GREATER`**. For citation grounding this is good (simpler, deterministic) but the case-fold and fuzzy paths are too lenient for a verbatim guard (see C). Source: repo `Alignment/index.ts` vs upstream resolver.py (both read).

### C. The "never a near match" contract: extraction grounding â‰  verbatim citation

- **Verbatim verification for citations must collapse to two outcomes: VERIFIED-with-location or NOT-FOUND** â€” a near match is a hallucination risk, not a weaker pass. This is the explicit design of the doc-haus reference (`verify-quote.ts`): "returns VERIFIED-with-location or NOT FOUND, never near match." Source: CAPTURE.md nugget `doc-haus#2` (`dochaus/tool/verify-quote.ts:37-56`).
- **This conflicts with promoting `match_fuzzy` (and arguably `match_lesser`/case-fold) to a VERIFIED verdict.** langextract's `MATCH_FUZZY` is an *approximate* grounding acceptable for entity extraction but unacceptable for a quote you will attribute to a court. The guard must accept only **normalization-equivalent exact** matches (whitespace/quote/Unicode folding with a preserved offset map) and map everything else to NOT-FOUND. Corroboration that exact-grep is the expected bar for verbatim quote retrieval (vs fuzzy n-gram tolerance for loose "verbatim"): CLERC notes "exact matches with grep are expected to achieve near-perfect accuracy" for direct quotes, while loose verbatim matching "produced non-verbatim matchesâ€¦ indicating a model's hallucinative tendencies." Sources: https://arxiv.org/pdf/2406.17186 (CLERC) and search synthesis on verbatim/grep exact-match.
- **doc-haus normalizes whitespace + curlyâ†’straight quotes but deliberately does NOT case-fold** ("no case fold") â€” stricter than the repo's `findLesser`, which lowercases. The verbatim guard should follow doc-haus (no case fold) so "Section" â‰  "section" cannot silently verify. Source: CAPTURE.md nugget `doc-haus#2`.
- **Output-side re-verification ladder** (re-check each emitted span against the *live* file: exact â†’ re-anchor nearest â†’ reject; drop quotes whose text no longer exists) is the staleness guard layered on top. Source: CAPTURE.md nugget `doc-haus#4` (`dochaus/plugin/legal.ts:214-232`).

### D. Normalization needs a per-character offset map, not index arithmetic

- **Unicode normalization changes string length, so a naive 1:1 offset map breaks.** `String.prototype.normalize("NFC"|"NFD"|"NFKC"|"NFKD")` can shorten or lengthen: `Ă±` (U+00F1, len 1) â†” NFD `n`+combining tilde (len 2); ligature `ď¬€` (U+FB00, len 1) â†’ NFKD `ff` (len 2); `cafĂ©` NFD inserts a combining accent shifting every later offset. "You cannot reliably map back-and-forthâ€¦ using simple index arithmetic." Source: https://developer.mozilla.org/en-US/docs/Web/JavaScript/Reference/Global_Objects/String/normalize
- **Therefore the verifier must build a normalized-index â†’ source-offset array** (one entry per *output* character), exactly the shape of the repo's `lowerWithSourceOffsets` `starts[]`/`ends[]` and doc-haus's `map.push(i)`. The extension is to (1) swap case-folding for whitespace-collapse + curly-quote/dash folding + optional `NFKC`, and (2) keep emitting the *raw* source slice (never the normalized text) as the verbatim `quote`. Sources: repo `Alignment/index.ts` (`lowerWithSourceOffsets`) + CAPTURE.md `doc-haus#2` (`normalizeWithMap`).

### E. Cross-chunk + cross-page straddle: verify against the reconstructed full text, not per-chunk

- **Chunk overlap reduces but does not eliminate boundary straddle.** Standard RAG practice repeats the last ~50-100 tokens of one chunk at the start of the next so a claim split across the boundary survives, but "important information may span two chunks." Page-based chunking treats each PDF page as one chunk. Sources: https://optyxstack.com/rag-reliability/rag-chunking-strategy-chunk-size-overlap-document-structure-recall and https://langcopilot.com/posts/2025-10-11-document-chunking-for-rag-practical-guide
- **Per-chunk offsets cannot express a straddling citation â€” you need a global offset basis.** Anthropic's own guidance confirms the failure mode: to cite individual RAG chunks "put each RAG chunk into a plain text document," whereupon `char_location` indices are **chunk-local** and a quote spanning two chunk-documents is two citations, not one. The straddle solution is therefore to verify against either (a) the original full document text, or (b) a reconstructed full text (chunks concatenated in order with overlap removed), keeping a `chunk â†’ [globalStart,globalEnd)` map so any verified span is expressible in whole-document offsets. Source: https://platform.claude.com/docs/en/docs/build-with-claude/citations ("if you want Claude to be able to cite specific sentences from your RAG chunks, you should put each RAG chunk into a plain text document").
- **Page straddle is handled with a `[[PAGE_BREAK]]` sentinel embedded inside the verbatim quote.** The reference contract emits quotes like `{page:"41-42", quote:"â€¦[[PAGE_BREAK]]â€¦"}` â€” the model returns the page range AND an in-quote marker at the page boundary. To verify, treat `[[PAGE_BREAK]]` as a normalization token: strip/whitespace-fold it out of BOTH the candidate quote and the page-joined source (recording the boundary in the offset map), match the residual verbatim text, then report `start_pageâ€“end_page`. Source: CAPTURE.md nugget `mike#3` (`backend/src/lib/chatTools.ts:120-136`).
- This mirrors Anthropic's `page_location {start_page_number (1-indexed), end_page_number (exclusive)}` as the result shape for a page-straddling citation. Source: https://platform.claude.com/docs/en/docs/build-with-claude/citations

### F. Canonical VERIFIED-with-location result shape (Anthropic Citations API)

- **Three location variants, all half-open, map cleanly onto repo primitives:** `char_location {start_char_index (0-indexed), end_char_index (exclusive)}` for plain text; `page_location {start_page_number (1-indexed), end_page_number (exclusive)}` for PDF; `content_block_location {start_block_index, end_block_index (exclusive)}` for custom content. `char_location` â‰… `TextAnchor {startChar, endChar}`. Source: https://platform.claude.com/docs/en/docs/build-with-claude/citations
- **`cited_text` is guaranteed verbatim** because "the API parses citationsâ€¦ and extracts `cited_text` directly, citations are guaranteed to contain valid pointers to the provided documents"; it also "does not count toward your output tokens." This is the external benchmark for the beep guard's VERIFIED output: a typed span + the raw source slice (not the model's paraphrase). Sources: https://platform.claude.com/docs/en/docs/build-with-claude/citations and corroboration https://simonwillison.net/2025/Jan/24/anthropics-new-citations-api/
- **Default chunking is sentence-level** (plain text + PDF); custom-content blocks are used as-is with no further chunking â€” i.e. the producer controls citation granularity by choosing the block boundary. Source: https://platform.claude.com/docs/en/docs/build-with-claude/citations
- **Gotcha: Citations are incompatible with structured outputs** â€” enabling both returns HTTP 400, because citations interleave citation blocks with text and cannot satisfy a strict JSON schema. Relevant if the guard wires citation grounding through the same LLM call that also emits structured extraction. Source: https://platform.claude.com/docs/en/docs/build-with-claude/citations
- **`search_results` content blocks** are a first-class RAG citation primitive (pass retrieved chunks with built-in citation support) â€” an alternative to round-tripping offsets yourself when the LLM is Claude. Source: https://platform.claude.com/docs/en/build-with-claude/search-results (linked from citations doc).

### G. Plainâ†”markup offset round-trip = eyecite `SpanUpdater` (BSD-2-Clause â€” port-safe)

- **LICENSING CORRECTION (material):** the CAPTURE.md caution lumps "eyecite/courtlistener â€¦ AGPL," but the **eyecite library is BSD-2-Clause, "Copyright (c) 2020, Free Law Project."** Only the CourtListener *application* (`cl/citations/annotate_citations.py`) is AGPL. The annotate / `SpanUpdater` *algorithm* is therefore safe to port with attribution; prefer it over reading the AGPL CourtListener source. Source: https://github.com/freelawproject/eyecite/blob/main/LICENSE
- **`annotate_citations(plain_text, annotations, source_text="", unbalanced_tags="unchecked"|"skip"|"wrap", use_dmp=True, annotator=None, offset_updater=None)`** inserts before/after markup around each `((start,end), before, after)` annotation. When `source_text` (the original markup) differs from the cleaned `plain_text` the citations were parsed from, it "intelligently reconcile[s] the differencesâ€¦ using a diffing algorithm, ensuring that each annotation is inserted in the correct location." Source: https://freelawproject.github.io/eyecite/
- **`SpanUpdater` builds a piecewise offset map from diff ops** between the two text versions: `"="`(match)â†’push offset + `partial(shift_offset, delta)`; `"+"`(insert)â†’advance delta; `"-"`(delete)â†’absolute `partial(replace_offset, new_offset=offset+delta)`. Parallel `self.offsets[]`/`self.updaters[]`, queried by `bisect` (`index = bisect(self.offsets, offset) - 1`). Spans use `bisect_right` on start, `bisect_left` on end (boundary-insertion semantics). Default diff backend is `fast_diff_match_patch.diff(timelimit=0, checklines=False, cleanup="No")`; `use_dmp=False` falls back to difflib `SequenceMatcher`. Source: https://raw.githubusercontent.com/freelawproject/eyecite/main/eyecite/annotate.py
- **Markup-corruption guard:** before inserting, it checks `is_balanced_html(span_text)`; `unbalanced_tags="skip"` drops the annotation (logs a warning) rather than emit invalid HTML, `"wrap"` repairs by wrapping tags. CourtListener's production call uses `full_span()` for `Id`/`Supra` citations (whole-citation span, avoids splitting a nested tag) vs `span_with_pincite()` otherwise, plus `offset_updater=plain_to_markup, unbalanced_tags="skip"`. Sources: https://freelawproject.github.io/eyecite/ and CAPTURE.md nugget `courtlistener#3` (`cl/citations/annotate_citations.py:77-128`).

### H. Lexical-specific round-trip (annotate without corrupting the node tree)

- **Do not splice serialized HTML; map offsets to the node tree.** Lexical serializes text via `$rootTextContent()` / `RootNode.getTextContent()` (aggregates all `TextNode` text). Selection points are `{node, offset}` where, for text nodes, `offset` is the char position *within that TextNode*. Sources: https://lexical.dev/docs/api/modules/lexical_text and https://lexical.dev/docs/concepts/selection
- **`$findTextIntersectionFromCharacters(root, targetCharacters)` exists but is deprecated** ("has never worked correctly and will be removed") â€” do NOT depend on it. Instead, walk the node tree once, recording each `TextNode`'s `[globalStart, globalEnd)` + node key while accumulating `getTextContent()` lengths; this is your plain-textâ†”node map. Source: https://lexical.dev/docs/api/modules/lexical_text
- **Round-trip recipe:** (1) serialize node tree â†’ plain text + per-node offset/key map; (2) verbatim-verify the quote against the plain text â†’ global char span; (3) map global span back to `{anchorNode/offset, focusNode/offset}` via the map; (4) apply a Lexical mark/decorator node over the `RangeSelection` â€” markup integrity is preserved by construction because you mutate nodes, never strings. The eyecite `unbalanced_tags` problem is the warning for any system that does string-splice annotation instead. Source: synthesis of https://lexical.dev/docs/concepts/nodes + https://lexical.dev/docs/concepts/selection + eyecite annotate behavior above.

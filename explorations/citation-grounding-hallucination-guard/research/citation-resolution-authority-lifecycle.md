# citation-resolution-authority-lifecycle

> Scope: how a parsed citation resolves to an authority, and how the candidate->resolved RESOLUTION lifecycle (NO_CITATION->FOUND->RESOLVED + FAILED_AMBIGUOUS/FAILED) is modeled net-new without forking the reused epistemic `ClaimLifecycle`/`ClaimGate` — covering eyecite resolution, the reporter/court crosswalk, the CourtListener lookup-driver wrap, and where unresolved/ambiguous parses are persisted with eyecite context.

## Findings

### Two distinct status enums live in CourtListener — do not conflate them

The capture nugget (`cl/citations/models.py:11-55`, states `NO_CITATION=1 FOUND=2 RESOLVED=3 FAILED_AMBIGUOUS=4 FAILED=5`) and the lookup-API result `status` field are **two different status vocabularies**, and the net-new design must keep them separate:

1. **Persistence/resolution-model status** — `BaseUnmatchedCitation.STATUS` is a `SmallIntegerField` with five values: `NO_CITATION = 1`, `FOUND = 2`, `RESOLVED = 3`, `FAILED_AMBIGUOUS = 4`, `FAILED = 5`. This is the *background-job resolution lifecycle* for a citation that was parsed out of an opinion but did not match the search index at parse time; a later task tries to resolve it. (https://raw.githubusercontent.com/freelawproject/courtlistener/main/cl/citations/models.py)
2. **Lookup-API per-citation result status** — the `/api/rest/v4/citation-lookup/` response sets each citation's `status` to an **HTTP-status-valued** code: `200` found, `300` Multiple Choices (ambiguous), `400` reporter not recognized, `404` valid citation not in DB, `429` too many. (https://wiki.free.law/c/courtlistener/help/api/rest/v4/citation-lookup)

These map onto each other but are emitted in different surfaces. The net-new beep model should pick **one** canonical resolution-state vocabulary (the 5-state model is the better fit for a persisted lifecycle) and treat the HTTP-valued codes purely as a driver-level wire detail to normalize at the CourtListener boundary.

### The 5-state UnmatchedCitation lifecycle and how each state is assigned

`BaseUnmatchedCitation` (abstract, inherits `BaseCitation`) carries `status`, `citation_string` (TextField: "The unparsed citation string in case it doesn't match the regular citation model"), `court_id` (TextField: "A court_id as identified by eyecite from the opinion's context"), `year` (nullable SmallIntegerField from eyecite context), plus inherited `volume`, `reporter`, `page`, `type` fields. The `create_from_eyecite_base()` classmethod assigns status: defaults to `NO_CITATION`; sets `FAILED_AMBIGUOUS` when `has_multiple_matches` is true; sets `FAILED` when the citation already exists in the `Citation` table; and extracts year/reporter-type/volume/page/court from eyecite's `FullCaseCitation` object. The STATUS help-text strings are the canonical semantics: `NO_CITATION` = "does not exist in the search_citation table"; `FOUND` = "now exists ... but hasn't been processed for opinion updates"; `RESOLVED` = "resolution task was run successfully" (opinion HTML updated); `FAILED_AMBIGUOUS` = "resolved to more than 1 cluster"; `FAILED` = "We couldn't resolve the citation". (https://raw.githubusercontent.com/freelawproject/courtlistener/main/cl/citations/models.py)

Concrete subclasses bind the citing context: `UnmatchedCitation` has a `citing_opinion` ForeignKey to `Opinion`; `UnmatchedCitationFromRECAPDocument` has a FK to `RECAPDocument`. Both add a DB index and a uniqueness constraint on `(citing_source + citation coordinates)`, and both expose a `create_from_eyecite()` factory delegating to the base. (https://raw.githubusercontent.com/freelawproject/courtlistener/main/cl/citations/models.py) — This is the canonical "where unresolved/ambiguous parses are persisted with eyecite context": a row per (source-document, parsed-citation) carrying the raw `citation_string`, the eyecite-derived `court_id`/`year`/`reporter`/`volume`/`page`/`type`, and the resolution `status`.

### eyecite is the parse+resolve engine; the lookup API only resolves full case citations

eyecite (BSD-2-Clause — safe to port/reimplement, unlike AGPL CourtListener) splits the work into two phases the beep model should mirror:

- **Parse** — `get_citations(plain_text, remove_ambiguous=..., tokenizer=..., markup_text=..., clean_steps=...)` returns a list of `CitationBase` subclasses: `FullCaseCitation`, `ShortCaseCitation`, `SupraCitation`, `IdCitation`, `ReferenceCitation`, `FullLawCitation`, `FullJournalCitation`, `UnknownCitation`. Each exposes `span()` (start/stop char offsets of matched text), `full_span` (boundaries incl. pre/post elements), `matched_text()` (raw extracted text e.g. `"1 U.S. 1"`), `corrected_citation()` (normalized variations; base impl returns `matched_text()`), and a `metadata` bag (`pin_cite`, `parenthetical`, `year`, `court`, `plaintiff`, `defendant`, `antecedent_guess`, `resolved_case_name_short`). (https://github.com/freelawproject/eyecite , https://raw.githubusercontent.com/freelawproject/eyecite/main/eyecite/models.py)
- **Resolve** — `resolve_citations(citations, resolve_full_citation=..., resolve_shortcase_citation=..., resolve_supra_citation=..., resolve_reference_citation=..., resolve_id_citation=...) -> Resolutions` where `Resolutions = dict[ResourceType, list[CitationBase]]`. Default `resolve_full_citation(full_citation) -> Resource` wraps each full citation in `@dataclass(frozen=True) class Resource(ResourceType): citation: FullCitation` — "a thin resource object that simply serves as a conceptual way to group citations with the same references together." `ResourceType = Hashable`. (https://raw.githubusercontent.com/freelawproject/eyecite/main/eyecite/resolve.py , https://raw.githubusercontent.com/freelawproject/eyecite/main/eyecite/models.py)

Resolution strategy per kind: `_resolve_shortcase_citation` matches by reporter+volume and refines via antecedent guessing when multiple full-cite candidates exist; `_resolve_supra_citation` resolves via antecedent guessing against plaintiff/defendant; `_resolve_id_citation` points at the previously-resolved resource and drops pin cites exceeding `MAX_OPINION_PAGE_COUNT` (150). **Critically: ambiguous or unresolvable citations are simply dropped — the output dict only contains citations whose `resolution` is truthy.** eyecite itself has no FAILED/AMBIGUOUS persisted state; that bookkeeping is exactly what CourtListener's `UnmatchedCitation` table adds on top. (https://raw.githubusercontent.com/freelawproject/eyecite/main/eyecite/resolve.py)

Note the asymmetry: eyecite *parses* id/supra/short/law/journal citations, but the **CourtListener lookup API explicitly does not resolve statutes, law journals, id citations, or supra citations and requires volume+page** — it only matches full case citations. (https://wiki.free.law/c/courtlistener/help/api/rest/v4/citation-lookup) So the net-new lifecycle must accommodate parsed-but-unresolvable kinds (the `NO_CITATION` bucket) for everything the resolver/API can't anchor to an authority.

### Reporter crosswalk: reporters-db (BSD)

reporters-db supplies the reporter normalization the `normalized_citations` field depends on. Primary structure `REPORTERS` is a dict of ~1,167 reporters with ~2,102 name variations; each entry carries `cite_type` (state, federal, neutral, specialty, specialty_west, specialty_lexis, state_regional, scotus_early), `editions` (citation-format -> {start date, end date, regexes}), `variations` (alt abbreviations), `name`, `mlz_jurisdiction`, `examples`, and optional `notes`/`href`/`publisher`. Crosswalk helpers: `VARIATIONS_ONLY` (canonical abbr -> variations), `EDITIONS` (edition abbr -> canonical reporter), `NAMES_TO_EDITIONS` (reporter name -> canonical abbr). Distributed as JSON, BSD-licensed. (https://github.com/freelawproject/reporters-db)

CourtListener's glue (`cl/citations/utils.py`) maps an eyecite reporter to its own DB: `slugify_reporter()` makes collision-aware slugs, `SLUGIFIED_EDITIONS` maps slug -> canonical edition, `get_canonicals_from_reporter()` disambiguates a reporter slug via the variation mappings, `map_reporter_db_cite_type()` converts reporters-db `cite_type` -> CourtListener Citation model type, and `get_years_from_reporter()` derives valid start/end years from edition data. It also filters with `filter_out_non_case_law_citations()` and `filter_out_non_case_law_and_non_valid_citations()` (drops cites lacking volume or page). (https://raw.githubusercontent.com/freelawproject/courtlistener/main/cl/citations/utils.py) — Per the repo tree-snapshot, this static reporter/edition vocabulary belongs in `goals/official-data-sync-foundation` (static-dataset ingest) feeding a `court-vocabulary-resolver`, NOT inline in the citation packet.

### Court crosswalk: courts-db (BSD)

courts-db resolves the eyecite-derived `court_id` from court strings/context. Two JSON files: `courts.json` + `variables.json` (regex templates). Per-court fields: `id` (CourtListener court id), `name`, `name_abbreviation`, `citation_string` (e.g. `"Mass."`), `regex` (array of patterns), `examples`, `dates` (start/end + notes), `court_url`, `type` (Trial/Appellate/Bankruptcy/AG), `system` (State/Federal/Tribal), `level` (COLR/IAC/GJC/LJC), `location`, `notes`. The `find_court(string, bankruptcy=..., date_found=...)` function returns matching court ids, filterable by date and bankruptcy. BSD-licensed. (https://github.com/freelawproject/courts-db) This is the lookup behind `UnmatchedCitation.court_id` "as identified by eyecite from the opinion's context."

### The CourtListener lookup-driver wrap: endpoint, schema, throttles

Endpoint: `POST https://www.courtlistener.com/api/rest/v4/citation-lookup/`, token auth (`Authorization: Token <token>`). Two input modes: (a) `text` block, **max 64,000 characters (~50 pages)**; or (b) a structured `volume` + `reporter` + `page` triad. No filtering/pagination/ordering/field-selection. (https://wiki.free.law/c/courtlistener/help/api/rest/v4/citation-lookup)

Per-citation response object (built in `cl/citations/api_views.py` and serialized by `CitationAPIResponseSerializer`): `citation` = `citation.matched_text()`; `normalized_citations` = `[citation.corrected_citation()]` (array of canonical forms, e.g. `"576 US 644"` -> `"576 U.S. 644"`); `start_index` & `end_index` (char positions in submitted text); `status` (HTTP-valued, see below); `error_message` (optional); `clusters` (array of matching CourtListener case objects when found). (https://raw.githubusercontent.com/freelawproject/courtlistener/main/cl/citations/api_views.py , https://wiki.free.law/c/courtlistener/help/api/rest/v4/citation-lookup)

Status codes are Python `HTTPStatus` constants: `HTTPStatus.OK` (200, single match), `HTTPStatus.MULTIPLE_CHOICES` (300, multiple clusters -> ambiguous), `HTTPStatus.BAD_REQUEST` (400, reporter not found), `HTTPStatus.NOT_FOUND` (404, no matches), `HTTPStatus.TOO_MANY_REQUESTS` (429, set when `idx >= settings.MAX_CITATIONS_PER_REQUEST`). (https://raw.githubusercontent.com/freelawproject/courtlistener/main/cl/citations/api_views.py)

Throttles (two independent limits): **60 valid citations/minute** and **250 citations/request**. Citations past 250 are parsed but not matched, status `429` ("Too many citations requested"). On the minute-throttle, the response carries a `wait_until` ISO-8601 timestamp; recommended pattern for >300 citations is "60 at a time via one call per minute." The maintainer confirms there is **no special `150` status code** — only the standard `429`. The throttle class pre-parses citations (`self.citations` is set by the throttle for performance), so eyecite parsing happens at the throttle boundary. (https://github.com/freelawproject/courtlistener/discussions/6895 , https://raw.githubusercontent.com/freelawproject/courtlistener/main/cl/citations/api_views.py)

Hallucination-guard framing: the API exists specifically as "a guardrail for those that are using AI to generate legal content" — submit a text blob, get back which citations actually exist; "Invalid and ambiguous citations will be identified." The normalization came from analyzing "over 50 million citations from American case law to identify every known reporter and its variations." (https://free.law/2024/04/16/citation-lookup-api/)

### How to model the RESOLUTION lifecycle net-new WITHOUT forking ClaimLifecycle/ClaimGate

The repo's reused machinery (verified in source):

- `@beep/shared-domain` `ClaimLifecycle` = `LiteralKit(["candidate","shape_valid","consistency_checked","admitted"])` — a *linear claim-admission* pipeline, product-agnostic, plus `ClaimLifecycleTransition` (`{from,to,reason}`, a pure value object that records but does not authorize a move). (`packages/shared/domain/src/values/ClaimLifecycle/ClaimLifecycle.model.ts`)
- `@beep/epistemic-domain` re-exports that vocabulary and owns the *mechanism*: `ClaimGateResult` = `ClaimGateVerdict.toTaggedUnion("verdict")({ admitted: {}, rejected: { violations: S.Array(ClaimGateViolation) } })`, projected from SHACL. (`packages/epistemic/domain/src/values/ClaimGate/ClaimGateResult.model.ts`)
- `CandidateClaim` (`BaseEntity.Class`) carries `lifecycle: ClaimLifecycle` + `snapshot` + `fixtureKey`; `EvidenceSpan` = `TextAnchorFields` + `Confidence`. (`packages/epistemic/domain/src/entities/CandidateClaim/CandidateClaim.model.ts`, `.../values/EvidenceSpan/EvidenceSpan.model.ts`)

The two lifecycles are **orthogonal**, which is exactly why citation-resolution must be net-new rather than a fork:

- `ClaimLifecycle` answers *"is this asserted claim admissible into the authoritative store?"* (candidate -> ... -> admitted, gated by SHACL).
- The citation RESOLUTION lifecycle answers a *different* question — *"does this parsed citation string anchor to a real authority, and unambiguously?"* (NO_CITATION -> FOUND -> RESOLVED, with FAILED_AMBIGUOUS / FAILED exits). Forcing this into the 4-state linear claim pipeline would corrupt the shared vocabulary (FAILED_AMBIGUOUS and FAILED have no home in a strictly-forward admission pipeline; eyecite "drops" them and CourtListener parks them in a side table).

Recommended net-new shape, reusing the *patterns* not the *vocabulary* (all primitives already exist in-repo):

1. **`CitationResolution` value** in the new citation packet: `LiteralKit(["no_citation","found","resolved","failed_ambiguous","failed"])` — mirror the 5-state CourtListener model, NOT placed in `@beep/shared-domain` (citation/IP-law vocabulary stays out of the epistemic/shared slice per the capture cautions). Add a `CitationResolutionTransition` value object analogous to `ClaimLifecycleTransition` (`{from,to,reason}`) for the audit trail.
2. **`CitationResolutionResult`** discriminated union via `…toTaggedUnion("outcome")`, mirroring `ClaimGateResult`: `resolved` carries the authority reference + `normalized_citations`; `ambiguous` carries the candidate cluster set (the FAILED_AMBIGUOUS payload); `unresolved` carries the eyecite context for persistence. This reuses the exact `ClaimGateResult` pattern without touching it.
3. **`UnmatchedCitation`-analog persisted entity** (`BaseEntity.Class`/`Table.make`) in the citation packet, NOT in epistemic: store `citationString` (raw `matched_text`), `span` as a `@beep/provenance` `TextAnchor`/`EvidenceSpan` (start/end char offsets — already the repo's char-span primitive), the eyecite context (`courtId`, `year`, `reporter`, `volume`, `page`, `citationType`), the `CitationResolution` status, the `normalizedCitations` array, and a FK to the citing source document. This is the direct beep analog of `UnmatchedCitation(citing_opinion FK)` and is the home for "where unresolved/ambiguous parses are persisted with eyecite context."
4. **Composition, not forking**: a `RESOLVED` citation whose verbatim span is verified becomes *evidence* feeding the existing `ClaimGate` → a `CandidateClaim` advances through `ClaimLifecycle`. i.e. `CitationResolution.resolved` is an upstream precondition; `ClaimLifecycle.admitted` is the downstream gate. The driver wrap normalizes CourtListener's HTTP-valued per-citation `status` (200/300/400/404/429) into the 5-state `CitationResolution` at the boundary (200→resolved, 300→failed_ambiguous, 400/404→no_citation/failed, 429→retry/throttle, not a resolution state).

## Open / Unverified

- **Character cap discrepancy (version-dependent).** The capture nugget (us-legal-tools SDK, `packages/courtlistener-sdk`) records a **10k char cap**; the official v4 docs state **64,000 characters**. Likely the SDK targets an older/v3 limit or a self-imposed conservative cap. The driver wrap should treat 64,000 as the current v4 ceiling but confirm against the live API and the SDK's pinned version. (https://wiki.free.law/c/courtlistener/help/api/rest/v4/citation-lookup)
- **Exact `wait_until` field name/spelling.** Sources render it as both `wait_util` (the WebSearch summary, likely an API typo carried verbatim) and `wait_until` (discussion summary). UNVERIFIED which the live v4 JSON actually emits — confirm against a real 429 response before coding the retry/throttle decoder.
- **`status` field type on the lookup result.** Docs describe per-citation `status` as "HTTP status code"; api_views uses `HTTPStatus.*` constants, but whether the wire JSON serializes the integer (200/300/400/404/429) or a string is not 100% confirmed from the fetched summaries — verify the serializer output shape (`CitationAPIResponseSerializer`).
- **`has_multiple_matches` provenance.** The `create_from_eyecite_base()` FAILED_AMBIGUOUS branch keys off `has_multiple_matches`, but whether that flag is computed inside the model method or passed in by the caller (the resolution task) was not pinned down — relevant to where ambiguity is decided in the beep port.
- **eyecite version pin.** Resolution behavior (e.g. `MAX_OPINION_PAGE_COUNT = 150`, reference-citation resolver presence) is from `main`; pin a specific eyecite release when porting, since the resolver signature set has grown over versions (ReferenceCitation resolver is relatively recent). (https://raw.githubusercontent.com/freelawproject/eyecite/main/eyecite/resolve.py)
- **CourtListener AGPL boundary.** courtlistener itself is AGPL (capture caution) — `models.py`/`api_views.py`/`utils.py` are *spec references only*, reimplement; eyecite (BSD-2), reporters-db (BSD), courts-db (BSD) are port-safe. The exact AGPL version/file headers were not re-fetched here; confirm before any code reuse.

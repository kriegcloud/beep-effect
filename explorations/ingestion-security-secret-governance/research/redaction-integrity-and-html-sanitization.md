# redaction-integrity-and-html-sanitization

Scope: output/ingest-integrity guards — PDF "x-ray" detection of recoverable text under failed redaction boxes (bounding boxes + honest blind-spot reporting) and a tag/attr-allowlist + safe-href sanitizer for untrusted fetched opinion HTML; survey x-ray/PDF tooling and DOMPurify/sanitize-html vs hand-rolled allowlist; decide TS-native vs Python-shell for x-ray and the `@beep/html` integration shape.

## Findings

### A. PDF x-ray (failed-redaction detection)

- **`freelawproject/x-ray` is the canonical tool.** It detects the most common bad redaction: a filled rectangle drawn over still-selectable text. Algorithm is four steps — (1) locate rectangles, (2) find text at the same coordinates, (3) render the rectangle to an image, (4) check whether the rectangle is a single uniform color (bad) vs. mixed content (a legitimately rasterized redaction). Source: <https://github.com/freelawproject/x-ray> and project page <https://free.law/projects/x-ray/>.
- **Output shape is exactly the bounding-box + recovered-text contract the wedge wants.** Keyed by page number, each finding is `{"bbox": [x_upper_left, y_upper_left, x_lower_right, y_lower_right], "text": "<text under the box>"}`; an empty result means no bad redactions found. CLI: `xray path/to/file.pdf` emits JSON. Python: `xray.inspect()` accepts a path (str/Path), an HTTPS URL, or raw `bytes`. Source: <https://github.com/freelawproject/x-ray>, <https://pypi.org/project/x-ray/>.
- **Current version v0.3.6, released 2025-12-24.** Notably, that release "Fixes CLI export to be JSON not Python object" (issue #189) — i.e. pre-0.3.6 the CLI emitted a Python `repr`, not valid JSON, so pin ≥0.3.6 if shelling the CLI. Source: <https://github.com/freelawproject/x-ray/blob/main/CHANGES.md>.
- **x-ray itself is BSD-2-Clause (permissive, portable), BUT it depends on PyMuPDF, which is AGPL-3.0.** `pyproject.toml` pins `PyMuPDF==1.24.14` and `requests>=2.26.0`, `requires-python >=3.10`, `license = "BSD-2-Clause"`. Source: <https://github.com/freelawproject/x-ray/blob/main/pyproject.toml>. (Note: the algorithm description circulating as "pdfplumber" is wrong — the engine is PyMuPDF/`fitz`.)
- **PyMuPDF / MuPDF AGPL-3.0 is the decisive licensing constraint.** Artifex dual-licenses MuPDF/PyMuPDF under AGPL-3.0 or a paid commercial license; under AGPL, **running it as a network service that users interact with triggers the source-disclosure obligation for your whole application** — a SaaS legal-ingestion runtime is squarely in scope. Source: <https://github.com/pymupdf/pymupdf> (license), corroborated <https://pymupdf.io/> and license-summary <https://pdfmux.com/blog/pymupdf-vs-pdfplumber/>. ADVERSARIAL CHECK: two independent sources agree on AGPL-3.0 + commercial dual-license and the network-use trigger.
- **mupdf.js (Artifex's TS/WASM port) is NOT an AGPL escape hatch.** It is the official TypeScript-first MuPDF binding, WASM-powered, with `toStructuredText` returning per-span bbox + font + text — technically ideal for a TS-native x-ray — but it ships under the **same AGPL-3.0 / commercial dual license**. Source: <https://github.com/ArtifexSoftware/mupdf.js/> and Artifex blog <https://artifex.com/blog/mupdfjs-with-npm>. So "TS-native via mupdf.js" inherits the exact AGPL problem as the Python path.
- **The permissive TS-native path is Mozilla pdf.js (Apache-2.0).** pdf.js exposes the two primitives x-ray needs: `page.getOperatorList()` yields path ops including `OPS.rect` / `OPS.constructPath` (rectangle geometry), and `page.getTextContent()` yields text `items` whose `transform` matrix carries x/y position. Overlap of a uniform-fill rect with text items at the same coordinates reproduces x-ray's core heuristic without AGPL. License: Apache-2.0. Sources: <https://github.com/mozilla/pdf.js/blob/master/LICENSE>, <https://github.com/mozilla/pdf.js/issues/16184> (OPS.rect coordinate transform), <https://mozilla.github.io/pdf.js/api/draft/module-pdfjsLib.html>. Permissive MIT alternative for geometry-only work: `pdf-lib` (no text-render, weaker for the "uniform color" check). UNVERIFIED: whether pdf.js's operator list reliably surfaces the *fill color* needed for the uniform-color test without a raster pass — may require rendering the rect region to a canvas/ImageData, which pdf.js supports but is heavier in Node.
- **x-ray is honestly scoped, and its blind spots are the "honest reporting" design lesson.** The docs explicitly say "Bad redactions take many forms… We'd love your help solving some of the tougher cases" — i.e. it claims only rect-on-text coverage, not "clean." The HN thread (with the creator) enumerates the blind spots a residue/honesty report should surface rather than silently pass: font-metric/kerning side-channels that leak word lengths, **prior PDF revisions reachable via `/Prev` pointers**, and image-encoded text. Quote: "I haven't seen any redaction system yet that protects against [glyph-spacing] this." Source: <https://news.ycombinator.com/item?id=46369923>.
- **"Reports unreachable binary parts honestly" maps to the doc-haus OOXML residue-check, not to x-ray.** x-ray does not scan DOCX/image/embedded binary parts; the honesty pattern (confirm scrub on reachable XML parts, and *report* unreachable binary parts — images/embeddings — rather than claiming clean) comes from doc-haus#13 (`dochaus/lib/redactions.ts`, MIT — adaptable). Carry the principle across both guards: the integrity report enumerates what was checked AND what could not be reached (image text, `/Prev` revisions, embedded objects).

### B. HTML sanitization for untrusted fetched opinion HTML

- **Hand-rolled regex/string-replace allowlists (the mike `sanitizeOpinionHtml` approach) are the textbook mutation-XSS (mXSS) failure mode for browser-rendered HTML.** mXSS exploits a *parser differential*: the sanitizer's parser sees benign markup, but the browser re-parses the same bytes into executable HTML. A regex sanitizer has effectively no parser, so it is maximally exposed. Even full HTML-aware sanitizers were bypassed — HTML Purifier, htmLawed, OWASP AntiSamy, jSoup, kses, and Mozilla Bleach all had mXSS CVEs. Sources: cure53 foundational paper <https://cure53.de/fp170.pdf>, Sonar <https://www.sonarsource.com/blog/mxss-the-vulnerability-hiding-in-your-code>, Checkmarx (Bleach) <https://checkmarx.com/blog/vulnerabilities-discovered-in-mozilla-bleach/>, OWASP XSS Prevention Cheat Sheet <https://cheatsheetseries.owasp.org/cheatsheets/Cross_Site_Scripting_Prevention_Cheat_Sheet.html>. CONCLUSION: a dependency-free regex allowlist is acceptable as a *server-side pre-filter/normalizer*, but it must NOT be the only/final defense for HTML that a browser will render.
- **DOMPurify is the mXSS-hardened gold standard, but it needs a real DOM.** v3.4.11 (released 2026-06-17), dual-licensed **Apache-2.0 OR MPL-2.0**, by cure53. It parses with a DOM and is explicitly hardened against mXSS, namespace confusion, DOM clobbering, and rawtext breakouts. In the browser it's native; on **Node it requires jsdom** and the docs warn that older jsdom has XSS holes and that **happy-dom is unsafe** ("will likely lead to XSS"). Config knobs are allowlist-shaped: `ALLOWED_TAGS`, `ALLOWED_ATTR`, `ALLOWED_URI_REGEXP` (default protocols: http, https, ftp, ftps, tel, mailto, callto, sms, cid, xmpp, matrix), plus `uponSanitizeElement`/`uponSanitizeAttribute` hooks for flag-not-block annotation. Source: <https://github.com/cure53/DOMPurify>.
- **DOMPurify-on-server has a real operational tax.** `isomorphic-dompurify` (the common Node wrapper) leaks the jsdom window across calls in long-running processes — after a few thousand `sanitize()` calls each call is ~an order of magnitude slower and heap climbs into hundreds of MB; mitigation is periodic `clearWindow()` (which also drops hooks/config, requiring re-`setConfig`). Sources: <https://www.npmjs.com/package/isomorphic-dompurify>, leak report <https://github.com/kkomelin/isomorphic-dompurify/issues/368>.
- **sanitize-html is the server-native pick (no DOM), MIT-licensed.** Current **v2.17.5** (actively published), MIT, built on **htmlparser2** (string-in/string-out, no DOM/jsdom needed) — ideal for an ingestion-time server pass. Config is the declarative allowlist the wedge describes: `allowedTags`, `allowedAttributes` (per-tag), `allowedSchemes` (**default `http, https, ftp, mailto` — note `tel` is NOT default**), `allowedSchemesByTag`, and `transformTags` (the hook to rewrite `<page-number>`→span markers and to rewrite/validate hrefs). The repo was **archived/retired into the ApostropheCMS monorepo**; releases still ship under `apostrophecms/sanitize-html`. Sources: <https://www.npmjs.com/package/sanitize-html>, <https://github.com/apostrophecms/sanitize-html>. CAVEAT: htmlparser2's parser is not the browser's, so sanitize-html does **not** carry DOMPurify's mXSS guarantee — it is a content allowlist, not an mXSS oracle.
- **Net: defense-in-depth beats either single library.** Server pre-sanitize (sanitize-html, no-DOM, at ingest/store) + browser-side DOMPurify at the actual render boundary (native, mXSS-hardened) is the layered pattern the 2026 guidance converges on ("sanitize-html on the server, DOMPurify in the browser, never trust HTML from users without sanitization"). Corroboration: <https://www.pkgpulse.com/guides/sanitize-html-vs-dompurify-vs-xss-xss-prevention-2026>.

### C. Repo grounding (live-tree, filesystem-verified)

- **`@beep/html` is NOT a "render boundary" or runtime sanitizer — the CAPTURE label is a misnomer.** It is a schema-first **AST of the WHATWG HTML spec**: every element (conforming + obsolete) modeled as an `effect/Schema` `TaggedClass`, unioned via `S.toTaggedUnion("_tag")` into `HtmlNode`; plus `ELEMENT_META` (per-element interface/conformance/content-category and **void/raw-text classification**) and a `GlobalAttributes` overlay value-typed from WHATWG/webref enumerations (`StandardGlobalAttributes` + `AriaAttributes` + `EventHandlerAttributes`). License MIT; data sourced from `data/webref` + `data/whatwg`. Files: `packages/foundation/modeling/html/src/{index,Html.model,Html.meta,Html.attributes,Html.nodes}.ts`. IMPLICATION: `@beep/html` is the ideal **source of truth for the tag/attribute allowlist** (derive the sanitizer allowlist from `ELEMENT_META` + `GlobalAttributes` instead of hand-maintaining a literal set, and reuse the void/raw-text flags), but the sanitizer logic itself is **net-new** and does not exist there yet.
- **The safe-href substrate already exists in `@beep/md` — reuse it, do not reinvent.** `packages/foundation/modeling/md/src/Md.utils.ts` exports `sanitizeUrlDestination` (multi-pass decode of HTML char-refs + percent-encoded bytes across all decode orderings, blocks `^(?:javascript|vbscript|data):`, returns `"#"` on any unsafe candidate), the `UnsafeUrlProtocolDestination` schema check, `escapeMarkdownDestination`, and `escapeHtmlUrlAttribute = flow(sanitizeUrlDestination, encodeUrlDestination, Html.escapeHtml)`. `@beep/ui/src/lib/url.ts` has a parallel client-side `sanitizeAnchorHref` (same multi-pass-decode + `javascript|vbscript|data` block). The opinion-HTML sanitizer's "safe-href rewriting" should call these rather than porting mike's href validator. NOTE: current `@beep/md` boundary deliberately does NOT block `file:`/`blob:`/`filesystem:` ("not treated as execution sinks here") — for *fetched untrusted opinion* HTML you likely want a stricter scheme allowlist (http/https/mailto/tel only), so configure, don't blindly inherit.
- **`Html.escapeHtml` lives in `@beep/utils` (`Html` namespace)** and is already used by the md render adapters (`Md.render.ts` builds `<a href="${escapeHtmlUrlAttribute(href)}">…`). `SafeRemoteHost` (the SSRF host allowlist, adjacent subtopic) is at `packages/foundation/modeling/schema/src/SafeRemoteHost.ts`.
- **No PDF x-ray / redaction-integrity code exists in the repo today** (`rg` for `xray|redact` in PDF context found only unrelated cause-redaction/log-redaction in `@beep/observability`). x-ray is genuinely net-new; the natural home is the `@beep/file-processing` retrieval boundary (`packages/foundation/capability/file-processing`), whose OCR/PDF-diagnostics scope is already spec-deferred per the live-tree snapshot.

### D. Decisions this research supports

- **x-ray: build TS-native on pdf.js (Apache-2.0), not Python-shell x-ray and not mupdf.js.** Rationale: both x-ray (via PyMuPDF) and mupdf.js carry AGPL-3.0 whose *network-service* trigger is fatal for a hosted legal SaaS; pdf.js gives the rect (`OPS.rect`) + positioned-text (`getTextContent`) primitives in-process with no Python sidecar and a permissive license. Treat shelling real x-ray as an optional, license-isolated CLI for offline/internal-only batch QA (where AGPL distribution isn't triggered), not the hosted path. Reproduce the JSON contract (`{page: [{bbox, text}]}`) so either backend is swappable.
- **HTML sanitizer: do NOT ship mike's hand-rolled regex allowlist as the render-boundary defense.** Layer it: (1) derive the tag/attr allowlist from `@beep/html` `ELEMENT_META`/`GlobalAttributes`; (2) server-side pre-sanitize at ingest with **sanitize-html** (MIT, no-DOM) doing the allowlist + `transformTags` `<page-number>` rewrite; (3) at the browser render boundary, run **DOMPurify** (native) as the mXSS-hardened final gate; (4) rewrite hrefs through the existing `@beep/md` `sanitizeUrlDestination`/`escapeHtmlUrlAttribute` with a tightened scheme allowlist. A bespoke regex pass is at most a normalizer, never the only barrier.
- **Honesty/flag-not-block carries across both guards.** x-ray findings are advisory char/box-span annotations (never silent drops), and the integrity report must enumerate unreached surfaces — image-encoded text, `/Prev` revisions, embedded binary parts (the doc-haus residue-check principle) — rather than claiming "clean."

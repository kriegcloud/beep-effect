# prompt-injection-detection-landscape

External prior art for a retrieval-boundary prompt-injection / instruction-override + examiner-disclosure detector, and where it should sit at the `@beep/file-processing` ingestion boundary.

## Findings

### Architecture decision (synthesis up top)

- **Recommended shape: a deterministic, three-pass, flag-not-block detector that emits char-span findings — no ML model on the hot path.** The three passes are: (1) invisible-Unicode normalization + detection, (2) regex heuristic banks (instruction-override / examiner-disclosure / exfiltration / persona-switch), (3) hidden-content detection via render-vs-extract diff for DOCX/PDF. This composes the in-repo doc-haus (MIT) and uspto_pfw_mcp (MIT) heuristics with the PhantomLint (BSD-3) rendering insight, and keeps everything local — the privilege-safe property the legal use case needs. ML/guardrail libs are positioned as an *optional* enhancement, not the spine, because the strongest one (LLM Guard's DeBERTa) is English-only and explicitly does not cover the indirect/document-embedded injection that matters here.
- **Where it sits: a pure transform inside `@beep/file-processing` ingestion, downstream of text extraction (Tika/libpff) and upstream of any LLM dispatch, writing advisory findings to `@beep/provenance`.** It is a detector, not a gate: findings are char-span annotations the lawyer reviews, never silent drops. This matches the CAPTURE flag-not-block contract (`doc-haus#5`, `LegalEase#3`) and is corroborated by every flag-oriented prior-art tool below.

### Regex heuristic banks (the portable spine)

- **Yelp `detect-secrets` is the canonical plugin shape to mirror, not a dependency to adopt.** Plugins inherit `detect_secrets.plugins.base.BasePlugin`; the common case is `RegexBasedDetector`, which exposes an `analyze_string(string, line_num, filename)` returning `PotentialSecret` objects — i.e. a regex bank + a per-line analyzer with positional output. https://github.com/Yelp/detect-secrets/blob/master/docs/plugins.md , https://github.com/Yelp/detect-secrets/blob/master/detect_secrets/plugins/base.py This is the exact "labeled regex bank → positioned finding" contract the in-repo `LegalEase#3` (`label+index` span) and `uspto_pfw_mcp#12` already follow; reimplement it as an Effect transform rather than porting Python.
- **`uspto_pfw_mcp` ships a patent-domain injection detector with exactly the category set this subtopic needs, and it is MIT-licensed (portable).** The repo describes a 70+ pattern system spanning instruction-override, prompt-extraction, persona-switching, patent-data exfiltration, **examiner-info disclosure**, and API-bypass, integrated via pre-commit + CI with a SHA256-fingerprinted `.prompt_injections.baseline` to separate known from new findings. https://github.com/john-walkoe/uspto_pfw_mcp License: MIT (confirmed on the repo). The examiner-disclosure category is the domain-specific piece with no equivalent in any general-purpose library — it must be hand-built, and this repo is the prior art to mine.
- **The baseline-fingerprint pattern (known-vs-new) is worth porting alongside the regexes.** `detect-secrets` popularized the `--baseline` audit workflow (a JSON ledger of accepted findings so CI only fails on *new* hits); `uspto_pfw_mcp` re-applies it with SHA256 per injection finding. https://github.com/Yelp/detect-secrets For an ingestion boundary that will see the same boilerplate legal phrasing repeatedly, a baseline of accepted spans prevents alert fatigue.
- **doc-haus (`#5`, MIT per CAPTURE) is the closest single source: `instruction-override` + `concealment` regex rules, invisible-Unicode strip-with-counts, raw-DOCX-XML vanish/white/tiny run reading, flag-not-block.** This is in-repo capture (`services/ingest/src/sanitize.ts:68-109`) and the recommended heuristic donor; combine its rule bank with uspto's examiner-disclosure category.

### ML / guardrail libraries (the alternative spine — evaluated, mostly rejected for the hot path)

- **Rebuff (protectai/rebuff): four-layer (heuristics → LLM detector → vector-DB of past attacks → canary tokens) but ARCHIVED read-only since 2025-05-16, and cannot run standalone — needs OpenAI + Pinecone/Chroma + Supabase.** https://github.com/protectai/rebuff/blob/main/README.md Heavy external dependency surface and unmaintained; its *layering idea* (cheap heuristics first, expensive checks later) is worth keeping, the library is not. The Guardrails-AI `detect_prompt_injection` validator is just a wrapper over the Rebuff library (Apache-2.0, also needs OpenAI + Pinecone). https://github.com/guardrails-ai/detect_prompt_injection
- **LLM Guard `PromptInjection` scanner = `protectai/deberta-v3-base-prompt-injection-v2`, Apache-2.0, but English-only and not for document/system content.** Score is a 0–1 maliciousness probability, default threshold 0.5, `MatchType` options for long inputs. https://protectai.github.io/llm-guard/input_scanners/prompt_injection/ The model card reports 95.25% acc / 95.49% F1, max 512 tokens, **and explicitly states it "does not detect jailbreak attacks or handle non-English prompts"** and is not recommended for system prompts (false positives). https://huggingface.co/protectai/deberta-v3-base-prompt-injection-v2 Crucially it targets *user-prompt* classification, not the indirect/embedded injection in retrieved documents — a poor fit for the retrieval-boundary threat model. ONNX export exists, so it *could* run locally via onnxruntime without Python if ever wanted as an optional second opinion.
- **NeMo Guardrails injection detection uses auditable YARA rules + a reject/omit action contract, Apache-2.0 — but it targets code/SQL/XSS/Jinja-template injection in model *output*, not document-embedded instruction-override.** https://docs.nvidia.com/nemo/microservices/latest/guardrails/tutorials/injection-detection.html , https://github.com/NVIDIA-NeMo/Guardrails The YARA-rule "auditable by security teams" framing is a good model for how to express the regex bank; the actual rule content is off-target for this wedge.
- **Lakera Guard is a commercial API (prompt-attack / data-leakage / content / unknown-link categories, ~98% claimed detection, sub-50ms), now owned by Check Point (acquired Sep 2025).** https://www.lakera.ai/lakera-guard , https://docs.lakera.ai/docs/prompt-defense A SaaS call sends document text off-box — disqualifying for privileged legal material. Listed for completeness, not adoption.

### Invisible-Unicode / bidi / zero-width normalization (deterministic pass 1)

- **The danger set is well-bounded and should be detected per-codepoint, not just NFKC'd.** Categories: Unicode Tags block **U+E0000–U+E007F** (ASCII-mirrored, fully invisible — `U+E0041`='A'); zero-width chars **U+200B/200C/200D/FEFF** and soft hyphen **U+00AD**; bidi controls **U+202A–U+202E**, **U+2066–U+2069**, plus **U+200E/200F**; C0/C1 controls **U+0000–U+001F / U+007F–U+009F**. https://labs.cloudsecurityalliance.org/research/csa-research-note-unicode-instruction-injection-ai-skills-20/ , https://blogs.cisco.com/ai/understanding-and-mitigating-unicode-tag-prompt-injection
- **ADVERSARIALLY VERIFIED: NFKC alone does NOT remove zero-width / tag / soft-hyphen characters — explicit range stripping or rejection is required.** Two independent sources agree: the CSA note recommends "apply NFKC … *and* strip or reject codepoints" (NFKC handles homoglyph/compatibility collapse only), and a separate analysis states plainly that "standard Unicode normalization forms (NFC, NFKC) do NOT remove invisible characters and stripping must be performed explicitly via character range filtering." https://labs.cloudsecurityalliance.org/research/csa-research-note-unicode-instruction-injection-ai-skills-20/ , https://ctx-guard.com/blog/invisible-prompt-injection So the pass is two-part: NFKC for homoglyph/confusable collapse, *plus* an explicit invisible-range scan that emits a char-span finding (count + offsets) before any stripping — preserving the flag-not-block contract. Cisco/Swanda's YARA heuristic (`\xF3\xA0[..]??`, flag at >10 occurrences per document) shows density-thresholding to cut false positives on the odd legitimate flag-emoji. https://blogs.cisco.com/ai/understanding-and-mitigating-unicode-tag-prompt-injection
- **The threat is real against models, not just theoretical.** The "Reverse CAPTCHA" study evaluates LLMs actually following invisible Unicode-encoded instructions embedded in normal-looking text, exploiting a perception gap where models read control characters humans cannot see. https://arxiv.org/html/2603.00164v1

### Hidden-DOCX-run / hidden-content reading (deterministic pass 3)

- **PhantomLint (arXiv 2508.17884, repo `tobycmurray/phantom-lint`, BSD-3-Clause, Python) is the strongest prior art for hidden-prompt detection and validates the render-vs-extract approach over brittle white-text regexes.** Core insight: render the document to images, OCR them, and treat "present in extracted text but absent from OCR" as hidden text — format-agnostic across white text, tiny fonts, vanish runs, metadata, and off-page positioning. https://arxiv.org/pdf/2508.17884 , https://github.com/tobycmurray/phantom-lint It is **flag-not-block and char-span native**: it emits `suspicious_phrases.txt` + `hidden_suspicious_phrases.txt` with Unicode-underline highlighting of the exact detected spans for human review. Dependencies: Tesseract + Poppler. Formats today: PDF + HTML (DOCX would be net-new). License (BSD-3) is port-friendly.
- **For DOCX specifically, the cheap deterministic signal is raw-OOXML run inspection: `<w:vanish/>` hides a run's contents at display time** (the doc-haus heuristic). https://www.datypic.com/sc/ooxml/e-w_vanish-1.html , https://ooxml.info/docs/17/17.3/17.3.2/17.3.2.41/ Microsoft documents the inverse (removing `w:vanish` runs), confirming `vanish` is the canonical hidden-text marker to scan for. https://learn.microsoft.com/en-us/office/open-xml/word/how-to-remove-hidden-text-from-a-word-processing-document This gives a tiered design: cheap OOXML-XML scan (vanish / near-background color / sub-threshold font size) on every DOCX, with the expensive PhantomLint render-OCR diff reserved for PDFs or as a second pass.

### Flag-not-block char-span findings contract

- **Every flag-oriented prior-art tool converges on the same contract: positioned span + label/category + concealment-type + confidence, surfaced for human review rather than dropped.** PhantomLint emits char-level highlighted spans split into "suspicious" vs "confirmed-hidden" tiers. https://github.com/tobycmurray/phantom-lint LegalEase (`#3`) returns `{label, match, index}` sorted by offset. detect-secrets returns `PotentialSecret` with positional metadata. https://github.com/Yelp/detect-secrets/blob/master/docs/plugins.md This is the schema to standardize in `@beep/file-processing`: a `Finding` with `{ span: [start,end], category, ruleId, concealment?, confidence, evidence }` written to `@beep/provenance`, advisory only.
- **OWASP LLM01:2025 backs detection-as-one-layer, not detection-as-gate.** It recommends input validation + context segregation + output filtering, and for RAG explicitly "sanitize and clearly delimit retrieved content before injecting it into the prompt context" — i.e. detection feeds delimiting/spotlighting, it does not unilaterally block. https://genai.owasp.org/llmrisk/llm01-prompt-injection/ The defense taxonomy (spotlighting/datamarking/delimiting, paraphrasing, retokenization) is catalogued by tldrsec. https://github.com/tldrsec/prompt-injection-defenses

### Licensing / IP cautions (matters for an IP-law shop)

- **Portable donors are clean: doc-haus MIT, uspto_pfw_mcp MIT, PhantomLint BSD-3, detect-secrets Apache-2.0, deberta-v3 model Apache-2.0.** All permit reimplementation/adaptation. mike/courtlistener (AGPL or unknown) per CAPTURE → reimplement, do not copy. https://github.com/john-walkoe/uspto_pfw_mcp , https://github.com/tobycmurray/phantom-lint , https://huggingface.co/protectai/deberta-v3-base-prompt-injection-v2
- **PATENT CAUTION: US12118471B2 (Preamble Inc, granted 2024-10-15) broadly claims prompt-injection *mitigation* via (a) a classifier that sanitizes/removes malicious commands, (b) RL-based trusted/untrusted token tagging, and (c) trusted/untrusted "data regions" with RL penalties.** https://patents.google.com/patent/US12118471 The RL-token-tagging and "incompatible token sets" claims are the aggressive ones. **Mitigation for this design: stay on the detector/flag side — regex + Unicode-range + render-diff that emits advisory char-span findings is not "tagging tokens as trusted/untrusted and training a model to refuse them," so it reads outside the independent claims.** Avoid building an RL-trained trusted/untrusted token classifier. (Claim-scope reading is the researcher's, not legal advice — see Open/Unverified.)

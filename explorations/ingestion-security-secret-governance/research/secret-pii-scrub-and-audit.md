# secret-pii-scrub-and-audit

Scope: pre-LLM secret/PII scrub (provider-key + PII regex banks → `[REDACTED]`/`[REDACTED_SECRET]`), span-grounded `findPiiMatches` (label+index), OOXML scrub across XML-escape variants incl. identity attrs, and a per-matter audit-row schema homed in `@beep/provenance` — surveyed against detect-secrets, gitleaks, and Microsoft/Data-Privacy-Stack Presidio.

## Findings

### Tooling survey: build-vs-port decision drivers

- **Microsoft Presidio is the reference PII engine, MIT-licensed, but ownership moved in 2025.** The canonical repo `github.com/microsoft/presidio` now redirects/transfers to the `data-privacy-stack` org; docs at `microsoft.github.io/presidio` return `301 Moved Permanently` to `presidio.dataprivacystack.org`, and the contact is now `presidio@dataprivacystack.org` ([docs redirect](https://microsoft.github.io/presidio/), [new docs host](https://presidio.dataprivacystack.org/anonymizer/)). License is MIT, copyright Microsoft Corporation ([LICENSE](https://github.com/microsoft/presidio/blob/main/LICENSE)). MIT permits porting heuristics/shapes; the stewardship move is a maintenance-risk signal worth noting before taking a runtime dependency vs. porting patterns. **UNVERIFIED:** exact transfer date and whether `microsoft/presidio` still receives releases post-transfer.
- **Presidio is a Python/REST engine (analyzer + anonymizer + image-redactor), not a JS library.** It runs locally or in containers and integrates via Python/.NET/REST ([overview](https://microsoft.github.io/presidio/)). For an Effect/TS repo this means porting the *contract* (recognizer span shape, operator semantics) rather than calling it inline — consistent with CAPTURE.md's "reimplement, do not copy" posture for the JS gold nuggets.
- **detect-secrets (Yelp) is Apache-2.0** and uses three plugin families: regex detectors (provider key patterns), Shannon-entropy detectors, and a keyword denylist; entropy thresholds are tunable (`--base64-limit` default 4.5, `--hex-limit` default 3.0, range 0.0–8.0) ([repo](https://github.com/Yelp/detect-secrets), [plugins doc](https://github.com/Yelp/detect-secrets/blob/master/docs/plugins.md)). Its defining feature is the **baseline-diff** model (only *new* secrets alert) and optional verifiability (live API check) since v0.12.4 ([DeepWiki advanced features](https://deepwiki.com/Yelp/detect-secrets/6-advanced-features)). The patent-domain injection detector in CAPTURE.md (`uspto_pfw_mcp#12`) is a detect-secrets plugin, so the plugin-regex-bank shape is the porting target.
- **gitleaks is MIT (the scanner binary)** and ships 100+ regex rules combining `regex + entropy + keywords` per rule; in v8.25.0 the config key `[allowlist]` was replaced by `[[allowlists]]`, and global allowlists outrank rule-scoped ones ([repo](https://github.com/gitleaks/gitleaks), [gitleaks.toml](https://github.com/gitleaks/gitleaks/blob/master/config/gitleaks.toml)). gitleaks-action ≥ v2.0.0 is *not* MIT (commercial license), but the rule library and binary remain MIT ([license discussion](https://github.com/gitleaks/gitleaks/discussions/1805)). gitleaks' `gitleaks.toml` is the best-maintained canonical regex bank to mirror.

### Canonical provider-key regex shapes (verified against gitleaks.toml + GitHub primary docs)

These are the authoritative shapes to seed the `SECRET_PATTERN_SOURCES` bank; the agentmemory nugget's patterns (CAPTURE.md `agentmemory#11`) are looser variants of the same families.

- **Anthropic:** gitleaks `anthropic-api-key` = `\b(sk-ant-api03-[a-zA-Z0-9_\-]{93}AA)(?:[\x60'"\s;]|\\[nr]|$)`, keyword `sk-ant-api03` ([gitleaks.toml](https://github.com/gitleaks/gitleaks/blob/master/config/gitleaks.toml)). Format is self-describing: `sk`=secret, `ant`=Anthropic, `api03`=3rd-gen scheme, ~95 base64url body chars, **no checksum** (only "works"/"doesn't"), total ≈108 chars ([Anthropic key anatomy](https://bip39-phrase.com/anthropic-api-key-format/)). Note the gold nugget's `sk-ant-[A-Za-z0-9\-_]{20,}` is intentionally looser (catches `sk-ant-oat01` OAuth keys too) at the cost of false positives.
- **OpenAI:** gitleaks `openai-api-key` = `\b(sk-(?:proj|svcacct|admin)-(?:[A-Za-z0-9_-]{74}|[A-Za-z0-9_-]{58})T3BlbkFJ(?:[A-Za-z0-9_-]{74}|[A-Za-z0-9_-]{58})\b|sk-[a-zA-Z0-9]{20}T3BlbkFJ[a-zA-Z0-9]{20})`, keyword `t3blbkfj` ([gitleaks.toml](https://github.com/gitleaks/gitleaks/blob/master/config/gitleaks.toml)). The literal `T3BlbkFJ` (base64 "OpenAI") is embedded mid-token; sub-prefixes `proj-`/`svcacct-`/`admin-` since the mid-2024 Projects launch; **no checksum**, validity only confirmable via API ([key format](https://vibekit.bot/openai-api-key-format), [length thread](https://community.openai.com/t/project-api-key-length-has-it-changed-from-48-to-156/920777)). Project-key bodies grew from 48 chars to ~155 in 2024 — length-anchored regexes are brittle; anchor on prefix + `T3BlbkFJ`.
- **GitHub:** `ghp_[0-9a-zA-Z]{36}` (PAT), `github_pat_\w{82}` (fine-grained), and the family `gho_`/`ghu_`/`ghs_`/`ghr_` ([gitleaks.toml](https://github.com/gitleaks/gitleaks/blob/master/config/gitleaks.toml)). Primary source: the `_` separator is deliberately **not** a Base64 char; the last 6 chars are a **CRC32 checksum (Base62, leading-zero padded)**, enabling false-positive rates ≈0.5% — a validator can verify the checksum offline ([GitHub Blog: new token formats](https://github.blog/engineering/platform-security/behind-githubs-new-authentication-token-formats/)).
- **AWS:** gitleaks `aws-access-token` = `\b((?:A3T[A-Z0-9]|AKIA|ASIA|ABIA|ACCA)[A-Z2-7]{16})\b`, entropy 3 ([gitleaks.toml](https://github.com/gitleaks/gitleaks/blob/master/config/gitleaks.toml)). Access key IDs are 20 chars; `AKIA`=long-term IAM/root creds ([AWS STS GetAccessKeyInfo](https://docs.aws.amazon.com/STS/latest/APIReference/API_GetAccessKeyInfo.html)). The gold nugget's `AKIA[0-9A-Z]{16}` is the AKIA-only subset.
- **Google/GCP:** gitleaks `gcp-api-key` = `\b(AIza[\w-]{35})(?:[\x60'"\s;]|\\[nr]|$)`, entropy 4 ([gitleaks.toml](https://github.com/gitleaks/gitleaks/blob/master/config/gitleaks.toml)).
- **Slack bot:** gitleaks `slack-bot-token` = `xoxb-[0-9]{10,13}-[0-9]{10,13}[a-zA-Z0-9-]*`, keyword `xoxb`. Slack prefixes are case-sensitive: `xoxb-` (bot), `xoxp-` (user), plus `xoxa/xoxr/xoxo/xoxs/xoxe` ([Slack token docs](https://docs.slack.dev/authentication/tokens/)); modern bot tokens have variable trailing segments, so trailing-length anchoring is unsafe.
- **JWT:** gitleaks `jwt` = `\b(ey[a-zA-Z0-9]{17,}\.ey[a-zA-Z0-9\/\\_-]{17,}\.(?:[a-zA-Z0-9\/\\_-]{10,}={0,2})?)`, keyword `ey` ([gitleaks.toml](https://github.com/gitleaks/gitleaks/blob/master/config/gitleaks.toml)). The gold nugget's `eyJ...` requires the leading `{"` (base64 `eyJ`) — narrower but equivalent in practice since JWT headers start `{"`.
- **Private key:** gitleaks `private-key` = `(?i)-----BEGIN[ A-Z0-9_-]{0,100}PRIVATE KEY(?: BLOCK)?-----[\s\S-]{64,}?KEY(?: BLOCK)?-----` ([gitleaks.toml](https://github.com/gitleaks/gitleaks/blob/master/config/gitleaks.toml)).

### Two-tier replacement tokens (`[REDACTED]` vs `[REDACTED_SECRET]`)

- The agentmemory nugget distinguishes **private-tag content** (`/<private>[\s\S]*?<\/private>/gi` → `[REDACTED]`) from **provider-key matches** (`SECRET_PATTERN_SOURCES` → `[REDACTED_SECRET]`) (CAPTURE.md `agentmemory#11`). This two-token scheme is defensible: it preserves *why* a span was scrubbed in the output itself (author-marked confidential vs. machine-detected credential), which an audit row can then key on. Presidio's analog is per-entity operator config (different `new_value` per `entity_type`) — see below.

### Span-grounded detection: `findPiiMatches` ↔ Presidio RecognizerResult ↔ `@beep/provenance` TextAnchor

- The LegalEase nugget's `findPiiMatches(text, patterns)` returns `{label, match, index}[]` sorted by `index` (CAPTURE.md `LegalEase#3`) — a label + char-offset span. This is structurally **the same contract** Presidio exposes: `RecognizerResult` has `entity_type, start, end, score, analysis_explanation, recognition_metadata`; `analyze()` returns `List[RecognizerResult]`; `to_dict()` = `self.__dict__`; spans are **half-open** (`text[result.start:result.end]`) ([recognizer_result.py](https://github.com/microsoft/presidio/blob/main/presidio-analyzer/presidio_analyzer/recognizer_result.py), [analyzer API](https://microsoft.github.io/presidio/api/analyzer_python/)).
- **Recommended span shape:** adopt `{label, start, end, score?}` (Presidio-aligned, half-open) over `{label, match, index}`. Reason: `@beep/provenance` already defines `TextAnchor` as a **half-open char range `[startChar, endChar)` + quoted `quote`** where `text.slice(startChar, endChar) === quote` (repo: `packages/foundation/modeling/provenance/src/TextAnchor.ts`). A PII finding that reuses `TextAnchorFields` (exported for spreading) gets re-sliceable, model-stable provenance for free and stays consistent with langextract/epistemic span semantics. `findPiiMatches`'s `index` is the `start`; `start + match.length` is the `end` — a trivial adapter.
- Presidio's `score` (confidence float) and `analysis_explanation` map to the optional confidence/reason that the epistemic slice already layers on top of `TextAnchor` (`EvidenceSpan` adds `Confidence`) — so a scored PII finding is a natural `TextAnchor` + score, not a new primitive.

### OOXML scrub across XML-escape variants incl. identity attributes

- XML defines exactly five predefined entities: `&amp; &lt; &gt; &quot; &apos;` ([W3C XML 1.0 §4.6](https://www.w3.org/TR/xml/), [entity reference list](https://en.wikipedia.org/wiki/List_of_XML_and_HTML_character_entity_references)). **Escaping differs by context:** in element text you must escape `&` and `<` (and `>` inside `]]>`); in an attribute value you must *also* escape whichever quote delimits the attribute (`"`→`&quot;` for double-quoted, `'`→`&apos;` for single-quoted) ([W3C XML 1.0](https://www.w3.org/TR/xml/), [attribute-escaping guide](https://learn-it-university.com/escaping-special-characters-in-xml-attribute-values-a-complete-guide/)).
- The doc-haus `xmlVariants(text)` (CAPTURE.md `doc-haus#13`) produces a `Set` of two encodings — a `textNode` variant (`&`,`<`,`>` escaped) and an `attribute` variant (additionally `"`→`&quot;`, `'`→`&apos;`) — and scrubs the needle in **every** form. This correctly covers both XML contexts above: the textNode form matches `<w:t>` body text, the attribute form matches values inside attributes. This is the right defensive shape because the same PII string appears differently-escaped depending on where Word serialized it.
- **Identity attributes are the load-bearing reason** to scrub attribute variants: OOXML comments (`comments.xml`, `CT_Comment` which *extends* `CT_TrackChange`) carry `w:author`, `w:initials`, and `w:date` as **attributes**, e.g. `<w:comment w:id="1" w:initials="User" w:author="Dan Battagin">` ([MS Learn: insert a comment](https://learn.microsoft.com/en-us/office/open-xml/word/how-to-insert-a-comment-into-a-word-processing-document), [datypic w:comment](https://datypic.com/sc/ooxml/e-w_comment-1.html)). Track-change marks (`w:ins`/`w:del`) likewise stamp `w:author`. A scrub that only walks `<w:t>` text leaks attorney/reviewer identity in metadata; scrubbing the attribute-escaped variant across all XML parts is what closes that leak.
- doc-haus is **MIT** (CAPTURE.md cautions) → adapt the `xmlVariants` + residue-check heuristics directly. Its residue check that reports *unreachable binary parts* (images/embeddings) rather than claiming "clean" is the honest-audit posture to preserve.

### Presidio anonymizer operator semantics (the scrub-output contract)

- Operators: `replace`, `redact`, `mask`, `hash`, `encrypt`, `keep`, `custom`, plus `surrogate_ahds`; deanonymize via `decrypt` ([anonymizer docs](https://presidio.dataprivacystack.org/anonymizer/)). **Default when no operator/`DEFAULT` key is given: `replace` producing `<ENTITY_TYPE>`** (e.g. `<PHONE_NUMBER>`). `redact` removes the span entirely (no token); `mask` replaces N chars with a masking char; `hash` uses a **random salt by default since v2.2.361**; `encrypt` is key-reversible ([anonymizer docs](https://presidio.dataprivacystack.org/anonymizer/)). `AnonymizerEngine.anonymize(text, analyzer_results, operators={ENTITY: OperatorConfig("replace", {"new_value": "<X>"})})` ([guide](https://explainx.ai/blog/microsoft-presidio-pii-detection-anonymization-guide-2026)).
- Design implication: the repo's `[REDACTED]`/`[REDACTED_SECRET]` tokens are Presidio's `replace` operator with a fixed `new_value` per category — keeping the placeholder (vs. `redact`'s full removal) means the LLM still sees that *something* was there, preserving sentence structure while denying the value. That is the right default for a pre-LLM prompt scrub.

### PII regex false-positive discipline (don't ship raw regex)

- Regex alone over-fires; production pipelines add a **validation stage**: Luhn check for credit cards (and BIN-range checks, e.g. Mastercard `51`–`55`), SSN structural rules (area ≠ `000`/`666`/`900–999`, group ≠ `00`, serial ≠ `0000`), and **keyword/context gating** so a pattern only fires near a contextual token ([PII validation survey](https://www.conduktor.io/glossary/pii-detection-and-handling-in-event-streams), [DLP predefined patterns](https://docs.versa-networks.com/Integrations_and_Solutions/SSE_Use_Cases/DLP_Predefined_Patterns_Quick-Reference)). gitleaks encodes the same idea as per-rule `entropy` + `keywords` thresholds. For a *redaction* (not alerting) use-case, over-redaction is safer than under-redaction, so the scrub can run looser than a scanner — but the **audit row must record score/validator outcome** so a human can spot over-redaction.

### Audit-row schema home: `@beep/provenance` (with an in-repo precedent in ai-metrics)

- `@beep/provenance` (`packages/foundation/modeling/provenance/src/index.ts`) is explicitly the "domain-agnostic provenance value models … where did this come from?" substrate and currently exports only `VERSION` + `TextAnchor`. It is the correct home for a `RedactionAuditRow` value: it already owns the span primitive the audit row references, and the `$ProvenanceId` identity composer is wired. The per-matter audit fields from doc-haus (`redacted_text`, `label`, `reason`, `author`, occurrence counts) map onto a `Model.Class` that embeds a `TextAnchor` (or `TextAnchorFields` spread) + matter id + actor + reason + timestamp.
- **In-repo precedent to mirror, not the gold nuggets:** `packages/tooling/library/ai-metrics/src/privacy.ts` already implements a pre-store secret scrub with a "redaction proof" schema. It defines regex banks `SECRET_ASSIGNMENT_PATTERN`, `AUTH_HEADER_PATTERN`, `BEARER_PATTERN`, `OPENAI_KEY_PATTERN` (`/\b(sk-[A-Za-z0-9_-]{8,})\b/gu`), counts matches, and emits `AiMetricsRedactionResult` (`S.Class` with `authHeaderCount`, `bearerTokenCount`, `openAiKeyCount`, `secretAssignmentCount`, `excludedRawTextFieldCount`, and a `safeForDerivedUi: S.Boolean` gate). This is exactly the "scrub-before-it-crosses-a-boundary + emit a counted proof" contract this subtopic needs, already in the repo's schema-first idiom — the new per-matter audit row should generalize `AiMetricsRedactionResult` (add span anchors + matter/actor) rather than invent a parallel shape, and the `safeForDerivedUi` boolean is the model for a `safeForPrompt` gate.

### Placement recommendation (scrub boundary)

- **Scrub at the retrieval/ingestion boundary in `@beep/file-processing`, audit-row + finding shape in `@beep/provenance`, regex banks as a shared schema/utility.** Rationale, cross-checked against CAPTURE.md netNew#5 and the live tree: (1) the prompt scrub must run *after* OOXML/text extraction but *before* any text reaches a shared store or an LLM driver — that is the file-processing retrieval boundary the injection detector (netNew#1) also sits on, so they share one pass; (2) the audit row and span findings are pure provenance values → `@beep/provenance`; (3) the provider-key + PII regex bank is a reusable constant set with no Effect deps → a `@beep/schema`-adjacent utility, co-located with the existing ai-metrics privacy regexes so there is **one** secret-pattern source of truth rather than two (the CAPTURE "two resolvers" caution applies equally to two regex banks).

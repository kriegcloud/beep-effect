# ssrf-and-fetch-hardening

Scope: SSRF defenses for user-supplied URLs (remote MCP servers, source ingestion) beyond the `@beep/schema` `SafeRemoteHost` allowlist — DNS-resolution + private/reserved-IP rejection, cloud-metadata blocking, `redirect:'manual'` guarded fetch, credential/hash stripping, header-name allowlisting — and where the guarded-fetch boundary lives in the Effect v4 HttpClient stack.

## Findings

### What the repo already has (baseline)

- `@beep/schema` `SafeRemoteHost` (`packages/foundation/modeling/schema/src/SafeRemoteHost.ts`) is a **pure, I/O-free schema guard**. It exposes `isBlockedRemoteHost` (predicate), `assertAllowedRemoteHost`/`assertAllowedRemoteUrl` (fail-closed Effect assertions raising typed `BlockedHostError`), an optional `allowlist`, and an **opt-in `resolve` hook** (`RemoteHostResolver = (hostname) => Effect<readonly string[], BlockedHostError>`) so driver packages inject `node:dns` rather than coupling a foundation schema to I/O. (in-repo source, read directly)
- It classifies the literal host as loopback (`127.`, `0.0.0.0`, `::1`, `::`), link-local (`169.254.`, `fe80:`), RFC1918/ULA (`10.`, `192.168.`, `172.16–31.`, `fc`/`fd`), `localhost`/`.localhost`, and cloud-metadata `169.254.169.254`, and **decodes IPv4-mapped IPv6** (`::ffff:c0a8:101`) back to dotted IPv4 so mapped private ranges classify identically (`extractMappedIpv4`). (in-repo source)
- The module `@remarks` **explicitly declares the residual gap**: even with a `resolve` hook, a DNS-rebinding attacker can return a public IP to the guard and an internal IP to the subsequent connection (TOCTOU). "eliminating that residual risk requires pinning the resolved IP at the HTTP client connection layer, which is out of scope for this schema-level guard." (in-repo source, `SafeRemoteHost.ts` lines 26-31)
- Consumers today: `packages/drivers/box/src/Box.streaming.ts` calls `assertAllowedRemoteUrl(url, { resolve: resolveRemoteHost })` where `resolveRemoteHost` wraps `dns.promises.lookup(hostname, { all: true })` — so **Box already closes the validate-time DNS gap but NOT the connect-time rebinding gap**. `packages/drivers/uspto/src/Uspto.service.ts` calls `assertAllowedRemoteUrl(downloadUrl, …)`. `packages/drivers/nlp-mcp/src/Streaming/DatasetLoader.ts` carries a **deliberately duplicated** literal-only blocklist (no resolver, no guarded fetch). (in-repo grep)
- **Net-new gaps confirmed by grep**: the repo has no `connect.lookup` pinned dispatcher, no `redirect:'manual'` guarded fetch loop, no header-name allowlist, and no URL credential/hash stripping anywhere. (in-repo grep across `packages/`, `apps/`)

### Core threat model & OWASP guidance

- OWASP's primary rule is **prefer allowlists over denylists**: "Deny-lists are bypass-prone. Prefer allow-lists." Denylists of RFC1918/loopback are insufficient standalone because attackers bypass via DNS rebinding, IPv6 representations, decimal/octal/hex IP encoding, and URL-encoded characters. [OWASP SSRF Prevention Cheat Sheet](https://cheatsheetseries.owasp.org/cheatsheets/Server_Side_Request_Forgery_Prevention_Cheat_Sheet.html)
- OWASP also advises: **don't accept a full URL from the user** — accept a validated host/IP and path separately; **disable HTTP redirect following** in the client to stop validation bypass; **disable risky URL schemes** (`file://`, `gopher://`, `data://`, `dict://`); use a parser resistant to encoding bypasses and **compare the parser's normalized output**, not the raw string. For JS, OWASP names the `ip-address` library as encoding-bypass-resistant. [OWASP SSRF Prevention Cheat Sheet](https://cheatsheetseries.owasp.org/cheatsheets/Server_Side_Request_Forgery_Prevention_Cheat_Sheet.html), [OWASP SSRF Prevention in Node.js](https://owasp.org/www-community/pages/controls/SSRF_Prevention_in_Nodejs)
- The canonical SSRF payload is the **cloud metadata service**, reachable from inside the host but not the public internet, used to steal instance credentials. [OWASP SSRF Prevention Cheat Sheet](https://cheatsheetseries.owasp.org/cheatsheets/Server_Side_Request_Forgery_Prevention_Cheat_Sheet.html)

### Cloud-metadata hosts to block (current `SafeRemoteHost` is incomplete)

- `169.254.169.254` is the shared metadata IP for **AWS, GCP, Azure, and DigitalOcean** (link-local; already covered by the repo's `169.254.` prefix). [HackTricks cloud-ssrf](https://github.com/b4rdia/HackTricks/blob/master/pentesting-web/ssrf-server-side-request-forgery/cloud-ssrf.md), [Cloud Metadata Dictionary (jhaddix)](https://gist.github.com/jhaddix/78cece26c91c6263653f31ba453e273b)
- **Not covered by the repo today**: Alibaba `100.100.100.200`; Oracle Cloud `192.0.0.192`; GCP DNS name `metadata.google.internal` (and `metadata`); and the broader reserved ranges OWASP/HackTricks list — **`100.64.0.0/10`** (CGNAT/shared, e.g. GKE/EKS), `192.0.0.0/24`, `192.0.2.0/24` (TEST-NET-1), `198.18.0.0/15` (benchmark), `198.51.100.0/24` (TEST-NET-2), `203.0.113.0/24` (TEST-NET-3), `224.0.0.0/4` (multicast), `240.0.0.0/4` (reserved/Class E). [HackTricks cloud-ssrf](https://github.com/b4rdia/HackTricks/blob/master/pentesting-web/ssrf-server-side-request-forgery/cloud-ssrf.md), [Cloud Metadata Dictionary (jhaddix)](https://gist.github.com/jhaddix/78cece26c91c6263653f31ba453e273b)
- Defense-in-depth at the cloud layer (not a substitute for the app guard): **AWS IMDSv2** requires a PUT-session token and rejects requests bearing `X-Forwarded-For`; GCP's metadata server auto-rejects requests carrying `X-Forwarded-For`. Migrate off IMDSv1. [OWASP SSRF Prevention Cheat Sheet](https://cheatsheetseries.owasp.org/cheatsheets/Server_Side_Request_Forgery_Prevention_Cheat_Sheet.html)

### undici / native fetch has NO built-in SSRF protection

- Tracking issue [nodejs/undici#2019](https://github.com/nodejs/undici/issues/2019) (open since 2023-03) confirms undici and Node native `fetch` (v18+) expose **no documented SSRF mechanism**. The `http.Agent`-based middleware that protects `axios`/`node-fetch` (`ssrf-req-filter`, `request-filtering-agent`) **does not apply to undici/native fetch**, because undici routes through a `Dispatcher`, not `http.Agent` — `fetch(url, { agent })` is silently ignored. The community recommendation is a **custom `connect.lookup` on an undici `Agent`** (a `{host}=>boolean`-style IP gate at connect time). [nodejs/undici#2019](https://github.com/nodejs/undici/issues/2019)
- Installed in this repo: **undici 7.28.0** (pinned by `@effect/platform-node@4.0.0-beta.91`); latest published undici is 8.5.0. (in-repo `node_modules` inspection)

### Effect v4 HttpClient is the correct interception boundary

- Effect v4 moved the HTTP client to `effect/unstable/http/HttpClient`; `@effect/platform-node/NodeHttpClient` provides the Undici-backed client. It exposes a **`Dispatcher` service** (`Context.Service<Dispatcher, Undici.Dispatcher>`), `makeDispatcher` (scoped `Agent`), `layerDispatcher`, `dispatcherLayerGlobal`, `layerUndici`, and `layerUndiciNoDispatcher`. `@effect/platform-node/Undici` re-exports the raw undici named exports (`Agent`, `setGlobalDispatcher`, …). **This means a custom pinned undici `Agent` can be injected by providing your own `Dispatcher` layer** — the seam for connect-time IP pinning already exists. (in-repo source `NodeHttpClient.ts`, `Undici.ts`)
- The Undici-backed `makeUndici` calls `dispatcher.request({ … origin: url.origin, path: url.pathname + url.search + url.hash … })` with **no `maxRedirections`**, so the Effect Undici client **does not follow redirects by default** — a 3xx is returned to the caller (effectively `redirect:'manual'`). Note it forwards `url.hash` into the path, so **stripping the fragment is meaningful**, and it uses `url.origin` (which drops `user:pass@` userinfo, but stripping/rejecting userinfo explicitly is still wanted so allowlist checks and logs don't see confusable credentials). (in-repo source `NodeHttpClient.ts` lines 143-177)
- `HttpClient` transform combinators available for the guard layer: `mapRequestEffect` / `mapRequestInputEffect` (effectful request transform — ideal home for `assertAllowedRemoteUrl` + URL normalization), `mapRequest`, `transformResponse`, `tapRequest`, `filterStatusOk`, and `followRedirects(maxRedirects?)`. [Effect HttpClient API](https://effect-ts.github.io/effect/platform/HttpClient.ts.html) (verified against in-repo `effect/unstable/http/HttpClient.ts`)
- **DO NOT use the built-in `HttpClient.followRedirects` for untrusted URLs.** Its implementation (in-repo `HttpClient.ts` ~line 1840) loops on `response.headers.location` via `HttpClientRequest.setUrl(request, new URL(location, request.url))` up to `maxRedirects ?? 10` and **(a) does not re-run any SSRF check on the new hop and (b) carries the original headers — including `Authorization`/`Cookie` — across cross-origin redirects.** That reintroduces the exact redirect-SSRF and credential-leak holes a guarded fetch must close. A guarded redirect loop must re-validate every `Location` and strip auth/cookie headers on cross-origin hops. (in-repo source)

### Redirect & credential-stripping hazards (verified against undici CVE)

- Redirect-SSRF is a live, common bug: a 30x `Location` pointing at loopback/metadata is dialed even though the *initial* URL passed validation. The fix is `redirect:'manual'` + re-validate each hop and own the cross-origin policy. [nodejs/undici#2019](https://github.com/nodejs/undici/issues/2019) and the SSRF-via-redirect advisories surveyed (e.g. Budibase [GHSA-fgqv-jh4g-pvg2](https://github.com/Budibase/budibase/security/advisories/GHSA-fgqv-jh4g-pvg2), protocol-change bypass write-ups).
- Credential stripping must cover **all** auth entries, not just `Authorization`: **CVE-2024-24758** — undici fetch stripped `Authorization` but **not `Proxy-Authorization`** on cross-origin redirect, leaking proxy creds. Affected ≤5.28.2 and 6.0.0–6.6.0; patched 5.28.3 / 6.6.1 (2024-02-16). The repo's undici 7.28.0 is past the fix, but the lesson stands for any hand-rolled redirect loop: strip `Authorization`, `Cookie`, **and** `Proxy-Authorization` on cross-origin hops. [undici GHSA-3787-6prv-h9w3 (CVE-2024-24758)](https://github.com/nodejs/undici/security/advisories/GHSA-3787-6prv-h9w3)

### The DNS-pinning pattern that closes TOCTOU (connect-layer)

- Canonical undici fix: a pinned dispatcher — `new Agent({ connect: { lookup: pinnedLookup } })` — where `pinnedLookup` (matching the `node:dns.lookup` callback signature) resolves once, runs every returned A/AAAA address through the internal-range classifier, and hands undici **only the validated addresses it then connects to**, so the validated IP and the connected IP are identical. Pass the agent as the request `dispatcher` (or set it as the `Dispatcher` service in the Effect layer). [nodejs/undici#2019](https://github.com/nodejs/undici/issues/2019)
- This is strictly stronger than `SafeRemoteHost`'s opt-in `resolve` hook (which validates a *separate* resolution from the one undici later performs). Both are wanted: the schema guard fails fast on obvious literals/known-bad names before a socket is opened; the pinned dispatcher is the only thing that defeats active DNS rebinding. [DNS rebinding SSRF walkthrough (2026)](https://aydinnyunus.github.io/2026/03/14/ssrf-dns-rebinding-vulnerability/), [OWASP SSRF Prevention Cheat Sheet](https://cheatsheetseries.owasp.org/cheatsheets/Server_Side_Request_Forgery_Prevention_Cheat_Sheet.html)

### Library survey (licensing for any port)

- **`request-filtering-agent`** (azu) — **MIT**, **v3.2.1, actively maintained (last publish 2026-06-29)**, Node 20/22, IP classification via `ipaddr.js`. Subclasses `http(s).Agent` and validates **at connection/socket time** (so it resists rebinding for `node:http`/`https`/`node-fetch`). Options: `allowPrivateIPAddress`, `allowMetaIPAddress`, `allowIPAddressList`, `denyIPAddressList` (allow precedes deny). **Caveat: `http.Agent` → does not work with undici/native fetch**, so it is a reference for the classifier + connect-time check, not a drop-in for Effect's Undici client. [request-filtering-agent (npm)](https://www.npmjs.com/package/request-filtering-agent), [azu/request-filtering-agent](https://github.com/azu/request-filtering-agent) (version/license/date verified via `npm view`)
- **`ssrf-req-filter`** (y-mehta) — **MIT**, **v1.1.1, stale (last publish 2024-05-11)**, ~22 stars, `http.Agent`-based (axios/node-fetch only). Its own README warns you must override **both** `httpAgent` and `httpsAgent` or be bypassed via **cross-protocol redirect** (cites Doyensec research). Same undici incompatibility. Lower-quality reference than azu's. [ssrf-req-filter (npm)](https://www.npmjs.com/package/ssrf-req-filter), [y-mehta/ssrf-req-filter](https://github.com/y-mehta/ssrf-req-filter) (version/license/date verified via `npm view`)
- **IP-classification primitives**: `ipaddr.js` (**MIT**, v2.4.0) — `.range()` returns `'private'|'loopback'|'linkLocal'|'reserved'|…`, used by azu's agent; `ip-address` (**MIT**, v10.2.0) — OWASP's recommended encoding-bypass-resistant parser. Either is a sound dependency for a robust CIDR-based classifier to replace the repo's current string-prefix checks (which miss `100.64.0.0/10`, `192.0.0.192`, `100.100.100.200`, multicast, Class E). (versions/licenses verified via `npm view`)

### Recommendation — where the guarded-fetch boundary lives

- **Keep `@beep/schema` `SafeRemoteHost` exactly as-is** (pure, I/O-free, the fail-fast literal+optional-resolve gate). Extend only its **range table** to add the missing reserved/metadata ranges (`100.64.0.0/10`, `192.0.0.192`, `100.100.100.200`, `metadata.google.internal`, `192.0.2/198.18/198.51.100/203.0.113`, `224.0.0.0/4`, `240.0.0.0/4`) — ideally by swapping the string-prefix checks for an `ipaddr.js`/`ip-address` CIDR classifier so encoding bypasses are normalized away.
- **Add one net-new `GuardedHttpClient` layer** (an `HttpClient` transform that wraps `NodeHttpClient.layerUndici`) in a platform-adjacent home that may import `node:dns`/`undici` — NOT in `@beep/schema`. It composes four things: (1) `mapRequestEffect` running `assertAllowedRemoteUrl` + HTTPS-only + strip userinfo + strip `#hash` + header-name allowlist before send; (2) a **manual redirect loop** (no built-in `followRedirects`) that re-runs the guard on every `Location` and strips `Authorization`/`Cookie`/`Proxy-Authorization` on cross-origin hops; (3) a **pinned undici `Agent({ connect: { lookup } })` provided as the `Dispatcher` layer** to defeat rebinding at connect time; (4) the existing `resolve` injection wired to that same lookup. This is the single net-new#4 deliverable; the three current consumers (Box already wires a resolver, USPTO, nlp-mcp's duplicated guard) collapse onto it.

## Open / Unverified

- **ssrf-req-filter exact mechanism UNVERIFIED at source level** — the GitHub raw `index.js` 404'd; whether it overrides `createConnection`/checks at socket-connect time (vs. literal-host only) is inferred from README, not read. (azu's connect-time behavior IS confirmed via its docs.) Low impact: it is a stale, lower-quality reference we would reimplement, not port.
- **Pinned-`connect.lookup` code shape is community-recommended, not officially documented** — undici#2019 remains open with no maintainer-blessed built-in; the `new Agent({ connect: { lookup } })` pattern is the accepted DIY fix but should be prototyped + unit-tested against a rebinding harness (resolve public → connect private) before relying on it.
- **Effect built-in `followRedirects` header/credential behavior** read from in-repo `HttpClient.ts` source (carries headers via `setUrl`, no per-hop re-validation) but **not cross-checked against an upstream Effect doc/changelog** — treat the "must hand-roll the redirect loop" conclusion as source-grounded but version-pinned to beta.91.
- **`metadata.google.internal` / `metadata` name blocking** depends on resolver behavior; blocking the name is belt-and-suspenders since it resolves to `169.254.169.254` (already covered), but split-horizon/custom DNS could differ — verify in the target deploy environment.
- Whether the new `GuardedHttpClient` lives in a fresh `@beep/http` foundation package vs. a `@beep/file-processing`/driver-shared module is an **align-stage decision**, not settled here.

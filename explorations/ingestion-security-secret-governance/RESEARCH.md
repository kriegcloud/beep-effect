## External Landscape

### 1. Prompt-injection / hidden-content detection
Raw: [`research/prompt-injection-detection-landscape.md`](research/prompt-injection-detection-landscape.md)

- **The defensible shape is a deterministic, three-pass, flag-not-block detector that emits char-span findings — no ML model on the hot path.** Passes: (1) invisible-Unicode normalize + detect, (2) regex heuristic banks (instruction-override / examiner-disclosure / exfiltration / persona-switch), (3) hidden-content render-vs-extract diff. OWASP LLM01:2025 backs detection-as-one-layer (sanitize + delimit + spotlight), not detection-as-gate (<https://genai.owasp.org/llmrisk/llm01-prompt-injection/>; defense taxonomy <https://github.com/tldrsec/prompt-injection-defenses>).
- **Regex-bank prior art is portable.** Yelp `detect-secrets` is the canonical plugin shape to *mirror* (a `RegexBasedDetector.analyze_string` → positioned `PotentialSecret`), Apache-2.0 (<https://github.com/Yelp/detect-secrets/blob/master/docs/plugins.md>). `uspto_pfw_mcp` ships a ~70-pattern patent-domain injection detector spanning instruction-override, prompt-extraction, persona-switching, exfiltration, **examiner-info disclosure**, and API-bypass, MIT-licensed and confirmed (<https://github.com/john-walkoe/uspto_pfw_mcp>) — the examiner-disclosure category has **no general-library equivalent and must be hand-built**. The `--baseline` known-vs-new SHA256 ledger pattern (detect-secrets) is worth porting to suppress alert fatigue on boilerplate legal phrasing.
- **Invisible-Unicode must be range-stripped, not just NFKC'd — adversarially verified across two sources.** NFKC collapses homoglyphs/compatibility only; it does **not** remove zero-width (U+200B/C/D/FEFF), soft hyphen (U+00AD), bidi controls (U+202A–E, U+2066–9), or Unicode Tags (U+E0000–E007F, ASCII-mirrored, fully invisible) — explicit codepoint-range scan/reject is required (CSA <https://labs.cloudsecurityalliance.org/research/csa-research-note-unicode-instruction-injection-ai-skills-20/>; <https://ctx-guard.com/blog/invisible-prompt-injection>; Cisco density-threshold YARA <https://blogs.cisco.com/ai/understanding-and-mitigating-unicode-tag-prompt-injection>). Models actually follow such hidden instructions ("Reverse CAPTCHA", <https://arxiv.org/html/2603.00164v1>).
- **Hidden-DOCX/PDF content: render-vs-extract is the strongest signal.** PhantomLint (arXiv 2508.17884, repo `tobycmurray/phantom-lint`, **BSD-3**, render→OCR→diff, flag-not-block char-span native) validates the approach over brittle white-text regexes; today PDF+HTML only, DOCX would be net-new (<https://arxiv.org/pdf/2508.17884>, <https://github.com/tobycmurray/phantom-lint>). The cheap DOCX signal is raw-OOXML `<w:vanish/>` run inspection (<https://ooxml.info/docs/17/17.3/17.3.2/17.3.2.41/>; MS hidden-text removal <https://learn.microsoft.com/en-us/office/open-xml/word/how-to-remove-hidden-text-from-a-word-processing-document>).
- **ML/guardrail libs evaluated and mostly rejected for the hot path.** Rebuff is **archived read-only since 2025-05-16** and needs OpenAI+Pinecone+Supabase (<https://github.com/protectai/rebuff/blob/main/README.md>). LLM Guard's DeBERTa-v3 (`protectai/deberta-v3-base-prompt-injection-v2`, Apache-2.0) is **English-only, user-prompt-only, explicitly not for jailbreak/system/document content** (<https://huggingface.co/protectai/deberta-v3-base-prompt-injection-v2>) — wrong threat model for retrieved documents; ONNX export makes it an *optional* local second opinion at most. NeMo Guardrails YARA injection-detection targets code/SQL/XSS in model *output*, off-target (<https://docs.nvidia.com/nemo/microservices/latest/guardrails/tutorials/injection-detection.html>). Lakera Guard is a commercial SaaS that sends text off-box — disqualified for privileged material (<https://www.lakera.ai/lakera-guard>).

### 2. Secret resolution + per-user vault
Raw: [`research/secret-resolution-and-per-user-vault.md`](research/secret-resolution-and-per-user-vault.md)

- **Ordered resolution is a first-class Effect feature — model it as ONE composed `ConfigProvider`, not hand-rolled `getX() ?? env`.** `ConfigProvider.orElse(self, that)` preferentially loads from one provider and falls back to `that` **only when `self` returns `undefined` (path not found)** (<https://effect.website/docs/configuration/>; API <https://effect-ts.github.io/effect/effect/ConfigProvider.ts.html>). `PlatformConfigProvider.layerDotEnvAdd` / `layerFileTreeAdd` add dotenv/file-tree (K8s-secrets) sources *as fallback* without displacing the active provider (the `*Add` vs non-`Add` distinction) (<https://effect-ts.github.io/effect/platform/PlatformConfigProvider.ts.html>). `Config.redacted` yields `Redacted<string>` so logs print `<redacted>` (<https://effect.website/docs/data-types/redacted/>). **CORRECTED (was "falls back on load error"; verified against installed source).** The live installed Effect source says the opposite of a load-error fallback: `node_modules/effect/src/ConfigProvider.ts:421-422` and `:451-452` state "The fallback only runs when the path is not found (`undefined`). A `SourceError` from `self` is not caught; it propagates immediately." **This kills the placeholder-rejection-via-refinement design**: an `S.filter`/refinement parse failure on a bad first-provider value raises a `SourceError`, which `orElse` propagates rather than treating as "missing", so a placeholder/too-short key in secure storage would **stop resolution instead of falling back** to env/dotenv. Catching all failures to force fallback would conversely mask real provider transport/auth errors. **Fix:** split "recoverable absent/placeholder" from "provider failure" — either normalize placeholders to `undefined` (absence) *inside* the secure-storage provider before `orElse`, or implement an explicit resolver service that tries providers sequentially and catches only typed `MissingSecret` / `PlaceholderRejected` errors. Required test: secure-storage placeholder → env fallback succeeds; secure-storage transport/auth failure → no fallback (error surfaces).
- **Per-user AES-256-GCM vault — Node `crypto` shape and the footguns.** Canonical: `iv=randomBytes(12)`, `createCipheriv("aes-256-gcm",key,iv)`, `getAuthTag()`; decrypt `setAuthTag` before `final()` (<https://nodejs.org/api/crypto.html>). NIST nonce is 96-bit; the hard rule is **uniqueness per key** — random 96-bit IVs risk birthday collisions, libsodium caps a single AES-GCM key at ~350 GB and warns nonce reuse "would totally destroy the security" (<https://doc.libsodium.org/secret-key_cryptography/aead/aes-256-gcm>). Mitigation: per-user/per-secret subkeys keep message counts tiny. **XChaCha20-Poly1305** (192-bit nonce) removes the entire nonce-collision class and is constant-time without AES-NI (<https://libsodium.gitbook.io/doc/secret-key_cryptography/aead/chacha20-poly1305/xchacha20-poly1305_construction>); libsodium is ISC-licensed (UNVERIFIED exact text).
- **KDF gotcha — Node scrypt defaults are 8× below the OWASP floor.** Node `scryptSync` defaults N=2^14, r=8, p=1, maxmem=32 MiB (<https://nodejs.org/api/crypto.html>); OWASP-2024 minimum is N=2^17/r=8/p=1 and prefers Argon2id for new systems (<https://cheatsheetseries.owasp.org/cheatsheets/Password_Storage_Cheat_Sheet.html>). Reconciliation: OWASP params target *low-entropy passwords*; a 32-byte random master key only needs domain-separation, so default N is acceptable **only if no key is ever passphrase-derived** — log this decision explicitly.
- **Secure-storage tier survey.** OS keychain (`keytar`) is **archived 2022-12-15** and is per-machine/single-OS-user — wrong shape for a multi-tenant server vault (<https://github.com/atom/node-keytar>, <https://snyk.io/advisor/npm-package/keytar>). **1Password SDK** (`@1password/sdk`, MIT, `client.secrets.resolve("op://vault/item/field")`) is one "secure-storage" first-provider candidate but is **beta, `v0.5.0-beta.1` (2026-06-16)**, pin and watch (<https://github.com/1Password/onepassword-sdk-js>). **CORRECTED — the repo already ships a 1Password substrate, not just MCP/skill wiring:** `@beep/onepassword-cli` (MIT, `packages/drivers/onepassword-cli/package.json`) exposes `probeReference` / `read` / `whoami` (`OnePasswordCli.service.ts:54-57`); `read` shells `op read <ref> --no-newline` and returns `Redacted.make(stdout)` (`:118-125`), `probeReference` returns non-secret metadata (`:127-134`). The shared domain already models the `op://vault/item/field` reference as `OnePasswordReference` (`packages/shared/domain/src/values/OnePasswordReference/OnePasswordReference.model.ts`, pattern at `:14-23`, `isOnePasswordReference` at `:72`). **The first provider should wrap the existing `@beep/onepassword-cli` driver rather than introduce a second 1Password integration**; only reach for `@1password/sdk` if a CLI-free runtime is a hard requirement, and record that reason explicitly. HashiCorp Vault transit-engine is encryption-as-a-service but heavy infra and **moved to BUSL-1.1 (non-OSS) Aug 2023** — OpenBao is the OSS fork (<https://developer.hashicorp.com/vault/docs/secrets/transit>; license UNVERIFIED).

### 3. SSRF + fetch hardening
Raw: [`research/ssrf-and-fetch-hardening.md`](research/ssrf-and-fetch-hardening.md)

- **OWASP: prefer allowlists; don't accept a full user URL; disable redirect-following; block risky schemes; compare the *normalized* parse.** Denylists are bypassed via DNS rebinding, IPv6 forms, decimal/octal/hex IP encoding, URL-encoding (<https://cheatsheetseries.owasp.org/cheatsheets/Server_Side_Request_Forgery_Prevention_Cheat_Sheet.html>; Node-specific, names `ip-address` <https://owasp.org/www-community/pages/controls/SSRF_Prevention_in_Nodejs>).
- **Metadata/reserved ranges the current guard misses.** `169.254.169.254` is shared AWS/GCP/Azure/DO; **also block** Alibaba `100.100.100.200`, Oracle `192.0.0.192`, `metadata.google.internal`, plus `100.64.0.0/10` (CGNAT/GKE/EKS), `192.0.2/198.18/198.51.100/203.0.113`, `224.0.0.0/4` multicast, `240.0.0.0/4` Class E (<https://github.com/b4rdia/HackTricks/blob/master/pentesting-web/ssrf-server-side-request-forgery/cloud-ssrf.md>; <https://gist.github.com/jhaddix/78cece26c91c6263653f31ba453e273b>). Defense-in-depth: AWS IMDSv2 / GCP reject `X-Forwarded-For`.
- **undici / native fetch has NO built-in SSRF protection, and the `http.Agent` middlewares don't apply to it.** `nodejs/undici#2019` (open since 2023-03) confirms `fetch(url,{agent})` is silently ignored because undici routes through a `Dispatcher`; the accepted fix is a **custom `connect.lookup` on an undici `Agent`** that pins the resolved-and-validated IP at connect time, defeating DNS-rebinding TOCTOU (<https://github.com/nodejs/undici/issues/2019>; rebinding walkthrough <https://aydinnyunus.github.io/2026/03/14/ssrf-dns-rebinding-vulnerability/>).
- **Redirect + credential-stripping hazards are CVE-backed.** A 30x `Location` to loopback/metadata is dialed even though the initial URL passed — fix is `redirect:'manual'` + re-validate every hop. Stripping must cover **all** auth: CVE-2024-24758 — undici stripped `Authorization` but **not `Proxy-Authorization`** on cross-origin redirect (patched 5.28.3/6.6.1) (<https://github.com/nodejs/undici/security/advisories/GHSA-3787-6prv-h9w3>; Budibase redirect-SSRF <https://github.com/Budibase/budibase/security/advisories/GHSA-fgqv-jh4g-pvg2>).
- **Library survey for port (classifier reference only, not drop-in).** `request-filtering-agent` (azu, **MIT, v3.2.1, actively maintained 2026-06-29**) validates at connect/socket time via `ipaddr.js` but is `http.Agent`-based → does not work with undici (<https://github.com/azu/request-filtering-agent>). `ssrf-req-filter` (MIT, v1.1.1, **stale 2024-05-11**) is a lower-quality reference (<https://github.com/y-mehta/ssrf-req-filter>). CIDR classifier primitives: `ipaddr.js` 2.4.0 MIT (`.range()`), `ip-address` 10.2.0 MIT (OWASP's encoding-resistant parser).

### 4. Secret/PII scrub + OOXML + audit
Raw: [`research/secret-pii-scrub-and-audit.md`](research/secret-pii-scrub-and-audit.md)

- **Microsoft Presidio is the reference PII engine (MIT) but is a Python/REST engine, and ownership moved in 2025** to the `data-privacy-stack` org (`microsoft.github.io/presidio` → 301 → `presidio.dataprivacystack.org`) (<https://github.com/microsoft/presidio/blob/main/LICENSE>; <https://presidio.dataprivacystack.org/anonymizer/>). Port the *contract* (recognizer span shape, operator semantics), not a runtime dep. Its `RecognizerResult{entity_type,start,end,score}` uses **half-open** spans (`text[start:end]`) — structurally identical to the in-repo `TextAnchor` (see inventory). Anonymizer operators: `replace` (default → `<ENTITY_TYPE>`), `redact`, `mask`, `hash` (random salt since v2.2.361), `encrypt`. The repo's `[REDACTED]`/`[REDACTED_SECRET]` tokens are `replace` with a fixed `new_value` — keeping a placeholder (vs full removal) preserves sentence structure for the LLM while denying the value.
- **Canonical provider-key regex shapes verified against `gitleaks.toml` (MIT binary) and primary docs** (<https://github.com/gitleaks/gitleaks/blob/master/config/gitleaks.toml>): Anthropic `sk-ant-api03-[a-zA-Z0-9_\-]{93}AA` (no checksum); OpenAI prefix + embedded literal `T3BlbkFJ` (bodies grew 48→~155 chars in 2024 — **length-anchoring is brittle, anchor on prefix+literal**, <https://github.blog/engineering/platform-security/behind-githubs-new-authentication-token-formats/>); GitHub `ghp_/github_pat_` with a verifiable **CRC32 checksum** in the last 6 chars (same GitHub blog); AWS `(AKIA|ASIA|...)[A-Z2-7]{16}`; GCP `AIza[\w-]{35}`; Slack `xoxb-`; JWT `ey...`; private-key block. `detect-secrets` (Apache-2.0) adds tunable Shannon-entropy + baseline-diff (<https://github.com/Yelp/detect-secrets>).
- **OOXML scrub must cover both XML escape contexts and identity attributes.** XML has exactly five predefined entities; element text escapes `&`/`<`/`>`, attribute values *also* escape the delimiting quote (`"`→`&quot;`) (<https://www.w3.org/TR/xml/>). doc-haus `xmlVariants` (MIT) scrubs the needle in both a textNode and an attribute variant — load-bearing because comments (`comments.xml`, `CT_Comment extends CT_TrackChange`) and track-changes stamp **`w:author`/`w:initials`** as attributes; a `<w:t>`-only scrub leaks attorney/reviewer identity (<https://datypic.com/sc/ooxml/e-w_comment-1.html>). Preserve doc-haus's honest residue check (report unreachable binary parts rather than claim "clean").
- **Don't ship raw regex — add a validation stage.** Luhn for cards, SSN structural rules, and keyword/context gating cut false positives; for a *redaction* use-case over-redaction is safer than under-, but the audit row must record score/validator outcome so a human can spot over-redaction (<https://www.conduktor.io/glossary/pii-detection-and-handling-in-event-streams>).

### 5. Redaction integrity (PDF x-ray) + HTML sanitization
Raw: [`research/redaction-integrity-and-html-sanitization.md`](research/redaction-integrity-and-html-sanitization.md)

- **`freelawproject/x-ray` is the canonical failed-redaction detector** (filled rect over still-selectable text; output `{page:[{bbox,text}]}`). It is **BSD-2 but depends on PyMuPDF (AGPL-3.0)** — the decisive constraint (see Constraints). v0.3.6 (2025-12-24) fixed the CLI to emit JSON not a Python `repr` — pin ≥0.3.6 if shelling (<https://github.com/freelawproject/x-ray>, <https://github.com/freelawproject/x-ray/blob/main/CHANGES.md>). It is honestly scoped (rect-on-text only); the creator's HN thread enumerates blind spots to *report* not silently pass: font-metric/kerning side-channels, prior revisions via `/Prev`, image-encoded text (<https://news.ycombinator.com/item?id=46369923>).
- **The permissive TS-native x-ray path is Mozilla pdf.js (Apache-2.0).** `getOperatorList()` yields `OPS.rect` geometry; `getTextContent()` yields positioned text — overlap of a uniform-fill rect with text reproduces x-ray's heuristic with no Python sidecar and no AGPL (<https://github.com/mozilla/pdf.js/blob/master/LICENSE>; <https://mozilla.github.io/pdf.js/api/draft/module-pdfjsLib.html>). **mupdf.js is NOT an AGPL escape hatch** — same dual license as PyMuPDF (<https://github.com/ArtifexSoftware/mupdf.js/>). UNVERIFIED: whether pdf.js exposes fill-color for the uniform-color test without a raster pass (likely needs canvas `ImageData`) — spike first.
- **Hand-rolled regex HTML allowlists are the textbook mXSS failure mode.** mXSS exploits a parser differential; a regex sanitizer has no parser so is maximally exposed — even HTML Purifier, AntiSamy, jSoup, Bleach had mXSS CVEs (cure53 <https://cure53.de/fp170.pdf>; Sonar <https://www.sonarsource.com/blog/mxss-the-vulnerability-hiding-in-your-code>; OWASP XSS cheat sheet <https://cheatsheetseries.owasp.org/cheatsheets/Cross_Site_Scripting_Prevention_Cheat_Sheet.html>). A regex pass is acceptable as a *server-side normalizer*, never the final defense.
- **Layered sanitization is the 2026 consensus.** Server-side **sanitize-html** (MIT, v2.17.5, htmlparser2, no-DOM — `allowedTags`/`allowedAttributes`/`allowedSchemes` default `http,https,ftp,mailto` with **`tel` NOT default**/`transformTags`) at ingest, plus browser-side **DOMPurify** (v3.4.11 2026-06-17, Apache-2.0/MPL-2.0, cure53, mXSS-hardened) at the render boundary (<https://www.npmjs.com/package/sanitize-html>; <https://github.com/cure53/DOMPurify>). DOMPurify-on-Node needs jsdom (happy-dom is unsafe) and `isomorphic-dompurify` leaks the jsdom window across calls — periodic `clearWindow()` (<https://github.com/kkomelin/isomorphic-dompurify/issues/368>).

## Constraints

### Licensing gravity (reimplement-not-copy)

### Auth / secret / offline boundaries
- **Privilege-safe = local-only on the hot path.** No SaaS detection (Lakera) and no off-box LLM call for privileged legal material; ML guardrails are optional local (ONNX) second opinions at most. The whole wedge is the *defensive ingestion gate*, deliberately deterministic.
- **Secret values never serialize.** Use `Config.redacted`/`RedactedFromValue` so logs print `<redacted>`; the per-user vault decrypts only at point of use, exposing provenance `source: "env"|"user"|"vault"`. - **Pre-LLM scrub boundary** sits *after* OOXML/text extraction but *before* any text reaches a shared store or LLM driver — the same `@beep/file-processing` retrieval pass the injection detector occupies (one pass, two checks).

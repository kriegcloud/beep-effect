# offset-preserving-char-span-chunking

Scope: design the offset-preserving char-span chunker contract between `@beep/md` and `@beep/langextract` — clause-aware sectionization, accumulated UTF-16-correct char offsets, ~2000-char chunks carrying `charStart/charEnd`, multibyte-safe slicing, and round-trip interplay with the existing `GroundedExtraction`/`AlignmentStatus` + `@beep/provenance` `TextAnchor` invariant without re-implementing langextract.

## Findings

### The round-trip invariant already exists in-repo and is UTF-16-based

- `@beep/provenance` `TextAnchor` is the canonical "where in the source" value: `{ startChar: NonNegativeInt, endChar: NonNegativeInt, quote: S.String }`, a half-open `[startChar, endChar)` range, with the documented invariant that "given the source text, `text.slice(startChar, endChar)` should reproduce `quote`." It carries `isWellOrdered` (`startChar <= endChar`) and no confidence/claim semantics. Source: `/home/elpresidank/YeeBois/projects/beep-effect/packages/foundation/modeling/provenance/src/TextAnchor.ts`.
- `@beep/langextract` `GroundedExtraction` already carries a `span?: Contract.Span` (`{ start, end }` as `NonNegativeInt`, half-open) plus `alignmentStatus`, `matchedText`, `text`, `label`. `AlignmentStatus = ["match_exact","match_lesser","match_fuzzy","unaligned"]`. Source: `/home/elpresidank/YeeBois/projects/beep-effect/packages/foundation/capability/langextract/src/Extraction/index.ts` (lines 118-224).
- **Load-bearing fact: those offsets are UTF-16 code-unit indices, not code points and not bytes.** JS `String.prototype.slice`, indexing and `.length` all count UTF-16 code units — MDN: "The `length` data property of a `String` value contains the length of the string in UTF-16 code units" and "each Unicode character may be encoded as one or two code units, so it's possible for the value returned by `length` to not match the actual number of Unicode characters" (`"😄".length === 2`, `[...emoji].length === 1`). https://developer.mozilla.org/en-US/docs/Web/JavaScript/Reference/Global_Objects/String/length . The existing langextract aligner confirms this: `spanFromMatch` sets `end = NonNegativeInt.make(start + matchedText.length)` where `matchedText.length` is a UTF-16 count and `start` comes from `sourceText.indexOf(...)` (also UTF-16). Source: `.../langextract/src/Alignment/index.ts` (lines 71-80, 231-234). **Therefore the chunker MUST emit `charStart/charEnd` as UTF-16 offsets into the exact same JS string langextract aligns against, or the `text.slice(start,end)===quote` round-trip breaks.**

### The chunker is a windowing layer that composes with — not duplicates — langextract's aligner

- langextract grounds a candidate by direct UTF-16 string search (exact `indexOf` → case-insensitive "lesser" with per-codepoint offset table → word-window fuzzy Levenshtein), failing closed to `unaligned`. The aligner already iterates the source by code point (`for (const segment of sourceText)`) while accumulating UTF-16 offsets (`sourceStart += segment.length`), so it is rune-aware on input but UTF-16-indexed on output. Source: `.../langextract/src/Alignment/index.ts` (lines 82-128). **Design consequence:** the chunker should NOT rebuild any alignment/CharInterval machinery. It produces `{ chunkText, charStart, charEnd }` windows; langextract aligns a candidate *within* a chunk's `chunkText` (offset 0..N), and the document-absolute `TextAnchor` is recovered by `chunkStart + span.start` / `chunkStart + span.end`. This is exactly how Google's reference langextract composes chunk-local token intervals into document-global `CharInterval`s (see below), so the two-layer split is the canonical shape.

### Google langextract (the ported reference) tracks char offsets per-token, aggregates to chunk spans — Apache-2.0

- `langextract/chunking.py` carries an Apache-2.0 header (port-safe to study/adapt). Core structures: `TokenInterval{start_index (incl), end_index (excl)}`, `CharInterval{start_pos, end_pos}`, and a `TextChunk` dataclass holding `token_interval`, a `document` ref, and lazily-computed `_char_interval`. `get_char_interval(tokenized_text, token_interval)` reads `start_token.char_interval.start_pos` and `final_token.char_interval.end_pos`, i.e. each token carries its own `char_interval` and a chunk's span = first/last token bounds. Source: https://github.com/google/langextract/blob/main/langextract/chunking.py (via fetch) and https://deepwiki.com/google/langextract .
- `ChunkIterator` boundary priority: (1) sentences are the atomic unit (`SentenceIterator` via `find_sentence_range`), (2) when exceeding `max_char_buffer`, prefer breaking at newline markers, (3) never split a token; a single oversized token becomes its own chunk. Accumulate sentences/tokens until the next would exceed `max_char_buffer`, then emit. https://zread.ai/google/langextract/13-document-chunking-strategy .
- **`max_char_buffer` default is 1000 chars**, with a recommended 1000-2000 for 32K+ context models. https://deepwiki.com/google/langextract — this directly anchors the packet's "~2000-char chunks" target as the upper end of langextract's own recommended band (UNVERIFIED whether 1000 vs 2000 is optimal for beep's legal corpus; see Open). NOTE: Google's offsets are Python `str` indices = Unicode code points (Python 3 strings are code-point sequences), so porting its *numbers* to JS requires a code-point→UTF-16 reinterpretation; re-deriving offsets against the JS string (below) sidesteps the mismatch entirely.

### Re-derive offsets by search, don't trust arithmetic accumulation (the LangChain lesson)

- LangChain's `RecursiveCharacterTextSplitter` exposes `add_start_index=True`. Its implementation does NOT trust running arithmetic; it re-finds each chunk: `offset = index + previous_chunk_len - self._chunk_overlap; index = text.find(chunk, max(0, offset)); metadata["start_index"] = index`. The index is in characters (Python code points). Source: https://raw.githubusercontent.com/langchain-ai/langchain/master/libs/text-splitters/langchain_text_splitters/base.py .
- Pure arithmetic offset accumulation is the documented bug surface: `start_index` goes wrong (even negative) when `chunk_overlap` is a token/word count while `previous_chunk_len` is characters (unit mismatch). Issues: https://github.com/langchain-ai/langchain/issues/18972 (custom len fn → wrong start_index), https://github.com/langchain-ai/langchain/issues/17642 (tiktoken + overlap offset in chars not tokens), https://github.com/langchain-ai/langchain/issues/16579 (inconsistent add_start_index). **Design consequence:** beep's chunker should ground each chunk's `charStart` by searching the source (like the existing `findExact`/`indexOf` path) rather than summing slice lengths, and keep one consistent unit (UTF-16). This mirrors beep's existing aligner and is robust to whitespace normalization that a sectionizer may apply.

### Multibyte/rune correctness: never split a surrogate pair (or grapheme) at a chunk boundary

- Slicing a JS string at a UTF-16 offset that falls *inside* a surrogate pair yields a lone surrogate: `"🤠".slice(0,1)` → invalid half-character; `encodeURIComponent` of it throws `URIError: URI malformed`. Real-world: lib0/CRDT `.slice` landing between an emoji's two code units produced orphaned surrogates; the fix was to detect a leading unpaired high surrogate and replace with U+FFFD. https://george.mand.is/2026/05/my-favorite-bugs-invalid-surrogate-pairs/ and https://vihan.org/blog/dealing-with-surrogates-in-javascript/ . **Consequence:** chunk boundaries must land on code-point boundaries — iterate with `for...of`/`[...str]` (code-point iterator, already the pattern in the aligner) when choosing a cut, never an arbitrary `i++` into `charCodeAt`. A boundary that splits a surrogate breaks the `quote` round-trip because the chunk's last `quote` char would be a lone surrogate.
- Grapheme clusters (user-perceived characters: `"👩🏾‍🌾"` is `.length` 7, `"😵‍💫"` is `.length` 5) are the *visually* safe boundary; `Intl.Segmenter({granularity:"grapheme"})` is the standard JS tool to segment them, since `.split("")`/`.length` operate on code units. https://developer.mozilla.org/en-US/docs/Web/JavaScript/Reference/Global_Objects/String/length and https://dev.to/ayc0/intlsegmenter-dont-use-stringsplit-nor-stringlength-dh6 . For span fidelity the hard requirement is code-point safety (no lone surrogates); grapheme-cluster safety is a nice-to-have for human-readable chunk edges. The lawyergpt#6 baseline (`runes := []rune(text)`, Go) is code-point-correct but Go `[]rune` indices are NOT UTF-16 — porting it directly would mis-offset against `TextAnchor`; it is a "skip" for offsets, baseline only (matches CAPTURE's P3·skip rating).

### Cross-stack precedent that char-offset chunking is the norm (offsets are first-class chunk metadata)

- LlamaIndex `TextNode` carries `start_char_idx`/`end_char_idx` tracking each node's character position in the source; `SentenceSplitter` prefers complete sentences/paragraphs. https://docs.llamaindex.ai/en/stable/api_reference/node_parsers/sentence_splitter/ and https://developers.llamaindex.ai/python/framework-api-reference/node_parsers/sentence_splitter/ .
- Chonkie's `RecursiveChunker` returns `Chunk` objects with `start_index`/`end_index` (position in original text), `token_count`, optional overlap context; rules expressed as a `RecursiveRules` hierarchy of `RecursiveLevel` delimiters. https://docs.chonkie.ai/oss/chunkers/recursive-chunker . Both confirm the contract shape (`{text, startIdx, endIdx}`) the packet wants, and both make offsets a returned field rather than a recomputed afterthought.
- LSP is the strongest standards precedent for "offsets need an explicit, agreed unit": `positionEncoding` (LSP 3.17+) defaults to and mandates `utf-16` for backward compat; UTF-8 counts code units as bytes, UTF-32 == code points. "Prior to 3.17 offsets were always based on UTF-16"; example "a𐐀b" → offsets 0,1,3 because 𐐀 is two UTF-16 units. https://microsoft.github.io/language-server-protocol/specifications/lsp/3.17/specification/ . This validates pinning beep's chunker to one declared unit (UTF-16, matching `TextAnchor`) instead of leaving it implicit.

### ~2000-char chunk size + overlap: tunable, with current (2026) guidance

- Common starting point is RecursiveCharacterTextSplitter at ~400-512 tokens with 10-20% overlap; a Jan-2026 systematic analysis found a "context cliff" near ~2,500 tokens where answer quality drops, and that the default `chunk_overlap` "adds no measurable benefit" — treat overlap as a metric-tuned knob, not a copied default. https://www.digitalapplied.com/blog/rag-chunking-strategies-2026-retrieval-quality-playbook and https://langcopilot.com/posts/2025-10-11-document-chunking-for-rag-practical-guide . (~2000 chars ≈ ~500 tokens at the common 4 chars/token heuristic, i.e. comfortably below the cliff.)
- Document-aware (clause/heading-respecting) chunking moves recall more than swapping embedding models — up to a ~9% recall gap between best/worst chunking on the same corpus/retriever — supporting the packet's clause-aware sectionizer over naive fixed slicing. https://optyxstack.com/rag-reliability/rag-chunking-strategy-chunk-size-overlap-document-structure-recall and Chroma's evaluation https://www.trychroma.com/research/evaluating-chunking . "Human readability rule": a chunk that makes sense to a human standalone usually makes sense to the model (Pinecone, June 2025, via the digitalapplied playbook).

### Proposed contract shape (synthesis, for align/shape stages)

- `@beep/md` produces normalized markdown text (`packages/foundation/modeling/md`, `Md.model.ts`/`Md.render.ts`/`Md.utils.ts`). The chunker takes that single source string and emits ordered chunks `{ chunkText, charStart, charEnd, breadcrumb?, sectionPath? }` where `charStart/charEnd` are UTF-16 offsets such that `source.slice(charStart, charEnd) === chunkText` (the same invariant as `TextAnchor`). `breadcrumb` = `docName › section` prefix (doc-haus#11 / netNew #3) prepended only for embedding, not stored in `chunkText`, so offsets stay exact.
- Sectionizer detects boundaries by clause-number / markdown heading / all-caps heading (doc-haus#11), accumulating absolute offsets; slicer caps at ~2000 UTF-16 chars, snapping cuts to code-point (ideally sentence) boundaries. A chunk-local `GroundedExtraction.span` composes to a document-global `TextAnchor` via `{ startChar: chunk.charStart + span.start, endChar: chunk.charStart + span.end, quote: matchedText }` — zero new alignment code, reusing `@beep/langextract/Alignment` and `@beep/provenance/TextAnchor` verbatim. (Existing in-repo barrels; do not duplicate.)

## Open / Unverified

- **Optimal chunk size for beep's legal corpus is UNVERIFIED.** langextract recommends 1000-2000 chars; the 2026 RAG playbooks favor ~400-512 tokens (~1600-2000 chars) and warn of a ~2500-token cliff — but none is measured on patent/office-action prose. The ~2000-char target is a defensible default, not an empirically tuned value; needs a retrieval-eval pass during shape/build.
- **Overlap policy UNVERIFIED for this corpus.** Jan-2026 analysis says default overlap "adds no measurable benefit" while other sources say overlap preserves cross-boundary clauses; langextract's own chunker emits non-overlapping chunks by default. Decision (overlap 0 vs ~10%) should be metric-driven, not copied. Note overlap complicates the offset round-trip only if duplicated text is re-searched — re-derivation via search (not arithmetic) keeps it sound.
- **Grapheme-cluster vs code-point boundary requirement is a design choice, not yet decided.** Code-point safety (no lone surrogates) is mandatory for the `quote` round-trip; grapheme-cluster safety (no split ZWJ emoji/combining marks) is cosmetic for chunk edges. `Intl.Segmenter` availability/perf at ~1M-char documents is UNVERIFIED for this path.
- **doc-haus#11 source (the primary port reference) is a private/local clone (MIT per CAPTURE) and was NOT web-fetchable.** Its exact `sectionize()`/`chunkSection()` signatures are taken from CAPTURE.md, not re-read from source here; confirm against the local clone before porting.
- **Google langextract's token-level CharInterval machinery is heavier than beep needs and is NOT recommended to port.** beep's existing direct-`indexOf` UTF-16 aligner already satisfies the round-trip without a tokenizer; the chunker only needs accumulated absolute offsets. Flagged as a design assertion to confirm in align/shape, not an established repo decision.
- **`Contract.Span` (from `@beep/nlp/Handoff`) exact field/encoding** was inferred from usage in the aligner (`{start,end}` NonNegativeInt, half-open) rather than read from its own source file; verify the `@beep/nlp/Handoff` definition before finalizing the compose mapping `TextAnchor ⇄ Span`.

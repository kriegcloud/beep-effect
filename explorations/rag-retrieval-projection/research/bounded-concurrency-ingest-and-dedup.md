# bounded-concurrency-ingest-and-dedup

Scope: re-express semaphore/pool ingestion as Effect `Stream.mapEffect` (bounded parallelism) + per-chunk transactional persistence + 202-then-process async shape, plus an AGPL-clean MinHash/LSH evidence-cluster dedup and a depth-capped citation-graph BFS projection traversal — all grounded against this repo's Effect 4.0.0-beta.91 substrate.

## Findings

### Bounded-concurrency ingestion as Effect primitives

- **The repo is on `effect@4.0.0-beta.91`** (root `node_modules/effect/package.json`). In v4 the semaphore is a **top-level `Semaphore` module** (`Semaphore.make(n)` returning `Effect<Semaphore>`, with `.take(n)`, `.release(n)`, `.withPermits(n)(effect)`), not the v3 `Effect.makeSemaphore`. The repo already uses exactly this idiom: `packages/tooling/tool/cli/src/commands/Graphiti/internal/ProxyServices.ts:914` does `const forwardSemaphore = yield* Semaphore.make(config.concurrency)` then `yield* forwardSemaphore.take(1)` / `yield* forwardSemaphore.release(1)`; `packages/drivers/duckdb/src/DuckDb.service.ts:308/328` uses `Semaphore.make(1)` as a connection mutex. (in-repo, grounded)
- **The lawyergpt Go pattern (semaphore + per-file goroutine, `numSemaphore = ceil(len(files)/2)`) maps directly onto `Stream.mapEffect(f, { concurrency: n })`.** `mapEffect` accepts `options?: { readonly concurrency?: number | "unbounded"; readonly unordered?: boolean; readonly bufferSize?: number }` — a bounded `concurrency: n` runs up to n effects at once with built-in backpressure, making a manual semaphore unnecessary for the steady-state pipeline. Signature verified against the API reference: https://effect-ts.github.io/effect/effect/Stream.ts.html and the docs page https://effect.website/docs/stream/operations/ . (verified, two sources)
- **`Stream.mapEffect` with bounded concurrency preserves source order by default**; results are "emitted downstream in their original order," and order is only relaxed when `unordered: true` is passed. https://effect.website/docs/stream/operations/ . Corroborated by the Effect-vs-RxJS comparison: Effect's `mapEffect` bridges the `mergeMap` (concurrent, unordered) / `concatMap` (ordered, no concurrency control) gap by offering concurrent-and-ordered execution. https://advancedweb.hu/rxjs-how-to-make-an-ordered-mergemap-operator/ . NOTE: a fast-model gloss of the raw `Stream.ts` API doc inverted this ("order not preserved unless unordered=false"); the authoritative docs page is correct — order IS preserved with bounded concurrency. (verified; one contradictory gloss flagged)
- **`numSemaphore = ceil(len(files)/2)` is an anti-pattern to drop, not port** — it scales concurrency with input size rather than the resource budget. The idiomatic replacement is a fixed `concurrency: n` sized to the actual constrained resource (DB connection-pool size / embedding-model throughput). This is exactly what the `Stream.mapEffect` concurrency arg encodes. lawyergpt source snippet from CAPTURE.md `api/main.go:176-298`; sizing guidance: prefer a bounded number over `concurrency: "unbounded"`, which spawns a fiber per element with no backpressure (https://effect.website/docs/concurrency/basic-concurrency/). (analysis grounded in primary docs)
- **Canonical pipeline shape:** `Stream.fromIterable(chunks)` → `Stream.mapEffect(persistChunk, { concurrency: n })` → `Stream.runDrain`. `Stream.mapEffect` is already used in-repo at `packages/agents/server/src/AssistantTurn/AnthropicTurnKernel.ts:145`; `Stream.runForEach`/`runDrain`/`runCollect` are the terminal runners (v4 `runCollect` now returns `Effect<A[]>`, not `Effect<Chunk<A>>` — a v4 breaking change). v4 change confirmed: https://effect.website/blog/releases/effect/40-beta/ and gist summary in https://gist.github.com/kevinmichaelchen/84c0ce72b5e33b39062822dcf6c7f595 . (in-repo + verified)

### Per-chunk transactional persistence

- **`@effect/sql` exposes `withTransaction`, which wraps an effect in `BEGIN`/`COMMIT` and automatically rolls back on failure OR interruption**, with nested transactions implemented via `SAVEPOINT`s on databases that support them (Postgres/PGlite do). https://deepwiki.com/Effect-TS/effect/6-data-persistence and savepoint-implementation detail in https://github.com/Effect-TS/effect-smol/issues/2216 . The "per-chunk transaction" requirement = call `withTransaction` (or Drizzle `.transaction`) inside the `mapEffect` body so each chunk's `resource + embedding` insert commits or rolls back independently, and a poisoned chunk interrupts only its own fiber's transaction. (verified)
- **The repo already has both transaction shapes to reuse:** Drizzle's `.transaction(...)` at `packages/workspace/server/src/aggregates/Thread/ThreadStore.repo.ts:376`, and a hand-rolled `withTransaction` (BEGIN/COMMIT/ROLLBACK, including nested) at `packages/drivers/duckdb/src/DuckDb.service.ts:256-301`. PGlite + Drizzle is the local-first authority store (tests at `packages/workspace/server/test/integration/ThreadStoreDrizzleRepository.pglite.test.ts`). The embedding vector is a **rebuildable projection**, so the authoritative transaction is the chunk+offset row; the pgvector/projection write can be a separate idempotent upsert. (in-repo, grounded)
- **Effect's transaction guarantee is interruption-safe**, which composes cleanly with `mapEffect` concurrency: if the parent stream is interrupted, each in-flight chunk transaction rolls back via the `withTransaction` finalizer rather than leaking a half-written row. https://deepwiki.com/Effect-TS/effect/6-data-persistence . (verified)

### 202-then-process async shape

- **Canonical async request-reply:** validate synchronously and return `400/422` for bad input; return **`202 Accepted` only for valid work**, with a `Location` header pointing at a status endpoint and a `Retry-After` header for poll cadence; the status endpoint returns `200` with a state body while running and **`303 See Other`** to the result resource on completion. https://learn.microsoft.com/en-us/azure/architecture/patterns/asynchronous-request-reply and https://restfulapi.net/http-status-202-accepted/ . (verified, two sources)
- **Effect-side "return 202 then process in a pool":** fork the ingestion stream off the request fiber. `Effect.forkDaemon` detaches to the global scope so the background work outlives the HTTP handler (the literal "fire-and-forget after 202" recipe), while `Effect.forkScoped`/`forkIn` tie the worker's lifetime to a managed `Scope` for graceful shutdown. The Effect maintainers' own answer to "fire and forget long running process / return 202" is `forkDaemon`. https://github.com/Effect-TS/website/issues/263 and fibers docs https://effect.website/docs/concurrency/fibers/ . (verified)
- **Backpressured hand-off:** enqueue accepted jobs onto an Effect `Queue` (a full back-pressured `Queue.offer` suspends; fork the offer to avoid blocking the request fiber) and have a long-lived consumer drain it through the bounded `Stream.mapEffect`. https://effect.website/docs/concurrency/queue/ . For durable/restart-safe semantics beyond a single process, `@effect/workflow` + cluster is the heavier option (UNVERIFIED for this repo's needs — see Open). (verified for in-process; durable variant unverified)

### AGPL-clean MinHash/LSH evidence-cluster dedup

- **Licensing is the decisive constraint and it cuts in our favor.** The courtlistener source (`cl/citations/group_parentheticals.py`) is **AGPL-3.0** (freelawproject/courtlistener) → reimplement from spec, do NOT copy. BUT the underlying library it wraps, **`datasketch`, is MIT-licensed** (https://github.com/ekzhu/datasketch , https://pypi.org/project/datasketch/), so the *algorithm itself* can be cleanly re-implemented/ported from the MIT datasketch source rather than reverse-engineered from the AGPL caller. Reimplement the LSH math from datasketch (MIT), and reimplement the *clustering policy* (constants, tokenization, representative selection) from the courtlistener spec without copying its code. (verified)
- **courtlistener clustering spec (reimplement, don't copy):** `num_perm = 64`, `SIMILARITY_THRESHOLD = 0.5`; a single empty `MinHashLSH(threshold=0.5, num_perm=64)` and empty `MinHash(num_perm=64)` are pre-seeded once and `deepcopy`'d per use to avoid re-paying RNG/permutation seeding; tokens are produced by stripping non-alphanumerics, lowercasing, dropping stopwords + leading gerunds, then stemming; clusters are connected components found by BFS over the similarity graph; the representative ("most-described") parenthetical is the node with the most neighbors above `BEST_PARENTHETICAL_SEARCH_THRESHOLD = 0.2`. Source (read for spec only): https://github.com/freelawproject/courtlistener/blob/main/cl/citations/group_parentheticals.py . (verified)
- **MinHash math for a clean-room TS port (from datasketch, MIT):** signature uses `num_perm` random permutations `h_i(x) = ((a_i * x + b_i) mod p) & maxHash` with `p = _mersenne_prime = (1<<61) - 1`, `maxHash = _max_hash = (1<<32) - 1`, `hash_range = 1<<32`; coefficients `a ∈ randint(1, p)` (nonzero), `b ∈ randint(0, p)`; per element update is `phv = (a*hv + b) % p & maxHash` then `hashvalues = min(phv, hashvalues)`; estimated Jaccard = `count(hv_A == hv_B) / num_perm`. Default `num_perm = 128`. https://raw.githubusercontent.com/ekzhu/datasketch/master/datasketch/minhash.py and https://ekzhu.com/datasketch/minhash.html . The `(a*x+b) mod p` permutation is the standard Carter–Wegman strongly-universal hash family. https://en.wikipedia.org/wiki/Universal_hashing . NOTE: with `p = 2^61-1`, JS `number` is unsafe (>2^53); a TS port must use `BigInt` (or 64-bit Wasm) for the multiply-mod. (verified; BigInt caveat flagged)
- **LSH banding spec (from datasketch, MIT):** split `num_perm` into `b` bands of `r` rows (`b*r ≤ num_perm`); a band collides when all r minhashes match (prob `s^r`), so a pair becomes a candidate with **`P(s) = 1 - (1 - s^r)^b`** (the S-curve). https://ekzhu.com/datasketch/lsh.html and Stanford CS345 LSH notes http://infolab.stanford.edu/~ullman/mining/2006/lectureslides/cs345-lsh.pdf . `(b, r)` are chosen for a target threshold by `_optimal_param`, a grid search over `b∈[1,num_perm]`, `r∈[1,num_perm/b]` minimizing `false_positive_weight * FP + false_negative_weight * FN`, where `FP = ∫_0^t [1-(1-s^r)^b] ds` and `FN = ∫_t^1 [1-(1-(1-s^r)^b)] ds` (weights default `(0.5, 0.5)`). Insert hashes each band's slice into a per-band hashtable keyed by the band hash; query unions candidate keys across all bands. https://github.com/ekzhu/datasketch/blob/master/datasketch/lsh.py . (verified)
- **Dedup sits before the candidate→approved gate:** collapse near-duplicate evidence/holding assertions, rank the most-described idea higher, then pass survivors to the ClaimGate. This is a pre-gate ranking step, not a retrieval-feeds-LLM path (the explicit beep anti-pattern). (from CAPTURE.md, in-repo policy)

### Citation-graph BFS projection traversal (uspto-patents-mcp#2)

- **Spec (port the shape):** depth-capped BFS over the patent citation graph, direction `forward | backward | both`; per-level frontier ceiling `frontier.slice(0, 50)` as a CPU budget; depth cap 2; emit an explicit edge list `edges.push({ from: cur, to: c.patent_id })` that maps directly onto a FalkorDB/Cypher projection. Loop guard `for (d < depth && frontier.length > 0)`. (from CAPTURE.md `src/patentsview.ts:89-149`; the exact upstream TS file path could not be re-fetched — see Open. The bounded-BFS shape — visited set + per-level frontier cap + `{from,to}` edge accumulation — is a standard, well-formed graph traversal.)
- **CRITICAL data-source correction — do NOT wire this to PatentsView's legacy API.** The legacy `api.patentsview.org` is **sunset: discontinued as of 2025-05-01, now returning `410 Gone`** (announced Oct 2024, originally slated Feb 2025, extended to May 1 2025). https://patentsview.org/data-in-action/patentsview-ends-support-legacy-api and https://patentsview.org/data-in-action/support-legacy-api-end-february-2025-switch-patentsearch-api-now . The replacement is the **PatentSearch API** (formerly "Elasticsearch/Beta/Search API"); USPTO also runs the Open Data Portal at `api.uspto.gov`. https://data.uspto.gov/support/transition-guide/patentsview . This matches the repo's live note (tree-snapshot): the `packages/drivers/uspto` driver targets ODP `api.uspto.gov` and must **NEVER** use PatentsView (sunset). So port the BFS *traversal logic*, but source citation edges from the ODP/PatentSearch driver, not the dead PatentsView endpoint. (verified; aligns with in-repo routing)
- **Re-express the BFS in Effect:** each level's frontier expansion is itself a bounded fan-out — `Effect.forEach(frontier, fetchCitations, { concurrency: n })` (or a level-wise `Stream.mapEffect`) — so the same bounded-concurrency primitive that drives ingestion also drives the projection traversal, feeding the same retrieval/projection tier. (analysis grounded in the concurrency primitives above)

The traversal SHAPE is standard and sound; only the provenance of the specific snippet is unverified.
- **Durable / restart-safe 202 processing** via `@effect/workflow` + cluster (vs a plain in-process `forkDaemon` + `Queue`) was not verified for this repo's footprint. For local-first single-process ingestion, in-process fork+queue is almost certainly sufficient; durable execution is likely over-engineering until ingestion spans process restarts. Needs an align-stage decision.
- **`bufferSize` default + exact backpressure tuning** for `Stream.mapEffect` (how deep the internal queue buffers before upstream pull stalls) was not pinned to a primary number; defaults exist but the value wasn't quoted from source. Treat as a tuning knob, not a correctness concern.
- **MinHash `BigInt` performance** for `(a*x+b) mod 2^61-1` over large corpora is unverified — a pure-BigInt TS port may be a hotspot; a 32-bit-prime variant (`p = 2^31-1`) or Wasm SIMD may be needed. Benchmark before committing `num_perm=64` over the full evidence set.
- **Whether to keep courtlistener's `num_perm=64 / threshold=0.5` verbatim** vs tuning for legal evidence-assertion text is unverified; these constants were chosen for short parenthetical holding-summaries and may need recalibration for prior-art assertion spans. Re-derive `(b, r)` from the chosen threshold via the `_optimal_param` grid search rather than hardcoding.

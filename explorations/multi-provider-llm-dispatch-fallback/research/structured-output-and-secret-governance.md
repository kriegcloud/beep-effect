# structured-output-and-secret-governance

Scope: how `response_model`-style structured output and single-source tool schemas bind to effect/Schema across providers, and whether the dispatch Layer should support a desktop-never-holds-the-key broker indirection for the solo-attorney app.

## Findings

### A. Structured output binding (`response_model` → effect/Schema) is ALREADY first-class — courtlistener#5 is alreadyCovered

- **courtlistener#5 pattern (the thing to replicate).** `call_llm(system_prompt, user_prompt, model="openai/gpt-4o-mini", response_model: type[BaseModel] | None = None, temperature=0.0, max_completion_tokens=1000, api_key=None)` builds a client with `instructor.from_provider(model, api_key=api_key)` and returns a validated Pydantic instance when `response_model` is set, else raw dict/str. A single `"provider/model"` string selects the backend. Source (raw file): https://raw.githubusercontent.com/freelawproject/courtlistener/main/cl/lib/llm.py
- **License: courtlistener is AGPL-3.0 — port the PATTERN, never the code.** `LICENSE.txt` is the full GNU AGPLv3 ("Copyright 2010, Brian Carver and Michael Lissner ... GNU Affero General Public License ... version 3 or later"). GitHub's license API reports `NOASSERTION` only because the file name/preamble defeats auto-detection; the body is unambiguously AGPLv3. Source: https://github.com/freelawproject/courtlistener/blob/main/LICENSE.txt — so the `response_model` ergonomics are a design reference only.
- **Instructor's model (what `response_model` actually does).** The Pydantic model defines the schema AND the prompt; for providers with native structured outputs it uses the server-side feature, otherwise it injects a JSON-emitting prompt and validates + retries. Source: https://python.useinstructor.com/concepts/models/ and https://python.useinstructor.com/
- **The effect-v4 equivalent already exists in the vendored tree:** `LanguageModel.generateObject({ prompt, schema })` where `schema` is a `Schema.Codec<unknown, ObjectEncoded, ...>`; the result `GenerateObjectResponse.value` is the DECODED application type (not raw JSON), and `.text` carries the raw text. Primary (vendored): `.repos/effect-v4/packages/effect/src/unstable/ai/LanguageModel.ts` (`generateObject` at L125-139, `GenerateObjectResponse` at L473-486). Public mirror: https://effect-ts.github.io/effect/ai/ai/LanguageModel.ts.html
- **Provider JSON-Schema subset differences are handled by a pluggable `CodecTransformer`, not by the caller.** `LanguageModel.CodecTransformer` rewrites a codec's encoded side to satisfy each provider's JSON-Schema constraints while preserving the decoded type; `make(...)` uses the provider's `codecTransformer` or `defaultCodecTransformer` when none is supplied. Invalid provider output fails as `AiError.InvalidOutputError`. Primary: `.repos/effect-v4/.../LanguageModel.ts` (L188-220, L728-743) and `.../internal/codec-transformer.ts`.
- **Two shipped provider transformers already exist:** `OpenAiStructuredOutput.toCodecOpenAI` and `AnthropicStructuredOutput.toCodecAnthropic`, each returning `{ codec, jsonSchema }`. They encode the provider-safe shape: tuples → objects with numeric-string keys, records → arrays of `[key,value]` pairs, optional props → required nullable props, `oneOf` → `anyOf`; OpenAI additionally flattens `allOf` (merging members) and merges multiple regex filters into one `pattern` because OpenAI structured output rejects `allOf`. Unsupported kinds THROW at conversion (no lossy schema). Primary: `.repos/effect-v4/.../OpenAiStructuredOutput.ts` and `.../AnthropicStructuredOutput.ts`. **Implication: the dispatch Layer does not re-implement `response_model`; it composes `generateObject` and selects the right transformer per provider.**

### B. Per-provider structured-output gotchas (the reason the codec transformers exist; what the dispatch fallback must respect)

- **OpenAI structured outputs (`strict: true`) is restrictive:** every object must set `additionalProperties: false`, every property must be listed in `required`, and `default` values are unsupported — you wrap/adapt schemas before calling. Primary: https://developers.openai.com/api/docs/guides/structured-outputs ; corroboration: https://community.openai.com/t/schema-additionalproperties-must-be-false-when-strict-is-true/929996 and the original launch post https://openai.com/index/introducing-structured-outputs-in-the-api/
- **Anthropic structured outputs is now GA (was beta).** Released public beta **2025-11-14** with beta header `structured-outputs-2025-11-13` and param `output_format`; it has since gone **GA** with the param relocated to `output_config.format` and beta headers no longer required (old header/param keep working for a transition period). It compiles the JSON schema into a grammar and constrains token generation (true constrained decoding), exposing JSON outputs (`output_config.format`) + strict tool use (`strict: true`). Supported on Claude Opus 4.8/4.6/4.5, Sonnet 4.6/4.5, Haiku 4.5, etc. Limits: `additionalProperties` must be `false`; NO recursive schemas, no external `$ref`, no numeric/length constraints, ≤20 strict tools, ≤24 optional params, ≤16 union-typed params (else 400 "Schema is too complex"). Primary: https://platform.claude.com/docs/en/build-with-claude/structured-outputs ; date corroboration: https://towardsdatascience.com/hands-on-with-anthropics-new-structured-output-capabilities/
- **Effect tracks this migration as an open item.** `@effect/ai-anthropic` currently EMULATES structured output by forcing a tool call (wraps the schema as a tool `input_schema` + `tool_choice`); issue #6091 proposes switching to Anthropic's NATIVE structured outputs. So the vendored Anthropic path may still be tool-call emulation, not native constrained decoding — verify against the actual vendored `@effect/ai-anthropic` build before relying on grammar-level guarantees. Source: https://github.com/Effect-TS/effect/issues/6091
- **Gemini is the schema outlier (validates mike#10's `normalizeSchema`).** Gemini/Vertex rejects an OBJECT function/output schema with empty `properties` ("should be non-empty for OBJECT type"); empty-param tools must be sent as `{ "type": "object", "properties": {} }` and Vertex strictly requires the `type: "object"` field; it also historically rejected `additionalProperties` (added support ~Nov 2025) and rejects `required`/`properties` on non-object types. Sources: https://discuss.ai.google.dev/t/gemini-api-2-0-flash-exp-requires-non-empty-properties-for-open-ended-object-parameters/64086 , https://github.com/BerriAI/litellm/issues/18442 , https://github.com/googleapis/python-genai/issues/1815 — **note: beep ships no Gemini driver today, so this gotcha is forward-looking, not blocking.**

### C. Single-source tool-schema portability (mike#10) is alreadyCovered by effect `Tool`/`Toolkit`

- **mike#10 is AGPL-3.0 — study only.** Repo `willchen96/mike` ("OSS AI Legal Platform") is AGPL-3.0 (`gh api repos/willchen96/mike/license` → `AGPL-3.0`). Its `backend/src/lib/llm/tools.ts` keeps ONE OpenAI-style tool def and maps it to Claude (`toClaudeTools`) + Gemini (`toGeminiTools` via `normalizeSchema`, omitting empty `parameters` that Gemini rejects). Source: https://github.com/willchen96/mike — port the single-source-of-truth idea, not the code.
- **Effect's canonical equivalent:** `Tool.make("name", { description, parameters: Schema.Struct({...}), success, failure? })` + `Toolkit.make(tool1, tool2, ...)` define ONE Schema-backed tool; each provider's `LanguageModel` translates that toolkit into the provider's native tool format and validates calls. Provider-defined/raw tools accept a raw `JsonSchema.JsonSchema` for `parameters` (used for MCP dynamic tools). Primary: `.repos/effect-v4/packages/effect/src/unstable/ai/Tool.ts` (L180-200, L429 `providerName`, L484-508 raw `jsonSchema`) and `.../Toolkit.ts` (L29-55 `Toolkit.make`). **Implication: the dispatch Layer authors tools once with `Tool.make` and lets the per-provider driver adapt them — mike#10's adapter is structurally already in the box.** The remaining net-new is only ensuring the Gemini empty-params normalization lands IF a Gemini driver is ever added.

### D. Local-first secret governance + the key-broker indirection decision (stenoai#2)

- **stenoai adapter pattern (the governance template), license MIT — reimplementable freely.** `ruzin/stenoai` ("AI-powered intelligence layer for confidential workflows ... government, defence, legal, CXOs") runs an Electron desktop + Python backend. In `adapter` mode "every AI request routes through the customer's org adapter, which holds the Anthropic key server-side; the desktop app never handles provider credentials directly. URL + JWT come from env vars set by Electron when a session is active; the model name is informational — the adapter uses its own configured default, so requests omit the model param." Other modes: `cloud` (Anthropic SDK / Bedrock Converse over HTTPS bearer / OpenAI + custom `base_url`), `local`/`remote` Ollama. Sources: https://raw.githubusercontent.com/ruzin/stenoai/main/src/summarizer.py , https://github.com/ruzin/stenoai , https://dev.to/ruzin_saleem_2b30cf073fd8/stenoai-privacy-focused-ai-meeting-intelligence-380-github-stars-1000-downloads-4o1m ; license MIT (`gh api repos/ruzin/stenoai/license` → `MIT`).
- **This is the same shape as LiteLLM's JWT→Virtual-Key mapping (canonical broker prior art).** Client presents a JWT (SSO/OIDC identity token) as `Authorization: Bearer`; the proxy reads a configured claim (`jwt_client_id_field`, e.g. `client_id`/`sub`/`azp`) and maps it to a server-held virtual key with per-user model/spend limits — "without giving each person a LiteLLM API key." The desktop never receives the real provider credential. Sources: https://docs.litellm.ai/docs/proxy/jwt_key_mapping and virtual-keys/BYOK pass-through (`forward_llm_provider_auth_headers: true`) https://docs.litellm.ai/docs/proxy/virtual_keys
- **Cloudflare AI Gateway is a second, managed instance of the same pattern (BYOK / authenticated gateway).** Provider keys are stored server-side; clients send a gateway token (`cf-aig-authorization` header) and never hold the upstream provider key. Source: https://developers.cloudflare.com/ai-gateway/configuration/authentication/ — useful as a hosted fallback target the openai-compat driver could point at.
- **The alternative — local OS-keychain storage — is viable but has a Linux trap.** Electron `safeStorage` encrypts via macOS Keychain / Windows DPAPI / Linux libsecret (kwallet/gnome-libsecret). Critical: on Linux with NO secret store available it silently falls back to a hardcoded-plaintext password (`getSelectedStorageBackend()` returns `basic_text`) — effectively unencrypted. Sources: https://www.electronjs.org/docs/latest/api/safe-storage and migration note https://freek.dev/2103-replacing-keytar-with-electrons-safestorage-in-ray . The repo already owns a `onepassword-cli` driver (`packages/drivers/onepassword-cli`) and an `onepassword-secret-refs` skill, giving a third local option: `op://` secret-reference resolution instead of raw at-rest keys.
- **RECOMMENDATION (decision): the dispatch Layer SHOULD support key-broker indirection — and the seam already exists, so it is cheap.** The right abstraction is an `ApiKeyResolver`/credential service the dispatch Layer depends on, with interchangeable Layers: (1) `env` (`Config.redacted`), (2) `os-keychain` / `op://`, (3) `broker` (fetch short-lived JWT/bearer, target an adapter `base_url`). For a privilege-sensitive solo-attorney app, broker indirection means provider keys never touch the desktop or the public repo — strictly better than client-side at-rest keys. Make the resolver pluggable rather than hard-coding broker mode, so single-user BYOK and managed-adapter deployments share one dispatch path. (Mirrors stenoai's adapter/cloud/local branch + LiteLLM's JWT→key indirection.)

### E. The repo already exposes the exact seams a broker resolver needs (net-new is small)

- **Key resolution is a single composable point.** The live Anthropic layer resolves its key with `apiKey: Config.redacted(ANTHROPIC_API_KEY_ENV)` (`AI_ANTHROPIC_API_KEY`) at `packages/drivers/anthropic/src/Anthropic.service.ts:36`. Swapping `Config.redacted(env)` for a resolver Effect is the entire change. Primary (in-repo): `packages/drivers/anthropic/src/Anthropic.config.ts` + `Anthropic.service.ts`.
- **The OpenAI-compat driver ALREADY tolerates the adapter/broker shape:** `apiUrl` is `S.optionalKey` (defaults to `https://api.openai.com/v1`, so it can point at an adapter/gateway base URL) and the bearer is conditional — `options.apiKey === undefined ? identity : HttpClientRequest.bearerToken(options.apiKey)` — so a request can carry a broker-injected/JWT bearer or no key at all (proxy injects auth). Primary: `packages/drivers/openai-compat/src/OpenAiCompatClient.service.ts:56,173-174`. Venice/xAI/Runpod/HubSpot/Sanity/m365 all already use `Option`-wrapped `Redacted` `HttpClientRequest.bearerToken`, so a bearer-token broker token is the repo's established auth idiom. **Therefore: the adapter/key-broker mode needs a new dispatch-level resolver service, NOT new driver code.**

# provider-registry-and-key-precedence-design

Scope: design a typed `PROVIDER_REGISTRY` + user-key>env>CLI key-precedence as an Effect Layer with typed config errors, API-key prefix auto-detect, and structured `api_key_required` errors — clean-room reimplementation of the unknown-license harvest-mcp#1 `ProviderFactory`.

## Findings

### Vendored Effect substrate (the canonical primitives this design must build on)

- The repo vendors `effect@4.0.0-beta.91` (`node_modules/effect/package.json`). All design choices below are validated against that exact tree, not against published v3 docs, because several v3 helpers named in the gold nuggets (notably `Layer.orElse`) are **absent** in this beta. (in-repo: `node_modules/effect/package.json`)
- **`Layer.orElse` does NOT exist in beta.91.** Grepping `node_modules/effect/dist/Layer.d.ts` for `orElse` returns nothing; the layer-level error-recovery surface is `Layer.catchCause` and `Layer.catchTag` (plus `Layer.provide`). The CAPTURE's gold nuggets (Juris.AI#1) and the GOLD_SYNTHESIS recommendation both proposed `Layer.orElse` for the fallback chain — that recommendation is **stale against the vendored runtime** and must not be ported verbatim. (in-repo: `node_modules/effect/dist/Layer.d.ts`; cross-ref CAPTURE.md line 35, 58)
- **The canonical Effect-v4 multi-provider fallback primitive is `ExecutionPlan` + `Effect.withExecutionPlan`**, and the official doc example is *literally* LLM-provider fallback over `LanguageModel.LanguageModel` layers: `ExecutionPlan.make({ provide: layerBad, attempts: 2, schedule: Schedule.spaced(3000) }, { provide: layerGood })`, then `Effect.withExecutionPlan(effect, ThePlan)`. Each step provides a `Context`/`Layer` and may set `attempts`, `schedule`, and a `while` predicate; "the runtime tries steps in order until the workflow succeeds or the plan is exhausted." (in-repo primary: `node_modules/effect/dist/ExecutionPlan.d.ts` lines 1-100, 188-216) — this supersedes `Layer.orElse` and directly serves netNew #1 (ordered fallback) and netNew #4 (declarative retry/round-robin policy) without a BAML runtime.
- **`Config.orElse(() => fallbackConfig)` is the key-precedence primitive.** Its doc states it "catches **all** `ConfigError`s (not just missing data)" and the fallback receives the error and returns a new `Config` — distinct from `Config.withDefault`, which only fills missing data. So `userKeyConfig.pipe(Config.orElse(() => envKeyConfig))` expresses "try user-supplied key, else env" with typed `ConfigError` propagation. (in-repo: `node_modules/effect/dist/Config.d.ts` lines 278-300, 407)
- **`ConfigProvider.orElse` + `ConfigProvider.layer` is the cleaner "precedence as an Effect Layer" expression.** A user/CLI key map can be wrapped as a `ConfigProvider` and chained `ConfigProvider.orElse(envProvider)`, then handed to the runtime via `ConfigProvider.layer(self): Layer.Layer<never, E, ...>`. `ConfigProvider.constantCase` normalizes lookup keys (e.g. `openai.apiKey` → `OPENAI_API_KEY`). This lets the whole user>env>CLI chain be a single provided Layer rather than imperative precedence code. (in-repo: `node_modules/effect/dist/ConfigProvider.d.ts` lines 355, 584, 744)
- Config building blocks confirmed present in beta.91: `Config.redacted` (wraps value in `Redacted<string>`), `Config.option` (`Config<A> → Config<Option<A>>`), `Config.withDefault`, `Config.nested`, `Config.map`, `Config.mapOrFail`. (in-repo: `node_modules/effect/dist/Config.d.ts` lines 120, 199, 407, 500, 1275; effect docs https://effect.website/docs/configuration/)
- **`effect/unstable/ai` ships in beta.91** with `LanguageModel`, `Model`, `Tool`, `Toolkit`, `Response`, `AiError`, `OpenAiStructuredOutput`, `AnthropicStructuredOutput`. The dispatch Layer should target `LanguageModel.LanguageModel` as its universal provider interface (what `ExecutionPlan` steps provide), exactly as the four drivers already do. (in-repo: `node_modules/effect/dist/unstable/ai/`)

### How the four drivers already expose provider Layers (the registry's `factory` target)

- Each driver already exposes a `LanguageModel` namespace whose `.layer(options)` returns `Layer.Layer<LanguageModel.LanguageModel, never, ProviderService>` and a `.model(name)` returning an `AiModel.make(providerId, name, layer)` value. Example: `export const layer = (options): Layer.Layer<LanguageModel.LanguageModel, never, XAi>` and `model = (modelName, config?) => AiModel.make("xai", modelName, layer(...))`. (in-repo: `packages/drivers/xai/src/XAiLanguageModel.service.ts` lines 176-230; `packages/drivers/openai-compat/src/OpenAiCompatLanguageModel.service.ts` lines 1-42) — so the registry `factory` is a thin `(opts) => XAiLanguageModel.layer(opts).pipe(Layer.provide(XAi.layer))`, NOT a hand-rolled HTTP client. This is the "DISPATCH layer ABOVE the drivers" the CAPTURE calls net-new (CAPTURE.md line 53).
- Drivers read their own key via `Config.redacted("XAI_API_KEY").pipe(Config.option)` inside the service `layer`, then map failures with `Effect.mapError(XAiError.config)`. The provider's *own* env var name + default model already live in driver config modules: `ANTHROPIC_API_KEY_ENV = "AI_ANTHROPIC_API_KEY"`, `ANTHROPIC_DEFAULT_MODEL = "claude-opus-4-6"`; xAI uses `XAI_API_KEY`. (in-repo: `packages/drivers/xai/src/XAi.service.ts` lines 1017-1041; `packages/drivers/anthropic/src/Anthropic.config.ts` lines 26-46) — the registry should READ these constants, not re-declare them, so per-provider default-model resolution (netNew #2) cannot drift from the driver. Note the Anthropic env var is non-standard (`AI_ANTHROPIC_API_KEY`, not `ANTHROPIC_API_KEY`); the registry must source `requiredEnvVar` from the driver constant to avoid the exact "fallback inherits wrong config" class of bug the gold nuggets warn about (agentmemory#4, CAPTURE.md line 36).

### Typed error pattern the structured `api_key_required` error must follow

- The repo's typed-error convention is `TaggedErrorClass<Self>($I\`Name\`)(tag, fields, annotation)` from `@beep/schema`, used by every driver (e.g. `VeniceAIError`, `RepairError`). Errors carry schema-validated fields and a `_tag`, and ship static smart constructors (`VeniceAIError.config()`, `.fromDescriptor(...)`). (in-repo: `packages/drivers/anthropic/src/Anthropic.errors.ts` lines 31-40; `packages/drivers/venice-ai/src/VeniceAI.service.ts` lines 544-624; `packages/foundation/modeling/schema/src/TaggedErrorClass/`) — the structured `api_key_required` error is therefore a `TaggedErrorClass` (e.g. `ProviderApiKeyMissing`) with fields `{ provider, requiredEnvVar, setupInstructions, nextActions, registrationUrl }`, NOT a bare `throw` and NOT a plain MCP content block. The MCP-facing "graceful structured content" rendering (GOLD_SYNTHESIS `mcp-uspto/src/lib/config.ts:32`) becomes a *serializer over* this typed error at the tool boundary, keeping the dispatch Layer pure-Effect.
- Provider error `reason` enums are modeled with `LiteralKit([...])` + `S.toTaggedUnion`/`$I.annoteSchema` (Venice `VeniceAIErrorReason`). The registry's provider-name union and key-source enum (`user` | `env` | `cli`) should be `LiteralKit` literals, giving exhaustive `Match` dispatch and schema validation for free. (in-repo: `packages/drivers/venice-ai/src/VeniceAI.service.ts` lines 192-220)

### API-key prefix auto-detect (the `sk-` vs `AIza` heuristic) — current facts, overlaps, and deprecations

- **OpenAI**: modern keys are `sk-proj-` (project-scoped, default since mid-2024), plus `sk-svcacct-` (service account) and `sk-admin-` (admin); legacy `sk-` / `sk-None-` are being phased out (replacing legacy "since April 2024"). Body is ~155 char base64url, no checksum, validity only confirmable by an API call. (secondary, multi-source: https://vibekit.bot/openai-api-key-format ; https://airbyte.com/agentic-data/openai-api-key ; GitGuardian confirms "replacing legacy API keys since April 2024" + prefixed=true but does NOT print the exact prefix string: https://docs.gitguardian.com/secrets-detection/secrets-detection-engine/detectors/specifics/openai_project_apikey)
- **Anthropic**: production keys are `sk-ant-api03-` (~108 char total); OAuth tokens issued by first-party CLIs are `sk-ant-oat01-` (subscription-billed, NOT per-token API). No checksum. (secondary, multi-source: https://vibekit.bot/anthropic-api-key-format ; https://bip39-phrase.com/anthropic-api-key-format/)
- **Google Gemini**: legacy "Standard" keys are the well-known `AIza...` Google API key shape, BUT Google is mid-transition: new AI Studio keys now issue as "Auth keys" bound to a Cloud service account, some surfaced with an `AQ.` prefix. **Hard deprecation: Standard `AIza` keys will be REJECTED starting September 2026.** Gemini auth uses the `x-goog-api-key` header (NOT `Authorization: Bearer`) and env vars `GEMINI_API_KEY` or `GOOGLE_API_KEY`. (primary: https://ai.google.dev/gemini-api/docs/api-key — confirms `x-goog-api-key`, the env vars, the Auth-key transition, and the Sept-2026 Standard-key rejection; corroborated: https://dev.to/rapls/i-created-a-gemini-api-key-and-got-aq-instead-of-aiza-3dp3)
- **xAI**: keys are reported as `xai-` prefix; official quickstart confirms `Authorization: Bearer $XAI_API_KEY` and base URL `https://api.x.ai/v1` but the fetched quickstart page text does NOT itself print the `xai-` prefix. (base URL + bearer = primary: https://docs.x.ai/developers/quickstart ; `xai-` prefix = secondary: https://www.apideck.com/blog/how-to-get-your-grok-xai-api-key ; https://docs.litellm.ai/docs/providers/xai)
- **OpenRouter**: `sk-or-v1-...` (the `sk-or-` discriminator deliberately distinguishes it from raw OpenAI `sk-`). **DeepSeek**: `sk-` followed by ~48 hex chars — i.e. *collides* with OpenAI's legacy `sk-`. **Mistral**: opaque token with NO standardized identifiable prefix. (secondary, multi-source: https://openrouter.ai/docs/api/reference/authentication ; https://api-docs.deepseek.com/ ; https://docs.mistral.ai/admin/security-access/api-keys)
- **Design consequence — prefix detection is ADVISORY, never authoritative, and needs longest-prefix-first matching.** `sk-ant-` ⊂ `sk-`, `sk-or-` ⊂ `sk-`, and OpenAI-legacy/DeepSeek both bare `sk-` → naive first-match `sk-` would misroute Anthropic/OpenRouter/DeepSeek keys. The detector must (1) order patterns most-specific-first (`sk-ant-`, `sk-or-`, `sk-svcacct-`, `sk-admin-`, `sk-proj-`, `xai-`, `AIza`/`AQ.` before bare `sk-`), (2) treat a match as a *suggestion* that an explicit provider selection overrides, and (3) NEVER reject a key solely because its prefix is unrecognized (Mistral has none; Google's format is changing under a hard 2026 deadline; OpenAI/Anthropic add new prefixes over time). Prefix auto-detect is best used only to *disambiguate which env var a loose key belongs to* when the user pasted one key without naming a provider — it is not a validity check.

### Recommended clean-room shape (reimplementing harvest-mcp#1, not porting it)

- `PROVIDER_REGISTRY`: a typed record keyed by a `LiteralKit` provider-name union, each entry `{ requiredEnvVar, defaultModel, keyPrefixes, factory }` where `requiredEnvVar`/`defaultModel` are **imported from the driver config modules** (single source of truth) and `factory: (opts) => Layer.Layer<LanguageModel.LanguageModel, ProviderConfigError, HttpClient>` wraps the existing `<Driver>LanguageModel.layer(...).pipe(Layer.provide(<Driver>.layer))`. (grounded on `XAiLanguageModel.layer` + `XAi.layer`, in-repo.)
- Key resolution as a Layer: build a per-provider `Config<Redacted<string>>` = `userKey.pipe(Config.orElse(() => cliKey)).pipe(Config.orElse(() => envKey))` (precedence order is a documented decision point — see Open below), provide a `ConfigProvider.orElse` chain via `ConfigProvider.layer`, and surface a missing key as a typed `ProviderApiKeyMissing` (`TaggedErrorClass`) mapped from `ConfigError`, carrying `requiredEnvVar` + `setupInstructions` + `nextActions` + `registrationUrl`. (grounded on `Config.orElse`, `ConfigProvider.layer`, `TaggedErrorClass`, in-repo.)
- Fallback/round-robin/retry: express as `ExecutionPlan.make(...steps)` where each step's `provide` is a registry `factory` Layer and `schedule`/`attempts` come from shared `Schedule` policies (fuse with `effect-orchestration-patterns`, per CAPTURE.md line 59); run via `Effect.withExecutionPlan`. Each step uses its OWN `defaultModel` from the registry (netNew #2). (grounded on `ExecutionPlan`, in-repo.)

## Open / Unverified

- **Precedence direction is unresolved in the source material.** The gold nuggets contradict themselves: harvest-mcp#1 snippet says "CLI > params > env"; Juris.AI#1 says "user map > env"; netNew #3 specifies "user-key > env > CLI". This is an explicit ALIGN decision, not a fact to inherit. Recommend `user-supplied (per-request) > CLI flag > env` for a solo-attorney desktop app (interactive override beats ambient config), but log it in DECISIONS.md.
- **Exact OpenAI prefix strings** (`sk-proj-`/`sk-svcacct-`/`sk-admin-`) are multi-source secondary but NOT printed on the GitGuardian primary page fetched, and OpenAI has no published canonical key-format spec page. UNVERIFIED against an OpenAI-owned doc.
- **`xai-` prefix** is secondary only; the official xAI quickstart page text fetched did not contain it. UNVERIFIED on primary.
- **Gemini `AQ.` new-prefix string** is from a primary-doc-adjacent dev report + Google forum threads; `ai.google.dev` confirms the Auth-key transition and Sept-2026 Standard-key rejection but does not print the literal `AQ.` prefix. Treat `AIza`/`AQ.` detection as advisory and time-boxed by the Sept 2026 deadline.
- **`AiModel.make` / `LanguageModel` execution-plan interop fine print** — the ExecutionPlan doc example provides `Layer<LanguageModel.LanguageModel>` directly; whether mixing different providers' `AiModel` values inside one plan needs a unifying `LanguageModel` requirement (vs per-step provider Service requirement leaking into `R`) needs a compile check before SHAPE. Not yet verified end-to-end.
- **Gemini is not yet a repo driver.** The four existing drivers are Anthropic/openai-compat/xai/venice-ai (CAPTURE.md line 53); a `gemini`/Google entry in the registry is net-new driver work, and its `x-goog-api-key` (non-bearer) auth + 2026 key transition make it the highest-maintenance registry entry. Flag before committing the registry to include it in V1.

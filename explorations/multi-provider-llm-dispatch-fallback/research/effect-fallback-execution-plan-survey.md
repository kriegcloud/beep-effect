# effect-fallback-execution-plan-survey

> Scope: inventory what `effect/unstable/ai` + vendored `effect@4.0.0-beta.91` already give us for provider fallback/retry/load-balancing, fix the reuse-vs-build boundary so the dispatch Layer wraps existing `@beep/{anthropic,openai-compat,xai,venice-ai}` drivers without duplicating retry code, and map BAML/Instructor/agentmemory fallback+circuit-breaker idioms onto Effect primitives.

## Findings

### The fallback engine already exists in the vendored effect — it is `ExecutionPlan`, not `Layer.orElse`

- The repo pins `effect@4.0.0-beta.91` (the effect-smol / v4 line; the AI surface lives under the explicitly-named `effect/unstable/ai` namespace) plus `@effect/ai-anthropic`, `@effect/ai-openai`, `@effect/ai-openrouter` all at matching `4.0.0-beta.91`. Source: `/home/elpresidank/YeeBois/projects/beep-effect/package.json` lines 27-29, 157.
- **`Layer.orElse` does NOT exist in effect v4.** A direct export grep of `node_modules/effect/dist/Layer.d.ts` finds no `orElse`; the Layer error-handling surface is `catch` / `catchCause` / `catchTag` / `fresh` only. So the CAPTURE.md open question "Layer.orElse vs ExecutionPlan vs Schedule" resolves decisively: **the canonical multi-provider fallback mechanism is `ExecutionPlan`**, not a Layer combinator. Source: `/home/elpresidank/YeeBois/projects/beep-effect/node_modules/effect/dist/Layer.d.ts` (no `export declare const orElse`).
- `ExecutionPlan` is a first-class built-in module (`@since 3.16.0`, vendored into v4): "Describes ordered fallback steps for running effects or streams… The runtime tries steps in order until the workflow succeeds or the plan is exhausted. This module also supports merging plans and reading metadata for the active step and attempt." Source: `/home/elpresidank/YeeBois/projects/beep-effect/node_modules/effect/src/ExecutionPlan.ts` lines 1-11.
- A plan is built with `ExecutionPlan.make(...steps)` where each step is `{ provide: Layer | Context, attempts?, schedule?, while? }`. "If `attempts` is omitted, the plan will only attempt once, unless a schedule is provided." Applied via `Effect.withExecutionPlan(effect, plan)` or `Stream.withExecutionPlan(stream, plan)`. The canonical doc example provides `layerBad` 2×/3× then falls back to `layerGood`. Source: `node_modules/effect/src/ExecutionPlan.ts` lines 65-108, 164-239.
- Adversarial confirmation from the official docs + dated blog: the AI-specific canonical pattern is `ExecutionPlan.make({ provide: OpenAiLanguageModel.model("gpt-4o"), attempts: 3, schedule: Schedule.exponential("100 millis"), while: (e) => e._tag === "NetworkError" }, { provide: AnthropicLanguageModel.model("claude-3-7-sonnet-latest") })` applied with `Effect.withExecutionPlan` — "fallback to alternative providers when the primary provider has downtime… keeping your business logic provider-agnostic." Sources: https://effect.website/docs/ai/planning-llm-interactions/ ; https://effect.website/blog/effect-ai/ (published 2025-04-01).
- `while` is a per-step predicate over the **error input** (`(input) => boolean | Effect<boolean>`); only matching errors keep retrying/falling back, so non-retryable errors fail fast. The plan's type machinery removes matched errors from the channel and accumulates each step's `provides` as a requirement. Source: `node_modules/effect/src/ExecutionPlan.ts` lines 118-127, 209-239.
- Extra surface useful for a dispatch Layer: `ExecutionPlan.merge(...plans)` concatenates per-provider plans into one ordered plan (so each driver can own its own sub-plan); `ExecutionPlan.CurrentMetadata` (`@since 4.0.0`) exposes `{ attempt, stepIndex }` at runtime for telemetry/"which provider served this turn"; `captureRequirements` satisfies plan requirements from the surrounding context. Source: `node_modules/effect/src/ExecutionPlan.ts` lines 356-416.

### The repo already CONSUMES this machinery — net-new is a multi-provider plan, not retry code

- `@beep/anthropic` already ships single-provider plans built with `ExecutionPlan.make`: `makeAnthropicTurnPlan()` / `AnthropicTurnPlan` (acquisition + retry) and `makeAnthropicRepairPlan()`, both with `schedule: Schedule.exponential(...)` and `while: (e) => AiError.isAiError(e) && e.isRetryable`. Source: `/home/elpresidank/YeeBois/projects/beep-effect/packages/drivers/anthropic/src/Anthropic.service.ts` lines 102-123 ; `…/Anthropic.repair.ts` lines 96-107.
- The consuming boundary is already wired: `AnthropicTurnKernel` runs `Stream.withExecutionPlan(AnthropicTurnPlan, { preventFallbackOnPartialStream: true })`, and `@beep/anthropic`'s repair path runs `Stream.withExecutionPlan(makeAnthropicRepairPlan(), { preventFallbackOnPartialStream: true })`. So the streaming-fallback plumbing exists; the **net-new is swapping the single-provider plan for a multi-provider one** whose ordered steps are the four existing driver layers. Source: `/home/elpresidank/YeeBois/projects/beep-effect/packages/agents/server/src/AssistantTurn/AnthropicTurnKernel.ts` lines 116-129 ; `…/Anthropic.repair.ts` line 156.
- `preventFallbackOnPartialStream` is a real `Stream.withExecutionPlan` option: "By default, a failing step can fallback even after emitting elements; set `preventFallbackOnPartialStream` to fail instead of mixing partial output with a later fallback." This is the gotcha for cross-provider streaming fallback — a token-streamed answer must not silently splice a second provider mid-stream. Source: `/home/elpresidank/YeeBois/projects/beep-effect/node_modules/effect/dist/Stream.d.ts` lines 8000-8094.

### All four drivers already expose `LanguageModel` Layers — the dispatch Layer wires them, does not touch internals

- Each driver already returns `Layer.Layer<LanguageModel.LanguageModel, …>` from a `.layer(options)` factory: anthropic `makeAnthropicLanguageModelLayer` (via `@effect/ai-anthropic`), `openai-compat` `OpenAiCompatLanguageModel.layer` (`Layer.Layer<LanguageModel.LanguageModel, never, OpenAiCompatClient>`), `xai` `XAiLanguageModel.layer` (`…never, XAi`), `venice-ai` `VeniceAiLanguageModel.layer` (`…never, VeniceAI`). Sources: `packages/drivers/anthropic/src/Anthropic.service.ts` lines 57-66 ; `packages/drivers/openai-compat/src/OpenAiCompatLanguageModel.service.ts` lines 951, 1002 ; `packages/drivers/xai/src/XAiLanguageModel.service.ts` line 208 ; `packages/drivers/venice-ai/src/VeniceAiLanguageModel.service.ts` line 201.
- Because every step in `ExecutionPlan.make` takes `provide: Layer`, the dispatch layer is literally `ExecutionPlan.make({ provide: anthropicLayer, … }, { provide: openaiCompatLayer, … }, …)` — **REUSE the existing driver layers as steps; do not duplicate their retry/HTTP code.** (Confirms CAPTURE alreadyCovered + caution "reuse anthropic/openai-compat retry code, do not duplicate".)
- No multi-provider router/registry exists yet (grep for `ProviderRegistry|dispatchLanguageModel|fallbackChain|LanguageModelRouter` across `packages/**` returns nothing) — so this is a genuine net-new Layer, not a dup. Only the single-provider Anthropic kernel uses `withExecutionPlan` today.

### `Model.make` solves the agentmemory "fallback inherits primary's model" bug structurally (netNew #2)

- `effect/unstable/ai/Model.make(provider, modelName, layer)` wraps a driver Layer and *automatically provides* `Model.ProviderName` + `Model.ModelName` services into context. "A Model can be used anywhere its underlying Layer can be used. It also provides the current provider name and model name through the Effect context." Source: `/home/elpresidank/YeeBois/projects/beep-effect/node_modules/effect/dist/unstable/ai/Model.d.ts` lines 1-116.
- Therefore each `ExecutionPlan` step provides its OWN `Model` carrying its OWN model name — a fallback step cannot inherit the primary's model name. This is exactly the agentmemory `#778` failure (fallback inherited primary's model → 404 every call, tripping the breaker) made structurally impossible. **REUSE `Model.make` per step** rather than building a `defaultModelFor(providerType)` switch. (Maps netNew #2 onto an existing primitive; the only build part is reading each provider's OWN env default, e.g. `ANTHROPIC_MODEL`.)

### AiError taxonomy is already provider-normalized with `isRetryable` — the `while` predicate is a one-liner (netNew #3)

- `effect/unstable/ai/AiError` defines a normalized tagged union with `isRetryable` on each: `NetworkError`, `InternalProviderError`, `RateLimitError`, `InvalidOutputError`, `StructuredOutputError` → `isRetryable === true`; `AuthenticationError`, `QuotaExhaustedError`, `ContentPolicyError`, `InvalidRequestError` → `isRetryable === false`. Source: `/home/elpresidank/YeeBois/projects/beep-effect/node_modules/effect/dist/unstable/ai/AiError.d.ts` (per-class `_tag` + `get isRetryable` + doc lines `// true|false`).
- The drivers already map provider errors into this union, and the repo's plans already key on it: `while: (e) => AiError.isAiError(e) && e.isRetryable`. So the dispatch layer's retry/advance predicate is reused verbatim. netNew #3's "structured `api_key_required` / setupInstructions error" maps onto `AuthenticationError` (non-retryable → fails fast, never wastes fallback budget); only the human-facing setup-instruction surfacing is build. Source: same file + `Anthropic.service.ts` line 107.

### Per-provider tool dispatch + structured output are already abstracted (alreadyCovered + netNew #5)

- `effect/unstable/ai/Toolkit` (`Toolkit.make(...tools)`, `.merge`, `.toLayer`, `.handle`) is the single-source tool-definition abstraction, and `LanguageModel.generateText` / `streamText` take a per-call `toolkit` option — the provider-specific JSON-schema translation lives below this boundary. This covers the mike#10 "one OpenAI-style tool def → Claude/Gemini" adapter need: **do not build a tool-schema adapter; use Toolkit.** Sources: `node_modules/effect/dist/unstable/ai/Toolkit.d.ts` lines 40-270 ; `node_modules/effect/dist/unstable/ai/LanguageModel.d.ts` lines 13, 54-81.
- `LanguageModel.generateObject({ schema, … })` binds an `effect/Schema` and returns a decoded typed object (adding the schema's `DecodingServices` to the requirement channel) — this IS netNew #5 (`response_model` → `effect/Schema`-validated structured output). REUSE `generateObject`; the Instructor/courtlistener `response_model` pattern is already native. Source: `node_modules/effect/dist/unstable/ai/LanguageModel.d.ts` line 564.

### Reuse-vs-build boundary (the decision this survey exists to make)

- **REUSE (already in effect, MIT-licensed + vendored, or already in repo):** the fallback engine (`ExecutionPlan` + `Effect/Stream.withExecutionPlan`), retry/backoff (`Schedule.exponential` / `spaced` / `jittered` / `recurs`), the streaming-partial guard (`preventFallbackOnPartialStream`), per-provider model identity (`Model.make` → `ProviderName`/`ModelName`), the normalized retryable-error taxonomy (`AiError.isRetryable`), per-provider tool dispatch (`Toolkit`), schema-validated output (`generateObject`), and all four `@beep/*` driver `LanguageModel` layers. effect itself is MIT (`node_modules/effect/LICENSE`), so the engine needs no porting.
- **BUILD (net-new, thin, sits ABOVE the drivers):** (1) user-key > env > CLI **key-precedence resolver** — *not* an effect primitive (each driver only reads `Config.redacted(ENV)`, e.g. `AnthropicClient.layerConfig({ apiKey: Config.redacted(ANTHROPIC_API_KEY_ENV) })`, `Anthropic.service.ts` lines 34-37); (2) **assembly** of the ordered multi-provider `ExecutionPlan` from the per-driver layers + their OWN env default models; (3) the typed `PROVIDER_REGISTRY` + prefix auto-detect + human setup-instruction surfacing layered on `AuthenticationError`; (4) round-robin / circuit-breaker **only if wanted** (see next finding).

### Effect has NO built-in CircuitBreaker or round-robin — ExecutionPlan is strictly ORDERED fallback

- There is no `CircuitBreaker` module in effect v4 (no `node_modules/effect/dist/CircuitBreaker*`; case-insensitive grep finds nothing). `Pool` exists; no `RateLimiter` module surfaced. So the agentmemory circuit-breaker idiom has no drop-in primitive — it would be net-new (a `Ref`-held breaker state gating an `ExecutionPlan` step's `while`, or `Schedule`-based cooldown). Source: filesystem check of `node_modules/effect/dist/`.
- `ExecutionPlan` advances steps **in fixed declaration order**; it has no round-robin / load-balancing mode. BAML's `provider round-robin` (load balancing across clients) therefore does NOT map onto `ExecutionPlan` — it is net-new if desired (a rotating/random step selector or custom Layer). BAML's `provider fallback` (ordered) maps 1:1 onto ordered `ExecutionPlan` steps. Sources: `node_modules/effect/src/ExecutionPlan.ts` lines 1-11 (tries steps "in order") ; https://docs.boundaryml.com/ref/llm-client-strategies/fallback ; https://docs.boundaryml.com/ref/baml/client-llm.

### External fallback-chain idioms re-expressed in Effect (BAML / Instructor / circuit-breaker)

- **BAML fallback** (`client<llm> X { provider fallback options { strategy [A, B, C] } }`): "attempt the first client, and if it fails, try the second, and so on"; strategy list "cannot be empty"; fallbacks nest. This is exactly `ExecutionPlan.make({provide: A}, {provide: B}, {provide: C})`, and BAML nesting maps to `ExecutionPlan.merge`. **Config-pattern reference only — do NOT adopt the BAML runtime** (BAML is Apache-2.0 per CAPTURE; the docs page states no license). Source: https://docs.boundaryml.com/ref/llm-client-strategies/fallback.
- **BAML semantics nuance (adversarial):** in BAML, a `retry_policy` attached to a *fallback* client "will test the fallback itself, **after** the entire strategy has failed" — i.e. retry wraps the whole ordered chain. `ExecutionPlan` is the inverse: each step interleaves its OWN `attempts`+`schedule` and THEN advances. A faithful re-expression therefore puts per-client retry inside each step (matching BAML's per-client `retry_policy Exponential { max_retries strategy { exponential_backoff … } }`) and relies on step ordering for the fallback, rather than wrapping the chain in an outer retry. Sources: https://docs.boundaryml.com/ref/llm-client-strategies/fallback ; CAPTURE TalentScore#2 snippet.
- **Instructor** `from_provider("provider/model")` + `response_model` + `max_retries`: one provider/model string selects the backend, a Pydantic model validates output, retries on validation/API failure. Effect equivalent: `Model.make(provider, model, layer)` (string identity) + `generateObject({ schema })` (validated output) + `ExecutionPlan` step `attempts`/`schedule` (retries). Instructor itself "doesn't explicitly implement circuit breakers" — production guides bolt on PyBreaker/Polly/Resilience4j separately, mirroring the Effect situation (build the breaker around the plan). Sources: https://python.useinstructor.com/getting-started/ ; https://python.useinstructor.com/concepts/retrying/ ; https://www.getmaxim.ai/articles/retries-fallbacks-and-circuit-breakers-in-llm-apps-a-production-guide/.

## Open / Unverified

- **effect API stability (date-sensitive):** `effect/unstable/ai` is in the explicitly-named `unstable` namespace and the repo is on a BETA (`4.0.0-beta.91`); `ExecutionPlan` shape, `Model.make`, and the `AiError` union may shift before GA. The pinned beta was current as of 2026-06-29. UNVERIFIED whether a stable `effect/ai` (non-unstable) path lands before this work ships.
- **Load-balancing primitives:** confirmed `Pool` exists and `RateLimiter`/`CircuitBreaker` modules do NOT exist as files in beta.91. Did not exhaustively verify there is no `Effect.makeSemaphore`-style helper usable for concurrency-capped dispatch — treat round-robin/rate-limit/circuit-breaker as BUILD until a primitive is confirmed.
- **BAML license for the config patterns:** the BAML docs pages cite no license; CAPTURE records BAML as Apache-2.0. Re-expressing BAML's declarative strategy *shape* in Effect config is pattern-reference (low risk), but verify the Apache-2.0 status from the BoundaryML repo `LICENSE` before quoting any BAML grammar verbatim in shipped docs.
- **Cross-provider streaming fallback UX:** `preventFallbackOnPartialStream: true` makes a mid-stream provider failure surface as an error rather than splicing a second provider. Whether the product wants fail-hard vs. restart-on-new-provider for streamed turns is a design decision, not settled by the framework — flag for the align stage.
- **`while` over the full driver error channel:** the existing predicate is `AiError.isAiError(e) && e.isRetryable`, but driver layers can also fail with `Config.ConfigError` (missing key). Confirm whether a missing-key on the PRIMARY should advance to the fallback step (treat as retryable-for-advance) or fail fast — current Anthropic predicate would fail fast on `ConfigError`. Not yet decided.

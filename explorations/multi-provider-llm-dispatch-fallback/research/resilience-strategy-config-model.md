# resilience-strategy-config-model

> Scope: shape declarative resilience strategies (ordered-fallback, round-robin/load-balance, named constant/exponential retry policies, circuit breaker) as an Effect-config surface that resolves to `ExecutionPlan` + shared `Schedule` policies — BAML/Instructor as config-pattern reference only, no foreign runtime.

## Findings

### Effect-native substrate (versions, module paths, typed retryability)

- The repo is on **`effect@4.0.0-beta.91`** (root `package.json` catalog; confirmed in installed `node_modules/effect/package.json` `"version": "4.0.0-beta.91"`). `ExecutionPlan` and `Schedule` are exported from the `"effect"` root; the AI surface (`LanguageModel`, `Tool`, `Toolkit`, `AiError`, `Response`) lives under `effect/unstable/ai`, and HTTP under `effect/unstable/http`. Source: local `package.json:157`; `packages/drivers/anthropic/src/Anthropic.service.ts:9-11`.
- **Ordered-fallback + named-retry is already a first-class Effect-native primitive: `ExecutionPlan`.** `ExecutionPlan.make(...steps)` takes a *non-empty array of steps* executed sequentially until one succeeds; each step is `{ provide: Context|Layer; attempts?: number; while?: (input) => boolean | Effect<boolean,E,R>; schedule?: Schedule<any,input,R> }`. Verified verbatim in vendored types `node_modules/effect/dist/ExecutionPlan.d.ts:109-112,188-190`. Multiple steps with *different `provide` layers* = ordered provider fallback; a single step with `attempts` + `schedule` = a named retry policy. API ref: https://effect-ts.github.io/effect/effect/ExecutionPlan.ts.html
- A plan is applied at the call boundary via **`Effect.withExecutionPlan(effect, plan)`** or **`Stream.withExecutionPlan(stream, plan, { preventFallbackOnPartialStream })`**. The repo already does exactly this for streaming forced-tool turns. Source: `packages/agents/server/src/AssistantTurn/AnthropicTurnKernel.ts:129`; `packages/drivers/anthropic/src/Anthropic.repair.ts:156`.
- **`ExecutionPlan.merge(...plans)`** composes existing plans into one (the equivalent of BAML *nested* fallback/round-robin clients). Verified in `node_modules/effect/dist/ExecutionPlan.d.ts:268-275`.
- **Typed retryability is the `while` hook.** The repo's canonical predicate is `(error) => AiError.isAiError(error) && error.isRetryable`, so auth/invalid-request failures fail fast and only Effect-AI-marked-retryable errors retry. This is the join point for the "failure-vs-defect" law (schema-validation failures should be **defects / `Effect.die`**, never retried). Source: `packages/drivers/anthropic/src/Anthropic.service.ts:107`; `Anthropic.repair.ts:106`; sibling exploration `explorations/effect-orchestration-patterns/CAPTURE.md` (law EF-31, `BamlParseError` → `Effect.die`).

### Schedule combinators = the named-retry-policy vocabulary (v4 API, with a migration trap)

- Verified present in vendored `node_modules/effect/dist/Schedule.d.ts` (beta.91): **`exponential(base: Duration.Input, factor?: number)`** (`factor` defaults to 2 — used as `Schedule.exponential(Duration.millis(250), 2)` in the repo), **`spaced(duration)`** = constant delay measured from end-of-execution, **`fixed(interval)`** = fixed clock-rate, **`recurs(times)`** = count cap, **`jittered(self)`** = anti-thundering-herd, plus `either`, `both`, `addDelay`, `andThen`, `forever`, `passthrough`, `delays`, `tapOutput`. Lines: `Schedule.d.ts:3172 (exponential), 4120 (spaced), 3312 (fixed), 3723 (recurs), 3623 (jittered), 2544 (either), 1012 (both)`. Repo usage: `packages/drivers/anthropic/src/Anthropic.service.ts:106`. Docs: https://effect.website/docs/scheduling/built-in-schedules/
- **MIGRATION TRAP (verified, load-bearing): in effect v4 beta.91 `Schedule.compose`, `Schedule.intersect`, and `Schedule.union` DO NOT EXIST.** A grep of `Schedule.d.ts` for `export declare const (compose|intersect|union)` returns nothing. v4 renames them to **`either`** (OR / union semantics — recurs if *either* schedule wants to, uses the **minimum** of the two delays) and **`both`** (AND / intersection semantics — recurs only while *both* want to, uses the **maximum** delay, stops when the first exhausts). Confirmed in the type-doc comments at `Schedule.d.ts:33-40` (both) and `:45-53` (either): "Compare with both, which provides intersection semantics (AND logic)" / "Compare with either which provides union semantics (OR logic)". Consequence: the sibling exploration's seed snippet `Schedule.exponential(...).pipe(Schedule.compose(Schedule.recurs(2)))` is v3/foreign-repo syntax and **will not compile** against beta.91 — the v4 equivalent is `Schedule.both(Schedule.exponential(...), Schedule.recurs(2))`.
- **Canonical v4 expressions for the two BAML retry strategies** (this is the core re-expression deliverable):
  - *constant_delay(delay_ms=D, max_retries=N)* → `Schedule.both(Schedule.spaced(Duration.millis(D)), Schedule.recurs(N))` (constant delay, bounded to N retries because `both` stops when `recurs` exhausts).
  - *exponential_backoff(initial_delay_ms=I, multiplier=M, max_delay_ms=Max, max_retries=N)* → `Schedule.both(Schedule.either(Schedule.exponential(Duration.millis(I), M), Schedule.spaced(Duration.millis(Max))), Schedule.recurs(N))`, optionally wrapped in `Schedule.jittered(...)`. The `either(exponential, spaced(Max))` clamps the growing delay at `Max` (either → min-of-delays), and `both(..., recurs(N))` caps the count. Combinator semantics cross-checked against https://effect.website/docs/error-management/retrying/ (cap via `either` with a fixed ceiling; jitter via `jittered`; count cap via the intersection combinator).

### BAML clients.baml as config-pattern reference (Apache-2.0; do NOT adopt the runtime)

- **BAML is Apache-2.0** (BoundaryML/baml, "100% open-source (Apache 2)"), so its *config grammar* is a safe shape reference and any reimplementation is license-compatible — but per the capture cautions, only the declarative shape is borrowed, never the Rust runtime. Source: https://github.com/BoundaryML/baml
- **Ordered fallback**: `client<llm> X { provider fallback options { strategy [ClientA ClientB ClientC] } }`, optionally `retry_policy MyRetryPolicy`, and clients nest (a fallback strategy member can itself be a fallback/round-robin client). Maps 1:1 to a multi-step `ExecutionPlan.make` (one step per strategy member) and `ExecutionPlan.merge` for nesting. Source: https://docs.boundaryml.com/ref/llm-client-strategies/fallback
- **Named retry policy**: `retry_policy Name { max_retries N; strategy { type constant_delay | exponential_backoff; delay_ms | initial_delay_ms; multiplier (default 1.5); max_delay_ms (default 10000) } }`; default strategy is `constant_delay(delay_ms=200)`; `max_retries` counts *additional* retries after the first attempt. A policy attaches to a client via the `retry_policy` field. This is the literal template for the named-policy registry → `Schedule` mapping above. Source: https://docs.boundaryml.com/ref/llm-client-strategies/retry-policy
- **Round-robin / load-balance**: `client<llm> X { provider round-robin options { strategy [ClientA ClientB ClientC] } }` with an optional `start` index (default `random(0, len(strategy))`, `0` in the Playground); it advances one client per invocation, **rotates the strategy list after each retry** (so retries hit *different* clients, not the same one), and nested round-robins keep independent rotation state. Source: https://docs.boundaryml.com/ref/llm-client-strategies/round-robin
- Instructor (the other capture reference) is a *structured-output* binding (`response_model` + `from_provider("provider/model")` + `max_retries`), not a resilience-strategy DSL; its only resilience surface is a flat `max_retries`. It belongs to the structured-output subtopic, not here — noted to avoid double-counting. (Capture: courtlistener#5, AGPL-3.0 → patterns only.)

### Round-robin and circuit breaker are the genuine net-new gaps

- **Effect has no round-robin / weighted-load-balance provider combinator.** `ExecutionPlan` steps are statically ordered (fallback only); round-robin needs *stateful rotation*. Re-expression: a `Ref<number>` index over a `NonEmptyArray` of provider layers, selected at build time via **`Layer.unwrap`** (the sibling exploration's netNew #2 — zero `Layer.unwrap` usages in the repo today), rotating per attempt with a random `start`, mirroring BAML's "rotate after each retry" semantics. This is a design synthesis (no single primary source): grounded in (a) verified absence of any Schedule/ExecutionPlan round-robin primitive, (b) BAML rotation semantics above, (c) the `Layer.unwrap` provider-selection pattern. Source: `explorations/effect-orchestration-patterns/CAPTURE.md` (research-squad#5, netNew #2).
- **Effect has no built-in circuit breaker.** Issue [Add Circuit Breaker #2843](https://github.com/Effect-TS/effect/issues/2843) is **open**; its PR [#2854](https://github.com/Effect-TS/effect/pull/2854) (ports the `CircuitBreaker` module from the ZIO **Rezilience** library, targets `next-minor`) is **open/unmerged** (last activity datapoint Oct 2024). So the circuit-breaker strategy is **NET-NEW** for this repo: a `Ref<CircuitState>` state machine with `Closed | Open | HalfOpen`, a failure-count threshold, a reset timeout (Clock/`Schedule`-driven), and a half-open probe — short-circuiting (fail-fast) while Open instead of dispatching. The sibling exploration already carries a concrete blueprint: `uspto_pfw_mcp#5` (CLOSED/OPEN/HALF_OPEN + retry budget + served-while-open cache + per-workload bulkhead pools). Source: `explorations/effect-orchestration-patterns/CAPTURE.md` (uspto_pfw_mcp#5).
- **Port-source licensing for the circuit breaker** (the only component with no Effect-native source): Rezilience (svroonland/rezilience, ZIO) and Opossum (nodeshift/opossum, Node) are **both Apache-2.0** — pattern-port-compatible, but reimplement idiomatically over `Ref`/`Clock`/`Schedule` rather than vendoring. Sources: https://github.com/svroonland/rezilience , https://github.com/nodeshift/opossum

### Fusion seam with effect-orchestration-patterns (avoid duplication)

- The two explorations split cleanly. **effect-orchestration-patterns owns the *primitive* shared `Schedule` policy library** — netNew #1: a centralized module exporting named policies (`llmRetry`, `networkRetry`, `databaseRetry`) plus `Layer.unwrap` provider selection. Those policies are **today duplicated inline per-driver** (`packages/drivers/anthropic/src/Anthropic.service.ts:106`, `Anthropic.repair.ts:105` both hand-roll `Schedule.exponential(Duration.millis(...), 2)`); a repo grep finds **no shared `@beep` RetryPolicy/Schedule foundation module**. Source: `explorations/effect-orchestration-patterns/CAPTURE.md` (netNew #1).
- **This subtopic owns the *declarative config surface above* those primitives**: a Schema-encoded named-strategy model that *resolves to* the shared `Schedule` policies and `ExecutionPlan` steps — it must *import*, not re-implement, the policy library. Concretely, the config model should be an `effect/Schema` tagged union of strategy tags (`ordered-fallback`, `round-robin`, `constant`, `exponential`, `circuit-breaker`) decoded into a runtime `ExecutionPlan`/`Schedule`/`Ref`-machine, per the repo's schema-first law (`S.toTaggedUnion` over a `_tag` discriminant). Named retry policies become a registry keyed by name → shared `Schedule`. (Schema-encoding shape is design synthesis aligned to repo conventions; the *primitives* it targets are the verified ones above.)
